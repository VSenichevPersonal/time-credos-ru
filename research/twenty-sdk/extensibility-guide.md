## 1. Модель данных и Custom Objects

### 1.1. Стандартные объекты (Standard Objects)

Twenty включает пять встроенных объектов:

| Объект | Назначение |
|--------|-----------|
| **People** | Контакты, лиды, партнёры |
| **Companies** | Организации, аккаунты |
| **Opportunities** | Сделки, продажи |
| **Notes** | Заметки, привязанные к записям |
| **Tasks** | Задачи, привязанные к записям |

**Ограничение:** синхронизация email и календаря работает только с People, Companies и Opportunities.

### 1.2. Ключевые понятия

- **Objects** — основные категории данных (таблицы)
- **Fields** — свойства объекта (столбцы): text, number, date, select, multi-select, relation и другие
- **Records** — конкретные записи (строки)

### 1.3. Создание Custom Objects

Путь: **Settings → Data Model → + New object**

При создании указываются:
- **Singular name** — имя в единственном числе (например, «Listing»)
- **Plural name** — имя во множественном числе (например, «Listings»)
- **Icon** — иконка для навигации
- **Description** — необязательное описание

После создания объект сразу появляется в боковой панели (sidebar).

### 1.4. Типы полей (Field Types)

Доступные типы полей:
- **Text** — текстовое поле
- **Number** — числовое поле
- **Date** — дата
- **Select** — выбор из списка (одиночный)
- **Multi-select** — множественный выбор
- **Relation** — связь с другим объектом
- И другие специализированные типы

Добавление поля: **Settings → Data Model → [объект] → + Add Field** → выбор типа → настройка → сохранение.

### 1.5. Связи между объектами (Relations)

Custom objects можно связывать с существующими сущностями (People, Companies и др.) через relation fields. Связи настраиваются через интерфейс Data Model.

### 1.6. Управление объектами

- **Деактивация** — объект скрывается из интерфейса, данные сохраняются
- **Удаление** — полное удаление объекта
- Настройка через **Settings → Data Model** (toggle)

### 1.7. Архитектура Custom Objects (backend)

Система использует три таблицы метаданных:

1. **DataSource** — описывает расположение и хранилище данных
2. **Object** — определяет custom object, связывает с DataSource
3. **Field** — описывает поля объекта, связывает с Object

При создании custom object через `/metadata` API:
1. Обновляется схема БД метаданных
2. Генерируется GraphQL-схема из обновлённых метаданных
3. Схема кешируется в GQL cache

Данные запрашиваются через `/graphql` endpoint, запросы проходят через Query Resolver.

## 2. REST API

### 2.1. Аутентификация

Все запросы требуют API-ключ в заголовке:

```
Authorization: Bearer YOUR_API_KEY
```

Создание ключа: **Settings → APIs & Webhooks → + Create key** → указать имя и срок действия. Ключ показывается только один раз при создании.

**Безопасность:** API-ключ даёт доступ к чувствительным данным. Через **Settings → Roles** можно ограничить права доступа для ключей.

### 2.2. Endpoints

| Окружение | Base URL |
|-----------|----------|
| **Cloud** | `https://api.twenty.com/` |
| **Self-Hosted** | `https://{your-domain}/` |

Основные пути:
- **Core API:** `/rest/` — CRUD-операции с записями
- **Metadata API:** `/rest/metadata/` — управление структурой данных

### 2.3. Операции

REST API поддерживает стандартные CRUD-операции:
- **Create** — создание записей
- **Read** — чтение записей с фильтрацией, сортировкой, пагинацией
- **Update** — обновление записей
- **Delete** — удаление записей
- **Batch** — пакетные операции (до 60 записей за запрос)

Не требуется указывать ID объекта в endpoint — API адаптируется к вашей data model.

### 2.4. Rate Limits

- **100 запросов в минуту** максимум
- **60 записей** за один batch-запрос

### 2.5. Интерактивная документация

Персонализированная API-документация доступна в **Settings → API & Webhooks** с playground для тестирования запросов и исследования схемы.

## 3. GraphQL API

### 3.1. Endpoint

- **Core API:** `/graphql/` — операции с данными workspace
- **Metadata API:** `/metadata/` — управление структурой workspace

### 3.2. Аутентификация

Аналогично REST API:

```
Authorization: Bearer YOUR_API_KEY
```

### 3.3. Возможности

GraphQL API поддерживает:
- **CRUD-операции** — создание, чтение, обновление, удаление записей
- **Batch operations** — пакетные операции
- **Batch upsert** — создание или обновление (эксклюзивно для GraphQL)
- **Relationship queries** — запросы связанных данных в одном вызове
- **Filtering** — фильтрация по полям
- **Pagination** — постраничная навигация (cursor-based)
- **Ordering** — сортировка результатов

### 3.4. Преимущества перед REST

- Batch upsert (создание или обновление) доступен только в GraphQL
- Возможность запрашивать связанные данные в одном запросе
- Гибкая выборка полей — запрашиваете только нужные данные

## 4. Metadata API

### 4.1. Назначение

Metadata API управляет структурой workspace и конфигурацией data model:
- Создание, изменение, удаление объектов
- Управление полями объектов
- Настройка связей (relations) между объектами

### 4.2. Endpoints

- **REST:** `/rest/metadata/`
- **GraphQL:** `/metadata/`

### 4.3. Использование

Через Metadata API можно программно:
- Создавать custom objects
- Добавлять/удалять/изменять поля
- Настраивать relation fields
- Получать информацию о текущей data model

## 5. Webhooks

### 5.1. Назначение

Webhooks отправляют уведомления в реальном времени при событиях в CRM — без необходимости polling.

### 5.2. Настройка

1. **Settings → APIs & Webhooks → Webhooks → + Create webhook**
2. Указать публично доступный URL
3. Сохранить — webhook активируется сразу

### 5.3. Поддерживаемые события

| Категория | Примеры |
|-----------|---------|
| Создание записи | `person.created`, `company.created`, `note.created` |
| Обновление записи | `person.updated`, `company.updated`, `opportunity.updated` |
| Удаление записи | `person.deleted`, `company.deleted` |

**Ограничение:** в текущей версии все типы событий отправляются на webhook URL без возможности фильтрации.

### 5.4. Формат payload

Webhook отправляет HTTP POST с JSON:

```json
{
  "event": "person.created",
  "data": { /* полные данные записи */ },
  "timestamp": "2026-03-14T10:30:00Z"
}
```

Endpoint должен ответить HTTP 2xx (200-299) для подтверждения получения.

### 5.5. Безопасность — проверка подписи

Каждый запрос содержит заголовки:
- `X-Twenty-Webhook-Signature` — HMAC SHA256 подпись
- `X-Twenty-Webhook-Timestamp` — временная метка запроса

Для валидации: вычислить HMAC SHA256 из webhook secret, объединив timestamp и payload, затем сравнить подписи.

### 5.6. Webhooks vs Workflows

| Webhooks | Workflows |
|----------|-----------|
| Исходящие уведомления | Пользовательская логика с фильтрацией |
| Автоматическая отправка всех событий | HTTP Request action для выборочных запросов |
| Для внешних систем | Webhook trigger для входящих данных |

## 6. Workflows — автоматизация

### 6.1. Назначение

Workflows позволяют строить произвольные автоматизации, адаптированные под бизнес-процессы:
- **Внутренняя автоматизация** — сокращение ручного ввода данных
- **Интеграция данных** — связь Twenty с внешними системами через API и webhooks

### 6.2. Создание Workflow

1. **Навигация** → Workflows → создать новую запись → задать имя
2. **Добавить Trigger** (триггер запуска)
3. **Настроить Actions** (действия)
4. **Тестирование** на тестовых данных
5. **Активация** — переключить toggle

### 6.3. Типы триггеров (Triggers)

#### Record-Based Triggers (на основе записей)

| Триггер | Описание | Рекомендации |
|---------|----------|-------------|
| **Record is Created** | Срабатывает при создании записи | Лучше для API/CSV-импорта; не рекомендуется для ручного создания из-за auto-save |
| **Record is Updated** | Срабатывает при изменении записи | Можно отслеживать конкретные поля |
| **Record is Updated or Created** | Оба сценария | Полезен, т.к. разные методы создания ведут себя по-разному |
| **Record is Deleted** | Срабатывает при удалении | Для cleanup-операций |

#### Manual Trigger (ручной запуск)

Активируется через `Cmd+K` или кнопки в навигации. Три режима:
- **Global** — не требует выбора записи
- **Single** — передаёт выбранные записи (soft limit: 100 запусков/мин)
- **Bulk** — требует Iterator action для пакетной обработки

#### Schedule Trigger (по расписанию)

Периодический запуск: минуты, часы, дни или cron-выражения.
**Важно:** расписание работает в UTC.

#### Webhook Trigger (внешний вызов)

Принимает GET/POST запросы от внешних сервисов с настраиваемой структурой body.

#### Поддерживаемые объекты

Триггеры работают не только со стандартными CRM-объектами, но и с: Workspace Members, Calendar Events, Messages (Emails), Tasks, Notes и другими системными объектами.

### 6.4. Типы действий (Actions)

#### Действия с записями (Record Actions)

| Действие | Описание | Лимиты |
|----------|----------|--------|
| **Create a Record** | Создание записи в выбранном объекте | — |
| **Update Record** | Обновление существующей записи | Нельзя искать по критериям — сначала Search Records |
| **Delete Record** | Удаление записи | Данные доступны в последующих шагах |
| **Search Records** | Поиск записей с фильтрацией и сортировкой | Максимум 200 записей |
| **Upsert Record** | Создание или обновление по критерию совпадения | Критерии: email (People), domain (Companies), ID, unique fields |

#### Управление потоком (Flow Actions)

| Действие | Описание |
|----------|----------|
| **Iterator** | Цикл по массиву записей — выполняет действия для каждой записи |
| **Filter** | Условный шлюз: если условие true — продолжить, иначе — остановить |
| **Delay** | Пауза: по длительности (дни/часы/минуты/секунды) или до указанной даты |

#### Коммуникации (Communication Actions)

| Действие | Описание |
|----------|----------|
| **Send Email** | Отправка шаблонных писем из синхронизированных почтовых ящиков. Поддержка переменных, форматирования, вложений. Пока только один получатель |
| **Form** | Сбор пользовательского ввода через настраиваемые поля (text, number, date, record, select) |

#### Интеграции (Integration Actions)

| Действие | Описание |
|----------|----------|
| **Code** | Исполнение JavaScript-кода с доступом к переменным предыдущих шагов. API-ключи встраиваются прямо в код |
| **HTTP Request** | Отправка API-запросов (GET, POST, PUT, PATCH, DELETE) с заголовками и параметрами |

#### AI Actions (в разработке)

| Действие | Описание |
|----------|----------|
| **AI Agent** | Анализ данных, классификация записей, генерация контента. Учитывает role-based permissions, расходует credits |

### 6.5. Ветвление (Branches)

- При нескольких соединениях от одного узла — **все ветки выполняются параллельно**
- Создание: правый клик на canvas → Add node → выбрать тип → соединить стрелкой
- **Условная логика:** через Filter nodes с взаимоисключающими условиями
  - Пример: `Stage = Won` → поздравление, `Stage = Lost` → follow-up task
- Ветки **не объединяются автоматически** — можно оставить раздельными или вручную соединить с общим downstream-действием
- **Ограничение:** Delay nodes не могут ждать завершения других веток

### 6.6. Рекомендации

- Давать шагам описательные имена
- Переиспользовать данные из предыдущих шагов
- Начинать с простых автоматизаций, усложнять постепенно
- Планировать логику workflow до реализации

## 7. Приложения (Apps) — расширение платформы

### 7.1. Архитектура

Приложения строятся с использованием **Twenty SDK**, который предоставляет типизированные компоненты с валидацией на этапе сборки и автодополнением в IDE.

### 7.2. Точки расширения (Extension Points)

#### Custom Objects через SDK

Создание через `defineObject()`. Объекты автоматически включают базовые поля:
- `id`, `name`, `createdAt`, `updatedAt`, `createdBy`, `updatedBy`, `deletedAt`

Требования:
- Уникальный `universalIdentifier` (стабилен между деплоями)
- Определения полей с индивидуальными идентификаторами
- Опциональные кастомные поля сверх базовых

#### Logic Functions

Исполнение backend-кода через `defineLogicFunction()`. Три варианта:

| Вариант | Назначение |
|---------|-----------|
| **Standard** | Переиспользуемая бизнес-логика с опциональными триггерами |
| **Pre-install** | Выполняется перед установкой приложения (валидация) |
| **Post-install** | Выполняется после установки (seeding данных, конфигурация) |

Типы триггеров для функций:

| Триггер | Описание |
|---------|----------|
| **Route trigger** | HTTP-endpoint под путём `/s/` |
| **Cron trigger** | Запуск по CRON-выражению |
| **Database event trigger** | Реакция на lifecycle-события объектов с опциональным отслеживанием конкретных полей |

#### Role-Based Permissions

Определение через `defineRole()`:
- Права на уровне объекта: read, update, soft delete, destroy
- Права на уровне поля: read, update
- Platform permission flags
- Роль функции определяет scope API-ключа

#### Front Components

React-компоненты через `defineFrontComponent()`:
- Рендерятся в изолированном контексте внутри Twenty
- Автоматическая синхронизация при разработке

#### AI Agent Skills

Создание через `defineSkill()`:
- Структурированные инструкции для AI-агентов
- Переиспользуемые промпты для специфических сценариев
- Кастомные иконки и описания

### 7.3. Типизированные клиенты (Generated Typed Clients)

Два автогенерируемых клиента для доступа к данным workspace:

| Клиент | Endpoint | Назначение |
|--------|----------|-----------|
| **CoreApiClient** | `/graphql` | Данные workspace |
| **MetadataApiClient** | `/metadata` | Конфигурация workspace |

Клиенты автоматически перегенерируются при изменении схемы.

### 7.4. Credentials и безопасность

Функции получают инъекцию credentials:
- `TWENTY_API_URL` — базовый URL API
- `TWENTY_API_KEY` — короткоживущий токен, ограниченный ролью функции

Заголовки запросов не форвардятся по умолчанию — нужно явно указать через `forwardedRequestHeaders`.

### 7.5. Tool-функции для AI Agent

Для регистрации функции как инструмента AI:
- Установить `isTool: true`
- Определить `toolInputSchema` (JSON Schema)
- Имя автоматически нормализуется в формат `logic_function_<name>`

### 7.6. Команды разработки

```bash
yarn twenty entity:add       # Генерация объектов, функций, компонентов
yarn twenty app:dev          # Сборка + синхронизация + генерация typed clients
yarn twenty function:execute # Ручной запуск pre/post-install функций
```

### 7.7. Типичный паттерн приложения

1. Custom object (через `defineObject()` с полями и enum)
2. Logic function с route trigger (HTTP-обработчик, создающий записи)
3. Role с ограничением прав доступа к объекту
4. Опционально: Tool-обёртка для AI-интеграции
5. Опционально: React-компонент для кастомного UI

Все entity используют единый паттерн `universalIdentifier` для портируемости между workspace.

## 8. Импорт данных через API

### 8.1. Когда использовать

API-импорт подходит когда CSV непрактичен, особенно для наборов > 50 000 записей.

### 8.2. Ограничения производительности

| Параметр | Значение |
|----------|----------|
| Максимум запросов | 100/мин |
| Записей за batch | до 60 |
| Пиковая пропускная способность | ~6 000 записей/мин |
| 100 000 записей | ~17 минут |

### 8.3. Порядок импорта (учёт зависимостей)

1. **Companies** (базовые)
2. **People** (ссылаются на Companies)
3. **Opportunities** (ссылаются на Companies и People)
4. **Tasks/Notes** (гибкие связи)
5. **Custom objects** (учитывать их зависимости)

### 8.4. Best Practices

- Группировать записи в batch по ~60 штук
- Минимальная задержка между запросами: **600 мс**
- Exponential backoff при rate-limit ответах
- Валидировать данные заранее
- Тестировать на 10-20 записях перед полным импортом
- Использовать batch upsert для предотвращения дубликатов

## 9. Паттерны расширения для CredosCRM

### 9.1. Рекомендуемый подход

На основе возможностей Twenty, для CredosCRM рекомендуются следующие паттерны расширения:

#### Custom Objects для B2B/B2G

- **Проекты** — для отслеживания интеграционных проектов
- **Продукты/Решения** — каталог предлагаемых решений
- **Тендеры** — для B2G-направления (госзакупки)
- **Контракты** — управление договорами

#### Интеграции через Workflows

- **Exchange** → Webhook trigger для входящих событий + HTTP Request для исходящих
- **АТС Билайн** → Webhook trigger для звонков + Code action для обработки
- **1С УНФ** → Scheduled workflow для периодической синхронизации + HTTP Request

#### API для интеграций

- **Metadata API** — программное создание custom objects при развёртывании
- **GraphQL batch upsert** — синхронизация данных из 1С без дубликатов
- **Webhooks** — уведомление внешних систем о событиях CRM

#### Приложения (Apps)

- Кастомные Front Components для специфического UI Кредо-С
- Logic Functions с Route triggers для интеграционных endpoint-ов
- Database event triggers для реакции на изменения в CRM
