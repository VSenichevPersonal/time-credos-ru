## 1. Что такое Twenty

Twenty — open-source CRM, предоставляющая «строительные блоки» для создания системы под конкретные потребности бизнеса. Философия проекта: универсальные принципы и общие паттерны важнее фиксированных списков функций.

### Ключевые возможности

| Категория | Функции |
|-----------|---------|
| **Коммуникация** | Синхронизация почты и календаря, привязка к записям CRM |
| **Организация данных** | Заметки (Notes), задачи (Tasks), привязка к записям |
| **Модель данных** | Кастомные объекты и поля, импорт/экспорт через CSV и API |
| **Визуализация** | Table views, Kanban boards, воронки продаж, дашборды |
| **Автоматизация** | Workflows, интеграция через API и webhooks |
| **AI** | AI-агенты, чат-бот (в разработке) |
| **Безопасность** | Ролевая модель доступа, SSO |

### Преимущества

- **Кастомизация** — полная подстройка под бизнес-процессы
- **Open-source** — сообщество, прозрачность, развитие
- **Self-hosting** — возможность развёртывания на своих серверах, отсутствие vendor lock-in

## 2. Модель данных (Data Model)

Data Model — «чертёж» организации клиентской информации. Три базовых концепции:

- **Objects** (объекты) — категории данных (People, Companies, Opportunities)
- **Fields** (поля) — свойства объектов (Name, Email, Phone)
- **Records** (записи) — конкретные экземпляры данных внутри объектов

> **Важно:** синхронизация почты и календаря работает только с объектами People, Companies и Opportunities.

### Принципы проектирования

**Когда использовать поля:** для вариаций, категорий, статусов и атрибутов существующих объектов.

**Когда создавать объекты:** когда сущность имеет собственный жизненный цикл, множественные связи, открытые возможности привязки или уникальные свойства.

**Вопросы при проектировании:**
- Это свойство или самостоятельная сущность?
- Будет ли несколько экземпляров на запись?
- Нужны ли связи с несколькими объектами?
- Есть ли собственные этапы жизненного цикла?

**Рекомендации:** начинать с простой модели, использовать Views для фильтрации вместо дублирования объектов, создавать новые объекты только когда поля исчерпали свои возможности.

### 2.1 Объекты (Objects)

#### Стандартные объекты (Standard Objects)

| Объект | Описание |
|--------|----------|
| **People** | Контакты. Хранит контактные данные и историю взаимодействий |
| **Companies** | Компании-аккаунты. Отрасль, размер, локация. Связан с People и Opportunities |
| **Opportunities** | Сделки. Отслеживает продвижение потенциальных продаж от prospecting до закрытия. Визуализация через Kanban pipeline |
| **Notes** | Заметки произвольной формы. Привязываются к People, Companies, Opportunities и другим записям |
| **Tasks** | Задачи и to-do. Привязываются к People, Companies, Opportunities и другим записям |

#### Кастомные объекты (Custom Objects)

Организации могут создавать объекты для уникальных сущностей бизнеса (например, Projects, Products, Tickets, Listings).

**Создание кастомного объекта:**

1. Перейти в **Settings → Data Model**
2. Нажать **"+ New object"**
3. Ввести название (singular и plural), выбрать иконку, добавить описание
4. Нажать **Save**

Объект сразу появится в боковой панели.

**Управление объектами:**

- **Деактивация:** Settings → Data Model → переключатель на объекте. Данные сохраняются, объект скрывается из workspace
- **Реактивация:** вернуть переключатель. Все данные восстанавливаются
- **Удаление:** доступно после деактивации (необратимо)

### 2.2 Поля (Fields)

Поля — «столбцы в таблице», хранящие различные типы данных.

#### Типы полей

| Тип | Назначение |
|-----|-----------|
| **Address** | Структурированный адрес (улица, город, страна, индекс) |
| **Array** | Список текстовых значений |
| **Boolean** | Чекбокс true/false |
| **Currency** | Денежные значения с кодом валюты |
| **Date** | Дата |
| **Date & Time** | Дата и время |
| **Domain** | Доменное имя сайта (для Companies) |
| **Email** | Email-адреса (основной + дополнительные) |
| **JSON** | Структурированные JSON-данные |
| **Links** | URL с лейблами (primary/secondary) |
| **Long Text** | Многострочный текст |
| **Multi-Select** | Множественный выбор из предопределённых опций |
| **Number** | Целые и десятичные числа |
| **Phone** | Телефонные номера с кодом страны |
| **Rating** | Звёздный рейтинг (шкала 1–5) |
| **Relation** | Связь между объектами |
| **Select** | Одиночный выбор из предопределённых опций |
| **Text** | Однострочный текст |

**Итого: 18 типов полей.**

#### Стандартные vs кастомные поля

- **Стандартные поля** — встроенные (First Name, Last Name и т.д.). Нельзя удалить, но можно деактивировать. Опции SELECT-полей (например, Stage у Opportunities) можно настраивать
- **Кастомные поля** — создаются пользователем для любого объекта

#### Создание кастомного поля

1. **Settings → Data Model** → выбрать объект
2. Нажать **"Add Field"**
3. Указать имя, тип, описание (опционально)
4. Сохранить

**Быстрый доступ:** нажать кнопку **+** в таблице объекта → "Customize fields"

#### Управление полями

- **Деактивация:** меню (⋮) → "Deactivate". Поле скрывается, данные сохраняются, доступ через API остаётся
- **Уникальность (Uniqueness):** можно настроить для предотвращения дубликатов (например, email). Система отклонит изменение, если дубликаты уже существуют
- **Реактивация/удаление:** деактивированные поля можно восстановить или удалить навсегда

#### Правила именования

- Singular и plural названия должны быть различными (для GraphQL API)
- Зарезервированные имена (Type, Application и др.) использовать нельзя
- Каждый объект требует одно основное текстовое поле (Main Display Field), отображаемое в крайнем левом столбце при связях

### 2.3 Связи (Relation Fields)

#### Типы связей

| Тип | Описание | Пример |
|-----|----------|--------|
| **One-to-Many** | Одна запись → много записей | Одна Company → много People (сотрудники) |
| **Many-to-One** | Много записей → одна запись | Много People → одна Company |
| **Multi-object** | Объект связан с несколькими типами объектов | Note привязана к Person, Company и Opportunity |
| **Many-to-Many** | Записи обоих объектов связаны множественно (через junction-объект) | Много People ↔ много Projects |

> **Many-to-Many** требует включения в **Settings → Updates → Lab**.

#### Встроенные связи

- People → Companies (Many-to-One)
- Opportunities → Companies (Many-to-One)
- Opportunities → People (Many-to-One)

#### Создание связи

1. **Settings → Data Model** → выбрать объект
2. **"+ Add Field"** → тип **Relation**
3. Указать целевой объект
4. Настроить имена полей на обеих сторонах связи
5. Выбрать тип связи
6. Сохранить

#### Рекомендации

- Планировать модель данных заранее
- Использовать описательные названия
- Учитывать направление «владения»
- Избегать избыточных и циклических связей (влияет на производительность)

## 3. Представления и воронки (Views & Pipelines)

**Views** — сохранённые конфигурации отображения данных. Каждое представление независимо хранит layout, фильтры, сортировку и видимые поля.

### 3.1 Типы представлений

| Тип | Описание | Применение |
|-----|----------|-----------|
| **Table View** | Табличный интерфейс (строки и столбцы) | По умолчанию для большинства объектов |
| **Kanban View** | Карточки, организованные по стадиям | Воронки продаж, проектные доски |
| **Calendar View** | Визуализация записей на календаре | Дедлайны, расписание, планирование |

#### Создание представления

**Способ 1 — из выпадающего меню:**
1. Открыть селектор видов (верхний левый угол)
2. Нажать **"Add view"**
3. Задать имя, выбрать тип layout
4. Настроить фильтры, сортировку, видимые поля
5. Сохранить

**Способ 2 — из существующего:**
Изменить текущий вид → **"Save as new view"**

#### Управление представлениями

- **Редактирование:** изменить фильтры, сортировку, столбцы → сохранить
- **Переименование/иконки:** меню рядом с названием вида
- **Перестановка:** перетаскивание видов в dropdown
- **Избранное (Favorites):** закрепление часто используемых видов
- **Удаление:** необратимое действие с подтверждением
- **Видимость:** workspace-wide или personal-only (кроме default views)

### 3.2 Фильтры и сортировка (Filters & Sorting)

#### Типы фильтров по полям

| Тип поля | Доступные операторы |
|----------|-------------------|
| **Text** | Equals, Contains, Starts with, Ends with, Is empty |
| **Number** | Equals, Greater than, Less than, Between, Is empty |
| **Date** | Equals, Before, After, Between, Is empty |
| **Select** | Equals, Is any of, Is empty |
| **Checkbox** | Is true, Is false |
| **Relation** | Equals, Is empty |

#### Добавление фильтров

1. Нажать кнопку **Filter** в панели инструментов
2. Выбрать поле
3. Выбрать оператор
4. Ввести критерий
5. Применить

**Множественные фильтры** комбинируются логикой **AND** (все условия должны выполняться). Удаление: кнопка **X** на каждом фильтре или «очистить все».

#### Сортировка

1. Нажать кнопку **Sort**
2. Выбрать поле
3. Выбрать порядок: ascending (A-Z, 0-9) или descending (Z-A, 9-0)

**Многоуровневая сортировка:** первичная сортировка имеет приоритет, последующие применяются к записям с одинаковыми значениями.

**Быстрая сортировка:** клик по заголовку столбца переключает ascending → descending → без сортировки.

**Сохранение:** конфигурации автоматически сохраняются в рамках вида.

### 3.3 Kanban-представления

Kanban-вид визуально отображает процесс: каждый столбец — стадия, каждая карточка — запись.

#### Основные функции

- **Перемещение карточек:** drag-and-drop между стадиями
- **Компактный вид (Compact View):** Options → toggle "Compact view" — скрывает детали полей
- **Агрегации колонок:** Count, Sum, Average, Min, Max. Клик по числу вверху столбца для настройки

#### Управление стадиями

- **Добавление:** Settings → Data Model → объект → настройка Select-поля, используемого для Kanban
- **Удаление:** наведение на название стадии → "Edit from settings" → Delete

#### Настройка отображения полей

Options (верхний правый угол) → Fields → переключение иконки «глаз» для показа/скрытия полей. Перетаскивание для изменения порядка.

#### Рекомендации

- Оптимальное количество стадий: **5–7**
- Использовать описательные названия стадий
- Показывать только необходимые поля
- Регулярно обновлять позиции карточек

### 3.4 Табличные представления (Table Views)

Записи отображаются в строках с настраиваемыми столбцами — «как в таблице Excel».

#### Управление столбцами

- Показывать/скрывать столбцы (поля)
- Менять ширину столбцов перетаскиванием
- Менять порядок столбцов перетаскиванием

#### Группировка (Grouping)

1. Options → Group
2. Выбрать Select-поле для группировки
3. Порядок групп: алфавитный, обратный алфавитный или ручной (перетаскивание)
4. Переключатель «глаз» для видимости каждой группы

> **Производительность:** рекомендуется не более **10–15 видимых групп** на один вид.

#### Сценарии использования

- Быстрый просмотр большого количества записей
- Массовое редактирование
- Одновременный просмотр множества полей
- Группировка компаний по типу, сделок по стадии, задач по статусу

### 3.5 Настройка воронки продаж (Sales Pipeline)

Воронка продаж — это **Kanban-вид объекта Opportunities**, где каждый столбец — стадия процесса продаж.

#### Шаг 1: Настройка стадий

**Settings → Data Model → Opportunities** → поле **Stage**

Рекомендуемые стадии:
- **New** — новые возможности
- **Qualified** — подтверждённое соответствие
- **Meeting** — активные обсуждения
- **Proposal** — предложение отправлено
- **Negotiation** — обсуждение условий
- **Closed Won** — успешное закрытие
- **Closed Lost** — неуспешное закрытие

> Оптимально: **5–7 стадий**. Слишком много — сложно сканировать, слишком мало — теряется видимость прогресса.

#### Шаг 2: Создание вида Pipeline

1. Перейти в **Opportunities**
2. Dropdown видов → **"+ Add view"**
3. Назвать "Sales Pipeline"
4. В Options выбрать layout **Kanban**

#### Шаг 3: Настройка вида

- **Ключевые поля:** Options → Fields → показать Company, Amount, Close Date, Owner
- **Агрегации:** клик по числу стадии → выбрать метрику (Sum, Count, Average) для поля Amount
- **Compact View:** опционально для упрощённого обзора

#### Шаг 4: Несколько видов воронки

| Вид | Фильтр | Видимость |
|-----|--------|-----------|
| **My Pipeline** | Owner = Me | Personal (unlisted) |
| **Team Pipeline** | Без фильтра | Workspace |
| **Closing This Month** | Close Date = текущий месяц, исключены Closed Won/Lost | Workspace |

#### Практики работы

- Обновлять сделки ежедневно
- Поддерживать реалистичные даты закрытия
- Оперативно перемещать или закрывать устаревшие сделки

## 4. Интеграция почты и календаря (Calendar & Emails)

### 4.1 Подключение аккаунтов

#### Google

1. **Settings → Accounts**
2. Нажать **"Continue with Google"**
3. Авторизовать доступ к Gmail и Google Calendar
4. Настроить параметры синхронизации
5. Завершить подключение

#### Microsoft

1. **Settings → Accounts**
2. Нажать **"Continue with Microsoft"**
3. Авторизовать Outlook и Microsoft Calendar
4. Настроить параметры аналогично Google

#### Другие провайдеры

- **Почта:** настройка через **SMTP**
- **Календарь:** настройка через **CalDAV**

> **Ограничение:** подключить можно только настоящие почтовые ящики. Email-алиасы с пересылкой на другой ящик подключить нельзя.

### 4.2 Настройки почты

#### Уровни видимости писем

| Уровень | Что отображается |
|---------|-----------------|
| **Metadata** | Только отправитель, получатель, дата |
| **Subject** | Тема + метаданные |
| **Everything** | Полное содержимое писем, включая вложения |

#### Автоматическое создание контактов

| Режим | Описание |
|-------|----------|
| **Disabled** | Отключено |
| **All** | Контакты из всех внешних переписок |
| **Sent only** | Контакты только из исходящих писем |
| **Auto-link to Company** | Автоматическая привязка к компании по домену email |

#### Выбор папок для синхронизации

Можно указать конкретные папки: Inbox, Sent, кастомные папки. Можно исключить Spam и Trash.

### 4.3 Работа почтового ящика (Mailbox)

#### Как работает синхронизация

- Система привязывает входящие/исходящие письма к записям **People**, **Companies** и **Opportunities**
- Обновление каждые **5 минут** после начальной загрузки
- Скорость импорта: ~**400 писем в минуту** (ограничение Gmail API)
- 1 000 писем ≈ 2.5 минуты, 10 000 писем ≈ 25 минут

#### Что синхронизируется

- Внешние письма с контактами вне организации
- Автоматическая привязка к существующим записям People и Companies
- Несколько email-адресов одного контакта привязываются к одной записи

#### Что НЕ синхронизируется

- Письма между коллегами (один домен)
- Рассылки и групповые email
- Письма из исключённых папок

#### Просмотр писем

Перейти к записи Person, Company или Opportunity → вкладка **Emails** → полные цепочки переписки с участниками и timestamps.

### 4.4 Настройки календаря

#### Видимость событий

- **Полная информация** — все детали события
- **Ограниченная** — только дата и участники

#### Участники встреч

При включении опции: автоматическое создание контактов для участников встреч, которых нет в CRM.

## 5. Автоматизация (Workflows)

Workflows позволяют создавать собственные автоматизации вместо адаптации к жёстким функциям.

**Основные сценарии:**
1. **Внутренняя эффективность** — минимизация ручного ввода и повторяющихся задач
2. **Интеграция данных** — связь Twenty с внешними системами через API и webhooks

**Создание workflow:**
1. Открыть **Workflows** в главном меню
2. Создать новую запись, дать имя
3. Настроить триггер, действия, протестировать
4. Активировать

### 5.1 Триггеры (Triggers)

Каждый workflow инициируется одним триггером. Поддерживаются стандартные объекты CRM (People, Companies, Opportunities) и расширенные (Workspace Members, Calendar Events, Messages, Tasks, Notes).

| Триггер | Описание | Применение |
|---------|----------|-----------|
| **Record is Created** | При создании новой записи. Подходит для записей из CSV, mailbox, calendar sync, API. Не рекомендуется для ручного создания (auto-save вызывает преждевременное срабатывание) | Новые лиды |
| **Record is Updated** | При изменении записи. Можно указать конкретные поля для отслеживания | Синхронизация данных |
| **Record is Updated or Created** | Объединяет оба события. Полезно для импортов (API/CSV заполняют все поля сразу) | Импорты |
| **Record is Deleted** | При удалении записи | Очистка, уведомления |
| **Manual Trigger** | Запускается пользователем через `Cmd+K` или кнопку в навбаре. Режимы: **Global** (без записи), **Single** (выбранные записи, лимит ~100 запусков/мин), **Bulk** (все выбранные записи одним вводом, требует Iterator) | Пользовательские действия |
| **On a Schedule** | Периодический запуск: минуты, часы, дни или cron-выражения. Работает в **UTC** | Отчёты, периодические задачи |
| **Webhook** | Внешний вызов через GET или POST запрос. Предоставляет уникальный URL, требует определения структуры POST body | Внешние интеграции |

### 5.2 Действия (Actions)

#### Действия с записями (Record Actions)

| Действие | Описание |
|----------|----------|
| **Create a Record** | Создаёт новую запись в выбранном объекте. Поля заполняются из предыдущих шагов или вручную |
| **Update Record** | Изменяет существующую запись. Выбор записи статически (dropdown) или динамически из предыдущих шагов |
| **Delete Record** | Удаляет запись. Данные удалённой записи остаются доступными в последующих шагах |
| **Search Records** | Поиск записей по условиям фильтрации. Максимум: **200 записей**. Совет: использовать Branches после поиска для обработки «найдено/не найдено» |
| **Upsert Record** | Создаёт или обновляет запись по критерию совпадения (email для People, domain для Companies, ID, Unique-поля) |

#### Управление потоком (Flow Actions)

| Действие | Описание |
|----------|----------|
| **Iterator** | Цикл по массиву записей из предыдущего шага. Доступ к полям текущего элемента через "Current Item" |
| **Filter** | Фильтрация по условиям. Если условие не выполнено — workflow останавливается |
| **Delay** | Пауза на указанную длительность (дни, часы, минуты, секунды) или до заданной даты/времени. Потребление кредитов: 1 при запуске, 0 во время ожидания |

#### Коммуникации

| Действие | Описание |
|----------|----------|
| **Send Email** | Отправка email из workflow. Требует подключённого email в Settings. Один получатель. Поддержка переменных, форматирования, списков, вложений. HTML-подписи не поддерживаются |
| **Form** | Форма для сбора пользовательского ввода (только для Manual trigger). Типы полей: text, number, date, record, select. Поля не могут быть обязательными |

#### Интеграции

| Действие | Описание |
|----------|----------|
| **Code** | Выполнение пользовательского JavaScript. Доступ к переменным из предыдущих шагов, возврат переменных для последующих шагов. API-ключи встраиваются прямо в код |
| **HTTP Request** | Запрос к внешнему API. Методы: GET, POST, PUT, PATCH, DELETE. Настройка headers и sample response для preview структуры |

#### AI (в разработке)

| Действие | Описание |
|----------|----------|
| **AI Agent** | Выбор агента, промпт с переменными. Возможности: анализ данных, классификация, генерация текста, принятие решений, взаимодействие с данными CRM. Потребляет кредиты в зависимости от модели |

#### Рекомендации по workflow

- Переименовывать шаги для ясности
- Использовать данные из предыдущих шагов (ссылки на переменные)
- Начинать с простых workflow
- Планировать логику перед построением

## 6. Навигация и советы по работе

### Основная раскладка

- **Центральная область** — отображение записей, создание/редактирование/удаление
- **Боковая панель (слева):** переключение workspaces, поиск, настройки, избранные виды, объекты, workflows
- **Side Panel (справа):** быстрый просмотр записи по клику. Можно развернуть на полный экран

### Command Menu (Cmd+K / Ctrl+K)

Также доступно через три точки в правом верхнем углу. Возможности:

- Создание новых записей
- Импорт/экспорт данных через CSV
- Создание новых видов
- Доступ к удалённым записям
- Просмотр горячих клавиш

### Поиск

Доступен через:
- Command Menu
- Верхняя часть навигационной панели
- Клавиша `/` для мгновенного фокуса

Поиск работает по всем объектам.

### Горячие клавиши

| Клавиша | Действие |
|---------|----------|
| `Cmd+K` / `Ctrl+K` | Открыть Command Menu |
| `/` | Фокус на поиске |

### Настройки (Settings)

Доступ через иконку в верхнем левом углу боковой панели:

- Подключение почты и календаря
- Настройка модели данных
- Настройка API и webhooks
- Управление разрешениями
- Приглашение участников команды
- Персональные настройки
- Биллинг и поддержка

### Общие советы

- Используйте **несколько видов** для одного объекта (личный pipeline, командный pipeline, фильтр по месяцу)
- **Favorites** — закрепляйте часто используемые виды
- **Группировка** в Table View — альтернатива Kanban для быстрого обзора
- **Agрегации** в Kanban — показывайте суммы, средние значения по стадиям
- **Compact View** в Kanban — для обзора на высоком уровне
- Начинайте с простой модели данных и усложняйте по мере необходимости
- Регулярно обновляйте воронку продаж — данные актуальны только при дисциплине
