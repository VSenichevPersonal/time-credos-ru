# Структура проектов и модели управления

Обновлено: 19.08.2025

В системе поддерживаются три модели управления проектами:

- **Каскадная** (Waterfall):
  - классический последовательный подход;
  - чёткое планирование всех этапов заранее.
- **Гибкая** (Agile):
  - итеративный подход с короткими циклами;
  - возможность адаптации к изменениям.
- **Гейтовая** (Stage-Gate):
  - разделение на фазы с контрольными точками ([подробнее](https://timetta.com/ru/docs/projects/conceptions/stage-gate));
  - формальные процедуры перехода между этапами.

Модели можно использовать по отдельности или комбинировать в рамках одного проекта. Такой подход обеспечивает гибкость, но требует работы с множеством сущностей и понимания связей между ними.

## Структура данных для управления проектами

![Структура проектов](https://timetta.com/entity-assets/47947296-b3c6-4183-9467-2ed87d8e89b7/structure1.png)

### Базовые сущности

- **Портфель** — набор проектов, ориентированный на достижение стратегических целей компании. Может включать отдельные проекты, программы и другие портфели.
  Пример: «Портфель блока ИТ».
- **Программа** — группа взаимосвязанных проектов с единым бюджетом и общей целью.
  Пример: «Внедрение ERP».
- **Проект** — временная деятельность, направленная на создание уникального продукта, услуги или результата в заданные сроки.
  Пример: «Внедрение финансового модуля ERP».

**Проект** — ключевая сущность в системе. К нему привязываются учёт времени, счета, акты, бюджет и другие данные.

### Сущности каскадной модели

- **Работа** (элемент ИСР, иерархической структуры работ) — задача, веха или этап, полученные в результате декомпозиции плана. Может отображаться в виде иерархии (структурная декомпозиция) или таймлайна (диаграмма Ганта).
- **Ресурсный план** — повременная оценка трудозатрат и себестоимости, необходимых для выполнения работ.

**ИСР** и **Ресурсный план** формируют основу для бюджета проекта. Другого способа корректно оценить трудозатраты и себестоимость в системе нет.

Создавать иерархию работ необязательно. Достаточно использовать одну автоматически сформированную суммарную **Работу** и учитывать для неё рабочее время, оценку трудозатрат и экономику.

### Сущности гейтовой модели

- **Модель проекта** — предопределённый набор фаз и контрольных точек. Можно создать несколько моделей и применять их к разным проектам.
- **Фаза** — этап проекта или программы. Выбирается из справочника «Фазы проектов».
- **Контрольная точка** — ключевой момент проекта, фиксирующий завершение важного этапа или достижение значимого результата.
  Этапная контрольная точка (гейт) обязательна для перехода к следующей фазе реализации.

**Контрольные точки** могут быть связаны с вехами в структуре работ. Это обеспечивает интеграцию гейтовой и каскадной моделей управления.

### Сущности гибкой модели

- **Задача** — атомарная единица работы. В Scrum должна быть выполнена в течение одного спринта.
- **Тип задачи** — определяет классификацию и жизненный цикл задачи.
- **Доска** — рабочее пространство для управления задачами.
- **Спринт** — фиксированный период работы (обычно 1–4 недели). Привязан к **Доске**.

Каждая **Задача** обязательно связана с проектом и его **Работой**. Это обеспечивает интеграцию каскадной и гибкой моделей управления.

#### Типы досок по отношению к проекту

- проектная — отображает задачи конкретного проекта;
- комбинированная — отображает задачи по фильтрам из разных проектов.

#### Форматы досок

- канбан-доска — набор колонок по состояниям. Можно создавать произвольные колонки;
- скрам-доска — предполагает организацию бэклога и использование спринтов. Задача может принадлежать только одному спринту.

#### Сценарии использования досок

- работа с доской и спринтами в рамках одного проекта. Например, проект разработки с выделенной командой;
- работа с объединённой доской и спринтами для нескольких проектов. Например, доска команды внедрения, ведущей несколько проектов.

## Дополнительные поля и классификаторы

С помощью настройки можно добавить дополнительные поля (например, номер договора, куратор) и классификаторы (тип, важность) в следующие сущности:

- проект;
- программа;
- портфель;
- работа проекта;
- задача;
- контрольная точка.

## Информационное наполнение проекта

Кроме структуры, проект включает следующее информационное наполнение:

- **Команда** — набор ресурсов: пользователей, универсальных ресурсов, подразделений, вакансий (*скоро*).
- **Бюджет** — включает оценку выручки, затрат, себестоимости, биллинга, а также тарифы, правила периодических затрат, финансовые риски.
- **Риски** — потенциальные события или условия, которые могут повлиять на достижение целей.
- **Документы** — документы в структурированном или слабо структурированном виде. Например, паспорт проекта, запросы изменений, регулярные отчёты.

## Версионирование проекта

**Версионирование** — создание альтернативных версий содержимого проекта. Цели использования версий:

- фиксация плана — мастер-версия;
- сценарный анализ — подготовка альтернативных вариантов проекта;
- процесс актуализации прогноза — регулярные статус-отчёты на определённый момент времени;
- работа с запросами на изменение — изменение в виде новой версии может быть «влито» в основной проект.
