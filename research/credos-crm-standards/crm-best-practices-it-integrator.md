## 1. Специфика CRM для ИТ-компаний и системных интеграторов

### 1.1. Особенности продаж в ИТ-интеграции

ИТ-компании и системные интеграторы работают в уникальной среде, где:

- **Длинный цикл сделки** — от 3 до 12 месяцев (в среднем 3–6 месяцев для SaaS/Technology-сектора). Решение о покупке в B2B редко принимается импульсивно.
- **Множество лиц, принимающих решения (ЛПР)** — в среднем 6,8 человек участвует в принятии решения по одной сделке (данные HubSpot). Среди них: CTO/ИТ-директор, CFO, служба безопасности, юристы, закупки.
- **Нелинейность воронки** — клиенты двигаются вперёд и назад между этапами, меняются ЛПР, возникают новые потребности.
- **Сложная оценка стоимости** — каждый проект уникален, требует предварительного пресейла, расчётов, пилотных проектов (POC).
- **Двойной сегмент: B2B + B2G** — работа с коммерческими компаниями и с госзаказчиками через тендерные процедуры (44-ФЗ, 223-ФЗ).
- **Послепродажное сопровождение** — внедрение, поддержка и развитие проектов как продолжение продажи.

### 1.2. Типичные воронки продаж

#### Воронка прямых продаж B2B

| Этап | Описание | Ключевые действия |
|------|----------|-------------------|
| **1. Выявление потребности** | Первичный контакт, квалификация лида | Оценка ICP, скоринг лида, обогащение данных |
| **2. Анализ потребностей** | Discovery-звонки, погружение в боли | Карта ЛПР, выявление болей, оценка бюджета |
| **3. Подготовка предложения** | Проектирование решения, расчёт ROI | КП с версионированием, калькуляция, ТЗ |
| **4. Презентация / Демо** | Демонстрация продукта, технический пресейл | Демо, POC, техническая презентация |
| **5. Переговоры** | Обсуждение условий, SLA | История переговоров, условия оплаты |
| **6. Согласование** | Внутренние согласования у клиента | Работа с 6–8 ЛПР, юридическая проверка |
| **7. Закрытие** | Подписание договора, старт проекта | Договор, оплата, передача в проект |

> **Важно:** «B2B-продажи нельзя уложить в простую воронку "заявка → встреча → договор"» — каждый этап требует конкретных действий и автоматизации.

#### Воронка тендерных продаж B2G

| Этап | Описание | Ключевые действия |
|------|----------|-------------------|
| **1. Мониторинг** | Отслеживание тендеров на ЕИС и площадках | Автоматический поиск, фильтрация по стоп-факторам |
| **2. Анализ ТЗ** | Оценка технического задания | Проверка соответствия, расчёт рентабельности |
| **3. Подготовка заявки** | Сбор документов, расчёт цены | Шаблоны документов, автозаполнение, чек-листы |
| **4. Подача заявки** | Оформление и отправка на площадку | Контроль дедлайнов, ЭЦП, проверка полноты |
| **5. Участие в торгах** | Торги / аукцион / конкурс | Стратегия ценообразования, мониторинг конкурентов |
| **6. Результат** | Выигрыш / проигрыш | Анализ причин, обратная связь |
| **7. Исполнение** | Реализация контракта | Переход в проектное управление |

#### Воронка послепродажного сопровождения

| Этап | Описание |
|------|----------|
| **Онбординг** | Передача от продаж в проектную команду |
| **Внедрение** | Этапы проекта, вехи, задачи |
| **Приёмка** | Тестирование, акты |
| **Поддержка** | SLA-обращения, тикеты |
| **Развитие** | Допродажи, расширение проекта |

### 1.3. Управление длинными сделками

Главная проблема длинных сделок — «зависание» без движения. CRM решает это через:

- **Автоматические напоминания** — задачи на каждом этапе с дедлайнами.
- **Уведомления о простое** — алерт, если сделка не двигалась N дней (рекомендуется: 7 дней для активной фазы, 14 дней для этапа согласования).
- **Контроль активности** — отчёты по менеджерам: количество касаний, звонков, встреч по каждой сделке.
- **Отчёты по «застрявшим» сделкам** — дашборд с визуализацией сделок без прогресса.
- **Взвешенный прогноз** — каждый этап имеет вероятность закрытия, что позволяет строить прогноз выручки.

### 1.4. Тендерные продажи B2G

Для системного интегратора тендеры — один из ключевых каналов. CRM должна:

- Интегрироваться с тендерными площадками (ЕИС, B2B-Center и др.).
- Фиксировать все документы и версии файлов в карточке тендера.
- Контролировать дедлайны подачи с автоматическими напоминаниями.
- Вести историю участия: выигрыши, проигрыши, причины.
- Автоматически проверять заполненность полей перед подачей.
- Рассчитывать стоимость, трудозатраты и рентабельность.

> «Каждая ошибка или задержка может стоить компании крупного контракта» — критичность контроля сроков в тендерах.

### 1.5. Управление проектами после продажи

После выигрыша тендера или закрытия сделки CRM должна обеспечить бесшовный переход:

- Автоматическое создание проекта из выигранной сделки.
- Разбиение на задачи с назначением ответственных.
- Связь проекта с карточкой клиента и сделки.
- Отслеживание хода работ на канбан-доске.
- Контроль бюджета и трудозатрат.
- Формирование актов и закрывающих документов.

## 2. Лучшие практики настройки CRM для профессиональных услуг

### 2.1. Структура воронок

**Принцип:** каждая воронка должна отражать реальный процесс принятия решений клиентом, а не внутренние этапы компании.

**Антипаттерн:** расплывчатые этапы вроде «Переговоры» или «В работе» — они маскируют реальный прогресс и мешают автоматизации.

**Рекомендуемая структура для системного интегратора:**

```
Пресейл-воронка:
  Лид → Квалификация → Анализ потребностей → КП → Демо/POC → Переговоры → Закрытие

Тендерная воронка:
  Мониторинг → Анализ ТЗ → Решение об участии → Подготовка → Подача → Торги → Результат

Проектная воронка:
  Старт → Обследование → Проектирование → Внедрение → Тестирование → Приёмка → Поддержка

Сервисная воронка:
  Обращение → Диагностика → Решение → Контроль качества → Закрытие
```

**Критерии перехода между этапами** должны быть чёткими и объективными:
- Квалификация → Анализ: подтверждён бюджет и сроки.
- КП → Демо: КП отправлено и получен отклик.
- Переговоры → Закрытие: согласованы все условия, договор на подписании.

### 2.2. Карточка клиента (Компания)

#### Обязательные поля

| Группа | Поля |
|--------|------|
| **Основное** | Название, ИНН, ОГРН, юр. адрес, факт. адрес, сайт |
| **Классификация** | Сегмент (B2B / B2G), отрасль, размер (СМБ / Крупный / Госсектор) |
| **Финансы** | Годовой оборот, кредитный рейтинг, условия оплаты |
| **Отношения** | Статус клиента (потенциальный / активный / бывший), источник привлечения |
| **ИТ-специфика** | Текущая ИТ-инфраструктура, используемые системы, стек технологий |
| **Закупки** | Тип закупок (прямые / тендер), ЭТП, аккредитация |
| **История** | Дата первого контакта, все сделки и проекты (связи) |

#### Поля для B2G-клиентов (дополнительно)

- Код ОКПД, вид закупки (44-ФЗ / 223-ФЗ)
- Бюджетный цикл, плановые закупки
- Реестровый номер организации
- История участия в тендерах данного заказчика

### 2.3. Карточка контакта (ЛПР)

| Поле | Описание |
|------|----------|
| **ФИО, должность** | Полные данные |
| **Роль в сделке** | Экономический покупатель / Технический эксперт / Пользователь / Согласующий / Блокирующий |
| **Уровень влияния** | Высокий / Средний / Низкий |
| **Отношение к компании** | Лоялен / Нейтрален / Скептичен |
| **Каналы связи** | Телефон, email, мессенджер, предпочтительный канал |
| **История взаимодействий** | Все звонки, встречи, письма — автоматически из интеграций |
| **Связи** | Кому подчиняется, на кого влияет, с кем конкурирует внутри |

> В B2B сделке критически важна «карта влияния» — визуализация связей между ЛПР, их ролями и отношением к вашей компании.

### 2.4. Сегментация: B2B vs B2G

| Параметр | B2B | B2G |
|----------|-----|-----|
| **Цикл сделки** | 3–6 мес. | 2–12 мес. (бюджетный цикл) |
| **Процесс закупки** | Прямые переговоры | Тендерные процедуры |
| **ЛПР** | CTO, CEO, CFO | Контрактный управляющий, комиссия |
| **Ценообразование** | Гибкое | НМЦ, СГОЗ, ограничения |
| **Документооборот** | Договор, акт | Контракт, ТЗ, заявка, протоколы |
| **Оплата** | По договору | Бюджетный процесс, казначейство |
| **Риски** | Отказ, смена приоритетов | Жалобы, ФАС, штрафы |

**Рекомендация:** вести отдельные воронки для B2B и B2G, поскольку процессы принципиально различаются.

### 2.5. Управление контактами и ЛПР

**Методология BANT для квалификации:**
- **Budget** — есть ли бюджет?
- **Authority** — кто принимает решение?
- **Need** — есть ли реальная потребность?
- **Timeline** — каковы сроки?

**Карта закупочного комитета** — обязательный элемент для каждой крупной сделки:

```
Спонсор проекта (CEO/CTO)
├── Технический эксперт (ИТ-директор)
│   └── Пользователи (отделы)
├── Финансовый согласующий (CFO)
├── Юрист
└── Закупки
```

Для каждого ЛПР фиксируется:
- Роль в процессе принятия решения.
- Степень влияния и лояльности.
- Персональные мотивы и болевые точки.
- История всех коммуникаций.

## 3. Метрики и KPI для CRM в ИТ-сфере

### 3.1. Метрики воронки продаж

| Метрика | Формула | Целевое значение (ИТ-интеграция) |
|---------|---------|----------------------------------|
| **Конверсия Лид → Квалификация** | Квалифицированные / Все лиды | 20–30% |
| **Конверсия Квалификация → КП** | КП отправлено / Квалифицированные | 40–60% |
| **Конверсия КП → Переговоры** | В переговорах / КП отправлено | 30–50% |
| **Конверсия Переговоры → Закрытие** | Выиграно / В переговорах | 25–40% |
| **Общий Win Rate** | Выигранные сделки / Все сделки | 15–25% |
| **Win Rate тендеров** | Выигранные тендеры / Поданные заявки | 20–35% |

### 3.2. Метрики цикла сделки

| Метрика | Формула | Ориентир |
|---------|---------|----------|
| **Средний цикл сделки** | Сумма дней всех сделок / Количество сделок | 90–180 дней |
| **Среднее время на этапе** | Дни на этапе / Количество сделок на этапе | Зависит от этапа |
| **Pipeline Velocity** | (Кол-во сделок × Средний чек × Win Rate) / Цикл сделки | Рост квартал к кварталу |
| **Время реакции на лид** | Время от получения лида до первого касания | < 4 часов |

### 3.3. Финансовые метрики

| Метрика | Формула | Описание |
|---------|---------|----------|
| **Средний чек сделки** | Общая выручка / Количество сделок | Трек по кварталам |
| **MRR** (Monthly Recurring Revenue) | Средний доход с клиента × Количество клиентов | Для сервисных контрактов |
| **ARR** (Annual Recurring Revenue) | MRR × 12 | Годовой рекуррентный доход |
| **Customer Lifetime Value (CLV)** | Средний чек × Частота покупок × Срок жизни клиента | Ключевая метрика для допродаж |
| **Customer Acquisition Cost (CAC)** | (Затраты на маркетинг + продажи) / Новые клиенты | CLV/CAC > 3 — хороший показатель |
| **Чистый новый доход** | Новые клиенты × Средний чек | За месяц / квартал |

### 3.4. Метрики для менеджеров

| Метрика | Описание |
|---------|----------|
| **Количество касаний** | Звонки, встречи, письма за период |
| **Конверсия по менеджерам** | Win Rate каждого менеджера |
| **Объём воронки** | Сумма сделок в работе у менеджера |
| **Потерянные сделки по причинам** | Классификация: цена, конкурент, отказ, бюджет, сроки |
| **Потерянные сделки по этапам** | На каком этапе чаще всего «отваливаются» |

### 3.5. Метрики удержания

| Метрика | Формула |
|---------|---------|
| **Renewal Rate** | Продлившие / Все на продлении |
| **Churn Rate** | Ушедшие / Все клиенты |
| **Churn по причинам** | Группировка по причинам ухода |
| **NPS** | Промоутеры − Критики (шкала 1–10) |
| **Доля допродаж** | Доход от существующих / Общий доход |

### 3.6. Дашборды — рекомендуемая структура

**Правило:** 5–8 метрик на дашборд, баланс опережающих (leading) и запаздывающих (lagging) показателей.

**Дашборд руководителя:**
- Pipeline Velocity (тренд)
- Прогноз выручки (взвешенный)
- Win Rate (тренд)
- Средний цикл сделки (тренд)
- Топ-10 сделок по сумме

**Дашборд менеджера:**
- Мои сделки по этапам
- Задачи на сегодня
- Застрявшие сделки (без активности > 7 дней)
- Конверсия по моим сделкам
- План / факт

**Дашборд тендеров:**
- Активные тендеры с дедлайнами
- Win Rate тендеров
- Средняя маржинальность выигранных
- Тендеры по статусам

## 4. Типичные ошибки внедрения CRM в ИТ-компаниях

### 4.1. Стратегические ошибки

#### Ошибка 1: CRM как «волшебная таблетка»
Руководство ожидает, что внедрение CRM само по себе решит все проблемы продаж. На практике CRM — это инструмент, который усиливает существующие процессы, но не заменяет их. Без продуманных процессов CRM лишь автоматизирует хаос.

#### Ошибка 2: Внедрение без анализа процессов
Компании «на ходу» создают воронки, не отражающие реальный процесс продаж. Результат — менеджеры не узнают свою работу в системе и саботируют её.

**Правильно:** сначала описать AS-IS процессы, затем спроектировать TO-BE, и только потом настраивать CRM.

#### Ошибка 3: Отсутствие спонсора проекта
Без активной поддержки руководства внедрение CRM обречено. Менеджеры воспринимают систему как «инструмент контроля сверху», а не как помощника.

### 4.2. Архитектурные ошибки

#### Ошибка 4: Неправильная объектная модель
Отгрузки, договоры и платежи реализованы как множественные поля в карточке вместо отдельных объектов. Данные загружаются медленно, аналитика невозможна.

#### Ошибка 5: Все поля — строковые
Даты записываются в разных форматах, числа — как текст. Фильтрация и аналитика не работают. Рекомендация: использовать типизированные поля (дата, число, список, связь).

#### Ошибка 6: Избыток функциональности
Перегруженный интерфейс отталкивает пользователей. Лучше скрыть неиспользуемые функции и раскрывать их по мере роста зрелости команды.

### 4.3. Процессные ошибки

#### Ошибка 7: Логические ошибки в бизнес-процессах
Непродуманные автоматизации создают замкнутые циклы. Отсутствие замещающих ролей приводит к зависанию согласований (менеджер в отпуске — сделка стоит).

#### Ошибка 8: Расплывчатые этапы воронки
Этапы вроде «В работе» или «Переговоры» без чётких критериев перехода. Менеджеры субъективно оценивают прогресс, прогнозы не работают.

### 4.4. Организационные ошибки

#### Ошибка 9: Недостаточное обучение
Разные роли (продажи, маркетинг, руководство) нуждаются в разном обучении. Единое «обучение за час» не работает.

#### Ошибка 10: Внедрение сразу в продуктивной среде
Без тестовой среды любое обновление рискует сломать существующие процессы и данные.

#### Ошибка 11: Отсутствие документации изменений
При развитии системы нет записей о том, что и зачем менялось. Через полгода никто не помнит логику настроек.

#### Ошибка 12: Нетерпение к результатам
CRM даёт отдачу не сразу. Если через месяц нет «волшебных результатов», руководство опускает руки вместо того, чтобы углубить настройку и вовлечение.

### 4.5. Интеграционные ошибки

#### Ошибка 13: Слабая интеграция с другими системами
CRM в изоляции от почты, телефонии, 1С и других систем заставляет менеджеров вводить данные вручную — и они перестают это делать.

#### Ошибка 14: Игнорирование прав доступа
Роли назначаются индивидуально, а не группами. При изменении оргструктуры приходится перенастраивать десятки пользователей.

## 5. Рекомендации для Кредо-С

Исходя из специфики компании (системный интегратор, 15–20 пользователей, B2B+B2G, Москва), предлагаются следующие приоритеты:

### 5.1. Воронки (реализовать в первую очередь)

1. **Основная воронка B2B:** Лид → Квалификация (BANT) → Анализ потребностей → КП → Демо/POC → Переговоры → Договор → Закрыто (выиграно/проиграно).
2. **Тендерная воронка B2G:** Мониторинг → Анализ ТЗ → Решение → Подготовка → Подача → Торги → Результат.
3. **Проектная воронка:** Старт → Обследование → Внедрение → Тестирование → Приёмка → Поддержка.

### 5.2. Карточка клиента — минимальный набор

- Название, ИНН, сегмент (B2B/B2G), отрасль, размер.
- Текущая ИТ-инфраструктура, используемые системы.
- ЛПР с ролями и уровнем влияния.
- История всех сделок и проектов.
- Источник привлечения.

### 5.3. Критические интеграции

- **Exchange** — автоматическая привязка писем к сделкам и контактам.
- **АТС Билайн** — звонки привязаны к карточке клиента, записи разговоров.
- **1С УНФ** — синхронизация счетов, оплат, актов.

### 5.4. KPI для старта (MVP дашбордов)

- Pipeline Velocity.
- Win Rate (B2B отдельно, B2G отдельно).
- Средний цикл сделки.
- Застрявшие сделки (без активности > 7 дней).
- Конверсия по этапам воронки.

### 5.5. План внедрения

1. **Аудит процессов AS-IS** (2 недели) — описать текущие процессы продаж.
2. **Проектирование TO-BE** (1 неделя) — спроектировать воронки и карточки.
3. **Базовая настройка** (2 недели) — воронки, поля, роли, права.
4. **Интеграции** (2–4 недели) — Exchange, АТС, 1С.
5. **Обучение** (1 неделя) — по ролям: менеджеры, руководство, проектный офис.
6. **Пилот** (2–4 недели) — работа 3–5 менеджеров, сбор обратной связи.
7. **Масштабирование** — подключение всех пользователей, дашборды, автоматизация.
