## ОЦЕНКА РИСКОВ

### 🟢 Где ошибок НЕ будет (есть точная модель)

| Компонент | Источник | Уверенность |
|-----------|----------|-------------|
| Структура БД (таблицы, поля, типы) | Timetta $metadata + Kimai Doctrine | 100% |
| Связи между сущностями | Оба источника | 100% |
| Формат данных (Decimal vs INT) | Оба источника → выбрали Decimal | 100% |
| REST API эндпоинты | Kimai Swagger + Timetta OData | 100% |
| Аутентификация | Kimai (SAML/LDAP/DB) + Timetta (OIDC) | 100% |
| Справочники и enum'ы | Оба источника | 100% |
| Meta Fields (key-value) | Kimai — готовая реализация | 100% |

### 🟡 Где риски минимальны (модель есть, логику восстанавливаем)

| Компонент | Риск | Как минимизируем |
|-----------|------|-----------------|
| Трекинг времени (begin→end) | Kimai даёт точную логику | Копируем Kimai + добавляем сетку Timetta |
| Инвойсинг | Kimai калькулятор открыт | Берём Kimai-логику |
| Бюджеты | Kimai BudgetTrait открыт | Копируем Kimai |
| Team ACL | Kimai many-to-many открыт | Копируем Kimai |
| Ставки (rates) | Kimai 4-уровневая модель открыта | Копируем Kimai |

### 🔴 Где риски высоки (чёрный ящик, пишем сами)

| Компонент | Почему риск | Как снижаем |
|-----------|------------|-------------|
| **State machines (Lifecycles)** | Только модель состояний, логика переходов неизвестна | Начать с простых статусов (Kimai), добавить машины позже |
| **AI-агенты** | Timetta агент — чёрный ящик в K8s | Отложить на фазу 3 |
| **P&L/финансовая отчётность** | Формулы расчёта скрыты | Начать с простого Kimai-инвойсинга |
| **Интеграция с 1С** | Формат обмена неизвестен | Отложить, использовать Dadata для старта |

## ПЛАН: НЕ УСЛОЖНЯТЬ

### Принцип: начать с Kimai-простоты, расти до Timetta-глубины

```
Фаза 1 (2 недели): ЯДРО — как Kimai, только лучше
  ✅ Customer → Project → Activity → Timesheet (4 уровня)
  ✅ begin/end + duration + Decimal часы
  ✅ REST API + Swagger
  ✅ Auth: DB + SAML + 2FA (Kimai)
  ✅ PostgreSQL (Timetta-выбор, надёжнее MySQL)
  ✅ Team ACL (Kimai)
  ✅ Meta Fields (Kimai)

Фаза 2 (2 недели): РАСШИРЕНИЕ — добираем Timetta-мощь
  ✅ Недельная сетка TimeSheet→Line→Allocation (Timetta)
  ✅ Бюджеты на всех уровнях (Kimai)
  ✅ Инвойсинг 5 форматов (Kimai)
  ✅ Ставки 4 уровня (Kimai)
  ✅ Плагинная система (Kimai)

Фаза 3 (2 недели): ПРОДВИНУТОЕ — если нужно
  ⚠️ State machines (Timetta)
  ⚠️ CRM: Deals, Contacts (Timetta-light)
  ⚠️ Отчёты и дашборды
```

## ВЕРДИКТ

```
✅ ДА, можем написать без критических ошибок
✅ ДА, можем сделать лучше обеих (взяв сильные стороны каждой)
✅ ДА, не усложняя (начав с Kimai-простоты, добавляя Timetta-глубину по потребности)

Ключевой инсайт:
Kimai — идеальный стартовый шаблон (открытый код, простота)
Timetta — идеальная целевая модель (глубина, экосистема РФ)
Наша система = Kimai + Timetta-фичи по потребности
```

### Почему не будет критических ошибок:
1. **Структура БД** — сверяем с двумя независимыми источниками
2. **Бизнес-логика трекинга** — Kimai даёт точный PHP-код
3. **API-дизайн** — Kimai Swagger + Timetta OData как референсы
4. **Auth** — Kimai SAML/LDAP/DB + Timetta OIDC как референсы
5. **Форматы/типы** — выбрали лучшее из обеих (Decimal, Meta Fields)

### Единственный реальный риск:
Сложность State Machines Timetta. Решение: начать с простых статусов (enum), добавить машины когда реально понадобятся.
