# REQ-0009 — Заявка на пресейл/пилот → согласование → план

**Статус:** PROPOSED (БЭКЛОГ — заказчик 2026-06-21)
**Приоритет:** 🟡 (в рамках функционала Планирования; после OLAP + ввод планов v2/аллокации)

---

## Сверка с лучшими практиками / референсами

| Источник | Паттерн |
|---|---|
| **Timetta** | Pre-sale/Opportunity → **предварительная бронь (tentative booking)** ресурсов → при выигрыше сделки конвертируется в план проекта. Мягкая бронь видна продажам как «если выиграем». |
| **Kantata/PSA** | Pipeline: запрос → review → soft-booking → hard-booking (confirmed) при win. |
| **Twenty native** | **Workflows** (defineLogicFunction: database-event / cron / triggers) + **Tasks** + **Opportunities** (CRM) — можно собрать согласование без своего движка. |
| **Наш CAPACITY §4** | уже заложен сценарий: проект категории Presale/Pilot с plannedEffort = «предварительная бронь», опц. флаг `tentative` (отделять цветом/фильтром). |

**Вывод:** паттерн = «opportunity/заявка → tentative-бронь → confirmed-план». Совпадает с Timetta. Реализуемо на нативных Twenty Workflows + Tasks (не переусложнять — не писать свой воркфлоу-движок).

---

## Суть (заказчик)

Продажник кидает **заявку на пилот/пресейл** → она **согласуется** (workflow + задачи) → если ОК → **встаёт в план** отдела или конкретных людей. Уровень детализации плана зависит от того, насколько детальна заявка — на этапе заполнения / согласования / принятия производством в работу.

---

## Замысел (когда возьмём)

1. **Заявка.** Объект `credosTimePlanRequest` ИЛИ переиспользовать CRM **Opportunity** (если общий workspace). Поля: инициатор(сейл), клиент, отдел(ы)/категория (Presale/Pilot), оценка часов, желаемые сроки, уровень детализации (отдел / конкретные люди), статус.
2. **Согласование — нативный Twenty Workflow** (database-event на создание/смену статуса) + **Task** ответственному (рук. отдела/производство): новая → на согласовании → согласована / отклонена / на доработку. Без своего движка.
3. **Конвертация в план (по стадии детализации):**
   - заполнение грубо (отдел, часы) → **tentative-бронь на отдел** (capacity, флаг `tentative`, отдельный цвет — уже в CAPACITY §4);
   - детализировано до людей → **tentative-аллокации по людям** (REQ-0004);
   - принято производством в работу → бронь становится **confirmed** (обычный план/проект), tentative снимается.
4. **Видимость продажам:** в Планировании видно «если выиграем» (tentative) отдельно от подтверждённого → продажи обещают клиенту реальный старт с учётом броней.

---

## Зависимости / границы
- Опирается на **REQ-0004** (аллокации по людям) для детального уровня + **CAPACITY §4** (tentative-бронь) для уровня отдела.
- Согласование — **нативные Twenty Workflows + Tasks** (изучить `research/twenty-sdk` logic/workflows перед дизайном). Не свой движок.
- Если Opportunity из CRM — зависит от прод-топологии (ADR-0005, синк/общий workspace).
- RBAC: кто согласует (рук. отдела/производство) — RBAC-волна.
- Не переусlöжнять: старт — грубая бронь на отдел + статус-workflow; детализация по людям — позже.
- Связь: REQ-0004, REQ-0007 (согласование-периоды — другой контур), CAPACITY_PLANNING §4, ADR-0005.
