# REQ-0005 — Обязательность этапа в записи трудозатрат (project + conditional stage)

**Статус:** PROPOSED (research + предложение; реализация по раздаче arch)
**Источник:** заказчик (через user) — запись обязана иметь проект; если у проекта есть этапы → обязан этап; «без этапа» — только осознанно
**Затрагивает:** `objects/credos-time-entry.object.ts`, `logic-functions/time-entry-api.logic.ts` (валидация), фронт грид (Dev 1 — пикер/дефолт)

## Требования заказчика

1. Запись трудозатрат **обязательно** привязана к **проекту**.
2. Если у проекта **есть этапы** → запись **обязательно** привязана к **этапу**.
3. **«Без этапа»** — допустимо, но **осознанно**: нельзя просто оставить поле этапа пустым (пустое ≠ выбор).
4. (Желательно) у проекта есть **«текущий этап»** — по умолчанию предлагается при вводе.

## Текущее состояние

`credos-time-entry.object.ts`: `project` и `stage` — оба `isNullable: true` (необязательны). Этап (`credosTimeStage`) уже обязан иметь проект (REQ no-orphan, isNullable:false + CASCADE). Статусы этапа: PLANNED/ACTIVE/ON_HOLD/DONE + startDate/endDate.

## Бенчмарк (research)

| Система | Под-проектное измерение | Обязательность | Дефолт |
|---|---|---|---|
| **Timetta** | `ProjectTask` (Работа) в `TimeAllocation` | `projectId` + `projectTaskId` **оба обязательны** (GUID без `?`). «Без задачи» нет | — |
| **Kimai** | `Activity` (≈ наш WorkType, не этап); `Project.globalActivities` (глобальные vs проектные) | Activity **обязателен** в записи | — |
| **Наше** | `Stage` (этап, ≈ Timetta ProjectTask) + `WorkType` (≈ Kimai Activity) | сейчас оба опциональны | нет |

Вывод: оба продукта **жёстко требуют** под-проектное измерение. Наш кейс мягче (этап обязателен **только если есть** у проекта) — это компромисс между строгостью Timetta и гибкостью.

## Ограничение SDK

Twenty SDK `defineField` — статичная `isNullable` (NOT NULL на уровне колонки). **Conditional-required («обязателен если у проекта есть этапы») на уровне схемы невозможен.** → гибридная защита: схема (где можно) + logic-функция (условная валидация) + UI (пикер/дефолт).

## Предложение

### 1. project — обязателен (схема)
`entry.project` → **`isNullable: false`**. Всегда. (Schema-change задеплоенного объекта → app sync; существующие записи проверить на null до миграции — см. «Проверка».)

### 2. stage — условно-обязателен (logic + UI), `isNullable: true` в схеме
Схема остаётся nullable (иначе сломаются записи проектов без этапов). Обязательность вводим в `/s/time-entry` (create/update):
- если у `project` есть этапы (`project.stages.length > 0`) → запись **должна** иметь либо `stageId`, либо осознанный отказ;
- иначе (этапов нет) → `stageId` остаётся пустым штатно.

### 3. «Без этапа» осознанно — 2 варианта (нужно решение arch)

**Вариант A (рекомендую) — boolean-маркер `noStage`:**
- новое поле `entry.noStage` (BOOLEAN, default false).
- инвариант (logic): `projectHasStages ⇒ (stageId != null) XOR (noStage == true)`.
- UI: пикер этапа со спец-пунктом «— Без этапа —», который ставит `noStage=true`. Пустой submit при наличии этапов → reject.
- ✅ различает «забыл» (null+noStage=false → ошибка) от «осознанно» (noStage=true). Аудируемо.
- ❌ +1 поле.

**Вариант B — без поля, UI-гейт:**
- UI не даёт submit пустого этапа при наличии этапов; «Без этапа» отправляет null.
- ❌ в данных «осознанно null» и «забыл null» неразличимы → требование 3 («осознанно») по сути только на фронте. Логика не сможет валидировать. Слабее.

→ **Рекомендация: A** (маркер `noStage`) — выполняет требование 3 на уровне данных, не только UI.

### 4. «Текущий этап» по умолчанию
- **Определение «текущего»:** этап проекта со `status = ACTIVE`; при нескольких — с минимальным `startDate`; если ACTIVE нет — этап, где `today ∈ [startDate, endDate]`; иначе — без дефолта.
- **Где считать:** чистая функция (бэк, переиспользуемо) `currentStage(stages, today)` — отдаю Dev 1 для пред-выбора в гриде + использую в logic при отсутствии явного выбора (опционально).
- UI (Dev 1): при выборе проекта пикер этапа пред-заполняется «текущим»; пользователь может сменить/выбрать «без этапа».

## План реализации (по раздаче arch, бэк-часть моя)
1. `entry.project` → isNullable:false (+ app sync).
2. (Вариант A) поле `entry.noStage` + UUID-конст.
3. `/s/time-entry`: валидация инварианта (project-has-stages ⇒ stage XOR noStage) на create/update; вернуть понятную ошибку.
4. `currentStage()` чистая функция + unit-тесты.
5. Контракт Dev 1: пикер этапа (обязателен при наличии этапов, дефолт = currentStage, спец-пункт «Без этапа»).
6. Seed: проверить, что записи проектов-с-этапами имеют stage (или noStage).

## Проверка (read-only, сервер, 2026-06-21)

| Метрика | Значение | Вывод |
|---|---|---|
| Записей всего | 422 | — |
| Без `projectId` | **0** | миграция `project` → NOT NULL **безопасна** ✅ |
| Проектов с этапами / этапов | 12 / 50 | — |
| Записей на проектах-с-этапами | 116 | — |
| Из них **БЕЗ stage** | **116 (все)** | ⚠️ при введении правила все 116 станут невалидны |

**Ключевой вывод:** правило «этап обязателен, если у проекта есть этапы» **ретроактивно ломает 116 существующих записей**. Поэтому rollout двухфазный:
1. **Сначала ретро-проставить** этим 116 записям этап — авто по `currentStage(stages, entry.date)` (этап, в чьи `[startDate,endDate]` попадает дата записи; иначе ACTIVE/ближайший) ИЛИ `noStage=true`, где не определяется. Сид-скрипт.
2. **Потом включить** валидацию в `/s/time-entry` (иначе новый сид/записи блокируются, а старые висят битыми).

Без фазы 1 включать нельзя — заблокируем правкой/пересидом.

## Открытые вопросы (к arch/заказчику)
- Вариант A (поле `noStage`) или B (только UI)? Рекомендую A.
- «Текущий этап»: по статусу ACTIVE или по датам? Предлагаю: ACTIVE приоритетно, иначе по датам.
- Ретро-проставление этапа существующим записям (≈420 записей, 50 этапов на 12 проектов) — нужно ли, или только для новых?
