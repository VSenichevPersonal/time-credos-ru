## Часть A — гейт правки плана (сейчас, P-D2 п.2)

**Задача:** `plannedEffort`/`startDate`/`endDate` проекта правит только руководитель (isManager / руковод отдела), не рядовой сотрудник.

**Факты:**
- PATCH этих полей под app-ролью (`default-role`) работает (`canUpdateAllObjectRecords: true` + per-object) — подтверждено (P-D2 п.1). ✅
- SDK поддерживает **field-level RBAC**: `defineRole.fieldPermissions` = `Record<fieldId, {canRead?, canUpdate?}>`.

**Варианты гейта:**
1. **Native field-RBAC (рекомендую):** запретить `canUpdate` на `plannedEffort`/`startDate`/`endDate` для роли рядового сотрудника; руководителю — разрешить.
   - ⚠️ Требует **app-defined user-роль «Сотрудник»** (сейчас её нет: не-менеджеры получают базовую workspace-роль вне контроля app). Без неё запрет повесить не на кого.
2. **Logic-function гейт** (`/s/project-plan` PATCH через функцию с проверкой `actor.isManager`) — но упирается в **нерешённый actor-резолв** ([REQ-0001](REQ-0001-approval-rbac-sod.md): `userWorkspaceId→employee` спуфится через client-param). Тот же корневой блокер, что у approval.
3. **UI-only гейт** (Dev1 прячет правку плана у не-руковода) — НЕ безопасность (REST обходится).

**Рекомендация Dev 2:** вариант 1 (native field-RBAC) + завести роль «Сотрудник». Решает гейт без actor-резолва (платформа сама проверяет по назначенной роли). **@arch/@CISO: подтвердите подход** — тогда реализую роль + fieldPermissions.

### Известное ограничение dev (текущий статус — как approval-guard)

**Сейчас гейт «план правит только руководитель» = чисто фронтовый** (Dev 1, по `isManager`), аналогично approval до закрытия REQ-0001:

- для v1 принимаем фронтовый гейт достаточным для dev: план — не финансовая операция с разделением обязанностей, риск ниже approval. Зафиксировано здесь и в DEV2_LOG. Ужесточение (field-RBAC / logic-guard) — отдельный трек по подтверждению arch/CISO.

## Часть B — credosTimePlanAllocation (v2, resource allocation / прогноз занятости)

**Суть (уточнено arch 22:55):** это **resource allocation / прогноз занятости**, не просто план проекта. Из ёмкости человека (напр. 50 ч/нед) часть предварительно бронируется: «сотрудник X занят N ч на проекте Y в период Z». Руковод распределяет (booking) ёмкость людей по проектам.

**Модель:** `credosTimePlanAllocation = { employee, project, period (ISO-неделя), plannedHours }` — аллокация **по СОТРУДНИКУ** (не только project×period).

**Производные метрики:**
- **Загрузка человека** = Σ его аллокаций; свободно = личная ёмкость − Σ аллокаций («у Иванова свободно 20 из 50»).
- **Загрузка проекта** = Σ аллокаций на проект (кто и сколько выделен).
- **Загрузка отдела** = Σ по людям отдела.
- **Capacity-доска:** загрузка = Σ аллокаций если заданы, **иначе fallback** на `project.plannedEffort` (v1, равномерно по команде).

**Связь:** питает [REQ-0003](REQ-0003-reports-aggregates-contract.md) (план «по людям» — закрывает открытый вопрос «allocation») + capacity-доску + DP-0001 «когда освободится».

**Статус:** PROPOSED, **v2** — после v1 (P-D1: грубая правка `project.plannedEffort`, Dev1). Объект + сид + интеграция в reports/capacity — отдельная итерация.

### Инвариант (КЛЮЧЕВОЙ, заложить навсегда)

**Бронь ≠ план-оценка ≠ факт.** Три РАЗДЕЛЬНЫЕ оси, намеренно НЕ связанные FK (как у Timetta, `booking-concept`):
- `credosTimeBooking` — **РЕЗЕРВ ёмкости** (ресурс-менеджер/РП: «держим Иванова под Альфу в июле»).
- `credosTimePlanAllocation` (Часть B) — **ОЦЕНКА трудозатрат** (прогноз РП, себестоимость).
- `credosTimeEntry` — **ФАКТ** (списанные часы).

Не смешивать. Бронь живёт своей осью: `employee × project × период × ёмкость × тип`. Это позволяет: soft-бронь пресейла без обещания плана/факта; конфликты за ресурс; «когда освободится».

### Модель `credosTimeBooking` (реализовано)

| Поле | Тип | Назначение |
|---|---|---|
| `label` | TEXT (labelIdentifier) | заголовок карточки («Иванов → Альфа, июль») |
| `type` | SELECT `SOFT`/`HARD` (default SOFT) | soft=предв./пресейл (не потребляет ёмкость), hard=подтв. (потребляет) |
| `hours` | NUMBER float | забронированная ёмкость на период (равномерно по раб. дням) |
| `startDate`/`endDate` | DATE_TIME | период брони |
| `note` | TEXT (nullable) | заметка/условие пресейла |
| `employee` | RELATION → Employee (M:1, CASCADE) | обратка `Employee.bookings` |
| `project` | RELATION → Project (M:1, CASCADE) | обратка `Project.bookings` |

Решения (минимально, без переусложнения):
- **Период = startDate/endDate** (не ISO-неделя) — зеркало `credosTimeDeptPlan`, проще; недельная нарезка — на стороне расчёта Demand (как dept-plan/project уже раскидываются).
- **Единица = часы** (как dept-plan/project). FTE%/детальный режим Timetta (ячейки по слотам) — follow-up, не MVP.
- **Овербукинг НЕ блокируется** (как Timetta) — показывается как конфликт (UI/расчёт Dev1).
- **Без billable** ([[no-billable-concept]]): bill-rate не вводим.

### Контракт «Demand с учётом брони» для Dev1 (follow-up)

Капасити-доска сейчас (`CAPACITY_PLANNING.md`) считает Demand из `project.plannedEffort`/`deptPlan`, раскидывая по неделям [startDate..endDate] равномерно по раб. дням. Бронь добавляется как **отдельный слой Demand**:

1. **Источник:** записи `credosTimeBooking` (relation `employee.bookings` / `project.bookings`). Раскидка `hours` по неделям периода — той же функцией раскида, что план (равномерно по раб. дням недели).
2. **HARD** → твёрдый Demand: суммируется к загрузке человека/проекта/отдела наравне с планом (потребляет ёмкость). Сплошная плашка.
3. **SOFT** → **отдельный/пунктирный** слой: НЕ суммируется в основную загрузку (не потребляет ёмкость), показывается параллельно как «потенциальная потребность» (полупрозрачно). Фильтр-тумблер «учитывать soft».
4. **Загрузка человека бронями** = Σ HARD-броней в неделю; свободно = личная ёмкость − Σ(HARD-бронь, и/или план — по выбору режима доски).
5. **Конфликт/овербукинг:** если Σ(HARD-бронь) недели > ёмкости — подсветить (не блокировать ввод).
6. **Связь с tentativeBookingEnabled:** существующий флаг управляет показом SOFT-слоя (когда выкл — soft не рисуем).

Замечание для Dev1/Dev2: бронь НЕ заменяет план-оценку в Demand — это параллельная метрика. Какой слой основной на доске (план vs hard-бронь) — режим-переключатель (как «Отделы/Люди» сейчас).
