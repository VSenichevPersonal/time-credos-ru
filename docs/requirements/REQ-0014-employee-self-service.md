# REQ-0014 — Личный кабинет сотрудника (Мои часы / Мои периоды / Welcome)

**Статус:** PROPOSED (БЭКЛОГ — заказчик 2026-06-21)
**Приоритет:** 🔴 ВЫСОКИЙ — крупнейшая дыра: система manager-focused, рядовой сотрудник (42 чел) «слепой».

## Контекст
Всё планирование/отчёты заточены под руководителя. Рядовой сотрудник не видит СВОИХ часов/статусов. Нужен employee-facing слой.

## Сверка Timetta/Kimai
Kimai/Timetta: у каждого юзера личный таймшит + «мои недели» со статусами + дашборд «мои часы за период». Стандарт.

## Состав
1. **Welcome/домашний экран** (front-component, раздел «Обзор»): для сотрудника — «мои часы за неделю/месяц», недобор, быстрый ввод, ссылки. Для руковода — его текущие сводки. Адаптируется по роли.
2. **«Мои часы»** — личный отчёт: факт/норма/недобор сотрудника за период, по проектам/категориям (переиспользует /s/reports с фильтром employee=я).
3. **«Мои периоды»** — история недель со статусами (Черновик/На согласовании/Согласовано/Отклонено). После отправки период не «исчезает» — виден в списке с возможностью открыть.

## БЛОКЕР: «кто я» (server-identity)
Песочница не отдаёт текущего юзера (RestApiClient под токеном роли app; нет `currentWorkspaceMember`/`me` в twenty-client-sdk/rest — Dev1 проверял). Без идентификации «мои» = небезопасно/невозможно отфильтровать.
- **Research (приоритет):** умеет ли front-component Twenty получить текущего пользователя (хук `useCurrentUser`/контекст/page-payload)? Если ДА — личный кабинет разблокируется БЕЗ полной server-identity (CISO-005). Если НЕТ — ждёт CISO-005 (RBAC-волна).
- Временно (до identity): селектор «кто я» (как REQ-0008) — но это компромисс, не настоящий личный режим.

## Связи
CISO-005 (server-identity), REQ-0008 (селектор «чей таймшит»), REQ-0001 (RBAC), /s/reports (byEmployee уже есть). Research-задача первой.
