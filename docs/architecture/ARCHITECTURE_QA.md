# Архитектура time.credos.ru — открытые вопросы и разборы

**Дата:** 2026-06-20
**Статус:** обсуждение. Принято первое решение — см. ADR-0001 (`docs/adr/0001-platform-and-data.md`). Остальное открыто.
**Назначение:** зафиксировать вопросы, варианты и аргументацию по архитектуре, чтобы не терять контекст. Решения принимаем позже.

## Вопрос 1. Twenty CRM (платформа) или отдельное приложение?

**Контекст:** Twenty CRM (форк CredosCRM1) имел смысл для CRM, потому что **CRM = CRUD над записями** — ровно то, для чего metadata-движок. Time tracking — другой зверь: его сердце не карточка записи, а **недельная сетка + таймер** (быстрый ежедневный ввод, клавиатура, инлайн-редактирование ячеек, живые суммы). Это высокоинтерактивный bespoke-UI.

### Вариант A — на Twenty CRM (metadata)
- 4 metadata-объекта через REST API (TimeSheet, TimeSheetEntry, Project, Activity)
- 2-3 query hook для бизнес-логики
- 2-3 кастомных виджета для недельной сетки
- Сроки: 1-2 недели, риск низкий
- Плюс: готовый RBAC, аудит, API, дашборды, одна кодовая база
- Минус: сетку времени всё равно пишешь руками — но внутри чужого layout-движка (борьба с фреймворком)

### Вариант B — отдельное приложение, общая БД + JWT
- Свой домен `time.credos.ru`, свой UI (сетка + таймер)
- Общие PostgreSQL / Redis / JWT с CRM, таблицы `timetracker_*`
- Читает справочники CRM (Company, Person, User) read-only
- Сроки: 8-10 дней
- Плюс: полная изоляция UI, свобода стека, отдельный продукт
- Минус: два деплоя, два приложения на поддержку, общий auth настроить

### Ключевой инсайт
Главную, самую интерактивную часть (сетка/таймер) пишешь руками **в обоих вариантах**. Вопрос: писать её в своём лёгком React-SPA или встраивать в тяжёлый layout-движок Twenty.

### Реальная ось выбора (НЕ «платформа vs свобода»)
| Если приоритет... | Вариант |
|---|---|
| UX ввода времени — дифференциатор; снаппи-сетка; отдельный продукт `time.credos.ru` | B (отдельное приложение) |
| Быстрее зашипить; time tracking «достаточно как записи»; одна кодовая база/инфра | A (Twenty metadata) |

### Решающий вопрос (ОТКРЫТ)
**`time.credos.ru` — отдельный продукт (свой UI, может в будущем продаваться) или внутренний инструмент Кредо-С (лишь бы работал)?**
- Первое → отдельное приложение.
- Второе → можно на Twenty.

## Вопрос 2. Сэкономим ли силы на Twenty, или усложним?

**Ответ: зависит от части. Часть ядра Twenty экономит, часть — усложняет.**

### Где Twenty РЕАЛЬНО экономит (бери оттуда)
| Кусок | Почему даром |
|---|---|
| Справочники Project / Activity / Rate | metadata-объект через REST = CRUD + API + список + карточка без кода |
| Авторизация, пользователи, RBAC, аудит | готово |
| Клиенты / контакты | уже есть Company / Person |
| Файлы, поиск, дашборды | готовые модули |

~60% обвязки. Чистая экономия.

### Где Twenty УСЛОЖНЯЕТ (пиши сам/отдельно)
| Кусок | Почему боль |
|---|---|
| Недельная сетка ввода | Twenty UI = карточка + таблица-список. Сетка 7×N с инлайн-вводом, табами, живыми суммами — против шерсти |
| Таймер (start/stop, фоновый) | mutation + состояние поверх их модели неудобно |
| TimeSheet → согласование | approval-флоу через query-hooks — вязко |
| Производные расчёты (часы×ставка, billable, бюджет) | в hooks, дебажить тяжелее своего кода |

Самые ценные, самые интерактивные ~40% (ради которых продукт существует) на Twenty идут тяжелее, чем в своём коде.

### Прямой ответ
- «Ядро» = данные + справочники + CRUD → Twenty экономит. Да.
- «Ядро» = сетка ввода + таймер + UX → Twenty усложняет.

### Гибрид (наиболее вероятное направление)
```
Twenty CRM (общая БД)              time.credos.ru (свой SPA)
├── Company / Person      ←читает─ │
├── Project   (metadata)  ←читает─ │  Недельная сетка   ← свой React
├── Activity  (metadata)  ←читает─ │  Таймер            ← свой React
├── Rate      (metadata)  ←читает─ │  Approval-флоу     ← свой код
├── RBAC / Auth / JWT     ──общий─ │
└── Аудит / Файлы         ←пишет── │  timetracker_entries (свои таблицы)
```
Справочники и обвязку — на Twenty-metadata (даром). Сетку/таймер/флоу — свой SPA на общей БД и JWT.
Цена: настроить общий auth + читать metadata-таблицы Twenty из своего приложения (пара дней).

Итог: на чистом Twenty упростишь данные, но усложнишь главное (ввод времени). Гибрид даёт экономию без боли.

## Вопрос 3. Единое пространство авторизации? Можно по AD?

**Установленный факт:** Twenty CRM (CredosCRM1) **уже умеет SSO** — SAML + OIDC стратегии:
- `engine/core-modules/sso`
- `auth/strategies/saml.auth.strategy.ts` + `oidc.auth.strategy.ts`
- UI: `settings/security/SSO`
- Прямого LDAP-коннектора нет (и не нужен — см. ниже).

### Единое пространство auth — реально
Правильная схема — **центральный IdP, оба приложения = его клиенты**. Логинишься раз у IdP, оба приложения доверяют токенам.
```
         ┌─────────────────────────┐
         │   IdP (источник правды)  │  ← Entra ID / Keycloak / AD FS
         │   умеет OIDC + SAML      │
         └───────┬─────────┬────────┘
            OIDC │         │ OIDC
        ┌────────▼──┐  ┌───▼──────────────┐
        │ Twenty CRM │  │ time.credos.ru   │
        │ (SSO готов)│  │ (свой OIDC-клиент)│
        └────────────┘  └──────────────────┘
              общий PostgreSQL / пользователи
```

### Про AD — развилка решает всё
| Что за «AD» | Как подключить | Сложность |
|---|---|---|
| Entra ID (Azure AD, облако) | OIDC/SAML напрямую в Twenty + в свой SPA. Из коробки | 🟢 легко |
| On-prem AD (контроллер в офисе) | apps на Railway (облако) НЕ видят LDAP офиса напрямую | — |
| → через AD FS | AD FS публикует SAML/OIDC в интернет, оба app — клиенты | 🟡 средне |
| → через Entra Connect | синк on-prem AD → Entra ID → дальше как облако | 🟡 средне |
| → через Keycloak в DMZ | Keycloak федерит LDAP, наружу даёт OIDC | 🟡 средне, гибко |

**Ключевое:** прямой LDAP с Railway к офисному AD — не вариант (нет сетевой видимости, дыра в безопасности). Нужен брокер, выставляющий AD в интернет по OIDC/SAML. Twenty это ест.

### Почему центральный IdP, а не «общий JWT Twenty»
Общий APP_SECRET связывает приложения жёстко и не даёт AD. Центральный IdP — настоящее «единое пространство», и AD прикручивается одной федерацией.

## Вопрос 4. Keycloak крутится где-то в инфре и смотрит наружу?

**Да.** Keycloak — посредник между двумя мирами: одной ногой внутрь (к AD по LDAP), другой наружу (OIDC к приложениям).
```
   ВНУТРЕННИЙ КОНТУР (офис/VPN)        │   ИНТЕРНЕТ (Railway, облако)
                                       │
   ┌──────────────┐                    │
   │  AD / LDAP    │                   │
   │ (контроллер)  │                   │
   └──────┬───────┘                    │
          │ LDAP (внутри, не наружу)   │
          ▼                            │
   ┌──────────────┐    OIDC/HTTPS      │   ┌──────────────┐
   │  Keycloak     │◄───────────────────────│  Twenty CRM  │
   │  (IdP-брокер) │    наружу только   │   └──────────────┘
   │              │◄───────────────────────│ time.credos  │
   └──────────────┘    443/TLS         │   └──────────────┘
          ▲                            │
   только OIDC-эндпоинты               │
   торчат наружу, LDAP скрыт           │
```

### Суть
- Внутрь: Keycloak ходит к AD по LDAP, читает юзеров/группы. LDAP наружу не выходит.
- Наружу: торчит только Keycloak по HTTPS/443, отдаёт стандартный OIDC. Приложения про AD не знают.
- Логин: юзер → app → редирект на Keycloak → Keycloak проверяет пароль у AD → токен → app пускает. Один вход = оба приложения.

### Нюансы
| Вопрос | Ответ |
|---|---|
| Где крутится Keycloak | В вашей инфре, где есть сетевой доступ к AD (тот же контур/VPN). Не на Railway |
| Что торчит наружу | Только `https://auth.credos.ru` (OIDC). LDAP-порты закрыты |
| Доступ Keycloak→AD | Внутренняя сеть / VPN. Снаружи DC не виден |
| Если нет своего сервера | Альтернатива — Entra ID (AD синкается через Entra Connect). Keycloak не нужен |
| БД Keycloak | Свой small Postgres (конфиг, сессии). Не путать с БД CRM |

### Две развилки по «где AD»
1. AD только локальный, есть свой сервер/VPS в контуре → Keycloak там, смотрит наружу.
2. Уже есть Microsoft 365 / готовы синкать в облако → Entra ID вместо Keycloak, проще, своего сервера не надо.

Keycloak = больше контроля, self-hosted (важно для 152-ФЗ / данные в РФ). Entra = меньше возни, но Microsoft-облако.

## Вопрос 5. Timetta или Kimai? Что копируем — гибрид?

**Ответ: ни то, ни другое целиком. Гибрид, но роли РАЗНЫЕ.** Не «выбрать одну» — у них разный статус.

| | Timetta | Kimai |
|---|---|---|
| Доступ | reverse-engineered (чёрный ящик) | открытый код AGPL |
| Что даёт | целевую модель (куда расти) | готовую логику (что копировать) |
| Копируем код? | НЕЛЬЗЯ — кода нет, только поведение | ДА — есть PHP-исходники (с оговоркой AGPL) |
| Лицензия | — | AGPL-3.0 ⚠️ вирусная |

### Принцип (из CAN_WE_BUILD_IT)
> Kimai — стартовый шаблон. Timetta — целевая модель.

- Структуру/логику берём из **Kimai** — только у него есть исходники. Трекинг (begin→end), инвойсинг, бюджеты, ставки 4 уровня, Team ACL, Meta Fields — готовая логика, проверена 832 тестами.
- Глубину/направление берём из **Timetta** — недельная сетка TimeSheet→Line→Allocation, Decimal-часы, справочники РФ, Lifecycle-машины. «Куда расти», модель-ориентир.

### Конкретно по кускам
| Фича | Откуда | Почему |
|---|---|---|
| Customer→Project→Activity→Timesheet | Kimai | открытая иерархия |
| Трекинг begin/end + duration | Kimai | точный код есть |
| Инвойсинг, бюджеты, ставки | Kimai | калькуляторы открыты |
| Team ACL, Meta Fields | Kimai | готовая реализация |
| Недельная сетка (grid) | Timetta | модель TimeSheet→Line→Allocation |
| Decimal-часы (не INT-секунды) | Timetta | удобнее для PSA |
| Справочники РФ, 152-ФЗ | Timetta | локальная специфика |
| State machines (Lifecycles) | Timetta — позже | чёрный ящик, старт с enum-статусов |
| PostgreSQL (не MySQL Kimai) | Timetta-выбор | надёжнее |

### ⚠️ ВАЖНО: AGPL-3.0 у Kimai
Прямое копирование кода Kimai → наш продукт обязан открыть исходники (вирусная лицензия, триггерится даже на SaaS).
**Поэтому:** из Kimai берём модель данных и алгоритмы как **референс** (идеи, формулы, структура таблиц — не лицензируются), но **переписываем свой код** под наш стек (Node/TS, не PHP). Юридически чисто + под наш стек.

### Прямой ответ
Гибрид с чёткими ролями:
- **Kimai = чертёж логики** (как работает трекинг/инвойсинг) → переписываем под себя, не копируем дословно (AGPL).
- **Timetta = чертёж амбиции** (сетка, глубина, РФ) → целимся поэтапно.
- Старт: Kimai-простота (4 уровня, enum-статусы) → рост до Timetta-глубины (сетка, бюджеты, машины) по потребности.

## Вопрос 6. В итоге Twenty или с нуля? Общая БД или интеграция?

**Контекст изменился:** на открытый вопрос «продукт или инструмент» дан ответ — **`time.credos.ru` = внутренний инструмент Кредо-С** (не отдельный продаваемый продукт). Это сдвинуло баланс.

### Что меняет ответ «внутренний инструмент»
| Аргумент | Если продукт | Если internal |
|---|---|---|
| Отдельный продаваемый продукт | за отдельное приложение | отпал |
| Поддержка инфры | терпимо ради продукта | меньше движущихся частей = лучше |
| Переиспользование CRM-данных | через интеграцию (граница) | внутри Twenty = бесплатно |
| UX ввода | дифференциатор | важен, но «достаточно» приемлемо |

**Главный сдвиг:** если трекер ВНУТРИ Twenty — вопрос «общая БД или интеграция» исчезает. Клиенты, юзеры, auth, RBAC уже там. Не нужен второй деплой, интеграционный слой, отдельный OIDC-клиент, вторая БД.

### Решение → на Twenty (см. ADR-0001)
```
Twenty CRM (одна платформа)
├── Project / Activity / Rate   ← metadata-объекты (даром: CRUD+API+RBAC)
├── TimeSheet / Entry            ← metadata-объекты
├── Company / Person / User      ← уже есть, без интеграции
├── Approval-флоу                ← query-hooks
└── Недельная сетка + таймер     ← ЕДИНСТВЕННЫЙ кастом (виджет в layout)
```
Цена: сетка/таймер — кастом-виджет в layout-движке Twenty (тяжёлый кусок). Выигрыш: всё остальное бесплатно.

### Что НЕ меняется
Auth — через центральный IdP (Keycloak/Entra). Twenty уже умеет SSO. Теперь IdP-клиент один (Twenty), а не два.

### Осознанный риск
Недельная сетка внутри Twenty-layout сложнее, чем в своём SPA. Для внутреннего инструмента приемлемо. **Точка пересмотра:** если ежедневный ввод окажется критично-неудобным и люди перестанут заполнять — выносим ТОЛЬКО виджет ввода в отдельный мини-SPA, читающий Twenty по API. По факту боли, не авансом.

## Вопрос 7. SDK-app vs core-патчи. Дата SDK. Миграция старого?

**Находка:** у Twenty есть официальный **Apps SDK** (`twenty-sdk`, `npx create-twenty-app`). Time tracking можно собрать как установочное app-приложение, БЕЗ правок ядра.

### SDK-примитивы (есть в форке, `packages/twenty-sdk` v0.7.0)
| `defineX` | Для нас |
|---|---|
| `defineObject` | TimeSheet / Project / Activity / Rate |
| `defineView` | списки таймшитов |
| `definePageLayout` | карточки + виджеты |
| `defineFrontComponent` | недельная сетка, таймер (React внутри Twenty) |
| `defineNavigationMenuItem` | пункты сайдбара (+ folder-группировка) |
| `defineLogicFunction` | approval/расчёты (route `/s/`, cron, db-event) |
| `defineRole` | RBAC приложения |
| `defineApplication` | манифест (+ `settingsCustomTab`) |

### Что получаем
App-пакет: изолирован, **upgrade-safe** (ноль записей в `core-changes.md`), удаляется/ставится как юнит, свой scope ролей/API, marketplace-ready.

### ⚠️ Даты и риск форка
- Twenty 2.0 GA + SDK: **21 апреля 2026**. v2.1.0 (24 апр) — первый production-ready. SDK молодой (~2 мес GA).
- **Форк CredosCRM1 создан 14 марта 2026** — за ~5 недель ДО 2.0 GA. SDK-файлы в форке от 13 марта = **пред-GA снимок**. Upstream с марта не подтягивали.
- Вывод: SDK в форке ранний → нужен **upstream-sync до 2.x** перед ставкой.

### SDK в CredosCRM задокументирован?
Да, но как **референс** (зеркало docs.twenty.com), НЕ как принятое решение:
- `twenty-reference/extensibility-guide.md` §7, `developers/extend/apps/{building,getting-started}.md`, `AGENT_docs/Twenty Apps.md`, `deepwiki/10-apps-extensibility.md`.
- Ни один существующий credos-модуль НЕ сделан через SDK (все на старом паттерне). Опыта у команды — ноль.

### СБИС-селектор продуктов (полная смена сайдбара)
НЕ нативен в SDK v0.7 (`NavigationMenuItemManifest` даёт пункты + folder, не рельсу-переключатель). Раздел-папка «Время» — нативно. Полная СБИС-рельса — только жёлтой зоной ядра. Отложено.

### Миграция существующих доработок на SDK?
**НЕТ.** Причины:
- Работают в проде (quotes, beeline, dadata, 1С, reports, AI, audit) — не трогаем.
- Часть **невыразима в SDK** в принципе: core-патчи рендереров (`FieldWidget` marker-dispatch, `FieldsWidget` viewId fallback, card-layout регистрации) — у SDK нет таких слотов.
- Интеграции (1С/Exchange/Билайн) — отдельный микросервис `credos-integrations`, не logic-functions.
- **Сосуществование:** старое = старый паттерн; новое (time tracking) = SDK-app. SDK только под новое/изолированное.

### Решение → ПРИНЯТО: time tracking через SDK-app (ADR-0002), с предусловием (upstream-sync + PoC)

## Вопрос 8. Разработка в изолированном репо и мердж в Twenty — или сразу в форке?

**Ответ: изолированный репо. И НЕ «мердж в форк», а УСТАНОВКА app в workspace.** Это ключевое преимущество SDK-пути.

### Суть
SDK-app — **самостоятельный пакет**, не часть монорепо Twenty. `npx create-twenty-app` создаёт отдельный проект. В Twenty он попадает через **install-механизм** (`application-install` / `application-registration` / `application-tarball` / marketplace), а НЕ через merge исходников в форк.

```
ЭТОТ репо (time-credos-ru)          Форк CredosCRM1
┌──────────────────────────┐       ┌──────────────────────┐
│ SDK-app (свой пакет)      │       │ Twenty + старые credos│
│ defineObject/View/...      │──build──▶ tarball ──install──▶│ workspace (app живёт тут)
│ front-компоненты, логика  │       │ ядро НЕ трогаем       │
└──────────────────────────┘       └──────────────────────┘
   разрабатываем здесь                устанавливаем туда
```

### Почему изолированный репо — правильно
- App декаплен от форка: разработка/git/PR/rollback — независимо.
- В Twenty не мерджим код — **устанавливаем собранный app**. Ноль merge-конфликтов с ядром.
- Это и есть отличие от старого паттерна (где код жил в форке + жёлтая зона).

### Что всё-таки попадает в форк (минимум, отдельно)
- Только если понадобится **СБИС-рельса** (полный продукт-переключатель) — отдельный маленький core-PR в форк (жёлтая зона). App это не требует — раздел-папка «Время» ставится самим app.
- Предусловие: форк должен быть на **2.x GA** (рабочий app-runtime для install). Сейчас пред-GA снимок → нужен upstream-sync.

### Вывод
Разрабатываем здесь (изолированно), в Twenty **устанавливаем**, не мерджим. Прямо в форк писать app НЕ нужно и хуже (теряется декаплинг). Подробности — ADR-0002.

## ЗАФИКСИРОВАННЫЕ РЕШЕНИЯ
- **ADR-0001** — Платформа на Twenty CRM, данные внутри платформы (без отдельной БД/интеграции), auth через центральный IdP. Триггер: `time.credos.ru` = внутренний инструмент.
- **ADR-0002** — Time tracking как SDK-app в изолированном репо (этот), установка в workspace (не merge в форк). Предусловие: upstream-sync форка до 2.x GA + PoC «hello-world» SDK. Существующие доработки НЕ мигрируем (сосуществование паттернов).
