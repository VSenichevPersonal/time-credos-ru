# CISO-007 — /s/reports: раскрытие данных всех сотрудников (P2)

**Статус:** OPEN · **Severity:** P2 · **Owner устранения:** Dev 2 · **Дата:** 2026-06-20

## Суть

Эндпоинт `/s/reports` (`reports.logic.ts`) возвращает агрегированные трудозатраты **всех** сотрудников (`byEmployee[42]`) любому аутентифицированному пользователю без проверки роли и без scope по отделу.

## Вектор

```
POST /s/reports
{ "from": "2026-01-01", "to": "2026-06-30" }
Authorization: Bearer <любой валидный сессионный токен пользователя>
```

Ответ содержит:
```json
{
  "byEmployee": [
    { "key": "<uuid>", "name": "Иванов Иван Иванович", "dept": "ОПИБ",
      "fact": 1120, "client": 840, "norm": 960, "util": 0.75, "under": -160 }
    // ... ещё 41 сотрудник
  ]
}
```

Поле `name` = ФИО сотрудника (ПДн). Поля `util`/`under` — чувствительные бизнес-метрики (переработка/недозагрузка конкретного человека).

## Почему P2, а не P3

1. **ПДн + бизнес-метрики:** ФИО сотрудника + его загруженность/недозагрузка — это HRдата, которую рядовой сотрудник не должен видеть по всей компании. Нарушение принципа минимальной привилегии (152-ФЗ: «обработка в объёме, необходимом для конкретной цели»).
2. **Системная широта:** 42 сотрудника × полная метрическая разбивка за любой период — один запрос.
3. **Смягчает:** dev-среда, внутренняя сеть, `isAuthRequired: true` (не публичный API).

## Корневая причина

`reports.logic.ts` не вызывает `resolveActor`/`resolveEmployeeId` — identity вообще не используется. Логика: «агрегат для дашборда менеджера» — но ограничение «только для менеджера» не реализовано.

Связь с CISO-005: серверной привязки userWorkspaceId→employee нет, поэтому role-check для scope по отделу физически невозможен без предварительного решения identity (та же системная причина).

## Scope filter injection (CISO-006 + reports)

CISO-006 применим и здесь: `from`/`to` из client params интерполируются в filter-строки (L108, L113) без валидации. Миграция на UUID/ISO-date валидацию закрывает оба.

## Требование

**R1 (краткосрочно, до прода):** добавить role-guard: вызов `/s/reports` разрешён только руководителю (`actor.isManager == true`). Для определения actor — ждёт CISO-005 (server-side identity).

**R2 (после CISO-005 resolution):** scope по отделу — `byEmployee` возвращает только сотрудников отдела руководителя, а не всей компании.

**R3 (быстрый фикс сейчас):** ограничить `byEmployee` payload до полей без ФИО для не-менеджеров ИЛИ отключить `byEmployee` в ответе пока нет role-guard:
```typescript
// Временный guard: если actor не менеджер — byEmployee = []
// (Пока нет server-side identity — логировать предупреждение и не раскрывать)
```

## DoD

1. Вызов `/s/reports` без `isManager`-роли → `{ ok: false, error: 'forbidden' }` или `byEmployee: []`.
2. Менеджер видит byEmployee своего отдела (после CISO-005).
3. `from`/`to` проходят ISO-date валидацию (CISO-006).
