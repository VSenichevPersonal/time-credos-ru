# CISO-010: Экспорт (F-F) + напоминания (F-E) — утечка ПДн

**Severity:** P2 (планируемая фича — требования перед реализацией)  
**Статус:** OPEN (pre-implementation)  
**Owner:** Dev 2 (backend export/cron), Dev 1 (UI-gate)  
**Связь:** REQ-0006 (табель/экспорт), F-E (cron), CISO-008 (absence SICK), 152-ФЗ ст. 10

## Контекст

F-F (экспорт CSV/табель) и F-E (cron-напоминания) — запланированные фичи (BACKLOG W6). Требования CISO перед реализацией.

## F-F: Экспорт CSV/табель (P2)

### Риски

**Риск 1 — Medical PII в файле (критично):**
Табель включает коды отсутствий: `Б` (больничный) = `SICK` из `credosTimeAbsence.absenceType`. Это медицинская ПДн (152-ФЗ ст. 10 спецкатегория). CSV-файл с `SICK` → файл с медпоказателем конкретного физлица. При неправильном хранении/пересылке — нарушение.

**Риск 2 — Роль-guard отсутствует:**
Если экспорт реализовать как button без auth-check, любой аутентифицированный может скачать табель всех 42 сотрудников (аналог CISO-007). Экспорт = разрыв ПДн-периметра (данные покидают систему как файл).

**Риск 3 — Интеграция 1С:ЗУП:**
Если CSV экспортируется для автоматического импорта в 1С — это новый канал передачи ПДн. Требует фиксации в ЛНА (реестр операций обработки, ст. 18.1).

**Риск 4 — Временный файл на сервере:**
Если генерация через logic-function с буферизацией → файл временно живёт в памяти/на диске. При app-token-scope — файл доступен функции глобально. Нужна немедленная стримовая отдача без персистирования.

### Требования к реализации

1. **Role-guard обязателен:** экспорт всего табеля — только `isManager`. Сотрудник может экспортировать только свои строки.
2. **Absence SICK — маскировка или ограничение:** в CSV для не-HR-роли код `Б` (больничный) → заменить на `Н` (неявка) или пустое. Детальный медкод только для HR/кадровик-роли (когда появится).
3. **Стриминг без сохранения:** HTTP streaming response (не temp-file на диске). Logic-function должна стримить CSV построчно.
4. **Аудит экспорта:** логировать: кто / когда / за какой период / сколько строк скачал. Минимум `approvedBy`-паттерн в `credosTimeExportLog` или в лог. До прода — обязательно.
5. **ЛНА:** экспорт табеля в 1С = новая операция обработки ПДн → зафиксировать в реестре до включения.

## F-E: Cron-напоминания (P3)

### Риски

**Риск 1 — Email-адреса (ПДн) в контексте logic:**
Cron-функция под `TWENTY_APP_ACCESS_TOKEN` читает WorkspaceMember.userEmail → отправляет письма. Если email'ы логируются/дампятся при ошибке → утечка в логи (CISO-001 паттерн в другом канале).

**Риск 2 — Раскрытие статуса коллег в письме:**
«Привет Иван, ты не заполнил табель за неделю» — ок. Но если письмо включает «твой отдел не заполнил: Петров, Сидоров» — раскрытие персональных данных коллег (кто не заполнил = косвенный трудовой показатель).

**Риск 3 — Массовая рассылка под app-token:**
Функция имеет доступ к email всех пользователей. Без scope по отделу/роли = потенциальный спам/раскрытие всем.

### Требования к реализации

1. **Контент письма — только про себя:** письмо не называет других сотрудников по имени. Только «ты / твой табель».
2. **Email не в логах:** try/catch вокруг отправки, email-адрес не включать в stack trace / error message.
3. **Scope по отделу:** руководитель получает reminder об своём отделе, не обо всех.
4. **Opt-out/preferences:** при появлении настроек — возможность отключить напоминания.

## Приоритет внедрения

- F-F export: перед реализацией — обсудить с arch risk 2 (маскировка SICK) и risk 3 (1С). Зафиксировать решение.
- F-E cron: risk 1/2 — архитектурные constraint'ы, Dev 2 закладывает при разработке.
