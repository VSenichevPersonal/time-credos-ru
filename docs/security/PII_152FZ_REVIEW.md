# Ревью 152-ФЗ: персональные данные модуля time

**Задача:** C-1 (волна 1). **Owner:** CISO. **Дата:** 2026-06-20.
**Статус:** ревью для arch. Дополняет [PII_INVENTORY.md](PII_INVENTORY.md) (ROPD-lite) и [RISK_REGISTER.md](RISK_REGISTER.md).

**Назначение:** оценить обработку персональных данных (ПДн) сотрудников в модуле time на соответствие ФЗ-152 «О персональных данных». Фокус ревью: (1) инвентаризация PII, (2) **юрисдикция хранения (ст. 18 — локализация в РФ)**, (3) риск-реестр 152-ФЗ, (4) минимизация на dev и меры для прода.

**TL;DR для arch:**
- Новый существенный риск — **152FZ-001 (P1): данные хранятся на Railway вне юрисдикции РФ** → прямое нарушение ст. 18.5 ФЗ-152 (требование локализации БД ПДн граждан РФ в прод). Для dev — приемлемо при обезличивании, для прод — **блокер запуска**.
- Подтверждаю и переношу в 152-ФЗ-контекст уже открытые: реальные ПДн в git (CISO-001), отсутствие правового основания обработки (согласие/ЛНА), приказ работодателя о трудозатратах как профилирование.
- Posture модуля по 152-ФЗ: 🟡 **MEDIUM** (был 🟢 LOW по общей безопасности — но 152-ФЗ-локализация и правовое основание поднимают планку для прода).

## 1. Инвентаризация ПДн (что обрабатывает модуль time)

### 1.1. Категория субъектов
**Работники Кредо-С** (15–20 активных, до 42 в сиде / 72 в Битрикс-реестре) — единственная категория субъектов ПДн. Клиентов-физлиц нет: `Company` = юрлица, `BillingLink` = реквизиты документов 1С (не ПДн).

Категория обработки по 152-ФЗ: **ПДн работников** (обрабатываются работодателем в рамках трудовых отношений — гл. 14 ТК РФ + 152-ФЗ). Спецкатегорий (ст. 10: здоровье, политика, религия) и биометрии (ст. 11) — **нет**.

### 1.2. Прямые идентификаторы (PII) — `credosTimeEmployee`

| Поле | Тип | ПДн | Категория 152-ФЗ | Источник |
|---|---|---|---|---|
| `firstName` / `lastName` / `middleName` | TEXT | **да** | ФИО — общедоступные в трудовых отношениях, но ПДн | Битрикс-реестр / Директум5 |
| `email` | TEXT | **да** | корп. email (`@credos.ru`) — ПДн (идентифицирует лицо) | Битрикс |
| `jobTitle` | TEXT | да (косв.) | должность | Битрикс |
| `workspaceMemberRef` | TEXT | идентификатор | псевдоним (ссылка на учётку Twenty WorkspaceMember) | платформа |
| `department` (rel) | RELATION | нет | организационная принадлежность | модель |
| `active` | BOOLEAN | нет | флаг активности | модель |

### 1.3. Поведенческие данные о работнике — `credosTimeEntry` (КЛЮЧЕВОЕ для 152-ФЗ)

Записи трудозатрат **сами по себе не PII**, но привязаны к конкретному работнику (`employee` → `credosTimeEmployee`) и в совокупности образуют **профиль производительности сотрудника**:

| Поле | Что раскрывает о работнике |
|---|---|
| `hours` (decimal) + `date` | сколько и когда работал — поведенческий/трудовой профиль |
| `project` / `stage` / `workType` | над чем работал — детализация деятельности |
| `description` («Состав работ», свободный текст) | ⚠️ может содержать произвольные ПДн/чувствительную инфу (вводит человек) |
| `status` / `approvedBy` / `approvedAt` | контроль и согласование (`approvedBy` = идентификатор руководителя-actor) |

**Вывод 152-ФЗ:** агрегат трудозатрат = **оценка эффективности/производительности работника**. Это чувствительные HR-данные (расчёт утилизации = client-hours/total-hours — KPI). Обработка с целью контроля производительности требует:
- **правового основания** (ст. 6) — для работника это обычно ЛНА работодателя + ознакомление под подпись, либо согласие;
- **информирования работника** о целях и составе обработки (ст. 18);
- хранения в пределах сроков и целей (ст. 5 — соответствие цели).

### 1.4. Связь с Битрикс-реестром

`credosTimeEmployee` наполняется из выгрузки Битрикс (`research/directum5/bitrix-users/roster.csv`, 72 чел) + Директум5 (`trudozatraty-dir5.xlsx`, 34k записей). Эти **сырые источники содержат реальные ПДн и трекаются в git** (см. Dev 2 [предложение #1] в SIGNALS) — расширяет периметр CISO-001 за пределы `seed-real.mjs`.

## 2. Где хранится + риск юрисдикции (ст. 18 152-ФЗ)

### 2.1. Места хранения ПДн

| Место | Что | Юрисдикция | Статус по 152-ФЗ |
|---|---|---|---|
| **Postgres воркспейса Twenty** | прод-БД ПДн (Employee + Entry) | Railway (по умолчанию **вне РФ**, см. 2.2) | ⚠️ **нарушение ст. 18.5 для прод** |
| **git-репозиторий** | реальные ФИО+email в `seed-real.mjs` + `research/*.csv,xlsx` | GitHub (вне РФ) | ⚠️ ПДн в VCS без основания (CISO-001) |
| **`.env` / Railway variables** | секреты (токены) — **не ПДн** | Railway | ✅ корректно (gitignored) |
| **локальные выгрузки** `research/*.xlsx` | первоисточник ПДн | локальная ФС / git | должны быть gitignored |

### 2.2. Риск Railway-юрисдикции — главный для прода

**Факт:** dev-сервер — Twenty 2.14 на **Railway** (`twenty-production-e5c5.up.railway.app`, проект «Twenty Credos Time», env production). Railway — американский PaaS; его регионы размещения — **US (us-west/us-east), Europe (EU-West), Asia (Singapore)**. **Региона на территории РФ у Railway нет.** В репозитории регион/юрисдикция размещения **нигде не задокументированы** (поиск по `devops/` пуст) — то есть по умолчанию ПДн физически лежат на серверах вне РФ.

**Норма:** ст. 18 ч. 5 ФЗ-152 — оператор обязан обеспечить **запись, систематизацию, накопление, хранение, уточнение и извлечение ПДн граждан РФ с использованием БД, находящихся на территории Российской Федерации**. Реестр локализации ведёт Роскомнадзор; нарушение — административный штраф (ст. 13.11 КоАП, для юрлиц — повторно до 18 млн ₽ по новым нормам 2025).

**Оценка:**
- **Dev/staging с обезличенными данными** — риск приемлем (синтетические ФИО ≠ ПДн → ст. 18 не применяется).
- **Dev/staging с реальными ПДн (текущее состояние)** — уже **нарушение**: реальные 42 сотрудника на Railway вне РФ.
- **Прод на Railway** — **блокер запуска**: нельзя обрабатывать ПДн работников РФ в облаке вне РФ без локализации.

→ Заведено как **риск 152FZ-001 (P1)**, см. §3.

## 4. Минимизация: dev сейчас + меры для прода

### 4.1. Что обезличить/сделать на dev (приоритетно)

1. **Обезличить сид** (закрывает 152FZ-002 + снимает ст. 18 для dev): синтетические ФИО + домен `@example.test`, маппинг отделов сохранить. Реальные ФИО/email грузить из gitignored-источника в рантайме (план Dev 2 — поддерживаю). **После обезличивания Railway-dev перестаёт быть нарушением** (синт. данные ≠ ПДн).
2. **`.gitignore` + `git rm --cached`** на `research/**/*.xlsx`, `research/**/roster.csv` и аналоги (предложение Dev 2 #1). Переписывание истории — на усмотрение arch (приватный internal-repo), минимум — не плодить новые коммиты с ПДн.
3. **Расширить pre-commit secret-scan** на ФИО/email-паттерны (`@credos\.ru`) — предложение DevOps. Блокировать новые коммиты с реальными ПДн.
4. **Минимизация полей:** пересмотреть необходимость `middleName` и дублирования `email` (есть `workspaceMemberRef` → email можно тянуть из WorkspaceMember). Принцип ст. 5 — не хранить лишнее.
5. **Ограничить `description`:** в UI-подсказке/политике явно запретить ввод ПДн третьих лиц и чувствительных сведений в свободный текст «Состав работ».

### 4.2. Меры для прода (до выхода в прод — обязательны)
