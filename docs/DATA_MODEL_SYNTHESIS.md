## 2b. Классификация работ (билляность / утилизация) — ГЛАВНАЯ метрика

Отдельное измерение от Department и Activity — **характер работы**. Задаёт billable vs non-billable и считает **утилизацию (эффективность)** = доля часов «на клиента».

| Категория (WorkCategory) | Billable | Что | В Директум 5 |
|---|---|---|---|
| **Эффективные работы на клиента** | ✅ да | проектная работа по договору | проект клиента (своя орг) |
| **Пресейл** | ❌ нет | работа до договора (инвестиция в сделку) | — (нужно добавить) |
| **Пилот** | ❌ обычно нет | пилотный проект | — |
| **Внутренние проекты** | ❌ нет | напр. «Внутренняя автоматизация КРЕДО-С» | псевдо-проект под КРЕДО-С ООО |
| **Инфраструктура** | ❌ нет | обслуживание своей ИТ-инфры | псевдо-проект |
| **Самообучение** | ❌ нет | «Обучение/повышение квалификации», «Самостоятельное изучение» | тип работ без клиента |

**Реальность подтверждает:** в выгрузке внутреннее/обучение = записи на свою орг (КРЕДО-С ООО — 11 555 из 34k). То есть классификация уже де-факто есть, но неявная.

**Куда в модели:** `Project.category` (категория проекта) — основное место (внутреннее/пресейл/пилот = проекты со своей категорией, как в реальных данных). Самообучение — либо псевдо-проект «Обучение» категории training, либо `TimeEntry` без проекта с категорией. `billable` выводится из category (Client → true).

**Зачем критично:**
- Метрика **утилизации** на сотрудника/отдел/период = client-hours / total-hours — ключевой KPI PSA.
- Связь с инвойсингом (фаза 2): billable-часы → счёт.
- Пресейл/пилот отдельно → видно стоимость инвестиций в сделки.

```
Project.category: enum (Client | Presale | Pilot | Internal | Infrastructure | Training)
   → billable = (category === Client)
TimeEntry наследует категорию проекта (или своя, если без проекта — самообучение)
```

> Связь с §4: Service (типовая услуга из каталога) применима только к category=Client/Presale/Pilot (клиентские). Internal/Infrastructure/Training — без Service, только Activity.

## 3. Единая модель (SDK `defineObject`)

```
Department (Отдел)  ── ОВ / ОИБ / ОПИБ / ТЦ / ОПР
   │
   ├──< Employee (Работник)        — принадлежит отделу (= workspaceMember CRM)
   │
   │     approvalRequired: bool   — согласование вкл/выкл на уровне ОТДЕЛА (дефолт)
   │
   └──< Project (Проект)  ★ РАЗВИТАЯ сущность: свой справочник + карточка
          │  code, name           — собственная идентичность (НЕ = заказ 1С)
          │  ↑ Company (CRM, read) — клиент (Организация+юр.лицо; для внутр. = КРЕДО-С)
          │  ↑ manager → Employee  — РП
          │  ↑ Service (опц.)      — типовая услуга из каталога (только клиентские)
          │  category: enum        — Client | Presale | Pilot | Internal | Infrastructure | Training
          │  billable: bool        — выводится: (category === Client)
          │  status, startDate, endDate
          │  approvalRequired: bool|null — переопределяет отдел (null = наследует Department)
          │
          ├──< Stage (Этап)        — подуровень проекта
          │      code, name, status, startDate, endDate
          │
          ├──< BillingLink (связь с 1С) ◇ M:N, синхронизация ПОЗЖЕ
          │      externalSystem='1С', docType: Order|Payment|Act
          │      externalId, number, date, amount
          │      (1 проект ↔ N документов 1С; 1 документ 1С ↔ N проектов)
          │
          └──< TimeEntry (Запись трудозатрат)   ← атом = строка Директум 5
                 ├── date: Date
                 ├── hours: Number (DECIMAL — 0.5/0.75/8)   ← Timetta + реальность
                 ├── description: Text (Состав работ, свободный)
                 ├── employee: Relation → Employee
                 ├── stage: Relation → Stage (nullable)   — списание на этап
                 ├── activity: Relation → Activity (Тип работ, nullable)
                 ├── (billable: наследует Project.category, override возможен)
                 └── (status: enum — Draft/Submitted/Approved — фаза 2)

Activity (Тип работ / вид работ)   — справочник, scope: Department | global
Service (Типовая услуга)           — справочник услуг по каталогу, scope: Department
```

### Объекты и роли

| Объект | Источник | Поля-ядро | Заметки |
|---|---|---|---|
| **Department** | новое | code, name, approvalRequired | **5** значений (ОВ/ОИБ/ОПИБ/ТЦ/ОПР), сид |
| **Company** | CRM (read) | — | НЕ дублируем, читаем из Twenty |
| **Employee** | CRM workspaceMember + dept | →department | 36 чел в выгрузке ОВ |
| **Project** | Директум 5 + каталог | code, name, →company, →department, →service?, status | 172 проекта (ОВ) |
| **Service** | каталог услуг | name, →department | справочник «типовая услуга» |
| **Activity** | Директум 5 + каталог | name, →department?, billable? | 69 (ОВ) + ИБ-наборы |
| **TimeEntry** | строка выгрузки | date, hours, description, →employee, →project, →activity | сетка/таймер сверху |

## 3a. Project — развитая сущность и развязка с 1С (важно)

**Project — самостоятельная развитая сущность** (свой справочник + карточка), а НЕ техническая ссылка на заказ 1С или что-то ещё. Своя идентичность (code/name), свой жизненный цикл, свои этапы.

### Этапы (Stage)
Проект делится на **этапы**. Трудозатраты списываются на проект или на конкретный этап (`TimeEntry.stage`, nullable). Этап — подуровень проекта (code, name, статус, даты).

### Развязка с 1С — many-to-many (заложить сейчас, синхронизировать потом)

Project **не матчится 1:1** с финансовыми документами 1С:

```
1 Проект            ↔  2-3 заказа 1С        (проект бьётся на несколько заказов)
1 заказ 1С / оплата ↔  2-3 Проекта          (один заказ покрывает несколько проектов)
```

Поэтому связь — **M:N через junction `BillingLink`**, а не прямой FK:

| BillingLink (связь проект ↔ документ 1С) | |
|---|---|
| project | → Project |
| externalSystem | '1С' |
| docType | Order / Payment / Act |
| externalId, number, date, amount | реквизиты документа 1С |

- M:N возникает естественно: у проекта — N связей; один `externalId` (документ 1С) — в N связях (на разные проекты).
- **Синхронизацию с 1С НЕ делаем сейчас** — только закладываем структуру связи, чтобы потом подтянуть оплаты и акты без переделки модели.
- Существующая 1С-интеграция CRM (`credos-integrations`) — будущий источник наполнения BillingLink.

### Зачем так
- Учёт времени (Project/Stage) и финансы (заказы/оплаты/акты 1С) — **разные плоскости**, связанные гибко.
- Позволит сводить: трудозатраты по проекту ↔ выручка/оплаты по связанным документам 1С → рентабельность проекта (фаза с финансами).
- Не привязываемся к структуре 1С жёстко — Project живёт своей жизнью.

## 6. Решения по данным и согласованию (зафиксировано 2026-06-20)

| # | Вопрос | Решение |
|---|---|---|
| 1 | Миграция/данные | **Сид-данные, период 2026-01-01 … 2026-06-30.** ОВ — реальная выборка из выгрузки (3501 запись). ОИБ/ОПИБ/ТЦ — **сгенерированные мок-данные** (выгрузок нет). См. `SEED_DATA_PLAN.md` |
| 2 | Выгрузки др. отделов | **Нет.** Генерируем мок по каталогу услуг (виды работ) + правдоподобные проекты/сотрудники/часы |
| 3 | Service vs Activity | **Требует уточнения.** На старте — единый справочник Activity (scope=Department); Service отложен |
| 4 | Согласование (approval) | **Да, но отключаемое.** Флаг `approvalRequired` на **Department** (дефолт), переопределяется на **Project** (`null` = наследует отдел). Позволяет: отдел с согласованием / без; отдельный проект — исключение |
| 5 | Признак отдела при сиде | ОВ — по работнику (мэппинг); мок-отделы — задаются при генерации |
| 6 | Проект ↔ Company >1 орг | уточнить позже (рамочные договоры vs шум); на старте Project→1 Company |

### Approval — поведение
```
Запись попадает под согласование, если:
  Project.approvalRequired ?? Department.approvalRequired === true
Иначе — учёт «плоский» (как Директум 5), статус не применяется.
```
Старт: внутренние/обучение (Internal/Training) — обычно без согласования; клиентские проекты — с согласованием. Настраивается per Department/Project.

## 7. Методология: как Директум 5 ложится на Timetta и Kimai

Анализ: чем по СУТИ учёт в Директум 5 является и как соотносится с двумя референсами.

### 8.1. Природа трёх подходов

| | Директум 5 | Kimai | Timetta |
|---|---|---|---|
| **Парадигма** | плоский журнал (бухгалтерия часов) | список записей (entry-centric) | периодный таймшит (period-centric) |
| **Атом** | строка: дата+часы+проект+тип+работник | Timesheet: begin/end или duration | TimeAllocation: часы за день в строке |
| **Контейнер** | нет (плоско) | нет (фильтр по дате) | TimeSheet (неделя) → Line → Allocation |
| **Часы** | Decimal (0.5/0.75) | begin/end ИЛИ duration | Decimal (сетка) |
| **Согласование** | нет | нет (только lock/export) | да (lifecycle: Draft→Submitted→Approved) |
| **Билляность** | неявно (через проект/орг) | явно: billable + rates + invoice | billing types, tariffs, rate matrices |
| **Глубина** | минимум | средняя (бюджеты, команды, мета) | максимум (PSA: booking, P&L, ресурсы) |

### 8.2. Главный вывод: Директум 5 = ядро Kimai, развёрнутое плоско

Строка Директум 5 **один-в-один** = запись Kimai (duration-mode):

| Директум 5 | Kimai Timesheet | Совпадение |
|---|---|---|
| Организация | Customer | ✅ |
| Проект | Project | ✅ |
| Тип работ | Activity | ✅ |
| Работник | User | ✅ |
| Состав работ | description | ✅ |
| Часы (Decimal) | duration | ✅ (Kimai тоже умеет duration-mode без begin/end) |
| — | begin/end, billable, rate, tags, meta | Директум 5 не использует |

**Вывод:** Директум 5 = Kimai-иерархия `Customer→Project→Activity→Timesheet` БЕЗ надстройки (rates/budget/invoice/teams). То есть это **подмножество Kimai**. Наша модель = ровно эта основа + надстройки по потребности.

### 8.3. Чего Директум 5 не имеет — и что добавляет каждый референс

```
Директум 5 (плоский журнал)
   │
   ├─ + от Kimai (надстройка над тем же атомом):
   │     billable-флаг, rates (4 уровня), budgets, invoicing, Team ACL, meta-fields
   │
   └─ + от Timetta (другая ПОДАЧА того же атома):
         TimeSheet-контейнер (неделя) → Line → Allocation = недельная СЕТКА,
         approval-lifecycle (статусы), Decimal-часы, справочники РФ
```

Ключ: **Kimai и Директум 5 хранят так же (плоско, по записи). Timetta — про ПОДАЧУ (недельный контейнер) и ПРОЦЕСС (согласование).** Это не противоречие — разные слои.

### 8.4. Наше методологическое решение (best-of-three)

| Слой | Берём из | Решение |
|---|---|---|
| **Хранение** | Директум 5 / Kimai | плоский `TimeEntry` (1 строка = 1 запись, Decimal-часы). Просто, проверено, мигрируемо |
| **Подача (UX)** | Timetta | недельная СЕТКА = группировка плоских TimeEntry по неделе/проекту на лету. Контейнер — представление, не таблица |
| **Процесс** | Timetta (lite) | approval — опциональный enum-статус на записи/периоде (фаза 2). Старт без него (как Директум 5) |
| **Билляность** | наше + Kimai | `Project.category` → billable → утилизация. Rates/invoice — фаза 2 (Kimai-логика) |
| **Иерархия** | все три совпали | `Company→Project→Activity→Entry` + Department + category |

**Итог методологии:** храним как Директум 5/Kimai (плоский журнал записей), показываем как Timetta (недельная сетка — это VIEW над записями, без отдельной таблицы-контейнера на старте), согласование и биллинг добавляем слоями по фазам. Недельный контейнер TimeSheet как отдельная сущность вводим, только если понадобится approval уровня периода (тогда он группирует записи недели).

### 8.5. Почему НЕ копируем Timetta-контейнер сразу
Timetta-модель `TimeSheet→Line→Allocation` тяжелее: жёсткая неделя, предсозданные строки, состояния. Директум 5 показал, что **реально работает плоский журнал** (34k записей, 11 лет). Поэтому: плоское хранение + сетка-представление = та же UX-выгода без тяжести контейнера. Контейнер — позже и только под approval.

## 7a. Локализация: справочники и UX/UI — РУССКИЙ (жёсткое правило)

**Все справочники, ярлыки и UX/UI — на русском.** Английский — только внутренние технические идентификаторы (значения enum, имена полей в коде), которые пользователь не видит. Каждый код всегда имеет русский label для отображения.

### Справочники (данные) — русский
Department, Activity (виды работ), Service (услуги), Project/Stage названия, описания — **только русские значения** (как в Директум 5 и каталоге услуг). Мок-данные — тоже по-русски.

### Enum: технический код (англ.) → отображаемый ярлык (рус.)

**Категория работ (Project.category):**
| код | ярлык UI |
|---|---|
| Client | На клиента (эффективные) |
| Presale | Пресейл |
| Pilot | Пилот |
| Internal | Внутренний проект |
| Infrastructure | Инфраструктура |
| Training | Самообучение |

**Статус записи/таймшита (status):**
| код | ярлык UI |
|---|---|
| Draft | Черновик |
| Submitted | На согласовании |
| Approved | Согласовано |
| Rejected | Отклонено |

**Документ 1С (BillingLink.docType):**
| код | ярлык UI |
|---|---|
| Order | Заказ |
| Payment | Оплата |
| Act | Акт |

### UX/UI — русский
Все надписи интерфейса (недельная сетка, таймер, кнопки, колонки, фильтры, карточка проекта, согласование, отчёты, пустые состояния, ошибки) — на русском. Дни недели, форматы дат — русская локаль. Иконки нейтральны.

> Реализация в SDK: значения enum — латиницей (id), отображение — через русские labels в `defineField`/`defineObject` и в front-компонентах. Никаких англоязычных строк в UI.

## 8. Связь с архитектурой
Эта модель ложится в SDK-app (ADR-0002): каждый объект — `defineObject`; справочники — сид через `definePostInstallLogicFunction`; Company читается из CRM по Core API. Мульти-отдел, Activity-scope, Project.category — обычные поля/связи, без правок ядра.
