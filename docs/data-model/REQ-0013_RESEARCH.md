**Автор:** Dev 2 (бэкенд/данные), режим RESEARCH

**Статус:** research-вывод (код не менялся)
**Требование:** [REQ-0013](../requirements/REQ-0013-multi-department-projects.md)
**Рекомендация:** **Вариант A** (join-объект с долями) — поэтапно, 2 под-волны.

## 0. TL;DR

- Сейчас связь жёсткая: `project.departmentId` (MANY_TO_ONE). Один проект = один отдел. Это противоречит реальности (внедрение + ИБ-аудит + техцентр на одном клиенте) и референсу Timetta (команда проекта из разных подразделений).
- **Рекомендую Вариант A:** новый join-объект `credosTimeProjectDepartment` (project × department × `plannedEffortShare` в часах). Это единственный вариант, который корректно раскладывает **план/загрузку** по отделам. MANY_TO_MANY без долей (Вариант B) ломает capacity — план либо дублируется на каждый отдел, либо не делится.
- **Миграция безопасна и обратима:** текущий `project.departmentId` → одна join-запись с долей = 100% `plannedEffort`. Поле `departmentId` оставляем на переходный период (становится «основной отдел»), расчёты переключаем на join с fallback на старое поле — потери данных нет.
- **Оценка: ~7-9 под-задач, 2 под-волны.** Отдельный крупный трек после волны-3, как и помечено в требовании. Можно поэтапно: сначала модель+миграция+capacity (главная боль), потом отчёты byDept и UI.

## 1. Текущее состояние (как есть)

### Модель
`credosTimeProject.department` — `MANY_TO_ONE` на `credosTimeDepartment`, `joinColumnName: 'departmentId'`, `onDelete: SET_NULL` (`src/objects/credos-time-project.object.ts:147-162`). Обратная сторона — `credosTimeDepartment.projects` (ONE_TO_MANY).

План проекта — единое число `plannedEffort` (FLOAT) на проекте. Дат две: `startDate`/`endDate`. Раскид плана по периодам — равномерно по дням (`plannedHoursInPeriod` в `calc-load.ts:186-202`).

### Где `departmentId` проекта используется в расчётах

**`calc-load.ts` (доска планирования / capacity):**
- `deptLoadCells` (стр. 227-244): `for (const p of projects) if (p.departmentId === dept.id) load += projectHoursInPeriod(p, period)` — **весь** `plannedEffort` проекта падает на ОДИН отдел.
- `employeeLoadCells` (стр. 250-281): то же по departmentId, плюс делёж поровну `/headcount`.
- `deptProjectLoads` (стр. 310-330): детализация — фильтр `p.departmentId !== dept.id`.

**`reports-calc.ts` (отчёты /s/reports):**
- `deptOfEntry` (стр. 201-205, и дубль в OLAP 414-418): отдел записи = **в первую очередь** отдел сотрудника (`empById...departmentId`), **fallback** = отдел проекта. То есть факт по отделам в отчётах раскладывается ПРЕИМУЩЕСТВЕННО через сотрудника, проект — только запасной вариант.
- `byDept` (стр. 259-271): агрегат факта/нормы по отделам; норма = `headcount × factor × baseNorm` (headcount вычисляемый = count активных сотрудников).
- `RawProject.departmentId` (стр. 33) — поле контракта, используется лишь в `deptOfEntry` fallback.

> **Важное наблюдение для оценки влияния на отчёты:** в отчётах **факт** уже сейчас раскладывается по отделу СОТРУДНИКА, а не проекта. Поэтому мульти-отдел проекта на `byDept`-ФАКТ влияет слабо (факт идёт за человеком). Главный удар REQ-0013 — по **ПЛАНУ/ЗАГРУЗКЕ** (capacity), где сегодня всё висит на одном departmentId проекта. Это сужает скоуп.

### Тип-контракт фронта
`CapProject` (`types.ts:26-34`): `departmentId: string | null` + `plannedEffort` — единая пара. Именно её предстоит заменить на список долей.

## 2. Вариант A vs Вариант B

### Вариант A — join-объект `credosTimeProjectDepartment` (РЕКОМЕНДУЮ)

Новый объект: `project` (MANY_TO_ONE → Project, CASCADE) × `department` (MANY_TO_ONE → Department, CASCADE/SET_NULL) × `plannedEffortShare` (FLOAT, часы доли отдела). Опционально `sharePercent` вместо часов — но **часы предпочтительнее** (см. ниже).

| Плюсы | Минусы |
|---|---|
| Загрузка отдела = Σ долей проектов отдела — **корректный capacity** | Новый объект + миграция + правка расчётов (≈main cost) |
| План делится по отделам без двусмысленности | Ввод сложнее: РП задаёт долю на каждый отдел (UI-вкладка) |
| Отчёт byDept-план раскладывается по факту участия | Нужна валидация Σ долей ≈ plannedEffort проекта (или авто-распределение остатка) |
| Расширяемо: позже на join можно повесить FTE/манагера отдела на проекте | Связь REQ-0011 (FTE) добавит ещё одну размерность |
| Согласуется с Timetta (часы по участникам/отделам) | — |

**Часы vs %:** держать **часы** (`plannedEffortShare`), а не процент. Причины: (1) `plannedHoursInPeriod` уже работает с часами — раскид не меняется, меняется только источник числа; (2) проект может вообще не иметь `plannedEffort` (есть такие — секция «без плана»), тогда % не от чего считать, а часы самодостаточны; (3) сумма долей-часов = план проекта естественно (можно валидировать или показывать расхождение). Процент — производная для UI (`share/plannedEffort`).

### Вариант B — MANY_TO_MANY project↔departments без долей

| Плюсы | Минусы |
|---|---|
| Проще: нет нового поля-доли, чистая M:N | **Ломает capacity:** как делить `plannedEffort` на N отделов? |
| Меньше UI (просто мультиселект отделов) | Либо весь план на каждый отдел (×N перегруз), либо /N поровну (неверно — отделы вкладываются неравно) |
| | byDept-план неинформативен (доля неизвестна) |
| | Через год всё равно придётся добавлять доли → переделка |

**Вывод:** B решает только «проект показать в нескольких отделах», но не главную бизнес-задачу — «сколько часов отдел реально набрал». Поскольку весь смысл capacity — «когда отдел освободится взять новый проект», B даёт неверные цифры загрузки. **Отклоняю B.**

> SDK-замечание: MANY_TO_MANY в Twenty SDK всё равно материализуется join-таблицей. То есть «простота» B иллюзорна — join-объект A почти той же стоимостью даёт доли. Раз структура одна, берём ту, что несёт долю.

## 3. Влияние на `calc-load.ts`

Сейчас единица раскида — проект (`CapProject` с одним `departmentId` + `plannedEffort`). Станет — **доля** (project×department×share).

**Минимально-инвазивный подход:** ввести новый тип `CapProjectDept` (или массив `departmentShares` внутри `CapProject`) и заменить точки фильтрации `p.departmentId === dept.id` на обход долей:

- `deptLoadCells` — вместо `if (p.departmentId === dept.id) load += projectHoursInPeriod(p, period)` → для каждой доли отдела: `load += plannedHoursInPeriod(share.plannedEffortShare, p.startDate, p.endDate, period)`. Даты остаются на проекте (один срок на проект — достаточно для v1; срок-на-отдел = future).
- `employeeLoadCells` — то же; делёж `/headcount` отдела сохраняется.
- `deptProjectLoads` (детализация) — фильтр по наличию доли отдела; `perPeriod` считается от `plannedEffortShare`, не от полного `plannedEffort`.
- `projectHoursInPeriod` — можно оставить как есть (работает от числа) или обобщить: уже есть приватный `plannedHoursInPeriod(effort, start, end, period)` — он-то и нужен, просто передаём долю вместо полного эффорта. **Раскид-логика не меняется** — это ключевое удешевление.

**Fallback / обратная совместимость:** если у проекта нет ни одной join-доли (старые данные до миграции / проект без отделов) — использовать `project.departmentId` + полный `plannedEffort` как одну виртуальную долю 100%. Так доска не падает и в переходный период работает на смешанных данных.

**REST-загрузка (capacity-rest.ts / use-capacity.ts):** добавить выборку `credosTimeProjectDepartment` (с полями project.id, department.id, plannedEffortShare) и собрать `Map<projectId, Share[]>`. Это +1 запрос/выборка, без изменения формы периодов/ёмкости.

## 4. Влияние на `reports-calc.ts` (byDept)

**Факт (byDept fact):** меняется МИНИМАЛЬНО. `deptOfEntry` уже ведёт факт за отделом сотрудника (проект — лишь fallback). Запись делает конкретный сотрудник конкретного отдела → его часы и так падают в его отдел. Мульти-отдел проекта тут ничего не ломает: если в проекте участвуют люди из ОВ и ОИБ, их факт и сейчас разносится правильно по их отделам.

**Затрагивается только fallback** `deptOfEntry`: сейчас при отсутствии сотрудника берётся `projById.get(projectId).departmentId` — одно значение. При мульти-отделе «отдел проекта» неоднозначен. Варианты:
- v1 (просто): оставить fallback на «основной отдел» проекта (`departmentId`, который сохраняем) — записи без сотрудника редки.
- future: разнести по долям (но факт без сотрудника делить по плановым долям — спорно; лучше не усложнять).

**План в отчётах:** `byProject.plannedEffort`/`budgetUsed` остаются на проекте целиком (бюджет проекта общий — это правильно). Если понадобится «план по отделам» в отчёте byDept — это новая колонка, считается из join (Σ долей отдела). Не входит в обязательный минимум v1.

**Вывод:** `reports-calc.ts` правок в обязательном минимуме почти не требует (факт идёт за сотрудником). Это снимает половину риска с REQ-0013 — его центр тяжести в capacity, не в отчётах.

## 5. Миграция (без потери данных, обратимо)

1. **Создать** `credosTimeProjectDepartment` (join) + обратные стороны (`Project.departmentShares`, `Department.projectShares`).
2. **Бэкфилл (post-install logic / разовый скрипт):** для каждого `credosTimeProject` с `departmentId != null` создать 1 join-запись `{ project, department: текущий, plannedEffortShare: plannedEffort ?? null }`. Это доля 100%.
3. **`project.departmentId` НЕ удалять сразу.** Переименовать смысл в «Основной отдел» (РП проекта / fallback для отчётов и записей без сотрудника). Удаление поля — отдельное решение позже (риск для существующих view/фильтров/сидов).
4. **Расчёты с fallback:** если join-долей нет — берём `departmentId` + полный `plannedEffort`. Так смешанные данные (часть мигрирована, часть нет) работают.
5. **Сид/демо:** добавить 1-2 мульти-отдел проекта в сид-данные (например, клиентский проект с долями ОВ 60% / ОИБ 25% / ТЦ 15%) для проверки доски.

**Обратимость:** пока `departmentId` жив и fallback работает, откат = просто не читать join. Данные старого поля не тронуты.

> SDK-нюанс: бэкфилл удобнее всего `definePostInstallLogicFunction` (как сиды справочников, см. DATA_MODEL_SYNTHESIS §8). Но он идемпотентен только при проверке «join уже есть?» — иначе повторный install задублит. Заложить guard по (projectId, departmentId).

## 7. Связь со смежными REQ

| REQ | Связь | Вывод |
|---|---|---|
| **REQ-0011 (FTE по отделам)** | Тот же паттерн join (employee×department×fte). Headcount отдела (FTE) и доля проекта — независимые оси (один — про штат, другой — про план). | Делать раздельно. Но **единый стиль join-объектов** — переиспользовать подход/миграцию. Capacity отдела = FTE-headcount × норма (REQ-0011) vs Σ долей-плана (REQ-0013) — это знаменатель и числитель ratio, не конфликтуют. |
| **REQ-0012 (план без проекта, deptPlan)** | Уже реализован (`credosTimeDeptPlan`, MANY_TO_ONE на department). | Не конфликтует: deptPlan — отдельное слагаемое загрузки отдела. После REQ-0013: `load = Σ долей проектов отдела + Σ deptPlans отдела`. Просто ещё одно слагаемое в `deptLoadCells`. |
| **REQ-0004 (plan allocation по людям)** | v2: аллокация employee×project×period×hours. Это **более гранулярный** уровень, чем доля отдела. | Иерархия: доля отдела (REQ-0013) = грубее, аллокация по людям (REQ-0004) = точнее. Fallback-каскад: если есть аллокации по людям → Σ их; иначе Σ долей отделов; иначе полный plannedEffort. REQ-0013 встаёт **средним слоем** этого каскада — закладывать совместимо. |

**Архитектурная рекомендация:** REQ-0011 и REQ-0013 — два join-объекта одного паттерна (×department с долей). Имеет смысл сделать их одной командой/стилем, чтобы миграция и расчёты были единообразны. REQ-0013 первее (центральнее для capacity-загрузки).

## 8. Сверка с Timetta (правило 8)

Из `research/timetta-kimai-timesheet-views.md` и DATA_MODEL_SYNTHESIS:
- Timetta — PSA с **командой проекта из разных подразделений**; ресурсы/часы распределяются по участникам/отделам (ресурсный план `createLinesFromResourcePlan`).
- Часы — Decimal, раскладываются по строкам (Line = Клиент>Проект>Работа), а не висят одним числом на проекте.
- Мульти-отдел — **норма для PSA**, а не исключение. Наш `plannedEffortShare` (часы доли отдела) — упрощённая (отдел-уровень, не человек-уровень) проекция Timetta-ресурсного плана.
- Kimai мульти-отдела на уровне модели не даёт (Customer→Project→Activity плоско), но Team ACL = команды. Долей часов по отделам у Kimai нет → берём паттерн Timetta.

**Подтверждение:** Вариант A (доли часов по отделам) соответствует целевому PSA-референсу Timetta. Вариант B (просто M:N) — ниже даже Kimai по аналитике плана.

## 9. Оценка волн, риски, поэтапность

### Под-задачи (≈7-9)

**Под-волна 13a (модель + capacity — главная ценность):**
1. Объект `credosTimeProjectDepartment` + UUID в SSOT + обратные стороны (Dev2).
2. index-view + nav для join (или скрытый технический — но REQ-0013 предполагает правку в карточке, значит видимый/встроенный) (Dev2).
3. Миграция/бэкфилл departmentId → join 100% (post-install, идемпотентно) (Dev2).
4. `calc-load.ts`: раскид по долям + fallback на departmentId; новый тип долей (Dev2).
5. `capacity-rest.ts`/`use-capacity.ts`: выборка долей + сборка Map (Dev2).
6. UI capacity: проект в нескольких строках отделов с долей; согласовать `CapProject` (Dev1).

**Под-волна 13b (UI карточки + отчёты + сид):**
7. Карточка проекта «Отделы» (список + доли + валидация Σ) (Dev1).
8. Сид: 1-2 мульти-отдел проекта (Dev2).
9. (Опц.) reports byDept: уточнить fallback `deptOfEntry` + опц. колонка «план по отделам» (Dev2).

### Риски
- 🔴 **Регрессия capacity:** доска — ядро, много мест читают `departmentId`. Митигация: fallback на старое поле + полное покрытие `calc-load.ts` unit-тестами (там уже 948 тестов в проекте; добавить кейсы долей).
- 🟡 **Двойной учёт плана:** если у проекта остался `plannedEffort` И появились доли, нельзя считать оба. Правило: есть доли → игнорируем целый `plannedEffort` для capacity (он становится «итого-бюджет» проекта). Чётко зафиксировать.
- 🟡 **Валидация Σ долей ≠ plannedEffort:** не блокировать (РП может задать частично), но показывать расхождение в UI. Бизнес-решение — не технический блок.
- 🟡 **Идемпотентность бэкфилла** (повторный install). Guard по паре (project, department).
- 🟢 **Отчёты:** риск низкий (факт идёт за сотрудником, см. §4).

### Поэтапность — ДА
13a (модель+миграция+capacity) даёт основную ценность и самодостаточна (UI карточки можно временно вести через стандартную таблицу join). 13b (карточка/сид/отчёты) — следом. Можно мёржить волнами без «всё или ничего».

## 10. Рекомендация

**Делать Вариант A** (join `credosTimeProjectDepartment` с долей в часах), **поэтапно (13a → 13b)**, **после волны-3**, как отдельный крупный трек. Объединить стиль с REQ-0011 (FTE) как однотипные join-×department. Миграция через post-install-бэкфилл с сохранением `departmentId` (→ «основной отдел») и fallback в расчётах — потери данных и риска регрессии минимизированы. Центр тяжести — `calc-load.ts` (capacity), отчёты затрагиваются слабо.
