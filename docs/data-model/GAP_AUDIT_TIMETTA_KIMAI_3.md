**Дата:** 2026-06-21
**Автор:** arch (арх-аудитор)
**База:** `research/timetta/OFFICIAL_DOCS_INDEX.md` + `research/timetta/docs/*` (200+ статей, скачаны локально), наши `docs/STATUS.md`, `docs/ROADMAP.md`, `requirements/REQ-0001..0019`, `docs/analysis/ANALYST_FINDINGS.md`, прошлые `GAP_AUDIT_TIMETTA_KIMAI{,_2}.md`, фактические объекты `apps/time/src/objects/`.

> Отличие от v1/v2: те сверялись с разбором вью Timetta/Kimai (UI-наблюдения). v3 сверяется с **официальными статьями-докладами** Timetta — там вскрылись сущности и правила, которых в UI-разборе не было видно (правила валидации как сущность, периоды таймшита как сущность, запросы ресурсов, баланс отсутствий, исключения расписаний, гранулярная финансовая RBAC-модель).

## ТОП-5 рекомендаций (оставшиеся пробелы, которых нет/слабо в плане)

1. **Признак billable / тип оплаты проекта (Fixed/T&M/non-billable).** Без него утилизация (ключевая метрика PSA) считается некорректно — по всем часам, а не по оплачиваемым. Дёшево: enum-поле на `credosTimeProject` + правка `/s/reports`. Источник: `application-of-utilization-rate`. **Приоритет 🟡, новый.**

2. **Booking soft/hard как сущность (не флаг).** `tentativeBookingEnabled` — лишь настройка; нет плашек брони человек×проект×период×(soft/hard). Это сердце планирования Timetta и пресейла (REQ-0009). Спроектировать в рамках REQ-0004, заложив инвариант **«план оценки ≠ резерв»** (Timetta их намеренно не связывает). **Приоритет 🔴, под-пробел REQ-0004.**

3. **Resource Gap как метрика + цветовая шкала.** Поверх капасити-доски добавить `Gap% = (Demand−Capacity)/Capacity` и цвета (синий/зелёный/оранжевый/красный по нормам ±5/10/15%). Дёшево, высокая ценность руководителю. Источник: `resource-gap`. **Приоритет 🟡, новый.**

4. **Правила валидации как данные + уровни Ошибка/Предупреждение.** Сейчас валидация хардкодом. Минимальный шаг: вынести лимит часов/день и отклонение от расписания в `credosTimeSettings` + поддержать уровень warning (пропустить) vs error (блок). Источник: `timesheet-validation-rules`. **Приоритет 🟡, под-пробел REQ-0015.**

5. **Заложить финансовую RBAC-модель заранее (cost-rate ≠ bill-rate, field-level).** Прежде чем вводить ставки/P&L (REQ-0002), спроектировать раздельный доступ: ProjectFinancial / UserFinancial / ResourceCurrentRate, области Мои/Все, ставка скрыта от рядового сотрудника. Иначе ретрофит дороже. Источник: `financial-permissions`. **Приоритет 🔴 (на этапе RBAC-волны), под-пробел CISO-005.**

> Прочее новое низкого приоритета (зафиксировано, вероятно вне MVP): исключения расписаний/индивидуальные графики, баланс отсутствий + заявочный workflow, запросы ресурсов + пулы, универсальные ресурсы (роли-заглушки), substitute-замещение согласующего, Workflow Assignment в RBAC, прогноз vs оценка плана.
