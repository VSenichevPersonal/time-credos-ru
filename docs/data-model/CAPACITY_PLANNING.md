# Планирование загрузки отделов (capacity)

**Дата:** 2026-06-20
**Статус:** принятая концепция (простой уровень — «Excel»)
**Зачем:** видеть, на сколько вперёд загружен каждый отдел, чтобы **продажи могли обещать клиенту реальный старт работ**.

---

## 1. Идея простыми словами

Две плоскости учёта:
- **Факт** — сколько часов уже списали (TimeEntry). Прошлое.
- **План (загрузка)** — сколько часов уже наобещано отделам вперёд. Будущее.

Планёрка отвечает на один вопрос: **«Когда отдел освободится настолько, чтобы взять новый проект?»**

```
Отдел ОВ:  [████████████░░░] загружен до конца августа, свободен с сентября
Отдел ОИБ: [██████░░░░░░░░░] есть окно уже с июля
```

Продажи смотрят → обещают клиенту старт.

---

## 2. Как считаем (просто)

### Ёмкость отдела (capacity)
```
ёмкость отдела за неделю = кол-во сотрудников × норма часов (≈40ч) × коэффициент (вычесть отпуска/накладные, напр. ×0.8)
```
Кол-во сотрудников — из реестра (Битрикс/Employee). Коэффициент — настраиваемый.

### Загрузка (план)
У проекта/этапа есть `plannedHours` + период (`startDate…endDate`).
План раскидывается по неделям периода → недельная загрузка отдела = сумма плановых часов всех его проектов в эту неделю.

### Свободно
```
свободно за неделю = ёмкость − загрузка
```
Первая неделя, где `свободно ≥ нужного на новый проект`, = возможный старт.

---

## 3. Что видит пользователь (Excel-уровень)

Таблица: **строки = отделы, колонки = недели/месяцы вперёд**, в ячейке — загрузка.

```
Отдел \ Период │ Июл │ Авг │ Сен │ Окт │ Ноя
──────────────┼─────┼─────┼─────┼─────┼────
ОВ            │ 95% │ 90% │ 40% │ 20% │  0%
ОИБ           │ 60% │ 30% │ 10% │  0% │  0%
ОПИБ          │ 80% │ 85% │ 70% │ 50% │ 30%
ТЦ            │ 50% │ 55% │ 40% │ 30% │ 10%
ОПР           │100% │100% │ 90% │ 80% │ 70%
```

- Цвет ячейки: зелёный (есть место) → жёлтый → красный (перегруз).
- Клик по отделу → список проектов, формирующих загрузку.
- Цель — наглядность, не точный ресурс-менеджмент.

---

## 4. Модель (минимум полей)

| Объект | Добавляем | Зачем |
|---|---|---|
| **Project** | `plannedHours`, период (startDate/endDate уже есть) | план для раскидки по неделям |
| **Stage** | `plannedHours` | план на уровне этапа (детальнее) |
| **Department** | `headcount` (или из Employee) + `capacityFactor` | ёмкость |
| (вычисляемое) | загрузка/свободно по неделям | не хранится, считается на лету |

**Пресейл-сценарий:** проект категории Presale/Pilot с `plannedHours` можно держать как «предварительную бронь» — тогда продажи видят, что будет, ЕСЛИ сделку закроют. Опционально: флаг `tentative` (предварительный) — такие отделять цветом/фильтром.

---

## 5. Границы (что НЕ делаем)

- НЕ ресурс-планирование по людям (кто именно занят) — только по отделам. Просто.
- НЕ диаграмма Ганта, НЕ зависимости задач.
- НЕ автобалансировка. Человек сам видит и решает.
- Точность — недельная/месячная, не часовая.

> Если позже понадобится планирование по конкретным людям (booking) — расширим. Старт: уровень отдела, как просили.
