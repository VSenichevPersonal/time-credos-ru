# ADR-0001. Платформа, данные и авторизация для time.credos.ru

**Дата:** 2026-06-20
**Статус:** Принято
**Контекст-документ:** `../../ARCHITECTURE_QA.md` (Вопросы 1–6)

---

## Контекст

Строим систему учёта проектов и времени для Кредо-С. Есть три актива:
- **Timetta** — reverse-engineered целевая модель (258 таблиц, lifecycles, РФ-специфика).
- **Kimai** — open-source (AGPL-3.0) с готовой логикой трекинга/инвойсинга.
- **Twenty CRM** (форк CredosCRM1) — metadata-driven платформа, уже в проде на Railway (RBAC, аудит, API, AI, 1С/Dadata, SSO).

Ключевой ответ, определивший решение: **`time.credos.ru` — внутренний инструмент Кредо-С, не отдельный продаваемый продукт.**

## Решение

1. **Платформа — Twenty CRM.** Time tracking строим внутри существующего форка CredosCRM1, а не отдельным приложением и не с нуля.
   - Объекты `Project`, `Activity`, `Rate`, `TimeSheet`, `TimeSheetEntry` — через metadata-движок (REST API), без кода в ядре.
   - Бизнес-логика (approval, расчёты часы×ставка/billable/бюджет) — через query-hooks.
   - Недельная сетка + таймер — единственный кастомный UI: виджет(ы) в layout-движке Twenty.

2. **Данные — внутри платформы. Без отдельной БД и без интеграционного слоя.**
   - Company / Person / User / RBAC уже в Twenty — переиспользуем напрямую.
   - Вопрос «общая БД vs API-интеграция» снят: при размещении внутри Twenty он не возникает.

3. **Авторизация — через центральный IdP** (Keycloak или Entra ID, выбор в ADR-0002), а не общий APP_SECRET. Twenty уже умеет SSO (SAML + OIDC). IdP-клиент один — сам Twenty.

4. **Источники модели — гибрид с ролями:**
   - **Kimai = референс логики** (трекинг, инвойсинг, ставки, бюджеты). Переписываем под наш стек, **не копируем код** (AGPL-3.0 вирусный).
   - **Timetta = целевая модель** (недельная сетка, Decimal-часы, справочники РФ, lifecycles — поэтапно).
   - Старт: Kimai-простота (4 уровня, enum-статусы) → рост до Timetta-глубины по потребности.

## Альтернативы (отклонены)

| Вариант | Почему нет |
|---|---|
| Отдельное приложение + общая БД | Главный смысл (отдельный продукт) отпал. Прямое чтение таблиц Twenty хрупко — схема генерируется metadata-движком |
| Отдельное приложение + API-интеграция | Лишний деплой, интеграционный слой, вторая БД, отдельный IdP-клиент — не оправдано для внутреннего инструмента |
| С нуля | Выкинуть готовые auth/RBAC/клиентов/интеграции CredosCRM1 |
| Копирование кода Kimai | AGPL-3.0 обязал бы открыть исходники (вирусная, триггерится на SaaS) |

## Последствия

**Плюсы:** одна платформа и кодовая база; данные/auth/RBAC/аудит бесплатно; ноль интеграции и второй БД; быстрый старт (metadata-объекты).

**Минусы / риски:**
- Недельная сетка внутри Twenty-layout сложнее, чем в своём SPA — плата за «одну платформу». Принято осознанно.
- **Точка пересмотра:** если ежедневный ввод окажется критично-неудобным и снизит заполняемость — вынести ТОЛЬКО виджет ввода в отдельный мини-SPA, читающий Twenty по API. По факту боли, не авансом.

## Открыто (не входит в это ADR)

- Выбор IdP: Keycloak (self-hosted, 152-ФЗ) vs Entra ID (облако M365) — зависит от того, AD локальный или уже в Entra. → **ADR-0002**.
- Размещение под Keycloak в контуре с доступом к AD (если on-prem).
- URL: поддомен `time.credos.ru` отдельным фронтом vs раздел внутри CRM (по умолчанию — внутри).
