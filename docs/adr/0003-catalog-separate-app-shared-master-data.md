# ADR-0003. Каталог услуг — отдельный SDK-app в контуре CRM; общие мастер-данные

**Дата:** 2026-06-20
**Статус:** Принято
**Зависит от:** ADR-0001 (платформа Twenty), ADR-0002 (SDK-app)
**Источник анализа:** `CredosITCatalog` (прототип Next.js+Prisma) — изучен; `../data-model/DATA_MODEL_SYNTHESIS.md`

---

## Контекст

Есть запрос на **Каталог услуг** (он же Knowledge Hub): `Направление → Категория → Услуга → {Документы, Продукты/Вендоры, Компетенции}` + матрица компетенций. Прототип `CredosITCatalog` (отдельное приложение Next.js+Prisma) — пока «фантазии», не продакшен.

Каталог пересекается с time-приложением по мастер-данным (Отдел, Сотрудник, Услуга). Возник вопрос: тащить ли каталог в контур time и как заложиться на рост.

Ключевой фактор размещения: **каталогом активно будет пользоваться отдел продаж** (подбор услуг в сделку, экспорт в КП, cross-sell, консультирование клиентов).

## Решение

1. **Каталог — отдельное SDK-app**, НЕ внутри time-приложения. Разные домены (знания/услуги vs учёт часов), разные аудитории, разный UX, свой ритм релизов.

2. **Размещение каталога — контур CRM** (а не time). Первичный пользователь — отдел продаж; каталог тесно интегрируется с CRM: Услуга ↔ Opportunity (сделка), экспорт в существующий модуль КП/Quotes CredosCRM, cross-sell.

3. **Общие мастер-данные** — определяются ОДИН раз как объекты Twenty/CRM, оба приложения ссылаются (не дублируют):
   - **Department (Отдел)** — общий справочник.
   - **Employee (Сотрудник)** — общий (= workspaceMember + отдел).
   - **Service (Услуга)** — **владеет каталог** (богатое определение: этапы, документы, трудоёмкость, статус); time и продажи **ссылаются**.

4. **Service — мост трёх контуров:**
   - Каталог — владеет определением (rich-контент, lifecycle, deliverables, estimatedDuration).
   - Продажи (CRM) — подбирают услугу в сделку/КП, cross-sell.
   - Time — ссылается на услугу для классификации проекта + берёт `estimatedDuration` в планирование загрузки.

5. **Что из каталога заложить в архитектуру time сейчас (для роста):**
   - `Service` в time — **ссылка на общий объект-услугу** (один id), а не свой отдельный список. Вводим как loose-FK (фаза 2), чтобы при появлении каталога просто связать.
   - `estimatedDuration` услуги → кормит `plannedHours` планёрки загрузки.
   - Стиль lifecycle-статусов согласовать (draft→active→deprecated→archived).
   - НЕ тащить в time: Компетенции, Продукты, Вендоры, Wiki/Landing-контент — домен каталога.

## Схема

```
            Twenty / CRM (общие мастер-данные)
            Company · Employee · Department · Service
       ┌──────────┬───────────────────────┬──────────┐
       │          │                       │          │
  ┌────┴─────┐  ┌─┴──────────────┐   ┌─────┴────────┐ │
  │ time(SDK)│  │ catalog (SDK)   │   │ CRM/Quotes   │ │
  │ часы/план│  │ услуги/знания   │   │ продажи, КП  │ │
  └────┬─────┘  └──────┬──────────┘   └─────┬────────┘ │
       └── Service ────┴── владеет ──────────┘  ссылаются
          (ссылка)        каталог            продажи+time
```

## Альтернативы (отклонены)

| Вариант | Почему нет |
|---|---|
| Каталог внутри time-app | разные домены/аудитории/UX — раздует оба, размоет ответственность |
| Каждое приложение со своей копией Услуги/Отдела/Сотрудника | дублирование, рассинхрон; Service должна быть одна |
| Оставить каталог на Next.js+Prisma отдельным стеком | расходится с экосистемой Twenty; нет общих объектов и единого входа |

## Последствия

**Плюсы:**
- Чистое разделение доменов; каждый app развивается независимо.
- Единые мастер-данные → нет дублей, аналитика сходится (часы по услуге ↔ продажи услуги).
- Каталог рядом с продажами (КП/сделки) — где он реально нужен.
- Service-мост связывает учёт времени, продажи и знания.

**Минусы / открытое:**
- Каталог требует переноса прототипа на Twenty SDK (если решим converge) — отдельная оценка.
- Нужно определить «владельца» мастер-объектов (Service/Department/Employee) в Twenty и права.
- Порядок: time-app первым (есть данные/потребность); каталог — следующей итерацией.

## Открыто
- Конвергенция прототипа CredosITCatalog → SDK-app (оценить позже).
- Точная модель Service как общего объекта Twenty (поля, владелец, RBAC).
- Связь Service ↔ Opportunity/Quotes в CRM.
