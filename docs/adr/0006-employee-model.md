# ADR-0006. Модель сотрудника — `credosTimeEmployee` + ссылка на WorkspaceMember

**Статус:** PROPOSED (Dev 2 → arch/CISO на review)
**Дата:** 2026-06-20
**Связано:** [ADR-0003](0003-catalog-separate-app-shared-master-data.md) (общие мастер-данные), ADR-0005 (прод-топология, 2 инстанса), CISO-004 (PII-видимость/дубль), задача R-EMP (arch)

## Контекст

Вопрос заказчика: «не используем системные справочники для работников — норм?»

В Twenty 2.14 нативно есть:
- **`Company`** — клиенты ✅ (используем нативный).
- **`WorkspaceMember`** — *приглашённые пользователи* воркспейса (поля: `name` FULL_NAME, `userEmail`, `userId`, `avatarUrl`, `locale`, `timeZone`…). Сейчас реальный 1 (`vs@credos.ru`).
- **`Person`** — *внешние контакты CRM* (emails, phones, `companyId`, jobTitle) — это контрагенты, не штат.

Штат Кредо-С = **72 сотрудника**, логируют часы. Из них пользователей системы — единицы. Нативного «реестра сотрудников компании» в Twenty нет: `WorkspaceMember` ≠ весь штат, `Person` = внешние лица.

## Решение

**Оставляем кастомный `credosTimeEmployee`** как профиль штатной единицы:
- доменные поля: `department` (relation), `capacityFactor`/норма, `isManager`, `active`, `jobTitle`;
- `workspaceMemberRef` (TEXT) → ссылка на нативный `WorkspaceMember` для тех, кто И сотрудник, И пользователь системы.

**Источник истины ФИО/email:**
- сотрудник-**пользователь** (есть `workspaceMemberRef`) → имя/email берём из `WorkspaceMember` (read), в `credosTimeEmployee` НЕ дублируем;
- сотрудник-**не-пользователь** (большинство, 71/72) → профиль хранит минимум ПДн (ФИО для отображения), без учётки.

Паттерн «**staff ≠ users**» — корректен и стандартен (как HR-объект поверх auth-юзеров).

## Альтернативы (отклонены)

1. **Только `WorkspaceMember`** — отклонено: пришлось бы заводить учётку (и лицензию) каждому из 72, включая тех, кто не работает в системе. Раздувает auth, ПДн, стоимость.
2. **Только `Person`** — отклонено: `Person` = внешние контакты под `Company`; смешивать штат с контрагентами CRM семантически неверно, ломает фильтры/представления.
3. **Extend `WorkspaceMember` доменными полями** (технически возможно — уже расширяем его relation-полями, см. `fields/workspace-member-*-projects.field.ts`) — отклонено как носитель ПРОФИЛЯ: применимо только к пользователям, не покрывает 71 не-юзера. Используем extend лишь для relation-связей (проекты), не для штатного профиля.

## Последствия

**+** Меньше копий ПДн (152-ФЗ): для юзеров имя из `WorkspaceMember`, не дублируется. Чёткое разделение штат/юзеры/контрагенты.
**−** Дубль для не-юзеров неизбежен (ФИО в профиле) — но это минимум, без учёток.
**Контроль (CISO-004):** общий/видимый профиль штата при ADR-0003 (catalog) → PII доступна каталогу/продажам без RBAC. Field-level RBAC на ПДн-поля — отдельный трек (CISO-003/004).
**ADR-0005 (2 инстанса):** при разделении CRM v1.19 / time 2.14 — синк сотрудников по API (как Company). Зафиксировать в ADR-0005.

## Действие

1. **Убрать дубль (миграция, P2):** для `credosTimeEmployee` с `workspaceMemberRef` — `firstName`/`lastName`/`middleName`/`email` перестать писать; читать из `WorkspaceMember`. Поля оставить nullable для не-юзеров.
   - UI/таймшит/approval: отображаемое имя — резолв: если `workspaceMemberRef` → WorkspaceMember.name, иначе профиль.
   - Оценка миграции: 1 затронутая запись сейчас (vs@credos.ru) → дёшево внедрить ДО масштабирования штата в юзеры.
2. Зафиксировать резолв ID в коде (см. REQ-0001: `userWorkspaceId`≠`workspaceMemberId`).
3. CISO review: field-level RBAC на остаточные ПДн-поля профиля.

> UUID-стабильность (ADR-0004): рефактор полей НЕ меняет опубликованные `universalIdentifier`. Удаление поля = миграция данных → согласовать с DevOps/arch.
