# Библиотека юзкейсов для прогона QA — apps/time

**Назначение:** подробный каталог сценариев для ручного/браузерного прогона QA. Проверяем три вещи одновременно: (1) **получается ли операция** (функционально), (2) **удобно ли** (число кликов, понятность, нет тупиков), (3) **полнота UX/UI** (состояния пусто/загрузка/ошибка, валидация, доступность).

**Опора:** реальный код `apps/time/src/front-components/*` + `logic-functions/*`, требования `docs/requirements/REQ-0001..0019`, эталоны Timetta/Kimai (`research/timetta-kimai-timesheet-views.md`). Правило 8 — все идеи сверены с референсами.

**Дата:** 2026-06-21 · **Срез прода:** см. `STATUS.md`.

---

## Как пользоваться

- Сценарии сгруппированы по модулям, нумерация `UC-<МОД>-NN`.
- Колонка **Статус** для прогона: `⬜` не прогнан · `✅` ok · `❌` баг · `⚠️` замечание.
- Каждый UC содержит: актора, предусловия, шаги, что видно, ожидаемый результат, критерий удобства/полноты, референс.
- Негативные/граничные сценарии помечены 🔴, happy-path — без метки.
- Известные нереализованные в MVP функции (отзыв согласования, акт, e-mail-уведомления) отмечены **[V2 — не в MVP]** — проверяем отсутствие тупика, а не работу.

### Роли (акторы)

- **Сотрудник** — `isManager=false`, заполняет свой таймшит, видит свой ЛК.
- **Руководитель / РП** — `isManager=true` (= head отдела, REQ-0018), согласует, планирует, видит чужие таймшиты и ПДн.
- **Админ** — настройки, справочники, тоггл `revealEmployeeNames`.

# Модуль TS — Ввод часов (таймшит)

Файлы: `front-components/grid/*`, логика `logic-functions/time-entry-api.logic.ts`.

| ID | Название | Актор | Статус |
|---|---|---|---|

### UC-TS-01 — Сотрудник вводит часы за неделю (happy-path)
- **Актор:** Сотрудник.
- **Предусловия:** есть профиль `credosTimeEmployee` (workspaceMemberRef резолвится), ≥1 проект; режим грида = «Неделя».
- **Шаги:**
  1. Открыть «Таймшит». 2. Убедиться, что режим = «Неделя». 3. Кликнуть ячейку Пн строки проекта (или дождаться автофокуса на первой пустой). 4. Ввести `8`, нажать Enter (вниз) или Tab (вправо). 5. Заполнить будни.
- **Что видно:** заголовок «Таймшит»; mode-switcher День/Неделя/Проект (активная — заливка, 600); week-header с днями «Пн 17…Вс 23» и нормами; левые колонки Проект/Вид работ; ячейки tabular-nums; кольцо фокуса на активной ячейке; SaveIndicator «Сохранение…» → «Сохранено»; footer: итог за день, недельный итог vs «План на неделю: 40 ч».
- **Ожидаемый результат:** записи `credosTimeEntry` (date, hours, status=DRAFT) сохранены, перезагрузка показывает те же часы; недельный итог окрашен (under/ok/over).
- **Удобство/полнота:** один ввод = ≤2 клика/нажатия (цифра + Enter); автофокус на первой пустой; нет лишних модалок.
- **Референс:** Timetta недельная сетка (Line→Allocation, decimal-часы inline); Kimai QuickEntry.

### UC-TS-02 — Ввод в режиме «День»
- **Актор:** Сотрудник. **Предусловия:** есть записи за выбранный день.
- **Шаги:** переключить mode-switcher → «День»; ввести часы в крупную ячейку строки; при необходимости добавить описание (inline U11).
- **Что видно:** только строки с записями за день + пустые добавленные; горизонтальный макет (цвет-точка | проект | вид работ | описание | часы); подитог дня, норма дня, хинт недобор/норма/переработка.
- **Ожидаемый результат:** часы и описание сохранены без смены дня.
- **Удобство/полнота:** день без записей → empty-state «Нет записей за этот день. Добавьте проект ниже.» (не пустой экран).
- **Референс:** Kimai day-view.

### UC-TS-03 — Ввод в режиме «Проект»
- **Актор:** Сотрудник.
- **Шаги:** mode-switcher → «Проект»; выбрать проект в автокомплите (dropUp); заполнить виды работ × 7 дней.
- **Что видно:** до выбора — «Выберите проект, чтобы заполнить виды работ за неделю»; после — сетка только по видам работ (одна левая колонка).
- **Ожидаемый результат:** часы по видам работ выбранного проекта сохранены.
- **Удобство/полнота:** проект без записей после выбора → «По проекту нет записей. Добавьте вид работ ниже.»
- **Референс:** Timetta группировка по проекту.

### UC-TS-04 — Добавление новой строки (проект + вид работ)
- **Актор:** Сотрудник.
- **Шаги:** в блоке «+ строка» выбрать проект в автокомплите «Проект (код или клиент)…», затем «Вид работ…»; нажать «Добавить».
- **Что видно:** недавние проекты (recent) сверху списка; виды работ ограничены отделом проекта; кнопка «Добавить» disabled пока не выбраны оба поля; при выборе проекта подставляется последний использованный вид работ (W3-5).
- **Ожидаемый результат:** строка появляется в гриде (в `extraRowKeys`), поля автокомплита очищаются.
- **Удобство/полнота:** ≤3 клика; не требует ввода часов сразу.
- **Референс:** Timetta «добавить строку», Kimai Recent/Favorites.

### UC-TS-05 🔴 — Add-row при активных фильтрах (строка не должна прятаться)

- **Ожидаемый результат:** «Добавить» даёт видимый эффект — строка не исчезает.
- **Удобство/полнота:** нет «немого» действия (баг-гард). Это регресс-сценарий.
- **Референс:** —.

### UC-TS-06 — Клавиатурная навигация по сетке
- **Актор:** Сотрудник.
- **Шаги:** активировать ячейку; перемещаться ↑↓←→, Tab/Shift+Tab; начать ввод цифрой; Enter — подтвердить и вниз; Esc — отмена.
- **Что видно:** подсветка активной ячейки (кольцо фокуса); ввод цифры сразу открывает редактор (seed-символ).
- **Ожидаемый результат:** курсор движется в границах сетки (clampCell), значения коммитятся.
- **Удобство/полнота:** мышь не обязательна; границы не «проваливаются».
- **Референс:** Timetta tab-навигация.

### UC-TS-07 — Удаление записи (0 / Delete)
- **Актор:** Сотрудник.

# Модуль VAL — Валидация ввода

Файлы: `grid/use-validation.ts`, `validation-toast.tsx`, `constants/validation.ts`, серверная `time-entry-api.logic.ts`.

### UC-VAL-01 🔴 — Превышение жёсткого лимита (ERROR, блок)
- **Актор:** Сотрудник. **Предусловия:** `maxHoursPerDay` (напр. 24).
- **Шаги:** ввести часы выше лимита.
- **Что видно:** toast ERROR (терракот, иконка «!», aria-live=assertive, не гаснет, кнопка ×): «Часы за день вне допустимого диапазона».
- **Ожидаемый результат:** запрос в API НЕ уходит (blocked), значение не сохранено.
- **Удобство/полнота:** ошибка явная, не исчезает до исправления.
- **Референс:** —.

### UC-VAL-02 — Переработка (WARNING, не блок)
- **Актор:** Сотрудник. **Предусловия:** `overtimeWarnHours` (напр. 12), значение между порогом и лимитом (13 ч).
- **Что видно:** toast WARNING (янтарь, иконка «△», aria-live=polite, авто-гаснет ~6 с).
- **Ожидаемый результат:** значение сохраняется (предупреждение информативное).
- **Удобство/полнота:** не мешает работе; гаснет само.
- **Референс:** Timetta предупреждение переработки.

### UC-VAL-03 — Недобор недельной нормы
- **Актор:** Сотрудник. **Предусловия:** `minHoursPerWeek`>0, `warnOnScheduleDeviation=true`.
- **Шаги:** заполнить неделю ниже нормы; перейти к отправке.
- **Что видно:** мягкое предупреждение (не блок).
- **Ожидаемый результат:** отправка возможна.
- **Удобство/полнота:** при `minHoursPerWeek=0` или выключенном тоггле — предупреждений нет.
- **Референс:** REQ-0015 pre-submit.

### UC-VAL-04 🔴 — Правка согласованной ячейки (locked)
- **Актор:** Сотрудник. **Предусловия:** день в статусе APPROVED.
- **Шаги:** кликнуть APPROVED-ячейку, попытаться ввести.
- **Что видно:** ячейка read-only (panelBg фон, textMuted, иконка замка LockGlyph, aria-disabled); toast WARNING «Согласовано — правка запрещена. Нужен отзыв согласования.»
- **Ожидаемый результат:** запрос не уходит (клиентский no-op); на сервере — `cannot_modify_approved`.
- **Удобство/полнота:** замок виден заранее; сообщение объясняет причину и выход.
- **Референс:** Kimai lock согласованного периода.

### UC-VAL-05 — Гибкий парсинг ввода часов
- **Актор:** Сотрудник.
- **Шаги:** ввести разные форматы: `1.5`, `1,5`, `.5`, `1:30`, `:45`, `1ч30м`, `90м`, `1h30`.
- **Ожидаемый результат:** все распознаются; шаг 0.25 ч (quantize); нереалистичное → откат.
- **Удобство/полнота:** прощает форматы; не требует строгого ввода.
- **Референс:** Timetta decimal-часы.

### UC-VAL-06 🔴 — Серверная валидация / инъекции (CISO-006)
- **Актор:** Сотрудник (попытка обхода).
- **Шаги:** отправить запрос с невалидными `workspaceMemberRef`/`id`/`from,to` (не-UUID, не-ISO).
- **Ожидаемый результат:** сервер отклоняет (guard inject), 400 с описанием; запись не создаётся.
- **Удобство/полнота:** граница безопасности; легитимный ввод не страдает.
- **Референс:** —.

# Модуль APR — Согласование

Файлы: `grid/approval-bar.tsx`, `use-approval.ts`, `approval-rest.ts`, логика `logic-functions/approval.logic.ts`. Требования REQ-0001, REQ-0007.

### UC-APR-01 — Сотрудник отправляет период на согласование (submit)
- **Актор:** Сотрудник. **Предусловия:** есть DRAFT-записи периода в проектах с `approvalRequired=true`.
- **Шаги:** заполнить неделю; в approval-bar нажать «Отправить на согласование (N)».
- **Что видно:** бейдж «Черновик» (нейтральный, тусклая точка); счётчик DRAFT в кнопке; при пробелах — GapsNotice (янтарь) «Не вся неделя заполнена по норме…».
- **Ожидаемый результат:** записи DRAFT → SUBMITTED; бейдж → «На согласовании» (янтарь); кнопка submit исчезает.
- **Удобство/полнота:** одна кнопка на весь период (REQ-0007 by period); пробелы предупреждают, но НЕ блокируют.
- **Референс:** Timetta «Отправить на согласование» (период целиком).

### UC-APR-02 — Чеклист пробелов перед отправкой (REQ-0015)
- **Актор:** Сотрудник.
- **Предусловия:** неделя заполнена частично (Пн-Чт по 8, Пт-Вс пусто).
- **Что видно:** GapsNotice «…2 дня не заполнено (недобор 16 ч). Проверьте перед отправкой.»
