## D1. СОСТОЯНИЯ (пусто / загрузка / ошибка+ретрай / нет-доступа / нет-данных-за-период)

### Контекст находок (из кода)
| Экран | Загрузка | Пусто | Ошибка | Нет-доступа | Нет-данных-за-период |
|---|---|---|---|---|---|
| grid (таймшит) | текст «Загрузка…» | «Нет записей. Начните печатать…» | ErrorState+retry | locked-ячейки 🔒 | — |
| capacity | текст «Загрузка данных планирования…» | «Нет отделов для планирования» | ErrorState+retry | режим «Планировать» только isManager | «Нет сотрудников для среза «по людям»» |
| reports/projects | **скелетон 7 строк** | «За выбранный период проектов…нет» | ErrorState+retry | — | по фильтру статуса/периода |
| reports/missing | **скелетон 6 строк** | «Все заполнили — недоборов нет» 🎉 | ErrorState+retry | «Напоминания выключены» 🔕 | — |
| my-time | текст «Загрузка…» | — | ErrorState (periods — **без** retry) | «Профиль не связан с сотрудником» | — |
| project-budget/summary/team | текст «Загрузка…» | разный | `Center` (текст, **без** retry/ErrorState) | — | — |
| calendar | текст «Загрузка календаря…» | «Нет данных календаря за {год}» | `Center` (текст, **без** retry) | — | — |

**Системные пробелы:** скелетоны лишь на 2 экранах; project-*/calendar/settings показывают ошибку как сырой текст `Center` без кнопки «Повторить»; нет единого empty-шаблона (emoji vs нет); состояние «нет прав» нигде не явное (только locked-ячейки и unlinked-профиль).

### Группа D1.A — Единый шаблон состояний
**D1.1** Нужен ли ЕДИНЫЙ визуальный шаблон для всех четырёх состояний (пусто/загрузка/ошибка/нет-доступа) во всём модуле, или допускаем разнобой по экранам?
— Гипотезы: (а) единый компонент `StateView` (иконка+заголовок+подсказка+опц.действие) на всё; (б) оставить как есть; (в) единый только для ошибки/пусто, загрузку — скелетоны.
— реф: impeccable empty-states «acknowledge → value → action»; Timetta — один стиль пустых/ошибочных экранов на весь продукт.

**D1.2** Загрузка: переходим везде на скелетоны (как reports/projects), или текст «Загрузка…» приемлем для лёгких экранов?
— Гипотезы: (а) скелетоны для всех таблиц/сеток (таймшит, capacity, my-time, project-team), текст — только для мелких блоков; (б) скелетоны везде; (в) оставить текст.
— реф: impeccable «Loading: reassuring» + skeleton снижает воспринимаемую задержку; Kimai/Timetta — skeleton-строки в таблицах.

**D1.3** Скелетон должен повторять структуру конкретной таблицы (колонки таймшита 9 шт / доска / отчёты) или достаточно generic «серых полос»?
— Гипотезы: (а) точная форма под каждую сетку; (б) generic-полосы; (в) generic + правильное число колонок.

**D1.4** Где показывать индикатор сохранения (save-indicator) и нужен ли он одинаковый на всех экранах с автосейвом (таймшит, capacity-план)?
— Гипотезы: (а) единый компактный «Сохранено / Сохраняю…» вверху; (б) только на таймшите; (в) toast.
— реф: impeccable «Saving your work…».

### Группа D1.B — Ошибка + ретрай
**D1.5** project-budget / project-summary / project-team / calendar / settings показывают ошибку текстом без кнопки «Повторить». Делать везде ErrorState с retry?
— Гипотезы: (а) да, ErrorState+retry на всех экранах с загрузкой; (б) retry не нужен где «открой карточку проекта»; (в) минимум — единый стиль текста.
— реф: impeccable «Network error: Check your connection and [action]»; ErrorState уже даёт это.

**D1.6** `my-time/my-periods` использует ErrorState БЕЗ `onRetry`. Это намеренно (нечего ретраить) или забыли?
— Гипотезы: (а) добавить retry; (б) намеренно; (в) заменить на единый шаблон.

**D1.7** Нужно ли различать типы ошибок текстом: сеть vs нет-прав vs пусто-на-бэке? Сейчас всё — «Не удалось загрузить…».
— Гипотезы: (а) три формулировки (сеть/доступ/сервер) по impeccable-шаблону; (б) одна универсальная; (в) две (сеть/прочее).
— реф: impeccable Error templates (Network/Permission/Server).

**D1.8** «Подробнее» раскрывает сырой текст исключения. Заказчику/пользователю это полезно или прятать совсем (только для админа)?
— Гипотезы: (а) оставить «Подробнее» всем; (б) показывать detail только менеджеру/админу; (в) убрать, логировать молча.

### Группа D1.C — Пустые состояния
**D1.9** Унифицировать тон пустых состояний: с emoji (🎉/🔕) или строго нейтрально без эмодзи?
— Гипотезы: (а) убрать emoji (product-register, спокойный тон); (б) оставить emoji как «человечность»; (в) emoji только для позитива (🎉 «все заполнили»).
— реф: impeccable product-register — спокойно; CLAUDE-инструкция «избегать emoji» в коммуникации (но это UI-контент — решает заказчик).

**D1.10** Каждое пустое состояние должно давать ДЕЙСТВИЕ (кнопку/ссылку), а не только текст? Сейчас только часть («Внесите трудозатраты…»).
— Гипотезы: (а) везде action в empty; (б) action где осмысленно (таймшит → «Добавить строку»); (в) текст достаточно.
— реф: impeccable «Empty states are onboarding moments → clear action».

**D1.11** Таймшит-empty «Нет записей. Начните печатать код проекта…» — текст понятен новичку? Нужна ли кнопка «Добавить строку» рядом?
— Гипотезы: (а) добавить явную кнопку; (б) оставить текст-подсказку; (в) показать пример строки-плейсхолдер.

### Группа D1.D — Нет-данных-за-период vs общее «пусто»
**D1.12** Различать ли «совсем нет данных» и «нет данных за выбранный период/фильтр»? Сейчас часто один и тот же текст.
— Гипотезы: (а) два разных текста (+ «Сменить период» в фильтр-варианте); (б) один текст; (в) различать только в reports.
— реф: impeccable — empty по фильтру должен подсказывать «ослабить фильтр».

**D1.13** В empty-по-фильтру нужна ли кнопка «Сбросить фильтры / Сменить период» прямо в пустом блоке?
— Гипотезы: (а) да, быстрый сброс; (б) пользователь сам идёт в тулбар; (в) только текст-подсказка.

**D1.14** capacity «Нет сотрудников для среза «по людям»» — это «нет данных» или «нет прав видеть людей» (ПДн/revealEmployeeNames)? Нужно различать причину?
— Гипотезы: (а) различать (нет данных vs скрыто настройкой ПДн); (б) один текст; (в) если скрыто — явно «ФИО скрыты настройкой».

### Группа D1.E — Нет-доступа / прав
**D1.15** Нужно ли явное состояние «Нет доступа к разделу» (роль без прав на capacity-план/reports/настройки), или раздел просто не показывается в меню?

**D1.16** «Профиль не связан с сотрудником» (unlinked) — текст и инструкция достаточны? Нужна ли кнопка-действие (напр. «Связать профиль» / «Написать админу»)?

**D1.17** locked-ячейки (🔒 «Согласовано — правка запрещена») — нужен ли на уровне СТРОКИ/недели сводный индикатор «строка согласована, read-only», а не только по ячейкам?
— Гипотезы: (а) pill статуса строки слева; (б) баннер сверху «Неделя согласована»; (в) только ячейки 🔒.

### Группа D1.F — CSV/экспорт и прочее
**D1.18** Экспорт (timesheet-grid Т-13, reports) выдаёт `textarea` с CSV вместо скачивания (ограничение Remote DOM). Это понятно пользователю? Нужен ли поясняющий текст «Скопируйте и вставьте в Excel»?

**D1.19** Нужен ли глобальный индикатор офлайн/потери связи (баннер «Нет соединения, изменения не сохраняются»)?

**D1.20** При частичной ошибке (часть данных загрузилась, часть нет) — показывать данные + предупреждение, или полностью блокировать экран ошибкой?

## D2. ДОСТУПНОСТЬ / КЛАВИАТУРА

### Контекст находок (из кода)
- `title` ≈ 76-92 вхождений vs `aria-label` ≈ 24-28 → подсказки/полные имена живут в `title`, недоступном с клавиатуры/touch.
- НЕТ `role=grid/gridcell/row` ни в одной таблице-сетке (таймшит, capacity, reports) — div+CSS-grid. Ячейки `tabIndex={-1}`, навигация через родительский `use-keyboard`.
- focus-ring есть (`inset 0 0 0 2px accentRing`) на сегментах/строках/ячейках.
- Esc закрывает поповеры (row-menu, copy-menu, cell-comment, approval-confirm), НО cheatsheet Esc не закрывает (только клик-вне).
- Контраст: `textFaint` на `weekendBg`/`todayCol`/heatmap-tint — на грани; placeholder `·` и норма дня приглушены вдвойне.

### Группа D2.A — Семантика сеток (role=grid)
**D2.1** Добавлять ли `role="grid"/"row"/"gridcell"` + `aria-rowindex/colindex` на таймшит-сетку, чтобы скринридер объявлял «строка 3, Среда, 8 часов»?
— Гипотезы: (а) да, полная grid-семантика на таймшит (приоритет); (б) на все три сетки (таймшит/capacity/reports); (в) отложить, не критично для desktop-инструмента.
— реф: impeccable harden + WCAG 2.1; самый весомый a11y-долг по аудиту.

**D2.2** Что объявлять при навигации по ячейке: «День, Проект, Вид работ, N часов» — полная координата или кратко? Это влияет на `aria-label` ячейки.
— Гипотезы: (а) полная координата; (б) только «N часов»; (в) координата при входе в строку, далее только значение.

**D2.3** capacity-доска и reports-таблицы — приоритет grid-семантики такой же, как у таймшита, или ниже (там в основном чтение, не ввод)?
— Гипотезы: (а) ниже приоритет; (б) равный; (в) reports — как таблица данных (role=table), capacity — grid.

### Группа D2.B — title vs aria-label
**D2.4** Системно перенести важный текст из `title` в `aria-label` (или дублировать)? Какие подсказки критичны (полное имя в ellipsis, «Согласовано — правка запрещена», тултип комментария)?
— Гипотезы: (а) все смысловые `title` → дублировать в `aria-label`; (б) только статусные/блокирующие; (в) оставить как есть.
— реф: impeccable «title недоступен с клавиатуры/touch»; аудит — системный пробел recognition.

**D2.5** Полные имена в усечённых ячейках (проект/вид работ/сотрудник, ellipsis) сейчас доступны лишь по `title` (hover мышью). Как раскрывать с клавиатуры/touch?
— Гипотезы: (а) `aria-describedby` + показ полного имени при focus строки; (б) tooltip по focus; (в) расширять строку по клику.
— реф: impeccable «Recognition not recall».

**D2.6** Иконка-кнопка комментария ✎ имеет `title`, но нужен ли ей явный `aria-label` («Комментарий к записи: …»)?
— Гипотезы: (а) да, aria-label обязателен для icon-button; (б) title достаточно; (в) и то и то.
— реф: impeccable «Icon buttons need aria-label».

### Группа D2.C — Навигация с клавиатуры
**D2.7** Подтверждаем модель навигации таймшита: стрелки = по ячейкам, Tab = вправо, Enter = вниз, Esc = отмена? (см. TIMESHEET_CELL_CONTROLS — уже решено, но влияет на a11y-объявления.)

**D2.11** copy-menu input не ловит Esc (по находке). Привести инпуты в поповерах к «Esc = отмена/закрыть»?
— Гипотезы: (а) да, единообразно; (б) Esc только очищает input; (в) оставить.

### Группа D2.D — Скринридер / aria-live

### Группа D2.E — Контраст / не-цветовые сигналы
**D2.14** Контраст `textFaint` для нормы дня и placeholder `·` на фоне выходного/сегодня/heatmap — поднимать кегль/тон до WCAG AA?
— Гипотезы: (а) поднять тон textFaint на цветных фонах; (б) увеличить кегль вторичных чисел до ≥10px; (в) оставить.

### Группа D2.F — Touch / hit-area

## D3. АДАПТИВ / МОБАЙЛ / УЗКИЙ ЭКРАН

### Контекст находок (из кода)
- Виджеты — ФИКС-размера (так задумано: front-component не должен иметь скролл, кроме canvas). CLAUDE: «responsive to its fixed widget height and width».
- Горизонтальный скролл уже есть: capacity-доска (`minWidth: NAME_WIDTH + periods*colWidth`), calendar month-table (`minWidth:640`), my-time/project-team (overflow auto).
- sticky-колонки имён в capacity/my-periods/project-team.
- Тулбары переполняются: grid/toolbar (6 контролов), capacity/board-toolbar (до 8 сегментов) — `flexWrap` спасает, но неаккуратно.
- НЕТ `@media`/CSS-медиазапросов (Remote DOM) — адаптив только через JS-логику без размеров вьюпорта.

### Группа D3.A — Стратегия адаптива в Remote DOM
**D3.1** Целевые форм-факторы модуля: только десктоп, десктоп+планшет, или нужен телефон?
— Гипотезы: (а) десктоп-приоритет, планшет «не ломается», телефон позже; (б) десктоп+планшет полноценно; (в) только десктоп.
— реф: TIMESHEET_CONTROLS F2 — заказчик подтвердил десктоп-приоритет; impeccable «detect input, not screen».

**D3.2** Раз `@media` недоступен, согласны ли на адаптив через JS-измерение контейнера (если доступен размер виджета) или фиксируем «один десктоп-лейаут + горизонтальный скролл на узком»?
— Гипотезы: (а) фикс-лейаут + h-scroll где не влезает; (б) JS-измерение и перекомпоновка (сложно в Remote DOM); (в) две статичные раскладки по порогу.

### Группа D3.B — Таймшит-сетка (9 колонок)
**D3.3** 9 колонок (Проект, Вид работ, 7 дней) при узком виджете: горизонтальный скролл, или скрывать/сжимать колонки?
— Гипотезы: (а) sticky левые 2 колонки + h-scroll по дням; (б) сжать дни до минимума без скролла; (в) переключатель «компактный вид».
— реф: CLAUDE запрещает скролл-вместо-адаптива; но 9 колонок физически широкие — нужен компромисс.

**D3.4** При нехватке ширины что важнее сохранить читаемым: дни (ввод часов) или Проект/Вид работ?
— Гипотезы: (а) дни в приоритете, Проект/Вид — ellipsis+подсказка; (б) проект важнее, дни уже; (в) равный приоритет.

**D3.5** Нужен ли альтернативный «список-вид» таймшита для узкого экрана (как Timetta mobile: день → список записей вертикально), вместо сетки?
— Гипотезы: (а) да, отдельный вертикальный вид для узкого; (б) только сетка+скролл; (в) режим «День» (day-view) и есть мобильный ответ.
— реф: Timetta — отдельный упрощённый мобильный вид.

### Группа D3.C — Capacity-доска (горизонтальный скролл)
**D3.6** Доска уже горизонтально скроллится (периоды × колонки). Sticky-колонка имени достаточна, или нужна и sticky-строка summary/период-хедер при вертикальном скролле?
— Гипотезы: (а) sticky имя + sticky хедер периодов; (б) только имя; (в) + sticky summary.

**D3.7** Гранулярность недели даёт МНОГО колонок (год = 52). Ограничивать видимое окно периодов (напр. квартал) с пагинацией, или бесконечный h-scroll?
— Гипотезы: (а) окно + переключение периода; (б) h-scroll без ограничений; (в) авто-гранулярность (год→месяцы).

**D3.8** Тулбар доски — до 8 сегментов (метрика 4 + срез 2 + гранулярность 2) в строку. Свернуть второстепенное (гранулярность) в меню «▾»?
— Гипотезы: (а) гранулярность в overflow-меню; (б) две строки тулбара; (в) оставить flexWrap.
— реф: аудит P2 «>4 видимых опций»; impeccable cognitive-load.

### Группа D3.D — Reports-таблицы
**D3.9** Detail-отчёт (40+ полей, 6 групп) — как показывать на узком: горизонтальный скролл, выбор видимых колонок, или группы-аккордеоны?
— Гипотезы: (а) выбор колонок (column picker) + h-scroll; (б) только h-scroll; (в) сворачиваемые группы.
— реф: impeccable — плотные таблицы лучше с настройкой колонок, чем бесконечный скролл.

**D3.10** KPI-карточки отчётов (4 в ряд) на узком — переносятся (flexWrap) или стек в 1 колонку с приоритетом главной (Утилизация)?
— Гипотезы: (а) стек, главная метрика крупнее; (б) 2×2; (в) flexWrap как есть.
— реф: аудит P3 «identical card grid» — усилить вес главной.

### Группа D3.E — Тулбары и сайдбар
**D3.11** grid/toolbar (6 контролов) — свернуть 3 кнопки копирования/заполнения в одно меню «Заполнить ▾» (overflow) для узкого?
— Гипотезы: (а) overflow-меню; (б) иконки вместо подписей; (в) две строки.
— реф: аудит P2 топ-1 быстрый выигрыш.

**D3.12** Сайдбар/навигация по вкладкам модуля (Записи/Мои часы/Планирование/Отчёты/Настройки) — как ведёт себя на узком: остаётся, сворачивается в «гамбургер», или это вне нашего контроля (host Twenty)?
— Гипотезы: (а) навигация хостовая, нас не касается; (б) внутренние табы свернуть в селект; (в) горизонтальный скролл табов.

**D3.13** flexWrap-тулбары переносятся «неаккуратно». Нужен ли явный порядок переноса (что уходит первым) или фиксированные группы?
— Гипотезы: (а) приоритезировать: важное остаётся, второстепенное в overflow; (б) фикс-группы; (в) оставить.

### Группа D3.F — Планшет vs десктоп
**D3.14** Если планшет в скоупе: контролы крупнее (≥44px touch), но это съедает плотность таблицы. Где компромисс — отдельный «touch-режим» или один размер?
— Гипотезы: (а) один desktop-размер, планшет терпит; (б) touch-режим по флагу; (в) хит-зоны ≥44 при визуале ≤24.

**D3.15** Inline-ввод в ячейку на планшете (экранная клавиатура) — поведение приемлемо, или нужен альтернативный нумпад/степпер?
— Гипотезы: (а) системная клавиатура; (б) числовой степпер +/−; (в) preset-чипы (8/4) основной ввод на touch.

## D4. UX-КОПИРАЙТИНГ / ТЕКСТЫ

### Контекст находок (из кода)
- Англицизмы в UI: `«Gap»` (сегмент доски, label), хотя в hint расшифровано. `FTE`/`headcount`/`bulk` — только в коде/комментариях (в UI нет).
- Неоднозначные имена: `«Заполнить 8×5»` (тулбар), `«Заполнить будни (8 ч)»` (меню строки), `«Только строки» / «Строки и часы»` (копирование) — три «заполнить» (см. TIMESHEET_CONTROLS A.3).
- Подтверждения: «Удалить все несогласованные записи недели?», «Неделя недозаполнена: N из M ч. Всё равно отправить?».
- Хорошо: «Согласовано — правка запрещена», placeholder «Что делали…», «Что исправить сотруднику…».
- em-dash «—» в hint — для русского легитимен (аудит P3 контекстный, не баг).

### Группа D4.A — Англицизмы и термины
**D4.1** `«Gap»` в сегменте метрик доски — переименовать на русский?
— Гипотезы: (а) «Дефицит/профицит» или «Баланс»; (б) «Разрыв»; (в) оставить «Gap» (профжаргон планирования).
— реф: impeccable terminology consistency; CLAUDE — русификация.

**D4.2** «Утилизация» (доля клиентских часов) — понятный термин для пользователя, или заменить («Загрузка по клиентам» / «Полезная загрузка»)?
— Гипотезы: (а) оставить «Утилизация» + hint; (б) «Полезная загрузка»; (в) «Доля клиентских часов».

**D4.3** Метрики доски «Свободно ч / Загрузка % / План ч / Gap» — терминология консистентна? Свести к одному словарю?
— Гипотезы: (а) выровнять формулировки (везде «… ч»/«… %»); (б) оставить; (в) добавить hint к каждой.

**D4.4** Если в UI появятся FTE/headcount (сейчас только в коде) — какие русские ярлыки? «Ставка/доля ставки», «Численность»?
— Гипотезы: (а) «Доля ставки (FTE)» / «Численность»; (б) скрывать термин, показывать число; (в) оставить англ. для HR-аудитории.

### Группа D4.B — Заполнение (три «заполнить»)
**D4.5** `«Заполнить 8×5»` — нотация «8×5» понятна (8 ч × 5 дней)? Переименовать?
— Гипотезы: (а) «Заполнить норму на всю неделю»; (б) «Заполнить будни (норма)»; (в) оставить «8×5».
— реф: TIMESHEET_CONTROLS F5 — предложено «Заполнить неделю для всех».

**D4.6** Три действия со словом «заполнить» (тулбар-все-строки / меню-одна-строка / чип-ячейки) — дать ТРИ разных однозначных имени?
— Гипотезы: (а) «Заполнить неделю для всех» / «Заполнить будни этой строки» / убрать чип; (б) свести к двум; (в) оставить.
— реф: impeccable «pick one term»; уже решено в TIMESHEET_CONTROLS F5.

**D4.7** «Будни» = строго пн-пт или «рабочие дни по производственному календарю» (с учётом переносов/рабочих суббот)? Текст должен это отражать?
— Гипотезы: (а) «рабочие дни (по календарю)»; (б) «будни» подразумевает календарь; (в) оставить «будни».

**D4.8** Жёстко указывать значение в ярлыке («8 ч») или «норма дня» (т.к. короткий день = 7/6 ч)? Сейчас и «8 ч», и «норма» вперемешку.
— Гипотезы: (а) везде «норма дня»; (б) показывать конкретное число; (в) «норма (8 ч)».

### Группа D4.C — Копирование недели
**D4.9** «Только строки» (hint «часы заново») — «заново» = «вводить вручную»? Переформулировать?
— Гипотезы: (а) «Скопировать строки без часов»; (б) «Только проекты и виды работ»; (в) оставить.
— реф: impeccable error/label clarity.

**D4.10** «Строки и часы» vs «Только строки» — пара названий читается как выбор? Сделать радиогруппой с явными описаниями?
— Гипотезы: (а) радио + описание под каждым; (б) две кнопки как есть; (в) одна кнопка + чекбокс «с часами».

### Группа D4.D — Подтверждения и деструктив
**D4.11** «Удалить все несогласованные записи недели?» — указывать число затрагиваемых записей («Удалить 12 записей…»)?
— Гипотезы: (а) добавить счётчик; (б) оставить; (в) счётчик + «Это нельзя отменить».
— реф: impeccable «Delete 5 items, name the destruction, can't be undone».

**D4.12** Деструктивные кнопки («Удалить строку», «Очистить») — текст подтверждения и кнопка называют действие (не «OK/Да»)?
— Гипотезы: (а) кнопка = «Удалить строку» (не «Да»); (б) «Да/Отмена»; (в) «Удалить» / «Отмена, оставить».
— реф: impeccable «Never OK/Yes/No; Cancel→Keep editing».

**D4.13** Различать «Удалить» (насовсем) и «Очистить» (обнулить часы, строка остаётся)? Тексты сейчас близки.
— Гипотезы: (а) «Удалить строку» vs «Очистить часы строки»; (б) оставить; (в) добавить hint к каждому.

**D4.14** Отклонение записей менеджером — placeholder «Что исправить сотруднику…» обязателен (нельзя отклонить без причины)? Текст подсказки достаточен?
— Гипотезы: (а) комментарий обязателен; (б) опционален; (в) обязателен + примеры причин.

### Группа D4.E — Статусы, валидация, тосты
**D4.15** Статусы записей — единый словарь: «Черновик / На согласовании / Согласовано / Отклонено»? Сейчас встречаются «несогласованные», «submitted», «draft».
— Гипотезы: (а) фикс-словарь из 4 статусов везде; (б) оставить; (в) + цвет/иконка к каждому.
— реф: impeccable terminology glossary.

**D4.16** Тосты валидации (переработка, недозаполнение) — формула «что/почему/как исправить» соблюдена? Пример: «Неделя недозаполнена: N из M ч».
— Гипотезы: (а) добавить «как» («Заполните оставшиеся дни»); (б) текущее достаточно; (в) кнопка-действие в тосте.
— реф: impeccable error formula.

**D4.17** «Отправить на согласование (N)» / «Согласовать (N)» — счётчик в скобках понятен (сколько записей)? Нужна ли подпись «записей»?
— Гипотезы: (а) «Отправить N записей»; (б) оставить «(N)»; (в) tooltip с деталями.

**D4.18** Сообщения сохранения: «Сохранено / Сохраняю…» — единый текст на всех экранах с автосейвом?
— Гипотезы: (а) единый словарь save-статусов; (б) по экранам; (в) + время «Сохранено в 14:32».

### Группа D4.F — Подсказки, hint, термины-объяснения
**D4.19** Hint у «Gap» очень подробный («Gap = спрос − ёмкость (▲ дефицит / ▼ профицит). Цвет по нормам ±5/15%»). Это для эксперта — нужна ли упрощённая версия для новичка?
— Гипотезы: (а) короткий hint + «Подробнее»; (б) оставить (аудитория — планировщики); (в) вынести в справку.

**D4.20** Нужен ли единый glossary-поповер (значок «?») с расшифровкой терминов (утилизация, gap, норма, FTE) на каждом экране?
— Гипотезы: (а) глобальная «?»-справка терминов; (б) inline-hint по месту (как сейчас); (в) и то и то.
— реф: impeccable teach/onboard.

**D4.21** «Ёмкость» / «Спрос» / «Загрузка» в доске — термины интуитивны для продаж/руководителя, или нужны пояснения при первом входе (onboarding-подсказка)?
— Гипотезы: (а) onboarding-тур первого входа; (б) hint по месту; (в) ничего.

**D4.22** em-dash «—» в русских hint — оставляем (для русского норма), но следим, чтобы не было `--`. Подтвердить правило?
— Гипотезы: (а) тире «—» норма для русского UI; (б) заменить на запятую/двоеточие; (в) оставить, запрет только на `--`.
— реф: аудит P3 — impeccable-правило про em-dash написано под англ., для русского неприменимо.
