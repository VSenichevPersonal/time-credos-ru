**Связано:** REQ-0004 Часть C (контракт + инвариант «оценка≠резерв»), REQ-0009 (пресейл→план), DP-0001 (capacity-доска).

## 0. Что есть сейчас (база, Dev 2 + Dev 1)

Объект `credosTimeBooking` (`apps/time/src/objects/credos-time-booking.object.ts`), 7 полей:

| Поле | Тип | Назначение |
|---|---|---|
| `label` | TEXT (labelIdentifier) | заголовок («Иванов → Альфа, июль») |
| `bookingType` | SELECT SOFT/HARD (default SOFT) | soft=пресейл (не потребляет ёмкость) / hard=подтв. (потребляет) |
| `hours` | NUMBER float | ёмкость на период, равномерно по раб. дням |
| `startDate`/`endDate` | DATE_TIME | период |
| `note` | TEXT | заметка/условие пресейла |
| `employee` | RELATION → Employee (M:1 CASCADE) | обратка `Employee.bookings` |
| `project` | RELATION → Project (M:1 CASCADE) | обратка `Project.bookings` |

**UX/интеграция уже есть:**
- index-view «Брони ресурсов» + nav (папка Трудозатраты) — CRUD через нативную таблицу/карточку Twenty.
- Capacity-доска: HARD → слой Demand (потребляет ёмкость, суммируется в load), SOFT → отдельный пунктирный слой (тумблер `tentativeBookingEnabled`). См. `calc-load.ts` (`sumBookingHours`, `buildBookingCtx`), `booking-marker.tsx` (▪HARD / пунктир SOFT под ячейкой), `cap-tokens.ts` (`bookingParts`).
- Овербукинг НЕ блокируется — показывается как конфликт (соответствует Timetta).

**Вывод:** ядро (резерв employee×project×период×ёмкость×тип) и инвариант «бронь ≠ план ≠ факт» заложены ПРАВИЛЬНО по Timetta. Доработка — поля 2-й волны + UX создания/конвертации.

## 1. ПОЛЯ — gap vs Timetta

Timetta-модель брони (по `booking-methods.md`, `booking.md`, `resource-request.md`): период · проект · ресурс · **тип (soft/hard)** · **метод** · ёмкость · **единица (часы/%/FTE/дни)** · **роль/универсальный ресурс** · ресурсный пул. Запрос ресурса несёт статус-воркфлоу + предпочтительных кандидатов + вероятность запуска.

| # | Поле-кандидат | Есть? | Нужно Кредо-С? | Приоритет | Комментарий |
|---|---|---|---|---|---|
| 1 | **`probability` (вероятность пресейла %)** | нет | **ДА, но на ПРОЕКТЕ, не на брони** | 🟡 P2 | Timetta привязывает вероятность к opportunity/проекту-пресейлу, не к каждой брони. Дублировать на брони — переусложнение. Рекомендация: probability как поле проекта-кандидата (REQ-0009), бронь его наследует. На SOFT-слое доски умножать на probability (взвешенный Demand) — опционально, follow-up. |
| 2 | **`status` (статус брони: предв./подтв./отменена)** | **частично** (есть `bookingType` SOFT/HARD) | спорно | 🟢 P3 / не делать | В Timetta базовом режиме статус = тип (Soft/Hard); отдельного «отменена» нет — отменённую бронь удаляют. SOFT→HARD = смена типа = «подтверждение». **Рекомендация: НЕ вводить отдельный status.** «Отмена» = удаление записи (CASCADE). Не плодить два параллельных enum'а. |
| 3 | **`role` / универсальный ресурс (бронь до назначения человека)** | нет (`employee` обязателен по смыслу) | **ДА** — ключевой для пресейла | 🟡 P2 | Timetta: на старте бронируют РОЛЬ («Программист»), позже Запросом на подбор заменяют на человека. У нас `employee` nullable по схеме — можно бронировать «дырку». Минимальный вариант: SELECT/TEXT `role` (роль-заглушка) + employee nullable; когда employee=null и role задан — это «универсальный резерв». Полноценный справочник ролей/грейдов/пулов — НЕ делать (биллинг-завязка, `[[no-billable-concept]]`). |
| 4 | **привязка к этапу/stage проекта** | нет | НЕТ | ⚪ skip | Timetta: «бронь на уровне проекта без детализации до работ» (`booking.md` §Ключевые принципы). Бронь намеренно грубее ресурс-плана. Привязка к stage противоречит концепции. Не делать. |
| 5 | **`createdBy` / `bookedAt` (кто забронировал/когда)** | нет (есть нативные `createdBy`/`createdAt` Twenty) | **ДА (через нативное)** | 🟢 P1-дёшево | Twenty даёт системные `createdBy`/`createdAt`/`updatedAt` из коробки. Рекомендация: НЕ заводить свои поля, а **вывести нативные в index-view и карточку** брони. Для централизованного подхода (ресурс-менеджер подтверждает) это база аудита. |
| 6 | **связь с проектом-кандидатом (пресейл)** | частично (`project` M:1) | ДА (на уровне проекта) | 🟡 P2 | Не отдельная связь у брони, а **тип/категория проекта** = Presale/Pilot (REQ-0009) + probability (#1). Бронь к такому проекту автоматически «пресейл-бронь». Не дублировать на брони. |
| 7 | **единица: `fte%` вместо/вместе с hours** | нет (только `hours`) | вместе, опционально | 🟡 P2 | Timetta: единицы часы/%/FTE/дни — пользователь переключает на доске, хранение в часах. **Рекомендация: хранить `hours` (SSOT), FTE%/дни — производные на UI** (`fte = hours / scheduleHours`). НЕ заводить отдельное поле fte — рассинхрон. Тумблер единиц на доске (display-only) = достаточно. |
| 8 | **`bookingMethod` (метод: ручной/фронтальный/% от расписания/...)** | нет | НЕТ для MVP | ⚪ skip / P3 | 5 методов Timetta = автораспределение часов по слотам в детальном режиме. У нас базовый режим (равномерно по раб. дням = «равномерное распределение»). Один метод покрывает 80%. Не вводить enum методов. |

**Итог по полям (рекомендация волнами):**
- **Сразу/дёшево (P1):** вывести нативные `createdBy`/`createdAt` в view+карточку (аудит «кто/когда»). Схему не трогаем.
- **2-я волна (P2):** `role` (роль-заглушка, TEXT/SELECT) + сделать `employee` фактически nullable в UX → бронь до назначения человека (универсальный резерв). `probability` — на ПРОЕКТЕ (REQ-0009), не на брони.
- **НЕ делать:** отдельный `status`, привязка к stage, поле `fte`, enum `bookingMethod`, bill-rate.

## 2. UX/UI — создание / конвертация / овербукинг

### 2.0. Жёсткое ограничение среды (учесть везде)
Front-component = **Web Worker / Remote DOM, НЕТ host-DOM** (`docs/design/UI_PLAYBOOK.md §0`). Крашат: `getBoundingClientRect`, `window.*`, `document.*`, нативный HTML5 drag, pointer-замеры. Значит:
- ❌ **Drag-and-drop плашки** по доске (как в Timetta) — **невозможен надёжно**. Не планировать.
- ❌ Модалки-оверлеи (product ban + нет portal). Создание/правка — inline или нативная боковая карточка Twenty.
- ✅ Доступно: React-хуки, `onClick`/`onKeyDown` на контейнере, `fetch` через RestApiClient, абсолютное позиц. по %.

### 2.1. Как СОЗДАВАТЬ бронь

| Вариант | Реализуемо в песочнице? | Оценка |
|---|---|---|
| Нативная таблица/карточка Twenty (есть сейчас) | ✅ из коробки | базовый путь, оставить |
| **Кнопка «+ Бронь» на строке сотрудника/проекта доски → inline-форма** (период, проект/сотрудник, тип, часы) | ✅ (форма = React-инпуты, POST `/rest/credosTimeBookings`) | **рекомендую как основной UX**. Контекст подставляется (employee/project из строки). Не drag, а «клик по ячейке периода → форма с пред-заполненным периодом». |
| Drag по таймлайну (растянуть период) | ❌ нет host-DOM pointer | НЕ делать. Замена: выбор start/end в форме + клик по колонкам периода (как «залить» в REQ-0015 заменил drag кнопкой). |
| Из карточки проекта (вкладка «Команда»/«Брони») | ✅ | Timetta бронирует «в рамках команды проекта». У нас уже есть `project-team` + view `credos-time-project-card-bookings`. Добавить кнопку «забронировать участника» там. |

**Паттерн «клик-ячейка вместо drag»:** на capacity-доске клик по ячейке периода в строке сотрудника → inline-форма с пред-заполненными employee + период колонки; пользователь правит часы/тип/проект. Это song-safe эквивалент drag-создания Timetta.

### 2.2. Как КОНВЕРТИРОВАТЬ SOFT → HARD (пресейл выигран)

- В Timetta базовом режиме конвертация = смена типа Soft→Hard в карточке брони (ресурс-менеджер подтверждает).
- **Рекомендация:** кнопка-действие **«Подтвердить бронь»** (SOFT→HARD) на карточке/строке брони — один PATCH `bookingType`. Дёшево, песочница-safe.
- Полный цикл REQ-0009 (заявка-пресейл → workflow-согласование → конвертация) = отдельный трек на нативных Twenty Workflows + Tasks; здесь только кнопка-переключатель типа как ручной MVP.
- При конвертации SOFT→HARD бронь начинает потреблять ёмкость → может вскрыть овербукинг (см. 2.3) — показать предупреждение, НЕ блокировать.

### 2.3. Овербукинг — видеть и разрешать
- Timetta (`booking.md`, `resource-gap.md`): овербукинг НЕ блокируется, конфликт виден, «не требует немедленного устранения».
- **Уже частично есть** (`calc-load.ts`: Σ HARD-бронь). Рекомендация-доработка:
  - подсветка ячейки когда `Σ(HARD-бронь + план) > ёмкости` — токен `over`/`overSoft` (уже в `cap-tokens.ts`).
  - бейдж/тултип «перебронирован на N ч» на строке сотрудника.
  - дублировать НЕ цветом (значок ⚠ + tabular-nums), как уже сделано для booking-marker (impeccable).
  - resource-gap метрика (Demand−Capacity) с цветовой шкалой Timetta (синий избыток / зелёный баланс / оранж-красный дефицит) — опц. P3, переиспользовать пороги из `resource-gap.md`.

### 2.4. Бронь vs план-аллокация — НЕ путать (визуально)
Инвариант REQ-0004: бронь (резерв) ≠ план-оценка ≠ факт. На доске уже разведено: план — заливка/число ячейки, бронь — подстрока-маркер (▪HARD / пунктир SOFT). **Сохранить разделение**, не сливать в одно число. Тумблер «слой»: план / hard-бронь / обе — режим-переключатель (как «Отделы/Люди»).

### 2.5. Календарь/таймлайн ресурса
- Timetta = горизонтальный календарь с плашками + масштаб день/неделя/месяц.
- У нас: capacity-доска (грид недели/месяцы) уже близка. Плашки-полосы по времени = Timeline-вид (DP-0006 R8: абсолютное позиц. по %, песочница-safe, БЕЗ drag).
- **Рекомендация:** оставить грид как основной; Timeline-полосы броней — опц. вид (тумблер), низкий приоритет.

## 3. Сценарии Кредо-С (кто пользуется)

| Сценарий | Роль | Как ложится на модель |
|---|---|---|
| **Пресейл-бронь до контракта** | РП / продажи | SOFT-бронь на проект-кандидат (Presale). Если человек не определён — `role`-заглушка (#3), employee=null. probability на проекте. Видна на SOFT-слое доски, не потребляет ёмкость. |
| **Резерв под проект (выигран)** | РП / ресурс-менеджер | Конвертация SOFT→HARD (2.2) ИЛИ создание HARD сразу. Потребляет ёмкость, входит в load. |
| **Перепланирование** | РП / руковод | Правка периода/часов брони (inline-форма, не drag). Овербукинг показывается, не блокирует (2.3). |
| **Балансировка загрузки отдела** | руковод отдела | Доска: видит Σ броней по людям, окна свободной ёмкости («свободен с», DP-0001), перебронированных. |

**Подходы Timetta (`booking-concept.md`):** децентрализованный (РП сами бронируют общую доску) vs централизованный (через Запросы ресурсов ресурс-менеджеру). **Рекомендация Кредо-С: децентрализованный** — проще, песочница-safe, не требует движка запросов/пулов. Запросы ресурсов + ресурсные пулы + универсальные справочники = НЕ делать (переусложнение, биллинг-завязка).

## 4. Roadmap (волнами)

**Волна 0 — есть в base ✅:** объект + 7 полей + index-view + nav + capacity-интеграция (HARD/SOFT слой, маркер, тумблер soft, конфликт-подсветка частично).

**Волна 1 — дёшево, без схемы (P1):**
- вывести нативные `createdBy`/`createdAt` в view брони + карточку (аудит).
- кнопка-действие «Подтвердить бронь» SOFT→HARD (1 PATCH).
- доострить овербукинг-подсветку + бейдж «перебронирован» на доске (логика уже частично в `calc-load.ts`).

**Волна 2 — UX создания + поля (P2, нужен Dev 2 для схемы):**
- inline-создание брони с доски (клик-ячейка → форма, БЕЗ drag) + кнопка из карточки проекта/команды.
- поле `role` (роль-заглушка) + UX «бронь без человека» (employee nullable) = универсальный резерв для пресейла.
- `probability` на ПРОЕКТЕ-кандидате (REQ-0009), взвешенный SOFT-Demand на доске (опц.).
- тумблер единиц на доске (часы/FTE%/дни — display-only, хранение в часах).

**Волна 3 — отложить / опционально (P3):**
- полный пресейл-цикл REQ-0009 (Workflow+Tasks согласование заявки).
- Timeline-вид броней (полосы по %, DP-0006 R8).
- resource-gap метрика с цветовой шкалой Timetta.

**НЕ делать (явно):** drag-and-drop плашек (песочница), отдельный `status`/`bookingMethod`/поле `fte`, привязка к stage, ресурсные пулы + Запросы ресурсов + справочники ролей/грейдов, любой bill-rate.
