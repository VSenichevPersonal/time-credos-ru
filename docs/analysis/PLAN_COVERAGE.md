## 1. БАГ: «0 в строке = ошибка» (корень найден)

**Код:** `project-plan-panel.tsx:116` `effort ? computePreview(...) : null`. `effort = parseEffort(hours)`. parseEffort на «0» возвращает **0** (валидно: `n ≥ 0`). НО `effort ?` — truthy-проверка → **0 falsy** → preview=null → панель ведёт себя как «объём не задан» → подсказка/блок «укажите объём». Пользователь видит ошибку при легитимном 0.

**Корень = W5C.27** (0 двусмыслен: валидное число vs «нет плана» vs удаление слота). Решение:
- **(а)** 0 = «нет плана/очистить» — трактовать явно, БЕЗ ошибки: пустое превью + нейтральная подсказка «план не задан» (не error). Заменить `effort ?` на `effort != null && effort > 0`.
- **(б)** запретить 0 с понятным текстом «объём должен быть > 0» (если 0 не имеет смысла как план).
Рекомендую **(а)** — 0/пусто = очистка плана (консистентно со слот-контрактом «0/пусто → удаление»), без error. **Зона Dev1 (panel). P2-баг.**

## 2. Стыковка с плановой трудоёмкостью — ЧТО ЕСТЬ / ЧЕГО НЕТ

**Три РАЗНЫХ числа (не путать, ось «внутри плана»):**
| Число | Поле/расчёт | Смысл | Статус |
|---|---|---|---|
| **plannedEffort** | `project.plannedEffort` | плановая трудоёмкость = бюджет часов проекта | ЕСТЬ (задаётся) |
| **Распланировано** (allocated) | Σ слотов (MANUAL) / Σ EVEN-раскид | сколько РАСПРЕДЕЛЕНО по периодам/людям | **частично** (reconcileSlots только в MANUAL-панели) |
| **factHours** | `project.factHours` | ФАКТ списано | ЕСТЬ |
| **budgetRemaining** | `plannedEffort − factHours` | освоение (план vs ФАКТ) | ЕСТЬ |
| **Coverage / остаток** | `plannedEffort − распланировано` | ВСЁ ли распланировано | **НЕТ** |

**Ключевой пробел:** `budgetRemaining` = план − **ФАКТ** (освоение). Это НЕ «всё ли распланировано». «Распланировано» = план − **allocated** (Σ слотов). Их путать нельзя:
- план 500, распланировано 500, факт 120 → план полностью РАСПЛАНИРОВАН, но освоен на 120.
- план 500, распланировано 300, факт 0 → 200 ч НЕ распланировано (дыра спроса), хотя budgetRemaining=500.

**Что есть:** `reconcileSlots` (plan-preview.ts:362) считает Σ слотов vs plannedEffort с допуском 1ч (`{sum, target, ok}`) — НО только в MANUAL-панели одного проекта, при ручном вводе. **НЕ показано:** на строке проекта (доска), для EVEN-режима, агрегатом.

## 3. Что добавить (стыковка + «сколько распланировано / всё ли»)

### 3.1 Индикатор coverage на проекте (строка доски + панель)
Показывать триаду для проекта: **«План M ч · Распланировано N ч · Остаток (M−N)»** + флаг «всё ли»:
- остаток = 0 (±допуск) → ✓ «распланировано полностью».
- остаток > 0 → «N ч не распределено» (жёлтый) — дыра спроса (W6C.29 unplanned-родственно).
- остаток < 0 → «переаллокация +N ч» (терракот) — слотов больше бюджета.

**Распланировано** = для MANUAL: Σ слотов проекта; для EVEN: Σ раскида = plannedEffort (по определению равномерный покрывает 100% → остаток 0). Т.е. EVEN всегда «распланирован полностью», MANUAL может иметь дыру/перебор. Это и есть ответ «всё ли».

### 3.2 Связь с ФАКТом (отдельная ось — не путать)
Рядом, но РАЗДЕЛЬНО: «Освоено (факт) 120 из 500» (budgetRemaining, есть). Три бара: **План → Распланировано → Факт** (как Float/Forecast budget burn). Память проекта: бронь ≠ план ≠ факт — здесь внутри плана: бюджет ≠ распланировано ≠ факт.

### 3.3 Пресейл (company-booking) — ПРОБЕЛ
`credos-time-booking` (пресейл = company без project) имеет ТОЛЬКО `hours` (на период), **НЕТ плановой трудоёмкости** (нет plannedEffort/estimate). → для пресейла «сколько распланировано / всё ли» посчитать НЕ от чего (нет target).
**Варианты (вопрос арху/заказчику):**
- (а) добавить `expectedEffort` (ожидаемая трудоёмкость) на booking — target для пресейл-coverage.
- (б) связать пресейл с Opportunity (CRM-сделка, W4A.26), брать estimate оттуда.
- (в) пресейл без coverage (только Σ часов брони, без «всё ли») — MVP.
Рекомендую (а) минимум (`expectedEffort` на booking) — даёт coverage и для пресейла без CRM-связи; (б) — когда появится Opportunity-связь.

## 4. Разведка (как у конкурентов)

- **Float/Forecast/Runn:** project **budget (hours)** + allocated + remaining + over — budget burn-бар. Allocated отдельно от actual.
- **Timetta:** `plannedEffort`/EstimatedHours (оценка) vs allocated (ресурс-план) vs ActualHours. Три слоя. Resource-plan показывает Σ vs бюджет.
- **Resource Guru:** project budget status. **Все** разделяют бюджет / распланировано / факт — наш пробел именно в «распланировано» (allocated) как видимом числе.
