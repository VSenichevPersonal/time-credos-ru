## Группа 1. Частично-согласованная неделя и редактируемость

**W5A.1 — Правка остальных строк недели, когда часть APPROVED.**
Неделя: строки проекта X = APPROVED, строки проекта Y = DRAFT. Lock сейчас построчный
(`cannot_modify_approved` бьёт только по APPROVED-записи). Подтверждаем, что:
- (а) DRAFT/REJECTED-строки той же недели **редактируемы** свободно (lock именно построчный, не на период) — **гипотеза по коду**;
- (б) при наличии хотя бы одной APPROVED строки вся неделя становится read-only (период-lock);
- (в) редактируемы, но с предупреждением «в неделе есть согласованные записи».
Реф: Timetta — lock на уровне строки таймшита, не периода.

**W5A.2 — Добавление НОВОЙ строки в неделю с APPROVED-строками.**
Lock ловит правку существующих APPROVED; создание новой записи в тот же период кодом не блокируется. Это:
- (а) допустимо — новая запись = DRAFT, согласуется отдельно (**гипотеза по коду**);
- (б) запретить добавление в период, где уже есть APPROVED;
- (в) разрешить, но новая запись сразу падает в очередь руководителя как «доп. после согласования».
Реф: Timetta — дозаполнение после approve = новый submit.

**W5A.3 — Статус «частично согласовано» в бейдже периода.**
Сейчас `calcPeriodStatus`: если есть SUBMITTED → весь период «SUBMITTED»; «APPROVED» только когда ВСЕ approved.
Кейс «3 approved + 1 draft» показывается как DRAFT (нет submitted/rejected). Нужно:
- (а) оставить как есть (агрегат по худшему открытому статусу) — **гипотеза по коду**;
- (б) ввести визуальный микс «3/4 согласовано» в бейдже (A4.21 уже про счётчики — уточняем, считать ли по DRAFT-остатку);
- (в) отдельный тон «частично» (не DRAFT, не APPROVED).
Реф: Timetta — прогресс-индикатор частичного согласования.

**W5A.4 — Submit при наличии уже APPROVED-строк в периоде.**
`draftIds` исключает APPROVED, submit отправит только DRAFT. Подтверждаем:
- (а) повторный submit «досылает» только новые DRAFT, не трогая APPROVED (**гипотеза по коду**);
- (б) запретить повторный submit пока в периоде есть APPROVED (период «закрыт»);
- (в) предупредить «часть периода уже согласована, отправляется N черновиков».
Реф: Kimai — incremental submit.

## Группа 2. Approve / Reject — гранулярность и батчи

**W5A.5 — Reject одной строки vs всего submit сотрудника.**
A4.13/G2.17 = «нет частичного reject, весь submit сотрудника — единое решение». Но `runResolve`
технически принимает произвольный список `ids` → можно отклонить подмножество. Уточняем границу:
- (а) UI всегда шлёт ВСЕ `submittedIds` сотрудника (frontend-инвариант, backend не ограничивает) — **гипотеза по коду** (`use-approval.ts` так и делает);
- (б) backend тоже должен запретить частичный reject (валидировать, что ids = полный submit);
- (в) разрешить частичный reject позже (отступление от A4.13).
Реф: Timetta — reject таймшита целиком.

**W5A.6 — Повторная отправка после REJECTED.**
`runSubmit` берёт `status[eq]:DRAFT`. REJECTED-запись в submit **не попадёт** — её сперва надо
вернуть в DRAFT. Где происходит REJECTED→DRAFT?
- (а) сотрудник правит REJECTED-ячейку → `time-entry-api` переводит её в DRAFT, потом submit (**нужно подтвердить, есть ли такой переход в коде**);
- (б) submit должен сам поднимать и REJECTED (`status[in]:DRAFT,REJECTED`) — текущая логика это НЕ делает (дыра);
- (в) отдельная кнопка «Исправить и переотправить» переводит REJECTED→DRAFT явно.
Реф: Timetta — после reject таймшит возвращается в редактирование.

**W5A.7 — Конкурентный approve двумя руководителями.**
`runResolve` читает статус по одному id, потом PATCH. Между read и write второй руководитель
может уже approve. Нет compare-and-set. Поведение:
- (а) допустимо — оба пишут APPROVED, последний `approvedBy` побеждает (idempotent по результату) — **гипотеза по коду**;
- (б) нужен optimistic-lock (если статус сменился между read/write — skip), вернуть «уже обработано N»;
- (в) запись в `approvedBy` первого победителя неизменна (не перезаписывать).
Реф: Timetta — single-resolver, повторное approve no-op.

**W5A.8 — Approve смешанного батча (часть уже APPROVED/REJECTED/DRAFT).**
`runResolve` пропускает всё, что `!== SUBMITTED` (`continue`). Возврат `{updated, skippedOwn}`,
но «пропущено по статусу» НЕ считается. Уточняем:
- (а) тихо пропускать не-SUBMITTED, в toast только `updated`/`skippedOwn` (**гипотеза по коду** — счётчик «skippedStatus» отсутствует);
- (б) добавить счётчик «пропущено N (уже решены)» в ответ и toast (A4.15 говорит про SoD/ошибки — уточняем, считать ли статусные skip);
- (в) считать смешанный батч ошибкой ввода.
Реф: Kimai — bulk-approve рапортует skipped.

**W5A.9 — Approve собственных записей руководителя в батче (SoD).**
SoD: `skippedOwn++` при `entry.employeeId === actor.employeeId`. Если руководитель approve очередь,
где есть его строки — они молча пропущены. Подтверждаем:
- (а) toast «согласовано N, пропущено M (ваши записи — согласует вышестоящий)» (A4.16) — **гипотеза по коду** (`skippedOwn` возвращается, текст уточнить);
- (б) UI вообще не показывает руководителю его строки в его же очереди (фильтр на фронте);
- (в) hard-error «в выборке есть ваши записи».
Реф: Timetta — SoD-фильтрация очереди.

## Группа 3. rejectComment, аудит-поля, повторные циклы

**W5A.10 — Судьба rejectComment при переотправке/approve.**
`setStatus` пишет `rejectComment` только для REJECTED, иначе `null`. Значит при DRAFT (recall/правка)
и при последующем APPROVED комментарий **затирается**. Уточняем намеренность:
- (а) да, при «оживлении» записи прежняя причина reject чистится (запись начинает цикл заново) — **гипотеза по коду** (комментарий в `setStatus` это и декларирует);
- (б) хранить историю reject-причин (не затирать, а вести лог) — для аудита повторных отклонений;
- (в) причина видна до первой правки, потом гаснет.
Реф: Timetta — комментарий ревизии хранится в истории таймшита.

**W5A.11 — approvedBy/approvedAt при revoke (APPROVED→SUBMITTED).**
`setStatus` для SUBMITTED обнуляет `approvedBy/approvedAt`. После revoke запись «как будто
не согласовывалась». Уточняем для аудита:
- (а) полное обнуление, кто/когда отозвал нигде не фиксируется — **гипотеза по коду** (это потеря аудита);
- (б) сохранять `revokedBy/revokedAt` отдельными полями (кто нажал Reopen);
- (в) писать запись в журнал событий, поля не трогать.
Реф: Timetta — Reopen фиксируется в истории с автором.

**W5A.12 — approvedBy/approvedAt при reject.**
`setStatus` пишет `approvedBy/approvedAt` и для APPROVED, и для REJECTED (одни поля). То есть
«approvedBy» на REJECTED-записи = тот, кто ОТКЛОНИЛ. Уточняем семантику:
- (а) поле трактуется как «resolver» (решивший), независимо approve/reject — **гипотеза по коду**;
- (б) переименовать/разделить `resolvedBy/resolvedAt` от `approvedBy` (approve-only);
- (в) оставить, но в UI подписывать «отклонил: …» для REJECTED.
Реф: Timetta — отдельные поля approver / последнего действия.

## Группа 4. recall сотрудника — границы и каскады

**W5A.13 — recall после того, как руководитель «открыл», но ещё не решил.**
A4.4 = recall разрешён «пока не открыт руководителем». Код разрешает recall ЛЮБОЙ SUBMITTED
(нет признака «открыто руководителем»). Уточняем:
- (а) recall разрешён всё время в SUBMITTED, признака «открыто» нет и не нужен — **гипотеза по коду**;
- (б) ввести флаг «взято в работу» (руководитель открыл) → recall блокируется;
- (в) recall всегда можно, но руководитель видит «сотрудник отозвал во время проверки».
Реф: Timetta — recall до решения; «в работе» не блокирует.

**W5A.14 — recall части строк одного submit.**
`runRecall` принимает произвольные `ids`. Сотрудник может отозвать 2 из 5 SUBMITTED-строк.
Это согласуется с «submit период целиком» (A4.1)?
- (а) recall построчный допустим (UI шлёт все, но backend не запрещает подмножество) — **гипотеза по коду**;
- (б) recall должен возвращать весь submit сотрудника за период (симметрия с submit);
- (в) частичный recall разрешён осознанно (правка одной строки без отзыва всего).
Реф: Timetta — recall таймшита целиком.

**W5A.15 — factHours/rollup при recall и revoke.**
Recall (SUBMITTED→DRAFT) и revoke (APPROVED→SUBMITTED) идут через `restPatch` поля `status`
напрямую в `approval.logic`, **минуя** `time-entry-api`. Триггеры rollup на `updated` сработают
(меняется запись), но `hours` не меняется. Подтверждаем:
- (а) recall/revoke factHours НЕ меняют (часы те же, меняется только статус) → rollup стабилен — **гипотеза по коду**;
- (б) factHours должен учитывать только APPROVED (тогда revoke уменьшает факт) — уточнить определение «факт»;
- (в) факт = все записи независимо от статуса (текущее поведение `computeFactHours`).
Реф: Timetta — факт по фактическим часам, не по статусу согласования.

## Группа 5. «Акт» (A4.27) и каскады отзыва

**W5A.16 — recall/revoke после создания Акта.**
A4.27 = кнопка «Создать акт». В коде реализации Акта не нашёл (только commit-месседж 8197a51).
Если Акт создан по APPROVED-записям, а потом revoke (APPROVED→SUBMITTED) — что с Актом?
- (а) revoke запрещён, если запись уже в Акте (Акт «замораживает» строки);
- (б) revoke разрешён, Акт помечается «требует пересоздания/устарел»;
- (в) Акт неизменен (снимок на момент создания), revoke на него не влияет — **наиболее вероятная гипотеза**.
Реф: Timetta/Директум — акт = снимок, ревизия часов требует нового акта.

**W5A.17 — что входит в Акт: только APPROVED или весь период.**
A4.28 = «часы по клиенту/проекту за период». Уточняем срез:
- (а) только APPROVED-записи периода (Акт = согласованное) — **гипотеза по смыслу акта**;
- (б) все записи периода независимо от статуса;
- (в) APPROVED + предупреждение, если есть несогласованные «N часов не вошли (не согласованы)».
Реф: Директум — акт по согласованным трудозатратам.

**W5A.18 — повторное «Создать акт» по тому же периоду.**
Идемпотентность кнопки A4.27:
- (а) запретить, если Акт за период уже есть (один акт = один период/клиент);
- (б) создавать дельта-акт по часам, добавленным после первого;
- (в) пересоздавать (старый помечается недействительным).
Реф: Директум/1С — нумерация актов, без дублей за период.

**W5A.19 — статус согласования как предусловие Акта.**
Можно ли «Создать акт», если в периоде есть SUBMITTED/REJECTED (не всё согласовано)?
- (а) блокировать кнопку, пока период не APPROVED целиком;
- (б) разрешить, акт только по APPROVED-части (см. W5A.17в);
- (в) предупредить и разрешить.
Реф: Директум — акт после полного согласования.

## Группа 6. submit — точные правила выборки и норма

**W5A.20 — submit при пустом периоде / все часы 0.**
`runSubmit` берёт DRAFT-записи проектов с `approvalRequired`. Если записей нет → `updated:0`.
UI (`canSubmit = draftIds.length > 0`) скрывает кнопку. Уточняем нулевой кейс:
- (а) нет DRAFT → кнопки нет, submit невозможен (нечего отправлять) — **гипотеза по коду**;
- (б) разрешить submit «пустой недели» (явное «ничего не делал»);
- (в) отдельное действие «подтвердить нулевой период».
Реф: Timetta — нулевой таймшит submit'ится (подтверждение отсутствия работ).

**W5A.21 — submit только проектов с approvalRequired — что с остальными.**
`tracked` = записи проектов, где согласование включено. Записи проектов без approvalRequired
**никогда** не идут в SUBMITTED (остаются DRAFT навсегда). Подтверждаем:
- (а) такие записи «всегда финальны» в DRAFT, факт по ним учитывается, согласования нет — **гипотеза по коду**;
- (б) им нужен отдельный терминальный статус «не требует согласования» (визуально не «черновик»);
- (в) показывать их как APPROVED-эквивалент (зелёные) без процесса.
Реф: Kimai — записи вне approval-flow считаются автоматически принятыми.

**W5A.22 — граница периода submit (from/to) vs неделя/месяц.**
`runSubmit` принимает произвольные `from/to`. UI шлёт границы текущего вида (неделя?). Уточняем
единицу согласования:
- (а) period = ровно отображаемая неделя (Пн–Вс), submit недельный — **гипотеза по A4.1+UI**;
- (б) согласование месячное (как Timetta-таймшит = месяц по умолчанию);
- (в) единица настраивается (неделя/месяц) — связать с reportPeriod из отчётов.
Реф: Timetta — таймшит-период обычно месяц; уточнить нашу единицу.

## Группа 7. lock APPROVED — точные границы

**W5A.23 — что именно блокирует APPROVED-lock.**
`cannot_modify_approved` срабатывает на upsert/delete записи в APPROVED (3 точки в `time-entry-api`).
Уточняем периметр:
- (а) блокируются только hours/description/дата/проект самой APPROVED-записи; статусные переходы
  (revoke) идут отдельным путём в обход lock — **гипотеза по коду**;
- (б) lock должен также запрещать смену проекта/даты у APPROVED (это «перенос» согласованного часа);
- (в) перенос APPROVED-записи на другой проект/дату = revoke + пересоздание (явный flow).
Реф: Timetta — согласованная строка read-only до Reopen.

**W5A.24 — REVOKED как отдельный статус (A4.20 отложен).**
A4.20 = «+REVOKED при реализации отзыва». Реализация revoke переиспользует SUBMITTED
(REVOKED не введён). Подтверждаем намеренность:
- (а) REVOKED не нужен, revoke = чистый возврат в SUBMITTED (очередь руководителя) — **гипотеза по коду**;
- (б) ввести REVOKED для отличия «отозвано после approve» от «впервые на согласовании» (UX/аудит);
- (в) различать через `approvedAt is null` + флаг wasApproved.
Реф: Timetta — Reopen возвращает в Pending без отдельного статуса.

**W5A.25 — удаление проекта со связанными APPROVED-записями.**
Проект удаляют, у него есть APPROVED-записи (и, возможно, Акт). `project-fact-rollup-deleted`
пересчитывает факт. Что со статусами/lock:
- (а) удаление проекта = soft-delete, APPROVED-записи остаются для аудита/акта — **уточнить, soft или hard**;
- (б) запретить удаление проекта, пока есть APPROVED-записи/Акт;
- (в) hard-delete каскадит записи (потеря согласованной истории) — нежелательно.
Реф: Timetta — проект с трудозатратами архивируется, не удаляется.

## Группа 8. агрегат-статус и уведомления

**W5A.26 — приоритет REJECTED в агрегате при частичном отклонении.**
`calcPeriodStatus`: одна REJECTED-строка → весь период «REJECTED» (баннер A4.23), даже если
3 строки APPROVED. Подтверждаем:
- (а) да, любой reject «красит» весь период в REJECTED (привлечь внимание к исправлению) — **гипотеза по коду**;
- (б) показывать «частично отклонено: 1 из 4» вместо чистого REJECTED;
- (в) REJECTED только если ВСЕ отклонены, иначе «требует внимания».
Реф: Timetta — отклонённый таймшит целиком в статусе Rejected.

**W5A.27 — уведомление сотруднику при revoke (APPROVED→SUBMITTED).**
A4.29 = in-app + email при смене статуса. Revoke возвращает согласованное в очередь. Сотрудник:
- (а) получает уведомление «согласование отозвано, требуется повторная отправка/проверка»;
- (б) revoke = действие руководителя, сотрудник узнаёт по бейджу без активного уведомления;
- (в) уведомление + причина (по аналогии с reject-comment — нужно ли поле причины для revoke?).
Реф: Timetta — Reopen шлёт нотификацию владельцу.

**W5A.28 — уведомление при recall — кому.**
Сотрудник отозвал submit (SUBMITTED→DRAFT). Руководитель, у которого запись была в очереди:
- (а) очередь просто перестраивается, отдельного уведомления руководителю нет — **гипотеза по коду**;
- (б) уведомить руководителя «сотрудник отозвал отправку»;
- (в) только если руководитель уже «открыл» запись (см. W5A.13).
Реф: Timetta — recall убирает из очереди без явного алерта.

**W5A.29 — порядок approve в массовом батче со смешанными статусами и SoD.**
`runResolve` идёт по `ids` последовательно; ошибка одного PATCH бросает исключение → handler
вернёт `{ok:false}`, но часть записей уже обновлена (нет отката). Уточняем:
- (а) частичное применение допустимо, повтор операции идемпотентен (уже APPROVED пропустятся) — **гипотеза по коду**;
- (б) нужна транзакционность «всё или ничего»;
- (в) продолжать при ошибке отдельной записи (collect errors, не бросать).
Реф: Kimai — bulk-операция собирает per-item результаты.

**W5A.30 — submit при наличии REJECTED + DRAFT в одном периоде.**
Период: 1 REJECTED (исправлена → DRAFT?) + 2 DRAFT. `calcPeriodStatus` покажет REJECTED
(приоритет), а `submit` отправит только DRAFT (REJECTED не в `draftIds`). Конфликт UX:
бейдж «отклонено», но кнопка submit шлёт не все. Уточняем:
- (а) сотрудник обязан сперва превратить REJECTED→DRAFT (правкой), потом submit подхватит — **гипотеза по коду** (зависит от W5A.6);
- (б) submit должен включать REJECTED (переотправка исправленного) — нужна правка `runSubmit`;
- (в) отдельная кнопка «Переотправить отклонённое».
Реф: Timetta — исправленный отклонённый таймшит отправляется повторно тем же submit.
