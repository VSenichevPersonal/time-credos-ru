**Волна:** 5, задание W5-C (аналитик)

**Формат:** `**W5C.x** вопрос — гипотезы (а)(б)(в) — реф: код / Timetta(T) / Kimai(K)`

## 0. Что УЖЕ реализовано (НЕ переспрашиваем — углубляем края)

Перед вопросами фиксирую фактическую реализацию (из кода), чтобы не дублировать решённое:

- **Уникальный ключ записи** — `UNIQUE(employeeId, projectId, workTypeId, date)` (`indexes/credos-time-entry-unique.index.ts`). Defense-in-depth: БД-индекс + приложенческий upsert-гард в `/s/time-entry` (`findExistingEntryIdByKey`). `date` — тип **DATE_TIME** (различает время суток), но гард нормализует по календарному дню через `dayBounds()` — диапазон `[деньT00:00:00.000Z, деньT23:59:59.999Z]`.
- **NULL-щель индекса** — PG: `NULL != NULL`, поэтому строки с `projectId`/`workTypeId = NULL` индексом НЕ ловятся; закрывает upsert-гард (`projectId[is]:NULL`). Комментарий утверждает «в данных NULL нет (0/422)».
- **factHours/budgetRemaining** — хранимые derived-поля проекта. Пересчёт идемпотентный (полный Σ из источника), вызывается из `/s/time-entry` И из 3 database-event триггеров (`created`/`updated`/`deleted`). `updated`-триггер слушает только `updatedFields: ['hours', 'projectId']`.
- **employeeCode (ПДн)** — формат `Сотрудник·<deptCode|deptId[0:4]|'—'>·<4 hex с конца id>` (`reports-calc.ts:156`). При `revealEmployeeNames=false` КОД заменяет ФИО во всех срезах (`reports.logic.ts`, fallback `false`).
- **Валидация часов** — `validateEntry` (`constants/validation.ts`): `hours<0|NaN|>maxHoursPerDay` → ERROR (блок); `>overtimeWarnHours` → WARNING (не блок). Пороги — из `credosTimeSettings`, дефолт max=24.
- **APPROVED-guard** — нельзя удалять/менять согласованную запись (`cannot_modify_approved`, CISO-011).
- **Дедуп перед индексом** — `scripts/dedup-entries.mjs`: survivor = APPROVED важнее, иначе ранний по createdAt/id; часы = Σ группы (по умолчанию) либо `--keep-hours`.

Ниже — **открытые края** этих механизмов.

## 1. Уникальный ключ и upsert-слияние

**W5C.1. При попытке создать дубль `(employee,project,workType,date)` без id — upsert-гард молча ОБНОВЛЯЕТ найденную запись (перезаписывает hours/description). Это ожидаемое поведение или нужно предупредить «запись на этот день уже есть, перезаписать?»?**
(а) молчаливый upsert (как сейчас) — повторный ввод просто правит существующую;
(б) предупреждение в UI перед перезаписью (риск затереть чужой/прежний ввод);
(в) слияние/суммирование часов (как `dedup` без `--keep-hours`), а не замена.
— реф: `time-entry-api.logic.ts:271-296` (upsert находит targetId → PATCH, текущие hours затираются). T — табель-ячейка одна на день; повторный ввод правит.

**W5C.2. Уникальность считается по `date` как DATE_TIME, но гард ловит по диапазону ДНЯ. Если запись создана прямым REST/гридом Twenty с временем `2026-06-21T15:30` (не полночь), а недельная сетка пишет фиксированное время дня — дубль в один день ловится гардом, НО БД-индекс (полное значение date) их РАЗЛИЧИТ и пропустит оба. Это реальный риск двойного счёта?**
(а) все пути пишут фиксированное время дня → проблемы нет (подтвердить, какое именно — полночь UTC?);
(б) грид Twenty/CSV могут писать произвольное время → нужен нормализатор date на день при записи (триггер/хук);
(в) считать уникальность по date::date (приведение к дню) на уровне индекса — но SDK это не умеет.
— реф: индекс по полному `date`; гард по дню (`dayBounds`). Расхождение семантик — корень потенциального дубля.

**W5C.3. Конкурентная правка ОДНОЙ ячейки двумя пользователями (например, сотрудник и руководитель массовым действием) — кто выигрывает? Last-write-wins без блокировок?**
(а) LWW молча (как сейчас — PATCH без версии/ETag) — последний перетирает;
(б) optimistic lock (поле version/updatedAt, конфликт → ошибка «запись изменена, обновите»);
(в) маловероятно на практике (один сотрудник = свои записи) — принять LWW осознанно.
— реф: `time-entry-api.logic.ts` PATCH без проверки версии. T — табель персональный, конкуренции мало; согласование — отдельный поток статусов.

**W5C.4. Гард делает 2-3 последовательных GET перед записью (resolve employee, find-by-key, read-status). Между find-by-key и PATCH/POST есть гонка: два параллельных запроса оба не найдут запись → оба POST → БД-индекс отклонит второй (500). Этот 500 ловится в `ok:false` или пользователь видит сырую ошибку?**
(а) handler оборачивает в `ok:false` с текстом ошибки БД (`time-entry-api.logic.ts:367`) — пользователь видит непонятный текст констрейнта;
(б) нужно перехватить именно unique-violation и вернуть дружелюбное «запись уже создана, обновите страницу»;
(в) ретрай гарда (повторный find→PATCH) при unique-violation.
— реф: `handler` try/catch отдаёт `e.message` (текст PG). Краёвый, но реальный при двойном клике «Сохранить».

**W5C.5. NULL-щель: комментарий говорит «в данных NULL нет (0/422)», но `projectId`/`workTypeId` nullable, и UI позволяет ввод без проекта? Если запись без проекта создаётся через грид Twenty (мимо гарда) дважды — индекс пропустит дубль. Допустимы ли записи без проекта вообще?**
(а) проект обязателен в UI → NULL только legacy/импорт, риск минимален;
(б) workType опционален (записи без вида работ норма) → NULL-дубли по workType реальны;
(в) запретить NULL на уровне объекта (`isNullable:false`) для project, закрыть щель индекса.
— реф: объект — `project`/`workType` nullable; индекс не ловит их NULL-комбинации.

## 2. Даты, таймзоны, границы суток/года

**W5C.6. `date` хранится как DATE_TIME (с временем и таймзоной), а вся отчётная логика режет по `iso.slice(0,10)` = ПЕРВЫЕ 10 символов ISO-строки (UTC-день). Запись, созданная вечером по МСК (UTC+3), напр. `2026-06-21T22:00` локально = `2026-06-21T19:00Z` — ок. Но `2026-06-21T23:00 МСК` = `2026-06-21T20:00Z` ок, а полночь-час `2026-06-22T01:00 МСК` = `2026-06-21T22:00Z` → попадёт в ПРЕДЫДУЩИЙ день в отчёте. Это известный риск сдвига суток?**
(а) все даты пишутся как полночь UTC выбранного дня (без реального времени) → сдвига нет;
(б) пишется реальный момент создания → отчёты/месячные бакеты могут отнести часы не в тот день/месяц;
(в) нужна единая политика: дата записи = календарный день в таймзоне воркспейса, время = 00:00 этой TZ.
— реф: `reports-calc.ts:144 dayKey = iso.slice(0,10)`, `monthKey = iso.slice(0,7)`; `dayBounds` использует `...Z`. Везде неявный UTC.

**W5C.7. Граница ГОДА: запись на 31 декабря / 1 января при сдвиге UTC может «уехать» в соседний год → попасть/выпасть из годового отчёта и месячного тренда (`computeTimeseries`, инвариант «Σ месяцев == год»). Как заказчик ожидает учитывать пограничные дни?**
(а) по календарю воркспейса (один TZ для всех) — зафиксировать TZ;
(б) граничные дни редки, погрешность приемлема;
(в) дата = чистый день без времени (DATE, не DATE_TIME) — убрать TZ-неоднозначность совсем.
— реф: `computeTimeseries` бакетирует `monthKey(e.date)`; запись без date в бакеты не попадает (тихо теряется из тренда, но остаётся в Σ?).

**W5C.8. Расхождение инварианта тренда: `computeTimeseries` пропускает записи БЕЗ `date` (`if (!m) continue`), а `computeReports`/`computeOlap` считают их в общий факт. Значит «Σ по месяцам fact» может быть МЕНЬШЕ годового факта при наличии записей без даты. Это допустимо?**
(а) записей без date не бывает (поле обязательное, default `now`) → инвариант держится;
(б) импорт может дать записи без date → тренд занизит, нужен учёт «без даты» отдельной строкой;
(в) запретить запись без date жёстко.
— реф: `reports-calc.ts:849 if (!m) continue` vs `computeReports` суммирует все entries с hours≠0.

**W5C.9. Фильтр периода в API использует строковое сравнение ISO (`date[gte]:from, date[lte]:to`) с `from='...T00:00:00.000Z'`, `to='...T23:59:59.999Z'`. Если запись хранит время вне этого окна из-за TZ — выпадет из недельной выборки. Сетка строится по тем же UTC-границам, что и хранение?**
(а) да, симметрично (и пишем, и читаем UTC-день) — консистентно;
(б) запись пишется в локальной TZ, читается в UTC → пограничные часы недели теряются/дублируются;
(в) нужно фиксировать смещение окна на TZ воркспейса.
— реф: `time-entry-api.logic.ts:162 dayBounds`, `:327 from/to`.

**W5C.10. Високосный год / короткий предпраздничный день (`dayType=SHORT`): норма берётся из часов производственного календаря (`d.hours`). Если в календаре 29 февраля или сокращённый день заведён с правильными часами — расчёт корректен. Кто и как гарантирует полноту календаря на стыке лет (нет дыр)?**
(а) календарь сидируется на год вперёд вручную/импортом — есть процесс;
(б) дыра в календаре → норма занижается молча (нет валидации полноты);
(в) нужна проверка «календарь покрывает весь период отчёта», иначе предупреждение.
— реф: `reports-calc.ts:208 workdays = calendar.filter(WORKDAY_TYPES)`; отсутствующий день = просто 0 в норме, без сигнала.

## 3. factHours-триггеры и rollup

**W5C.11. При УДАЛЕНИИ последней записи проекта `factHours` должен стать 0. Триггер `deleted` пересчитывает по `before.projectId`. Но `recalcProjectFactHours` собирает ВСЕ записи проекта курсором — если их 0, `computeFactHours([])=0`, ок. Подтвердить: budgetRemaining при факте 0 = plannedEffort (не null), а если плана нет — null. Это ожидаемо в UI-колонке?**
(а) пустой проект: factHours=0, budgetRemaining=plannedEffort или null — корректно;
(б) нужно различать «0 часов» и «нет данных» в колонке (тире vs 0);
(в) пустой/закрытый проект скрывать из факт-колонок.
— реф: `project-fact-rollup.ts:51-62`. Краёвый: проект существует, записей нет.

**W5C.12. Триггер `updated` слушает только `['hours','projectId']`. Если запись меняет `employeeId` (переназначение на другого сотрудника) — проект не меняется, факт проекта тот же, НО факт по СОТРУДНИКУ в отчёте сместится. Отчёты считаются on-read, так что ок. Подтвердить: НЕТ хранимого rollup по сотруднику/отделу, только по проекту?**
(а) rollup только factHours проекта; сотрудник/отдел всегда on-read → смена employee не требует триггера;
(б) если появится хранимый показатель по сотруднику — нужен триггер на `employeeId`;
(в) смена employeeId на записи вообще не предусмотрена UI.
— реф: `project-fact-rollup-updated.logic.ts:20 updatedFields`. Только проектный rollup хранимый.

**W5C.13. Двойной пересчёт: `/s/time-entry` сам зовёт `recalcProjectFactHours` И database-event триггер `created`/`updated` сработает на ту же мутацию. Идемпотентность защищает от неверного значения, но это 2x нагрузка + потенциальная гонка (оба читают-пишут параллельно). Приемлемо или дедуп пути нужен?**
(а) идемпотентность достаточна, 2x пересчёт терпим (редкие записи);
(б) гонка: триггер и /s/time-entry пишут factHours параллельно из разных снимков → возможен устаревший результат до следующей мутации;
(в) убрать пересчёт из /s/time-entry, оставить только триггеры (single source).
— реф: `time-entry-api.logic.ts:306,317` + `project-fact-rollup-events.ts`. Оба идемпотентны, но не сериализованы.

**W5C.14. Курсорный сбор записей проекта ограничен `for (i<1000)` × 200 = до 200k записей. Если проект превысит — факт занизится молча. Реалистичный потолок числа записей на проект и нужна ли защита от тихого обрыва?**
(а) нереалистично (проект < 200k записей) — потолок чисто страховочный;
(б) долгоживущий проект (годы × команда) может приблизиться — нужен мониторинг/алерт при достижении лимита;
(в) поднять/убрать лимит.
— реф: `project-fact-rollup.ts:90 for (let i=0; i<1000; i++)`.

**W5C.15. Триггер-обёртка `wrapEvent` глотает ошибку пересчёта (лог + `ok:false`), чтобы не валить обработку события. Значит при сбое REST (таймаут) factHours останется устаревшим БЕЗ повторной попытки. Нужен ли механизм наверстывания (повторный backfill по расписанию)?**
(а) post-install backfill + ручной запуск достаточно — редкие сбои переживём;
(б) нужен периодический reconciliation-проход (сверка хранимого factHours с Σ записей);
(в) ретрай в самом триггере при сетевой ошибке.
— реф: `project-fact-rollup-events.ts:50 wrapEvent` — ошибка не пробрасывается, дрейф возможен до следующей мутации.

## 4. ПДн-код, редакция, reveal

**W5C.16. `employeeCode` включает `deptCode` сотрудника: `Сотрудник·<deptCode>·<hex>`. При СМЕНЕ отдела сотрудника КОД МЕНЯЕТСЯ (другой deptCode), хотя hex-суффикс из id стабилен. В исторических отчётах один человек получит РАЗНЫЕ коды за разные периоды. Это приемлемо или код должен быть неизменным?**
(а) приемлемо — суффикс id разводит людей, deptCode лишь для читаемости группировки;
(б) ломает узнаваемость в кросс-периодных отчётах (один человек = два ярлыка) → код должен зависеть ТОЛЬКО от id;
(в) deptCode брать на момент записи, а не текущий.
— реф: `reports-calc.ts:160-163`. deptCode резолвится из ТЕКУЩЕГО departmentId.

**W5C.17. Hex-суффикс = 4 последних hex-символа id. При большой команде в одном отделе возможна КОЛЛИЗИЯ (два разных сотрудника → один код в рамках одного deptCode). Вероятность ~1/65536 на пару в отделе. Допустимо или нужен длиннее суффикс/проверка уникальности?**
(а) 4 hex достаточно (отделы малы, коллизия маловероятна);
(б) при отделе >50-100 чел риск растёт (парадокс дней рождения) → 6 hex или проверка;
(в) использовать табельный номер сотрудника вместо hex, если он есть.
— реф: `reports-calc.ts:162 .slice(-4)`. Коллизия = два человека сливаются в отчёте при reveal=false.

**W5C.18. Смена `revealEmployeeNames` (админ-тоггл) — применяется ли МГНОВЕННО во всех вью или есть кэш? Отчёт открыт у руководителя, админ выключил reveal — старые данные на экране остаются ФИО до перезагрузки?**
(а) каждый запрос /s/reports читает настройку заново → новые запросы сразу редактируют, открытые экраны — до обновления (приемлемо);
(б) нужен push/инвалидация открытых вью при смене тоггла;
(в) тоггл редко меняется, мгновенность не критична.
— реф: `reports.logic.ts:173 settings[0]?.revealEmployeeNames` читается на каждый вызов; фронт-кэш отдельно.

**W5C.19. При `reveal=false` КОД заменяет ФИО в `byEmployee`, OLAP-оси/фильтрах/крошках. А в DETAIL-отчёте и CSV-экспорте (`reports-detail.ts`, столбец `employeeName`) — редакция тоже применяется? Иначе экспорт сольёт ПДн в обход тоггла.**
(а) detail/export тоже редактируются по reveal — проверить (CISO-007 риск утечки);
(б) detail доступен только при reveal=true (роль) → редакция не нужна;
(в) ПРОБЕЛ: detail-экспорт отдаёт ФИО независимо от тоггла — баг приватности.
— реф: `reports-detail.ts:137` пишет `employeeName` в CSV; в grep редакции по detail не видно (только reports.logic срезы). ТРЕБУЕТ проверки Dev2.

**W5C.20. Миграция старых записей: поля `tags`, `approvedBy`, `approvedAt`, `rejectComment` добавлены позже (nullable, default null). Старые записи без них в отчётах/согласовании ведут себя как «нет тегов / не согласована»? Подтвердить, что null трактуется безопасно везде.**
(а) null → пустой тег-список / DRAFT-семантика — безопасно;
(б) где-то null приводит к падению/неверному срезу (напр. сортировка по дате согласования);
(в) нужен backfill default-значений.
— реф: объект — все 4 поля `isNullable:true, defaultValue:null`.

## 5. Жёсткое/мягкое удаление, восстановление, справочники

**W5C.21. Удаление записи — ЖЁСТКОЕ (REST DELETE), мягкого удаления/корзины нет. APPROVED удалять нельзя. Но DRAFT/REJECTED удаляются безвозвратно. Нужна ли корзина/восстановление (особенно при случайном массовом удалении)?**
(а) жёсткое удаление ок — записи дёшево пересоздать;
(б) нужна мягкая корзина (Timetta хранит, не удаляет физически);
(в) запрет удаления SUBMITTED тоже (только DRAFT).
— реф: `time-entry-api.logic.ts:199-218` hard DELETE; guard только на APPROVED.

**W5C.22. Связи `stage`/`workType` имеют `onDelete: SET_NULL`. При УДАЛЕНИИ вида работ из справочника все записи теряют `workTypeId` → меняется уникальный ключ (был `(emp,proj,wt,date)`, стал `(emp,proj,NULL,date)`). Это может создать КОЛЛИЗИЮ ключа (две записи на день с разными wt → один день с NULL) → нарушение уникального индекса при следующей мутации. Реальный риск?**
(а) виды работ не удаляют (только деактивируют) → не возникает;
(б) удаление wt → SET_NULL может породить дубли по новому ключу, индекс отклонит правки → нужна защита/архивирование вместо удаления;
(в) при удалении wt предварительно слить/перепривязать записи.
— реф: объект `workType onDelete: SET_NULL`; индекс по `workTypeId`. SET_NULL × unique = потенциальный конфликт.

**W5C.23. Удаление ПРОЕКТА — связь `project onDelete: CASCADE`: удалятся ВСЕ записи трудозатрат проекта (включая APPROVED, в обход guard'а CISO-011, т.к. каскад идёт на уровне БД). Это допустимо? APPROVED-часы из табеля/1С исчезнут безвозвратно.**
(а) проект не удаляют, только закрывают статусом → каскад не срабатывает;
(б) CASCADE на проекте обходит APPROVED-guard → риск потери согласованных часов, нужен RESTRICT/архив;
(в) при удалении проекта явно предупреждать «удалится N записей, из них M согласованных».
— реф: объект `project ... onDelete: CASCADE`. Guard в /s/time-entry не защищает БД-каскад.

**W5C.24. Удаление СОТРУДНИКА — `employee onDelete: CASCADE`: все его записи удалятся. Но в отчётах есть механизм «деактивированный сотрудник с записями» (`active=false`, грузим для резолва ФИО). Значит сотрудников ДЕАКТИВИРУЮТ, а не удаляют. Удаление сотрудника вообще запрещено в UI?**
(а) только деактивация, удаление недоступно → CASCADE теоретический;
(б) удаление возможно → теряются исторические часы + ломается резолв ФИО (код покажет UUID-хвост);
(в) запретить delete employee при наличии записей.
— реф: объект `employee onDelete: CASCADE`; `reports.logic` грузит inactive для резолва — намёк, что удалять не должны.

**W5C.25. Тег удалён из справочника `ENTRY_TAG_OPTIONS` (MULTI_SELECT): что с записями, у которых он проставлен? SELECT-опции — это enum в схеме; удаление опции из объекта при наличии данных — что делает Twenty SDK?**
(а) удалять опции тега нельзя при наличии данных (миграция упадёт) → только добавляют;
(б) запись сохранит «осиротевший» код тега, в отчётах покажется сырой UPPER_CASE без ярлыка;
(в) нужен процесс «переименование/архив тега», а не удаление.
— реф: `objects/credos-time-entry.object.ts:86 tags MULTI_SELECT options: ENTRY_TAG_OPTIONS`. Поведение SDK при удалении опции — уточнить у Dev2.

## 6. Округление, нулевые/пустые значения, дедуп

**W5C.26. Часы — FLOAT, decimals=2. Расчёты округляют `toFixed(2)`/`toFixed(4)`. На границе 0.005 округление в большую (`toFixed` использует банковское?/half-up зависит от движка). Σ многих записей может расходиться с суммой отображаемых округлённых значений (накопление ошибки). Допустима ли копеечная расходимость факт vs Σ строк?**
(а) расходимость в 0.01 ч приемлема — учёт грубый;
(б) нужна сверка «итог == Σ строк» (округлять итог, а не каждую строку);
(в) хранить часы как целые минуты/decimal, не float.
— реф: `reports-calc.ts` множественные `toFixed`; `validation` не округляет вход. FLOAT 0.1+0.2 классика.

**W5C.27. Пустая строка / 0 часов: расчёты делают `if (hours === 0) continue` — запись с 0 часов в факт НЕ попадает. Но СОХРАНЯЕТСЯ ли запись с hours=0 вообще (валидация пропускает 0, т.к. `0 >= 0`)? Зачем хранить запись 0 часов — занимает ключ `(emp,proj,wt,date)`, блокируя реальную запись?**
(а) 0 часов сохранять нельзя (минимум >0) → добавить ERROR в validateEntry на hours==0/пусто;
(б) 0 допустим (отметка «был, но не работал») — но тогда он держит уникальный ключ;
(в) пустое поле hours → удалять запись, а не сохранять 0.
— реф: `validation.ts:75 hours<0` — ноль проходит; `reports-calc.ts:332 if (hours===0) continue` — игнор в факте. Несогласованность: хранится, но не считается, и занимает ключ.

**W5C.28. Дедуп-скрипт: выбор между Σ часов и `--keep-hours` ручной (флаг при запуске). Если оператор ошибётся (дубли были КОПИИ, а запустил без `--keep-hours`) — факт удвоится у выживших. Как заказчик/оператор различает «копии» vs «разные сеансы» ПЕРЕД запуском, и есть ли откат?**
(а) различение по описанию/времени создания — оператор смотрит dry-run и решает;
(б) нужна эвристика в скрипте (одинаковый hours+description → копия → keep; разные → суммировать);
(в) скрипт одноразовый (уже выполнен), вопрос неактуален для прода — подтвердить, что больше не запускается.
— реф: `dedup-entries.mjs:93-114 planDedup keepHours`. Решение Σ vs keep — на операторе, отката нет (DELETE необратим).

## Сводка найденных РЕАЛЬНЫХ рисков целостности (для arch)

Приоритизированы по серьёзности (из чтения кода, не гипотезы):

1. **[ВЫСОКИЙ] CASCADE на project/employee обходит APPROVED-guard** (W5C.23/24). Удаление проекта/сотрудника физически снесёт согласованные записи в обход CISO-011 (guard только в /s/time-entry, каскад — БД-уровень). Потеря данных табеля/1С.
2. **[ВЫСОКИЙ] Детализированный отчёт/CSV-экспорт может игнорировать reveal=false** (W5C.19). В `reports-detail.ts` пишется `employeeName` без видимой редакции — потенциальная утечка ПДн в обход CISO-007. Требует проверки Dev2.
3. **[СРЕДНИЙ] Семантика date: индекс по DATE_TIME (полное время), гард по дню** (W5C.2). Запись с произвольным временем через грид/CSV не ловится БД-индексом как дубль → двойной счёт factHours. Корневое расхождение двух механизмов.
4. **[СРЕДНИЙ] SET_NULL по workType × unique-индекс** (W5C.22). Удаление вида работ из справочника → SET_NULL → возможна коллизия уникального ключа, блокирующая последующие правки записей.
5. **[СРЕДНИЙ] Запись 0 часов сохраняется, но не считается и держит уникальный ключ** (W5C.27). Несогласованность: validateEntry пропускает 0, расчёт игнорит, ключ занят.
6. **[СРЕДНИЙ] employeeCode меняется при смене отдела** (W5C.16) + коллизия 4-hex (W5C.17). Кросс-периодная узнаваемость ломается; малая вероятность слияния двух людей.
7. **[НИЗКИЙ] Инвариант тренда нарушается записями без date** (W5C.8): Σ месяцев < год.
8. **[НИЗКИЙ] Двойной пересчёт factHours (триггер + /s/time-entry) не сериализован** (W5C.13); идемпотентность спасает значение, но не гонку снимков.
9. **[НИЗКИЙ] Жёсткое удаление без корзины** (W5C.21); сбой пересчёта в триггере не наверстывается (W5C.15).

**Всего вопросов: 28** (W5C.1–W5C.28), 6 групп.
