# UI-вопросы W4-C: мобайл, онбординг, уведомления-центр, поиск, масштаб

**Волна:** 4, задание W4-C (аналитик)
**Статус:** УТОЧНЯЮЩИЕ ВОПРОСЫ ЗАКАЗЧИКУ
**Формат:** `**W4C.x** вопрос — гипотезы (а)(б)(в) — реф: Timetta(T)/Kimai(K)/impeccable(imp)`

## 0. Что УЖЕ решено (НЕ переспрашиваем — углубляем)

Эти темы получили **принципиальные** ответы в волнах 1–2 (UI_ANSWERS_2). W4-C идёт **глубже** — до реализуемости и открытых краёв, не повторяя принятое:

- **Уведомления (F4):** in-app сейчас (баннер+бейдж), центр **только для руководителя** (F4.3), email-дайджест **когда появится канал** (F4.1), единый ToastHost (F4.12), persona-preferences чек-листом (F4.18), бейджи «N не заполнили» / «N на согласовании» (F4.8/F4.20).
- **Поиск (G1):** видимая строка «Поиск…» в шапке — **не Cmd+K** (Cmd+K в Worker невозможен, нет глобальных слушателей, G1.1); проекты+сотрудники+отделы (G1.2); по таймшиту — через фильтр, отдельного нет (G1.4); мгновенный локальный фильтр (G1.9).
- **Онбординг (E3):** пустые состояния = онбординг, **полного тура нет** (E3.4); dismissible-подсказки 1–2 шт; серверный флаг «виден впервые» на User Twenty (E3.5, localStorage в RemDOM нет); cheatsheet по `?` (E4.11).
- **Мобайл/touch:** compact только desktop, на touch принудительно comfortable 44px (F1.6); preset-чипы как основной ввод на touch (D3.15); ≥44px hit-zone при touch (D2.17).
- **Масштаб:** sticky-header + REST-пагинация 60/стр (F1.8), виртуализация «если объёмы вырастут».

**Технический факт (важен для всех 5 тем):** фронт — React 19 на инлайн-стилях `style={{}}`, **нет @media, нет breakpoint-системы** в `tokens.ts`. Адаптив-логика встречается лишь точечно: `project-plan-panel.tsx:162` (`min(360px, calc(100vw-32px))`), `overflow:auto` + sticky-колонки в `weekly-grid`/`capacity-board`/`project-team`. Touch-таргеты сейчас 26–58px (ячейка дня 58×~28px, input панели 26px) — **ниже стандарта 44px**. RemDOM без `window`-слушателей, без localStorage, без `matchMedia`.

## 1. Мобайл / планшет (адаптив без @media)

**W4C.1. Заходят ли вообще с телефона/планшета заполнять таймшит, или это desktop-only инструмент?**
От ответа зависит весь объём адаптив-работы (сейчас её почти нет).
(а) только desktop — мобайл out of scope, не тратимся;
(б) планшет (РП на встрече смотрит доску ёмкости) — приоритет №2;
(в) телефон сотрудника «забил часы в дороге» — нужен реальный мобильный ввод.
— реф: T — desktop-first SaaS, мобильная версия урезана (просмотр); K — Bootstrap responsive из коробки. imp: «выясни реальное устройство до адаптива».

**W4C.2. Если планшет нужен (W4C.1б) — какой режим первичен: горизонтальный (landscape, как узкий desktop) или вертикальный (portrait, нужна перестройка)?**
(а) только landscape ~1024px — текущая сетка влезет с h-scroll;
(б) portrait ~768px — 9 колонок не влезут, нужен другой макет строки;
(в) оба.
— реф: imp responsive — «landscape планшет ≈ узкий ноут, portrait — отдельный кейс».

**W4C.3. Поскольку @media в RemDOM нет — адаптив придётся делать через JS (замер ширины контейнера от SDK). Это приемлемо, или хост-виджет даёт фиксированную ширину и адаптив не нужен вовсе?**
(а) хост даёт фикс-ширину (всегда desktop-окно) — адаптив не нужен;
(б) хост резиновый — нужен JS-замер ширины + 1–2 порога (узкий/широкий);
(в) есть SDK-сигнал «mobile/tablet/desktop» — использовать его, не мерить пиксели.
— реф: текущий код — нет ни одного замера ширины, только `100vw`/flex. Уточнить у Dev2 что отдаёт песочница.

**W4C.4. Сетка таймшита (9 колонок: Проект + Вид работ + 7 дней + Итого) на узком экране — какой паттерн?**
Сейчас: фикс-колонки, h-scroll, sticky левые. На телефоне 7 дней не видны разом.
(а) h-scroll по дням, sticky Проект (текущий подход, просто принять для узкого);
(б) «день-карточка»: переключатель дня, видишь один день столбцом проектов (мобильный паттерн ввода);
(в) аккордеон по проектам: проект → раскрыл → 7 дней внутри.
— реф: T mobile — ввод по одному дню; K — таблица со скроллом. imp — «на телефоне таблица → список карточек».

**W4C.5. Capacity-доска (heatmap отделы×периоды) на планшете — она в принципе для мобайла, или это «руководитель только за десктопом»?**
Доска: `minWidth = 240 + периоды×56 + 72`, при 16 неделях ≈ 1200px.
(а) доска desktop-only, на планшете — read-only с h-scroll;
(б) на планшете доска нужна интерактивной (РП планирует с iPad);
(в) на планшете — урезанный вид (меньше периодов, крупнее ячейка).
— реф: T — планирование desktop. imp — «heatmap при сужении ячейки нечитаема» (это же решено в F1.10).

**W4C.6. Touch-таргеты: подтверждаем подъём до 44px на touch (F1.6/D2.17), но это удваивает высоту ячейки (58×28 → 58×44+). На планшете строк меньше влезет. Компромисс?**
(а) строго 44px на touch, меньше строк — ок (точность важнее плотности);
(б) 40px компромисс (между desktop-32 и touch-44);
(в) 44px только у интерактивных контролов (кнопки/чекбоксы), ячейка-ввода открывает крупный input оверлеем.
— реф: Apple HIG 44px, Material 48dp. imp — «touch-target ≥44px, но input можно открывать модально».

**W4C.7. Preset-чипы (8ч / 4ч / норма) как основной ввод на touch (решено D3.15) — где они на узком экране: над сеткой (общие), в открытом инпуте ячейки, или контекстно по тапу ячейки?**
(а) панель чипов под выбранной ячейкой (поповер);
(б) фикс-ряд чипов внизу экрана (как клавиатура-аксессуар);
(в) чипы в самом инпуте ячейки.
— реф: imp — «на touch минимизируй набор цифр, дай быстрые значения». T — шаблоны заполнения.

## 2. Онбординг / первый вход

**W4C.8. Серверный флаг «видел впервые» (E3.5) — он один на весь модуль, или по каждой фиче (видел сетку / видел доску / видел отчёты)?**
(а) один флаг `onboardingSeen` — проще, но подсказки на доске не покажутся тому, кто вошёл сразу в отчёты;
(б) набор флагов per-фича (`seenGrid`,`seenBoard`,`seenReports`) — точнее, дороже;
(в) счётчик визитов на фичу (показывать подсказку первые N раз).
— реф: T — флаги завершённости на User. imp onboard — «контекст важнее одного welcome».

**W4C.9. Первый вход РУКОВОДИТЕЛЯ (не сотрудника) — отдельный сценарий? У него другой первый экран (доска/согласование/отчёты), не «Мои часы».**
(а) единый онбординг, роль не различаем;
(б) разные первые подсказки по роли (сотрудник → «заполни неделю», руководитель → «вот кто не сдал»);
(в) руководителю онбординг не нужен (он внедряет систему, обучен).
— реф: T — разные дашборды по роли. imp — «онбординг под job-to-be-done роли».

**W4C.10. Пустой workspace у НОВОГО сотрудника без бронирований/проектов — что показывать? (E3.11 дал CTA «Добавить проект»/«Скопировать неделю», но у новичка и копировать нечего.)**
(а) «Ещё нет проектов — обратитесь к руководителю» (тупик, но честно, E3.8);
(б) если есть SOFT/HARD-брони — предложить строки из них (E3.10);
(в) показать норму/расписание как заглушку «вот сколько часов от вас ждут».
— реф: T — пустой таймшит подтягивает ресурс-план. imp — «first-use ≠ no-results, разный текст».

**W4C.11. Где живёт «Справка / Помощь» постоянно (не только при первом входе)? Cheatsheet по `?` решён, но это только горячие клавиши.**
(а) только cheatsheet горячих клавиш по `?` — достаточно;
(б) пункт «Помощь» в меню → панель с краткими how-to (заполнить/отправить/что значит цвет);
(в) контекстные «(i)»-иконки у непонятных мест (цвета загрузки, статусы).
— реф: T — раздел справки + тултипы. K — внешняя docs. imp — «help в point-of-use, не отдельный раздел».

**W4C.12. Подсказка-баннер первого входа dismissible один раз (E3.7) — если сотрудник закрыл, а через месяц забыл как отправлять — как вернуть обучение?**
(а) никак, закрыл = закрыл (есть cheatsheet);
(б) пункт «Показать подсказки заново» в меню/настройках;
(в) подсказка возвращается при значимом изменении (новый статус REJECTED первый раз).
— реф: imp — «дай re-trigger обучения, не запирай навсегда».

## 3. Уведомления — центр (in-app) UX

**W4C.13. Центр уведомлений только у руководителя (F4.3) — это отдельная панель/дровер, или просто бейдж-счётчики на пунктах меню (F4.8/F4.20) без единого центра?**
(а) единый дровер «Уведомления» (колокольчик) с лентой событий;
(б) без центра — только бейджи на «Записи»/«Отчёты»/«Согласование» (дешевле, текущее направление);
(в) бейджи + по клику — мини-список под пунктом (не полноценный центр).
— реф: T — есть индикаторы-счётчики в навигации (`hasViewIndicator`), полноценного Slack-центра нет. imp — «бейдж достаточен, центр — когда событий много».

**W4C.14. Если центр нужен (W4C.13а) — где колокольчик: в шапке виджета или это пункт хост-приложения Twenty (вне нашего виджета)?**
(а) в шапке нашего виджета time;
(б) интеграция с notification-центром Twenty (если он есть) — единое место для всех модулей;
(в) оба не нужны — бейджи на месте действия (W4C.13б).
— реф: уточнить у Dev2 наличие notification-API в хосте. T — свой центр в приложении.

**W4C.15. Типы in-app событий — финальный список и приоритет. Сейчас разбросаны: «заполни» (F4.7), «отклонено» (F4.17), «согласовано» (F4.20), «перегруз». Перегруз (>ёмкости) — это уведомление или только цвет на доске?**
(а) 3 типа: напоминание-заполнить / отклонено / на-согласовании — перегруз только визуально на доске;
(б) +перегруз как уведомление руководителю («Иванов перегружен на след. неделе»);
(в) +«согласовано» сотруднику (позитивное подтверждение).
— реф: T States: Draft/Submitted/Approved/Rejected — 4 события ЖЦ. imp — «не уведомляй о том, что и так видно цветом».

**W4C.16. Бейдж-счётчик — что считает и когда обнуляется? «N не заполнили» (F4.8) меняется ежедневно сам; «N на согласовании» (F4.20) обнуляется по факту согласования.**
(а) бейдж = текущее состояние (живой счётчик, не «непрочитанное») — обнуляется когда проблема решена;
(б) бейдж = непрочитанные события (как мессенджер) — обнуляется по просмотру;
(в) гибрид: действия-к-исполнению = состояние; информационные = непрочитанное.
— реф: T `filter(read eq false)` — модель «непрочитанное». imp — «счётчик-состояние честнее счётчика-непрочитанного для to-do».

**W4C.17. Дайджест руководителю (F4.10) — пока канала email нет, как доставляется? On-demand при входе (F4.8 «on-demand дайджест») или это будущий email?**
(а) сейчас — экран/панель «Сводка» при входе руководителя (pull), email — позже;
(б) баннер в шапке «3 сотрудника не сдали неделю» со ссылкой;
(в) и то, и то.
— реф: F4.1/F4.2 — pull сейчас, push когда появится канал. T — email-шаблоны. K — flash-сообщения, дайджеста нет.

**W4C.18. Частота напоминаний сотруднику — настройка у кого: у сотрудника (раздражает его) или у руководителя/админа (политика компании)?**
(а) reminderDayOfWeek + «за N дней» настраивает руководитель/админ (F4.5/F4.6) — единая политика;
(б) сотрудник может приглушить частоту в своих preferences (F4.18);
(в) гибрид: админ задаёт политику, сотрудник может «не раньше N» в рамках.
— реф: T dueDate + настройки уведомлений per-user. imp — «дай контроль получателю, иначе игнор/выключение».

**W4C.19. Когда появится email-канал (F4.1) — какие 2–3 события первыми уходят в почту? Не всё, что in-app, нужно дублировать письмом.**
(а) только «период закрывается, не сдал» (сотруднику) + дайджест «кто не сдал» (руководителю);
(б) +«ваш таймшит отклонён» (важно, легко пропустить in-app);
(в) всё что in-app — дублировать (рискует стать спамом).
— реф: T — конфигурируемые email-шаблоны по событиям. imp — «email = только то, что требует действия вне приложения».

## 4. Поиск — UX вглубь

**W4C.20. «Поиск…» в шапке (G1.1/G1.3) — он глобальный (по всем сущностям сразу) или контекстный (на экране отчётов ищет в отчётах, на проектах — проекты)?**
(а) контекстный per-экран (проще, данные уже локальны, G1.9);
(б) глобальный «найди что угодно» — проекты+сотрудники+отделы из одного поля (G1.2), переход на сущность;
(в) контекстный на экранах + один глобальный в шапке модуля.
— реф: T — глобальный Search (term/kind/entityType) + searchVector в БД. K — поиск в пределах раздела. imp — «глобальный поиск окупается от ~сотен записей».

**W4C.21. Быстрый переход по результату (G1.16: клик проект→фильтр, клик имя→карточка) — на чём строится «карточка сотрудника/проекта»? У нас есть такие экраны или это переход в Twenty?**
(а) свои мини-карточки в модуле time;
(б) переход на карточку сущности в хосте Twenty;
(в) не карточка, а «применить как фильтр» везде (G1.16б) — карточки нет.
— реф: T — клик в карточку проекта/агента. Уточнить у Dev2 deep-link в Twenty.

**W4C.22. Cmd+K-заглушка: раз Cmd+K в Worker невозможен (G1.1), но привычка сильная — нужна ли видимая кнопка «Поиск» с подписью-хоткеем, который НЕ работает, или вообще не намекать на хоткей?**
(а) кнопка «Поиск» без хоткей-подписи (честно, ничего не обещаем);
(б) кнопка с «/» — фокус в поле по `/` (это ловится на контейнере, в отличие от Cmd+K) — рабочая альтернатива;
(в) только иконка-лупа.
— реф: G1.1 — Cmd+K нельзя, но контейнерный keydown можно (E4.10). `/` — частый паттерн (GitHub/Slack). imp — «не показывай неработающий хоткей».

**W4C.23. Что НЕ ищем сейчас (scope): записи времени отложены (G1.2 «позже»). Подтверждаем что поиск по тексту комментариев записей / по часам — не в первой версии?**
(а) да, только проекты/сотрудники/отделы; записи и комментарии — roadmap;
(б) нужен поиск по комментариям записей (РП ищет «что делал по проекту X»);
(в) нужен поиск записей по диапазону часов/дате (аудит).
— реф: T searchVector охватывает Project/Issue/Deal, не каждую запись времени. imp — «не раздувай scope поиска до данных, которые ищут редко».

**W4C.24. Подсветка совпадения (G1.19) и «найдено N из M» (G1.10) — при сотнях результатов лимитировать вывод «показать ещё» (G1.10). Сколько показывать сразу?**
(а) 20–30 с «показать ещё»;
(б) все (данные локальны, скролл) — но риск тормозов при 72×N;
(в) топ-10 + «уточните запрос».
— реф: K — пагинация результатов. imp — «первый экран результатов ≤ один скролл».

## 5. Производительность / масштаб

**W4C.25. Реальный объём через год: 72 сотрудника. Сколько проектов одновременно и сколько недель смотрит руководитель на доске разом? Это калибрует, где сломается.**
(а) ~30–50 активных проектов, доска до квартала (13 нед) — текущий код тянет;
(б) >100 проектов, доска до года (52 нед) — `deptLoadCells` O(отделы×проекты×периоды) станет тяжёлым;
(в) точно неизвестно — заложить запас.
— реф: текущий расчёт доски в `useMemo` (`capacity-board.tsx:48`), при 72×300×16 ≈ сотни тыс. операций на рендер. T — серверная агрегация для тяжёлого.

**W4C.26. Сетка таймшита грузит `limit:500` записей за период БЕЗ проверки hasNextPage (`time-rest.ts:147`). За неделю одного — норм, но «все сотрудники за неделю» может превысить 500. Где должна жить агрегация — фронт или бэк?**
(а) фронт грузит сырьё, считает сам (текущее) — до объёма N ок;
(б) тяжёлые срезы (доска, отчёты) — серверная агрегация, фронт только рисует (отчёты уже так, `reports-rest`);
(в) дорастить пагинацию (hasNextPage) на горячих запросах.
— реф: reports/olap уже считаются на `/s/reports` (server-side). T — OLAP/Reports на бэке. imp/архитектура — «не тяни 10k строк в браузер».

**W4C.27. Виртуализация списков (F1.8 «если объёмы вырастут») — порог «вырастут» какой? Сейчас весь DOM рендерится (нет virtual scroll нигде).**
(а) откладываем, 72 сотрудника не критичны для DOM;
(б) ввести виртуализацию реестров записей/сотрудников при >200 строк;
(в) сразу заложить (дорого, может не понадобиться).
— реф: imp/perf — «virtual scroll окупается от ~хундреда видимых строк». T — серверная пагинация вместо virtual.

**W4C.28. Отчёт за год (detail, 40+ полей × тысячи строк) — экспорт и просмотр. Просмотр в браузере всех строк или сразу «слишком много, уточните период / выгрузите файл»?**
(а) показывать с пагинацией/«показать ещё», экспорт — полный;
(б) при >N строк блокировать просмотр, предлагать экспорт;
(в) агрегат на экране, детализация только в файле.
— реф: reports уже server-side; экспорт ПДн логируется (G3.15). T — отчёт смотрят агрегированно, детализацию выгружают.

**W4C.29. Первая загрузка модуля (cold start): сейчас параллельно тянется много справочников (проекты 200, виды работ 200, сотрудники 200, отделы 50 — `time-rest`). Приемлемое время до интерактива?**
(а) ≤1–2с, можно грузить всё разом (текущее);
(б) >2с — нужен скелет + ленивая дозагрузка справочников;
(в) кэшировать справочники между визитами (они меняются редко).
— реф: imp/perf — «справочники кэшируй, не перезапрашивай каждый вход». Уточнить у Dev2 возможность кэша в RemDOM.

**W4C.30. Календарь тянет год курсор-пагинацией (`calendar-rest.ts:16`, PAGE=60, до 12 итераций = 12 запросов). При 72 сотрудниках это per-user или общий? 12 последовательных запросов — узкое место?**
(а) per-user, 12 запросов на свой год — терпимо;
(б) распараллелить страницы (не последовательно);
(в) серверный агрегат «год одним запросом».
— реф: текущий цикл `for page<12` последовательный. imp/perf — «параллель вместо waterfall запросов».

## 6. Гипотезы (рабочие предположения для проверки)

**ГИП-1. Мобайл — не приоритет, кроме планшета-просмотра у руководителя.**
Тайм-трекинг заполняют за рабочим ПК; РП может глянуть доску с планшета. Тогда: полноценный мобильный ВВОД (W4C.4б день-карточка) — out of scope; нужен лишь читаемый landscape-планшет + touch 44px на интерактиве. *Проверка: W4C.1, W4C.5.*

**ГИП-2. Центр уведомлений как отдельная панель — избыточен; хватает бейджей на месте действия.**
F4.8/F4.20 уже дают «N не заполнили»/«N на согласовании». Slack-центр — over-engineering для инструмента на 72 чел. Тогда W4C.13б (бейджи без центра) + дайджест-панель руководителю при входе (W4C.17а). *Проверка: W4C.13, W4C.17.*

**ГИП-3. Поиск — контекстный per-экран, глобальный отложить.**
Данные локальны, фильтры в сетке/реестрах уже есть (G1.4). Глобальный кросс-сущностный поиск окупается позже. Тогда W4C.20а + `/`-фокус как дешёвая замена Cmd+K (W4C.22б). *Проверка: W4C.20, W4C.22.*

**ГИП-4. Главный риск масштаба — доска ёмкости и сырьевые `limit:500`, не списки.**
`deptLoadCells` считается на фронте в `useMemo` (O(отделы×проекты×периоды)); сетка/команда грузят 500 записей без hasNextPage. При росте проектов это сломается раньше, чем понадобится virtual scroll. Тогда приоритет: серверная агрегация тяжёлых срезов (W4C.26б) > виртуализация (W4C.27а отложить). *Проверка: W4C.25, W4C.26.*

**ГИП-5. RemDOM-ограничения — сквозной фактор всех 5 тем.**
Нет `@media`/`matchMedia` (мобайл через JS-замер), нет `window`-слушателей (Cmd+K нельзя, `/` можно), нет localStorage (онбординг-флаг на сервере, кэш справочников под вопросом). Многие «очевидные» паттерны требуют обходного решения или SDK-сигнала. *Проверка: W4C.3, W4C.14, W4C.29 — все упираются в «что даёт песочница Dev2».*
