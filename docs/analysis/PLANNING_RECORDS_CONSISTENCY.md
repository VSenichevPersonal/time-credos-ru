# Планирование: создаёт ли «Планировать» записи плановых загрузок? (консистентность модели)

**Дата:** 2026-06-21
**Автор:** аналитик AITEAM
**Повод (заказчик):** «По результатам "Планировать" — создаются ли записи "плановые загрузки" (чтобы потом править)? Как править из планирования?»
**Связанные доки:** [PLANNING_MODELS.md](PLANNING_MODELS.md) (методы раскида, 35 вопросов), REQ-0004 (plan-allocation + booking), REQ-0010 (разрезы люди/проекты), REQ-0012 (план отдела без проекта), REQ-0013 (доли отделов), [CAPACITY_PLANNING.md](../data-model/CAPACITY_PLANNING.md).

> Этот документ узко про **модель хранения плана** (агрегат vs записи-слоты) и про разрыв между кнопкой «Планировать» и реестром «Плановые загрузки». Методы распределения (равномерно/фронт/ручной) разобраны отдельно в PLANNING_MODELS.md — не дублирую.

---

## 0. TL;DR — короткий ответ заказчику

- **НЕТ. «Планировать» НЕ создаёт записи «плановых загрузок».** Кнопка делает один **PATCH** существующей записи (проекта `credosTimeProject` ИЛИ плана отдела `credosTimeDeptPlan`): меняет три поля — `plannedEffort` (итого часов), `startDate`, `endDate`. Новых строк не появляется.
- **Раскид по периодам (неделя/месяц) — ВЫЧИСЛЯЕТСЯ на лету и НИГДЕ не хранится.** Функция `plannedHoursInPeriod` (calc-load.ts) каждый раз пересчитывает «сколько часов попадает в эту колонку» из `plannedEffort`+дат. Хранимых «слотов план×период» в системе нет.
- **Объект «Плановые загрузки» в сайдбаре** = `credosTimeDeptPlan` (REQ-0012). Это **отдельный** вид плана: плановая загрузка отдела **без проекта** (резерв/пресейл/прочее). Это НЕ «детализация плана проекта по периодам». Правится в реестре как обычная запись (label/часы/даты/отдел) и из доски той же кнопкой «Планировать».
- **Главный разрыв:** «Планировать» = агрегат (одно число часов + период на проекте/плане отдела). «Плановые загрузки» = независимый реестр записей **другого смысла** (план отдела без проекта). Они **не являются** «план → его редактируемые слоты». Записей-слотов «план проекта по месяцам» не существует — поэтому **режим «Вручную по месяцам» (фаза-2 WI-11) сейчас построить не на чем** (нет хранимого объекта, куда писать ручные значения по периодам).
- **Править план из планирования можно** на двух уровнях: проект целиком (часы+срок) и доля отдела/план отдела без проекта (часы+срок). **Нельзя:** задать разные часы по месяцам, задать план на конкретного человека.

## 1. Что делает «Планировать» СЕЙЧАС (точно по коду)

### 1.1 Поток сохранения

```
ProjectPlanPanel (project-plan-panel.tsx)         ← поповер «✎ План» на строке
   onSave(project.id, { plannedEffort, startDate, endDate })
        │
usePlanEdit.save / .saveDeptPlan (use-plan-edit.ts)
        │  PATCH + рефетч (НЕ create)
        ▼
patchProject(id, patch)      → PATCH /rest/credosTimeProjects/{id}   { plannedEffort, startDate, endDate }
patchDeptPlan(id, patch)     → PATCH /rest/credosTimeDeptPlans/{id}  { plannedEffort, startDate, endDate }   (capacity-rest.ts:285-301)
```

- **Создания записей нет нигде.** В `capacity-rest.ts` есть только `patchProject`/`patchDeptPlan` (PATCH). Ни одного `client().post(...)` для плана. Кнопка правит **уже существующую** запись (проект создаётся отдельно в реестре проектов; запись `credosTimeDeptPlan` — в реестре «Плановые загрузки»).
- Поповер сохраняет **3 поля целиком**: `plannedEffort` (часы за весь срок), `startDate`, `endDate`. Способ распределения «Равномерно» — единственный активный; «Вручную по месяцам» — **disabled-заглушка** (project-plan-panel.tsx:190-197, прямая отметка «фаза 2, нужно бэк-поле planMethod»).

### 1.2 Раскид по периодам — вычисляемый, не хранимый

`plannedHoursInPeriod(plannedEffort, startDate, endDate, period, spread)` (calc-load.ts:222-261):
- берёт `plannedEffort` и даты, считает долю, попадающую в колонку периода, **в момент рендера доски**;
- инвариант Σ по горизонту = `plannedEffort`;
- результат **никуда не сохраняется** — это чистая функция отображения.

Одна и та же функция раскидывает ВСЕ слои Demand: проекты (`projectHoursInPeriod`), доли отделов (`projectShareHoursInPeriod`, REQ-0013), планы отдела без проекта (`deptPlanHoursInPeriod`, REQ-0012), брони (`bookingHoursInPeriod`, REQ-0004 C).

**Вывод:** «правка раскида по периодам» сейчас технически невозможна — раскида как данных не существует, есть только формула из агрегата.

## 2. Объекты планирования: что хранят и как соотносятся

| Объект | Что хранит (поля плана) | Гранулярность | «Создаёт ли его Планировать?» | Правится |
|---|---|---|---|---|
| `credosTimeProject` (план проекта) | `plannedEffort`, `startDate`, `endDate`, `departmentId` | проект целиком | нет (PATCH существующего) | поповер «План» + реестр проектов |
| **`credosTimeDeptPlan`** = «Плановые загрузки» в сайдбаре (REQ-0012) | `plannedEffort`, `startDate`, `endDate`, `department`, `label`, `category` | отдел × период, **без проекта** | нет (PATCH существующего; создаётся в реестре/сидом) | реестр «Плановые загрузки» + поповер «План» |
| `credosTimeProjectDepartment` = «Доли отделов» (REQ-0013) | `plannedEffortShare` (часы доли), `project`, `department` | проект × отдел | нет (правится в реестре долей; поповер часы доли не трогает) | реестр «Доли отделов в проектах» |
| `credosTimeBooking` = «Брони ресурсов» (REQ-0004 C) | `hours`, `startDate`, `endDate`, `bookingType` (SOFT/HARD), `employee`, `project`/`company` | сотрудник × проект × период | нет (отдельный реестр «Брони ресурсов») | реестр «Брони ресурсов» |

Ключевые факты соотношения:
- **«Плановые загрузки» (credosTimeDeptPlan) ≠ слоты плана проекта.** Это самостоятельный вид нагрузки отдела БЕЗ проекта. Суммируется к загрузке отдела **наравне** с проектами, а не детализирует их (calc-load.ts:490-491). Заказчик может воспринимать название «Плановые загрузки» как «детализация плана» — это НЕ так.
- **Доли отделов (REQ-0013)** — единственная штатная декомпозиция плана проекта (проект → его отделы в часах). Но **без разбивки по периодам** (доля раскидывается по датам проекта той же формулой) и **не правится из поповера** «Планировать» (поповер трогает только `plannedEffort` проекта/плана отдела, не `plannedEffortShare`).
- **Брони** — отдельная ось (резерв), намеренно НЕ связана FK с планом-оценкой (инвариант Timetta «оценка ≠ резерв», REQ-0004 §B/C).
- **Плана на человека НЕТ.** Срез «по людям» (`employeeLoadCells`, calc-load.ts:501-539) делит нагрузку отдела поровну `1/headcount` — это вычисление, не хранимая аллокация. Объект `credosTimePlanAllocation` (REQ-0004 B) спроектирован, но НЕ реализован.

## 3. Можно ли СЕЙЧАС править план из планирования

| Разрез | Можно править? | Где / чем | Ограничение |
|---|---|---|---|
| Проект целиком (часы + срок) | ✅ да | поповер «План» на строке проекта → PATCH | только агрегат, без разбивки по периодам |
| План отдела без проекта (часы + срок) | ✅ да | поповер «План» на строке dept-plan + реестр «Плановые загрузки» | агрегат |
| Доля отдела в проекте (часы) | ⚠️ частично | реестр «Доли отделов» (НЕ из поповера) | вне доски планирования; нет периодов |
| **План по месяцам (разные часы по периодам)** | ❌ нет | — | нет хранимого объекта-слота; режим «Вручную» — заглушка |
| **План на конкретного человека** | ❌ нет | — | нет `credosTimePlanAllocation` (REQ-0004 B) |
| Бронь сотрудника | ✅ да (как резерв, не план) | реестр «Брони ресурсов» | отдельная ось |

**Разрыв сводится к двум дырам:**
1. **Нет слотов план×период** → нет ручной правки распределения, режим «Вручную по месяцам» висит заглушкой.
2. **Нет аллокации план×человек** → нет «плана на сотрудника» (REQ-0010 разрез «Люди» заблокирован).

## 4. Анализ консистентности: дубль смыслов или нет?

**Это НЕ дубль, но и НЕ «план → слоты».** Сейчас три уровня плана живут как три РАЗНЫХ агрегата:
- `project.plannedEffort` — план «есть проект, столько часов на срок»;
- `credosTimeDeptPlan` — план «отдел занят на столько-то БЕЗ проекта»;
- `plannedEffortShare` (доля) — «из плана проекта столько-то на этот отдел».

Все три раскидываются **одной** формулой `plannedHoursInPeriod` → модель раскида уже консистентна (хорошо). Проблема не в дублировании, а в **отсутствии слой-объекта для периодной детализации**: между «итого часов» и «колонками доски» нет хранимого промежуточного слоя, который можно было бы править руками.

**Для фазы-2 «Вручную по месяцам» нужен хранимый, редактируемый слот план×период.** Сейчас его нет → фаза-2 не на чем строить. Это и есть ядро вопроса заказчика «создаются ли записи, чтобы потом править».

### Сверка с Timetta (легальный референс)

Timetta `projects-components-projects-resource-plan.md` + `booking-scale.md`: ресурсный план хранится **записями-аллокациями** (член команды × работа × период), редактируется **по ячейкам периодов**; при смене масштаба (день/неделя/месяц) значения распределяются пропорционально рабочему расписанию. То есть «взрослая» модель — **план = записи-слоты**, агрегат вычисляется ИЗ них (а не наоборот, как у нас).

У нас инверсия: агрегат хранится, слоты вычисляются. Это проще (keep-it-simple) и достаточно для «равномерно», но **принципиально не даёт ручную правку по периодам и план на человека** без введения слот-объекта.

## 5. Рекомендованная консистентная модель (взвешенно)

Предлагаю **гибрид, поэтапно** — не ломать рабочий агрегат, добавить слотный слой только там, где он реально нужен.

### Уровень 0 (есть): агрегат + вычисляемый раскид
Оставить как основной режим «Равномерно». Дёшево, покрывает 80% кейсов. Ничего не трогаем.

### Уровень 1 (рекомендую как следующий шаг): слот-объект `credosTimePlanAllocation`
Единый объект-слот плана: `{ project (nullable), department (nullable), employee (nullable), periodStart, periodEnd | periodKey, plannedHours, source }`.
- **Когда заполнен** для проекта/отдела → доска и отчёты берут **сумму слотов** вместо вычисляемого раскида (как fallback-логика уже описана в REQ-0004 §B: «Σ аллокаций если заданы, иначе fallback на plannedEffort»).
- **«Планировать» в режиме «Вручную»** создаёт/правит эти слоты (по одному на период). В режиме «Равномерно» — НЕ создаёт, оставляет агрегат (раскид считается на лету). Так заказчик получает прямой ответ: «в ручном режиме записи создаются и правятся; в равномерном — нет, всё из агрегата».
- **Реестр** этих слотов виден отдельно («Плановое распределение» / вкладка на карточке проекта) — правится и оттуда, и с доски. Двусторонняя видимость = ответ на «чтобы потом править».
- **Синхронизация:** `plannedEffort` проекта = либо Σ слотов (если ручной режим, slots — источник истины), либо самостоятельное число (равномерный режим). Флаг `planMethod` на проекте (EVEN/MANUAL) решает, кто источник истины. Это убирает рассинхрон.

### Что НЕ делать (keep-it-simple)
- НЕ превращать `credosTimeDeptPlan` и `credosTimeBooking` в слоты — у них своя ось, они и так записи.
- НЕ создавать слоты автоматически в равномерном режиме (плодит сотни строк ради того, что считается формулой).
- НЕ вводить FTE/дни/% как единицы плана сейчас (PLANNING_MODELS.md, отдельное решение).

### Гранулярность слота — на решение заказчика
- проект × период (просто, без отделов/людей) — минимум для «Вручную по месяцам»;
- проект × отдел × период (стыкуется с REQ-0013) — средне;
- проект × сотрудник × период (полный Timetta, закрывает REQ-0010 «Люди») — максимум.

Рекомендую начать с **проект × период** (закрывает прямой запрос «по месяцам»), а отдел/человека добавить как опциональные поля того же слота, когда дойдут REQ-0013-детализация / REQ-0010.

## 6. Вопросы заказчику (18)

> Формат: вопрос → гипотезы (Г1..) → референс/связь.

**Модель хранения: агрегат vs записи-слоты**

1. **Должна ли кнопка «Планировать» создавать редактируемые записи распределения, или хватит «итого часов + срок» с авто-раскидом?**
   Г1 (рекомендую) гибрид: «Равномерно» = агрегат без записей, «Вручную» = создаёт слоты. Г2 всегда агрегат (записей не надо, ручной режим не нужен). Г3 всегда слоты (как Timetta, агрегат = Σ слотов).
   Реф: REQ-0004 §B, Timetta resource-plan.

2. **В равномерном режиме создавать слоты «про запас» или считать на лету (как сейчас)?**
   Г1 (рекомендую) на лету. Г2 материализовать (проще отчётам, но плодит строки).
   Реф: calc-load.ts `plannedHoursInPeriod`.

3. **Кто источник истины при ручном режиме: Σ слотов или `plannedEffort` проекта?**
   Г1 (рекомендую) слоты → `plannedEffort` пересчитывается из них. Г2 `plannedEffort` ведущий, слоты лишь подсказка. Г3 флаг `planMethod` решает per-проект.
   Реф: REQ-0004 «Σ аллокаций если заданы, иначе fallback».

**Гранулярность слота**

4. **Гранулярность ручного слота?**
   Г1 проект × период. Г2 (рекомендую старт) проект × период, с опц. отделом. Г3 проект × отдел × период. Г4 проект × сотрудник × период.
   Реф: REQ-0013, REQ-0010, Timetta booking-scale.

5. **Масштаб периода слота — фикс месяц или как на доске (неделя/месяц)?**
   Г1 (рекомендую) месяц (соответствует «по месяцам» в запросе). Г2 неделя. Г3 наследовать масштаб доски.
   Реф: buildPeriods, PLANNING_MODELS §1.2.

6. **Что при смене масштаба/срока проекта с заполненными ручными слотами — пересчитывать пропорционально или сохранять введённое?**
   Г1 (рекомендую, как Timetta) сохранять введённые значения, не пересчитывать. Г2 пропорционально перераспределить. Г3 предупредить и спросить.
   Реф: Timetta booking-methods «ручное — значения остаются».

**Редактирование: доска vs реестр**

7. **Где править ручное распределение — только из доски, только из реестра, или из обоих?**
   Г1 (рекомендую) из обоих (двусторонне). Г2 только доска. Г3 только реестр.
   Реф: запрос «чтобы потом править».

8. **Нужен ли отдельный реестр слотов в сайдбаре («Плановое распределение»), или они видны только на карточке проекта/доске?**
   Г1 (рекомендую) вкладка на карточке проекта + доска. Г2 отдельный пункт сайдбара. Г3 и то, и другое.
   Реф: текущий «Плановые загрузки» = credosTimeDeptPlan (другой объект).

9. **Inline-правка ячейки прямо в колонке доски (как Excel) или через поповер?**
   Г1 (рекомендую фаза-2) inline по ячейке. Г2 поповер со списком периодов. Г3 модалка-таблица.
   Реф: project-plan-panel.tsx (текущий поповер).

**Терминология / «Плановые загрузки»**

10. **Название «Плановые загрузки» в сайдбаре сейчас = план отдела БЕЗ проекта (credosTimeDeptPlan). Это совпадает с вашим пониманием, или вы ждали там детализацию плана проектов?**
    Г1 всё ок, это резерв/пресейл отдела. Г2 (вероятно) ожидалась детализация плана — нужен новый объект-слот + переименование. Г3 объединить оба понятия в один реестр с фильтром «с проектом / без».
    Реф: REQ-0012, credos-time-dept-plan.object.ts.

11. **Объединять ли план отдела-без-проекта (REQ-0012) и будущие слоты плана проекта в один реестр, или держать раздельно?**
    Г1 (рекомендую) раздельно (разный смысл). Г2 единый реестр «Все плановые загрузки» с типом-фильтром.
    Реф: calc-load.ts суммирует их как разные слои.

**Связь с REQ-0012 / 0013 / booking**

12. **Доли отделов (REQ-0013) должны редактироваться из доски «Планировать» (сейчас только из реестра)?**
    Г1 (рекомендую) да, добавить правку `plannedEffortShare` в поповер при раскрытии проекта по отделам. Г2 оставить в реестре долей. Г3 через будущие слоты (проект×отдел×период).
    Реф: project-plan-panel.tsx не трогает plannedEffortShare; calc-load.ts:333.

13. **Если введён ручной слот по периодам И есть доли отделов — как они сочетаются (слот делится по долям? слот задаётся уже по отделу)?**
    Г1 (рекомендую при гранулярности «проект×период») слот общий, делится по долям. Г2 при «проект×отдел×период» слот сразу по отделу, доли не нужны. Г3 запретить одновременно.
    Реф: REQ-0013, calc-load.ts projectDeptHoursInPeriod.

14. **Бронь (credosTimeBooking) остаётся отдельной осью от плановых слотов (инвариант Timetta), или их свести?**
    Г1 (рекомендую, как заложено) оставить раздельно (резерв ≠ план-оценка). Г2 свести бронь и план в один объект с типом.
    Реф: REQ-0004 §C инвариант «бронь ≠ план ≠ факт».

15. **План на конкретного человека (разрез «Люди», REQ-0010) — делать сейчас тем же слот-объектом (employee-поле) или отложить?**
    Г1 (рекомендую) одним объектом, поле `employee` опционально — заполняется когда дойдём до «Люди». Г2 отдельный объект `credosTimePlanAllocation` именно по людям. Г3 отложить.

**Синхронизация plannedEffort ↔ слоты**
