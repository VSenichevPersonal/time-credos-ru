## C1. СПРАВОЧНИКИ (реестры)

**Текущее состояние (факт из кода):** 7 реестров (Проекты, Виды работ, Этапы, Сотрудники, Отделы, Произв.календарь, Связи 1С) — стандартные Twenty index-view (колонки заданы во view-файлах). Импорт/массовая загрузка отсутствует. BillingLink (связь 1С) — пока ручной ввод, синхронизация заявлена на фазу 2.

**C1.3.** Реестр «Сотрудники» — колонки Фамилия/Имя/Должность/Отдел/Активен. С учётом CISO-007 (`revealEmployeeNames`) — кто и где видит ФИО в реестре? Должен ли реестр сотрудников вообще показывать ФИО рядовому пользователю или только руководителю/админу?

## C2. BOOKING / БРОНИ

**Текущее состояние (факт из кода):** объект `credosTimeBooking` — поля label / bookingType (SOFT|HARD, дефолт SOFT) / hours / startDate / endDate / note + relations employee (обязат., CASCADE), project (nullable, SET_NULL), company (nullable, SET_NULL). Создание/правка — стандартная модалка Twenty (нет спец-формы на доске). Часы раскидываются равномерно по рабочим дням периода. HARD входит в load (потребляет ёмкость), SOFT — отдельный пунктирный слой, показывается только при `tentativeBookingEnabled`. Овербукинг = `load > capacity` → conflict (только подсветка, не блокирует). Конвертация SOFT→HARD — только ручной правкой записи. Роль-заглушка (бронь без сотрудника) НЕ реализована (employee обязателен). Доска capacity (`capacity-board`) показывает срезы Отделы/Люди. На доске бронь — нередактируемый маркер (pointer-events:none).

### Группа 2.1. Создание брони

**C2.1.** Бронь сейчас создаётся только через общую модалку Twenty (как любой объект). Нужно ли создавать бронь прямо с доски capacity (клик по ячейке сотрудник×период → форма брони с предзаполненными сотрудником и датами)?
— Гипотезы: (а) хватит общей модалки/реестра; (б) нужен quick-create с доски; (в) drag по ячейкам для выбора периода (учесть: в Remote DOM нативного drag нет).
— Реф: Timetta/Float — создание брони кликом/протяжкой по таймлайну.

**C2.2.** Какие поля обязательны при создании брони? Сейчас по сути employee + bookingType + даты; hours/project/company опциональны. Должен ли быть обязателен хотя бы один из project/company (чтобы не было «висячих» броней)?
— Гипотезы: (а) как сейчас (можно без проекта и компании); (б) обязателен project ИЛИ company; (в) обязателен project всегда (компания — только при пресейле).

**C2.3.** `hours` — ёмкость на весь период, раскидывается равномерно по рабочим дням. Заказчик мыслит бронь в часах на период, в FTE % (например 50% занятости), или в днях?
— Гипотезы: (а) часы на период (как сейчас); (б) FTE % (доля занятости); (в) часы/день × дни.
— Реф: Float/Timetta — бронь в % загрузки или ч/день; Kimai — нет capacity-броней.

**C2.4.** Нужна ли проверка/предупреждение при создании брони, что период выходит за горизонт планирования (`planningHorizonWeeks`, дефолт 16) или пересекается с отсутствием сотрудника?
— Гипотезы: (а) не нужно; (б) предупреждать о выходе за горизонт; (в) предупреждать о пересечении с отпуском/больничным.

### Группа 2.2. SOFT → HARD конверсия

**C2.5.** Сейчас SOFT→HARD = ручное изменение поля bookingType в записи. Нужна ли явная кнопка «Подтвердить бронь» (SOFT→HARD) на карточке/доске?
— Гипотезы: (а) хватит правки поля; (б) кнопка «Подтвердить» на карточке брони; (в) кнопка прямо на маркере доски.
— Реф: Float — «tentative → confirmed» переключателем; Timetta — статус брони.

**C2.6.** При конверсии SOFT→HARD нужно ли что-то ещё, кроме смены типа? Например: привязать проект (если бронь была под компанию-пресейл), пересчитать часы, зафиксировать дату подтверждения/автора?
— Гипотезы: (а) только смена типа; (б) требовать проект при HARD; (в) логировать кто/когда подтвердил.

**C2.7.** Должна ли система предупреждать при конверсии SOFT→HARD, если это создаёт овербукинг (load превысит capacity)?
— Гипотезы: (а) молча разрешать (как сейчас, conflict только подсветкой); (б) предупреждение перед подтверждением; (в) запрет без подтверждения руководителем.

### Группа 2.3. Бронь под проект vs компанию-пресейл

**C2.8.** Модель допускает бронь с project, с company, или с обоими/ни с одним. Какие сценарии реальны у заказчика? (1) проект известен; (2) пресейл — только клиент; (3) совсем общий резерв сотрудника?
— Гипотезы: (а) только проект и пресейл-клиент; (б) бывает общий резерв без обоих; (в) всегда хотя бы клиент.

**C2.9.** Для пресейл-брони (company без project) — как она должна отображаться на доске и в отчётах? Отдельной категорией «Пресейл», цветом, или так же как проектная?
— Гипотезы: (а) отдельный цвет/слой «Пресейл»; (б) как обычная HARD; (в) пресейл всегда SOFT и показывается только пунктиром.
— Реф: Timetta — отдельный pipeline для пресейл-резервов.

**C2.10.** Когда пресейл превращается в проект — как «переносить» бронь? Менять company→project вручную, или нужен механизм «создать проект из пресейл-брони»?
— Гипотезы: (а) ручная правка project; (б) кнопка «преобразовать в проектную бронь»; (в) бронь автоматически подхватывает проект при его создании под клиента.

### Группа 2.4. Роль-заглушка (бронь без конкретного сотрудника)

**C2.11.** Сейчас employee обязателен — нельзя забронировать «нужен 1 senior-разработчик» без конкретного человека. Заказчику нужны брони на роль/должность-заглушку (TBD-ресурс)?

— Реф: Float/Timetta — placeholder/TBH (to-be-hired) ресурсы на роль.

**C2.12.** Если роль-заглушка нужна — как её потом «назначать» на реального сотрудника? Заменой поля, или бронь-заглушка остаётся как «потребность», а реальная бронь создаётся отдельно?

**C2.13.** Должна ли роль-заглушка потреблять/показывать ёмкость отдела (демонстрировать дефицит), или это чисто «список потребностей» вне расчёта capacity?

**C2.17.** Что должна показывать ячейка доски при овербукинге — насколько перегружено? Сейчас есть ratio/free и подсветка. Нужно ли явное «-12ч» / «140%» сверх ёмкости?
— Гипотезы: (а) подсветки+ratio хватит; (б) показывать дефицит в часах; (в) показывать % перегрузки.

**C2.18.** SOFT не входит в load (не создаёт conflict). Нужен ли отдельный индикатор «потенциальный овербукинг с учётом SOFT» (если все предварительные подтвердятся)?

## C4. ЭКСПОРТ / ПЕЧАТЬ

**Текущее состояние (факт из кода):** два CSV-экспорта через POST `/s/reports` — (1) detail (7 колонок: Дата/Сотрудник/Отдел/Проект/Вид работ/Часы/Статус), (2) timesheet-grid = табель Т-13 (сотрудник×день 1..31 + Итого, опц. буквенные коды Я/ОТ/Б/ДО/К/НН). Оба: разделитель `;`, BOM (`﻿`, UTF-8) для кириллицы в Excel/1С, имя `*_YYYY-MM-DD_YYYY-MM-DD.csv`. CISO-007: ФИО только при `revealEmployeeNames=true`, иначе ключ/пусто. Скачивание файла НЕ автоматическое — из-за Remote DOM (нет Blob/anchor.download) CSV выводится в выделяемый текстовый блок (выдели → скопируй → вставь в .csv). Выгрузка в 1С — формат готов (BOM+`;`), интеграции нет. Печать — НЕ реализована (только браузерный Ctrl+P).

### Группа 4.1. Скачивание файла (host-bridge)

**C4.1.** Главное ограничение: в Remote DOM нельзя инициировать скачивание файла напрямую. Сейчас CSV показывается текстом для копирования вручную. Это критичный блокер для заказчика, или копирование-вставка временно приемлемо?
— Гипотезы: (а) критично, нужен реальный download; (б) копирование приемлемо как временное; (в) приемлемо только для маленьких отчётов, для табеля — нужен файл.

**C4.2.** Если нужен реальный download — каким путём? Host-bridge action в Twenty («скачать файл»), генерация файла на бэке с ссылкой, или отправка на email?
— Гипотезы: (а) host-bridge download (требует доработки SDK); (б) бэк генерит файл → ссылка для скачивания; (в) экспорт на email/в облако.
— Замечание: в SDK сейчас нет метода download — это follow-up по host-bridge.

**C4.3.** Нужны ли форматы, кроме CSV? XLSX (родной Excel, без проблем с кодировкой/разделителем), PDF (для печати/подписи табеля)?
— Гипотезы: (а) CSV достаточно; (б) добавить XLSX; (в) добавить XLSX+PDF (PDF для подписанного табеля).
— Реф: Timetta — экспорт XLSX/PDF; Kimai — CSV/XLSX/PDF/HTML.

### Группа 4.2. CSV detail-отчёт

**C4.4.** Detail CSV = 7 колонок. Не хватает ли заказчику: код проекта/этап/группа работ/категория/биллируемость/комментарий записи?
— Гипотезы: (а) 7 хватает; (б) добавить этап+комментарий; (в) полный набор (40+ полей как в detail-отчёте экрана).
— Замечание: detail-экран уже показывает 40+ полей в 6 группах — CSV можно расширить до того же набора или дать выбор колонок.

**C4.5.** Нужен ли выбор колонок перед экспортом (галочки «какие поля выгрузить») или фиксированный набор?
— Гипотезы: (а) фиксированный набор; (б) выбор колонок; (в) пресеты («для 1С» / «полный» / «краткий»).

**C4.6.** Разделитель сейчас `;` (RU-Excel/1С). У некоторых пользователей Excel настроен на `,`. Нужен ли выбор разделителя, или `;` — жёсткий стандарт для заказчика?
— Гипотезы: (а) `;` жёстко (1С-совместимость); (б) выбор `;`/`,`/Tab; (в) `;` по умолчанию + XLSX как решение проблемы.

— Гипотезы: (а) обезличенный код/id; (б) табельный номер; (в) пусто (как сейчас).

### Группа 4.3. Табель Т-13

**C4.8.** Табель Т-13 сейчас — CSV сотрудник×день (часы или буквенные коды). Заказчику нужен именно унифицированный бланк Т-13 (XLSX/PDF с шапкой формы, реквизитами организации, подписями), или CSV-сетки достаточно для импорта в 1С:ЗУП?
— Гипотезы: (а) CSV для импорта в 1С достаточно; (б) нужен печатный бланк Т-13 (PDF); (в) и CSV (импорт), и PDF (подпись).
— Реф: 1С:ЗУП — типовая печатная форма Т-13.

**C4.9.** Буквенные коды (Я/ОТ/Б/ДО/К/НН) — опциональны (`codes=true`). Должны ли они быть включены по умолчанию для кадровиков? Полон ли набор кодов (нужны ли РВ-работа в выходной, С-сверхурочные, Г-госо­бязанности, ПР-прогул)?
— Гипотезы: (а) текущих 6 кодов хватает; (б) включить коды по умолчанию; (в) расширить набор кодов Т-13 до полного.

**C4.10.** Приоритет в ячейке табеля: при наличии и часов, и отсутствия показывается «Я 8» (явка). Это правильно, или нужна двухстрочная запись (верх — код, низ — часы, как в бумажном Т-13)?
— Гипотезы: (а) одна запись «Я 8» ок; (б) двухстрочная как в бланке; (в) две half-ячейки.

**C4.11.** Когда задан projectId, коды отсутствий в табеле не показываются (отсутствия не привязаны к проекту). Это понятно, или табель Т-13 должен всегда быть «по сотруднику целиком» без проектного фильтра?
— Гипотезы: (а) понятно, по проекту — без отсутствий; (б) Т-13 всегда полный (игнорировать projectId); (в) предупреждать пользователя «табель по проекту неполный».

**C4.12.** Нужны ли подытоги в табеле: итого по отделу, явки/неявки по видам (Σ отпускных дней, Σ больничных), как в кадровой сводке?
— Гипотезы: (а) Итого по строке хватит; (б) добавить подытоги по отделу; (в) добавить сводку по кодам.

### Группа 4.4. Выгрузка в 1С

**C4.13.** Формат под 1С готов (BOM+`;`), но интеграции нет. Какой механизм обмена нужен: ручная выгрузка файла → импорт в 1С вручную, авто-выгрузка по расписанию, или REST/API-обмен 1С↔Time?
— Гипотезы: (а) ручной файл → ручной импорт; (б) выгрузка по расписанию в папку/обмен; (в) двусторонний API-обмен.
— Реф: 1С:ЗУП — обмен через типовые правила/EnterpriseData.

**C4.14.** Что именно выгружать в 1С: табель Т-13 (для расчёта ЗП), detail трудозатрат (для биллинга/себестоимости проектов), или и то, и другое?
— Гипотезы: (а) только Т-13; (б) только трудозатраты по проектам; (в) оба потока.

**C4.15.** Как сопоставлять сотрудников/проекты между Time и 1С при выгрузке? Через externalCode/externalId (они есть в моделях), ФИО, или нужен явный маппинг-справочник?
— Гипотезы: (а) externalCode/externalId; (б) по ФИО+табельному; (в) отдельный маппинг-справочник.

**C4.16.** Должна ли выгрузка отдавать только согласованные (APPROVED) записи, или все? Для расчёта ЗП обычно только утверждённые.
— Гипотезы: (а) только APPROVED; (б) все с пометкой статуса; (в) настраиваемо.

### Группа 4.5. Печать

**C4.17.** Печать не реализована (только браузерный Ctrl+P). Что заказчику реально нужно печатать — табель Т-13, сводный отчёт, таймшит сотрудника?
— Гипотезы: (а) только Т-13; (б) Т-13 + сводка по проектам; (в) печать не нужна (всё через файл/1С).

**C4.18.** Если печать нужна — через печатный PDF (генерится на бэке, сразу пригоден к подписи) или через print-friendly HTML-вид (Ctrl+P из браузера)?
— Гипотезы: (а) PDF с бэка; (б) print-CSS вид; (в) PDF для Т-13, print-CSS для остального.
— Замечание: Remote DOM не имеет прямого доступа к window.print — печать вероятно через host-bridge/PDF.

**C4.19.** Нужны ли в печатной форме реквизиты организации, подписи (руководитель/кадровик/табельщик), номер и дата документа — то есть полноценный официальный бланк?
— Гипотезы: (а) да, полный бланк с реквизитами и подписями; (б) данные без официального оформления; (в) реквизиты подставлять из настроек организации.

### Группа 4.6. Общие вопросы экспорта

**C4.20.** Имя файла сейчас `timesheet-detail_2026-06-01_2026-06-30.csv`. Нужно ли включать в имя отдел/проект/фильтр (чтобы при множественных выгрузках не путаться)?
— Гипотезы: (а) период достаточно; (б) добавить отдел/проект в имя; (в) добавить дату-время выгрузки.

**C4.21.** Нужен ли экспорт прямо с экрана отчёта (текущий отфильтрованный вид → CSV/XLSX «как вижу»), а не только отдельные detail/табель?
— Гипотезы: (а) detail+табель достаточно; (б) «экспорт текущего вида» с любого среза; (в) экспорт каждой таблицы отчёта.
— Реф: Kimai/Timetta — «export current view».

**C4.22.** Нужна ли защита/логирование экспорта PII (кто выгрузил отчёт с ФИО, когда) — учитывая CISO-007/152-ФЗ?
— Гипотезы: (а) не нужно; (б) логировать факт экспорта с ФИО; (в) требовать повышенных прав для экспорта с ФИО.
