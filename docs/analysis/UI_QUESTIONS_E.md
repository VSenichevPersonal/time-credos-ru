## E1. Микро-взаимодействия

**Корень обсуждается отдельно** (TIMESHEET_CELL_CONTROLS.md, симптом 1): контролы в `hour-cell` привязаны к `active`, а один клик = «выбрать + сразу edit», поэтому кнопки живут доли мс и мышью недостижимы. Вопросы ниже — глубже: про паттерны замены drag, inline-edit, hover-reveal, тайминги, фокус, copy/paste.

### Группа E1.A — замена drag (drag в Remote DOM невозможен)

**E1.1.** Каким жестом заменить **drag-fill** (протянуть значение по дням недели, как в Excel/Timetta)? Drag по координатам в Remote DOM недоступен.
— Гипотезы: (а) кнопка/глиф «продлить на будни строки» (есть сейчас как ⇥, но недостижим мышью); (б) пункт меню строки ⋯ «Заполнить будни нормой/значением»; (в) «выделить диапазон ячеек кликами Shift+клик → одно значение на все»; (г) ввод в одну ячейку + чекбокс «применить ко всем будням».
— Реф: Timetta drag-fill заменяем на «Расставить по расписанию» (кнопка, не drag); Kimai «Fill week» в контекст-меню строки. impeccable interaction: «не полагаться на жест как единственный способ — всегда видимый fallback».

**E1.2.** Нужно ли **перетаскивание строк** (изменить порядок проектов/видов работ)? Drag-reorder невозможен.
— Гипотезы: (а) не нужно — порядок фиксирован (по проекту/алфавиту); (б) кнопки ↑/↓ в меню строки ⋯; (в) сортировка кликом по заголовку колонки.
— Реф: Timetta — строки сортируются автоматически, ручного reorder нет. Kimai — нет drag-reorder строк таймшита.

**E1.3.** Нужно ли **выделение диапазона ячеек** (выбрать Пн–Пт одной строки и задать значение / удалить разом)? Без drag — только клики.
— Гипотезы: (а) не нужно (bulk-fill покрывает); (б) Shift+клик задаёт диапазон от активной до кликнутой; (в) клик по заголовку дня выделяет колонку (все строки этого дня).
— Реф: табличный стандарт (Excel/Sheets) — Shift+клик диапазон; в Remote DOM реализуемо на координатах ячейки `(row,col)`, host-DOM не нужен.

### Группа E1.B — модель клик/выбор/редактирование

**E1.4.** Клик по ячейке = сразу inline-ввод (как решено в волне-1, F2) — но нужно ли **двойному клику** дать отдельную роль (напр. открыть форму записи с проектом/видом работ/комментарием)?
— Гипотезы: (а) нет, клик=ввод, всё; (б) двойной клик = расширенная форма дня (часы + комментарий + теги в одном поповере); (в) двойной клик = выделить весь текст в input для перезаписи.
— Реф: Timetta — клик = inline-ввод; расширенная карточка записи открывается отдельной иконкой, не двойным кликом.

**E1.5.** При входе в edit по клику — **курсор в конец** значения или **выделить всё** (чтобы перезапись цифрой стирала старое)? Сейчас `setDraft(fmtHours(value))`, поведение курсора не задано явно.
— Гипотезы: (а) выделить всё (быстрая перезапись — частый кейс); (б) курсор в конец (дозапись); (в) при печати цифры (seed) — заменить, при клике — выделить.
— Реф: Excel/Sheets — клик ставит курсор, печать стирает (F2 vs прямой ввод). Timetta — клик выделяет значение целиком.

**E1.6.** Поведение на **commit с невалидным вводом** (parseHours вернул null): сейчас тихо откатывает к старому значению (`setDraft(fmtHours(value))`) и НЕ выходит из edit. Это ожидаемо, или показать ошибку?
— Гипотезы: (а) тихий откат (сейчас); (б) красная рамка input + tooltip «введите часы: 8 / 8.5 / 8:30»; (в) shake-микроанимация + откат.
— Реф: impeccable interaction — «8 интерактивных состояний», error state не должен быть тихим. Но для частого ввода тихий откат менее назойлив.

### Группа E1.C — inline-edit паттерны

**E1.7.** Какие **форматы ввода** показывать как принимаемые (parseHours принимает `8`, `8.5`, `8,5`, `8:30`, `1ч30м`, `90м`)? Сейчас формат нигде не подсказан в UI.
— Гипотезы: (а) placeholder в пустой ячейке-input «8 / 8:30»; (б) подсказка в cheatsheet (?); (в) tooltip при первом фокусе; (г) не подсказывать (предполагать, что пользователь знает).
— Реф: Kimai/Timetta принимают `8`, `8.5`, `8:30`, но НЕ `1ч30м` — у нас формат богаче, тем важнее его проявить.

**E1.8.** **Комментарий к ячейке** (`description`) — открывать inline под input при вводе (Kimai-style) или вынести в меню строки/двойной клик? Сейчас иконка ✎ в ячейке недостижима мышью.
— Гипотезы: (а) поле комментария появляется под активной ячейкой в режиме edit; (б) пункт меню строки «Комментарий к [Пн]…»; (в) только в режиме День (где работает); (г) расширенная карточка записи.
— Реф: Timetta — комментарий через иконку при hover строки (не ячейки); Kimai — отдельный «+» → modal.

**E1.9.** Нужны ли **preset-чипы** (8ч / 4ч / последнее значение) в момент ввода? В волне-1 (F2) решено «оставить чип нормы, но показывать при фокусе input снизу» — нужны ли ещё пресеты кроме нормы дня?
— Гипотезы: (а) только норма дня; (б) норма + «повтор последнего» (↩); (в) настраиваемые пользователем пресеты (Kimai); (г) без пресетов, только ввод.
— Реф: Kimai — настраиваемые preset chips в панели снизу при фокусе.

### Группа E1.D — hover-reveal vs всегда-видимое (корень бага контролов)

**E1.10.** Действия строки (дублировать / заполнить / очистить / коммент) — **всегда видеть** триггер ⋯ или показывать **при наведении на строку**? Сейчас ⋯ всегда видим (надёжно), но «съедает» место.
— Гипотезы: (а) ⋯ всегда видим (сейчас, надёжно под Remote DOM); (б) показывать ⋯ при hover строки (JS-onMouseEnter, экономит место, но flaky); (в) ⋯ виден только у активной строки.
— Реф: Timetta — ряд кнопок появляется при hover строки. impeccable interaction: hover-only действия невидимы для клавиатуры/touch — нужен фокус-эквивалент. Под Remote DOM «всегда видимый» надёжнее.

**E1.11.** Норма-чип/действия в ячейке: окончательно **убрать из ячейки** (вариант C1) и держать только в меню строки + в момент edit? (Это закрывает корневой баг.)
— Гипотезы: (а) да, ячейка = только число/input (C1); (б) оставить, но показывать ТОЛЬКО когда input открыт, панелью снизу (C3); (в) оставить, увеличив ячейку до 72×40 (C2).
— Реф: волна-1 F4 рекомендует C1 основным; C3 — следующая волна.

**E1.12.** Индикатор «у ячейки есть комментарий» (точка 5px) — оставить как пассивный always-visible маркер? Достаточно ли он заметен и не путается ли с другими точками (save-индикатор, статус)?
— Гипотезы: (а) точка в углу (сейчас); (б) уголок-«загиб» ячейки (corner fold); (в) подчёркивание числа; (г) фоновый тинт ячейки.
— Реф: Timetta/Kimai — маленький dot в углу ячейки (совпадает с текущим).

### Группа E1.E — тайминг тултипов

**E1.13.** Сейчас все подсказки — нативный `title` (мгновенно по hover браузера, недоступен на touch/клавиатуре; 76 `title` против 24 `aria-label` — IMPECCABLE_AUDIT). Нужна ли **управляемая задержка** появления tooltip (напр. 400–600 мс), чтобы не мелькали при проходе курсора?
— Гипотезы: (а) оставить нативный title (0 задержки, но недоступен); (б) свой JS-tooltip с задержкой ~500мс (но host-DOM позиционирования нет → сложно); (в) подсказки-hint текстом под контролом (как в меню ⋯/копировать — уже есть); (г) ничего не менять.
— Реф: impeccable interaction — recognition not recall; title недоступен на touch. impeccable motion — tooltip 200–300мс на появление состояния.

**E1.14.** Нужен ли tooltip на **read-only (locked) ячейке** объясняющий ПОЧЕМУ нельзя править («Согласовано — отзовите согласование, чтобы изменить»)? Сейчас title = «Согласовано — правка запрещена», без «как разблокировать».
— Гипотезы: (а) добавить «как отозвать»; (б) оставить кратким; (в) показывать действие «Отозвать согласование» в меню строки locked-строки.
— Реф: impeccable — «No permissions» empty/blocked state объясняет why + how.

### Группа E1.F — focus-поведение

**E1.15.** После **commit (Enter/Tab)** фокус уходит на соседнюю ячейку (`onKey` → nav). А после commit по **blur (клик мимо)** — куда девается active/фокус? Сейчас blur коммитит и `setEditing(false)`, но active-координата не сбрасывается. Должна ли подсветка active оставаться?
— Гипотезы: (а) active остаётся (видно «где я был»); (б) клик вне сетки снимает active; (в) клик по другой ячейке переносит active туда.
— Реф: Sheets — клик вне таблицы снимает выделение; Timetta — выделение остаётся до следующего клика.

**E1.16.** При **смене недели / режима** (День↔Неделя↔Проект) куда ставить фокус? Сейчас автофокус первой пустой ячейки только при первой загрузке (`autofocusedRef`, один раз). После навигации по неделям автофокуса нет.
— Гипотезы: (а) автофокус первой пустой ячейки при каждой смене недели; (б) сохранять позицию `(row,col)` между неделями; (в) не автофокусить вовсе (мышь-приоритет); (г) фокус на «сегодня».
— Реф: Timetta — фокус на «сегодня» при открытии недели.

**E1.17.** Виден ли **focus-ring** при заходе в виджет клавиатурой (Tab с внешней страницы Twenty в наш виджет)? Активная ячейка — это div `tabIndex={-1}`, не попадает в Tab-порядок host-а; навигация только внутренней `use-keyboard`. Как пользователь понимает, что фокус «внутри сетки»?
— Гипотезы: (а) при входе подсветить первую ячейку; (б) видимая рамка вокруг всего виджета при фокусе; (в) подсказка «нажмите стрелку для навигации».
— Реф: impeccable interaction — focus ring обязателен, видимый при keyboard-входе; roving tabindex для групп.

### Группа E1.G — копировать / вставить значения

**E1.18.** Нужна ли **Ctrl/Cmd+C / Ctrl/Cmd+V** по ячейкам (скопировать часы из одной ячейки/строки, вставить в другую)? Доступ к системному clipboard из Remote DOM ограничен.
— Гипотезы: (а) не нужно (есть «копировать прошлую неделю» + bulk-fill); (б) внутренний clipboard виджета (запомнить значение в React-стейте, вставить Cmd+V в активную); (в) «копировать строку» дублирует строку (уже есть в ⋯).
— Реф: Timetta `copyLines*` — копирование на уровне строк, не ячеек; системный clipboard не используется. Excel — Ctrl+C/V ячеек (но это host-clipboard, нам недоступен напрямую).

**E1.19.** «Копировать прошлую неделю» (CopyMenu: только строки / строки+часы) — заказчик пользуется (волна-1 «да»). Нужна ли **выборочная вставка** (только конкретные дни, или только определённые проекты)?
— Гипотезы: (а) нет, копировать всё (сейчас); (б) выбрать дни-источники; (в) копировать из любой прошлой недели (не только предыдущей), с выбором даты.
— Реф: Timetta «копировать из недели …» с выбором источника.

**E1.20.** Нужен ли **«повтор последнего значения»** в строке (вставить значение, введённое в предыдущую ячейку этой же строки, одним нажатием)?
— Гипотезы: (а) не нужно; (б) клавиша (напр. Ctrl+D «заполнить вниз/вправо»); (в) preset-чип «↩ N» при фокусе (см. E1.9).
— Реф: Excel Ctrl+D/Ctrl+R; Kimai preset «повтор».

### Группа E1.H — обратная связь действия

**E1.21.** **Optimistic UI**: при вводе часов значение применяется сразу, save-индикатор «Сохранение…→Сохранено». При ошибке сохранения — что видит пользователь? Сейчас save-индикатор «Не сохранено» (терракот), но значение в ячейке уже изменено. Откатывать ли значение?
— Гипотезы: (а) оставить значение + «Не сохранено» + кнопка «Повторить»; (б) откатить к серверному; (в) пометить ячейку рамкой «не сохранено».
— Реф: impeccable interaction — optimistic для low-stakes, rollback при failure; но destructive/важное — не оптимистично.

**E1.22.** Деструктивные действия («Очистить неделю», «Удалить строку», «Очистить строку») — сейчас «Очистить неделю» под confirm (CopyMenu), «Удалить строку» — без confirm (меню ⋯). Нужен ли **undo-тост** вместо confirm (impeccable: undo > confirm)?
— Гипотезы: (а) undo-тост «Строка удалена · Отменить» (5с); (б) confirm на всё деструктивное; (в) текущая смесь (confirm на неделю, мгновенно на строку).
— Реф: impeccable interaction — «Undo > Confirm»; confirm только для необратимого/batch. «Очистить неделю» (batch) → confirm оправдан; «удалить пустую строку» → undo лучше.

## E2. Motion / анимации

**Текущее состояние (из IMPECCABLE_AUDIT + код):** переходы только на `color`/`background`/`box-shadow` (не на layout-свойствах — правило impeccable соблюдено), длительности 120–160мс, `save-indicator` имеет `transition: color 160ms ease-out`, `approval-bar` использует `cubic-bezier(0.22,1,0.36,1)` (ease-out-quint). Скелетон/shimmer НЕТ — загрузка показывается текстом «Загрузка…» (`Center`). Анимаций появления поповеров/меню/тостов НЕТ (мгновенно). `prefers-reduced-motion` не обрабатывается (нет доступа к media-query через host).

### Группа E2.A — что анимировать / что НЕ анимировать

**E2.1.** Нужна ли **анимация появления поповеров/меню** (cheatsheet, row-menu, copy-menu, reject-поповер, cell-comment)? Сейчас появляются мгновенно.
— Гипотезы: (а) оставить мгновенно (плотный инструмент, скорость > украшение); (б) fade+lift 150–200мс ease-out (impeccable: state changes 200–300мс); (в) только fade opacity (дешевле, без layout).
— Реф: impeccable motion — menu/tooltip 200–300мс ease-out; но «animation fatigue is real», для инструмента сдержанно.

**E2.2.** Подтверждаем правило **НЕ анимировать layout** (`width/height/top/left/margin`) — все переходы только opacity/color/box-shadow/transform? Это снимает риск дёрганья таблицы.
— Гипотезы: (а) да, жёсткое правило (соблюдается); (б) разрешить grid-template/FLIP для разворачивания строк, если понадобится.
— Реф: impeccable motion — «avoid animating layout-driving properties»; FLIP/grid-template-rows допустимы для reflow.

**E2.3.** Нужен ли **микро-feedback при commit ячейки** (короткая вспышка фона/чек 100мс «сохранено»)? Сейчас обратная связь только в общем save-индикаторе тулбара (не у самой ячейки).
— Гипотезы: (а) нет, save-индикатор достаточно; (б) короткий тинт ячейки 100–150мс при успешном save; (в) галочка-флеш в углу ячейки.
— Реф: impeccable motion — 100–150мс instant feedback; 80мс порог «мгновенно».

### Группа E2.B — easing / длительности

**E2.4.** Унифицировать **easing-токен**? IMPECCABLE_AUDIT #15: часть hover-переходов на `ease` (банится impeccable), часть на ease-out-quint. Свести к одному ease-out-quart `cubic-bezier(0.25,1,0.5,1)`?
— Гипотезы: (а) один токен `--ease-out-quart` для всех входов; (б) два токена (вход ease-out, выход ease-in); (в) оставить как есть (непринципиально).
— Реф: impeccable motion — «Don't use ease»; рекомендованный дефолт ease-out-quart; exit = ~75% от enter.

**E2.5.** Целевые **длительности** для нашего инструмента: hover/feedback 100–150мс, открытие меню/тоста 200мс, без entrance-анимаций >300мс. Подтверждаем «быстро и сдержанно»?
— Гипотезы: (а) 100/200, без длинных; (б) чуть живее (150/250); (в) минимум анимаций вообще.
— Реф: impeccable motion 100/300/500; для плотного product-инструмента — нижняя граница.

### Группа E2.C — переходы между режимами/вкладками

**E2.6.** Переключение режимов **День ↔ Неделя ↔ Проект** (mode-switcher) и вкладок «Мои часы ↔ Мои периоды» (Segmented) — нужен ли переход (crossfade содержимого) или мгновенная замена?
— Гипотезы: (а) мгновенно (сейчас); (б) crossfade 150мс (opacity, без layout); (в) slide (но это layout — против правила).
— Реф: impeccable motion — state toggle ease-in-out; но crossfade (opacity) предпочтительнее slide для таблиц.

**E2.7.** Анимация **подсветки активной ячейки** при переходе стрелками (сейчас `box-shadow inset` появляется мгновенно)? Плавный перенос рамки между ячейками?

### Группа E2.E — reduced-motion / чего нет

— Реф: impeccable motion — «Avoid animating everything»; анимировать то, что несёт смысл (появление сообщения = заметить его).

## E4. Горячие клавиши

**Текущий cheatsheet (`cheatsheet.tsx`, по «?»):** `↑↓←→` перемещение, `0–9` начать ввод, `Enter` подтвердить и вниз, `Tab` подтвердить и вправо, `Shift+Tab` влево, `Esc` отмена ввода, `0/Del` удалить запись, `Shift+Enter` часы на все будни строки, `?` — подсказка. Инлайн-поповер, не модалка; закрывается кликом-вне (НЕ по Esc — IMPECCABLE_AUDIT #9). `use-keyboard.keyAction` реально обрабатывает: стрелки, Tab/Shift+Tab, Enter (=вниз), цифры/`.`/`,` (seed), Delete/Backspace (=seed `0`). Глобальных шорткатов (открыть «?», сменить неделю, переключить режим, submit) — НЕТ; всё это только мышью. Слушателей на `window` нет (Remote DOM).

### Группа E4.A — навигация (стрелки / Tab / Enter / Esc)

**E4.1.** **Enter = вниз** (Excel-style, текущее). Подтверждаем, или Enter должен подтверждать на месте (без перехода)?
— Гипотезы: (а) Enter вниз (сейчас); (б) Enter вправо; (в) Enter подтвердить + остаться; (г) настраиваемо.
— Реф: табличный стандарт — Tab вправо, Enter вниз (совпадает с текущим).

**E4.2.** **Tab** — сейчас вправо в пределах строки, упирается в последнюю колонку (clamp, не переносит на след. строку). Нужен ли **перенос на следующую строку** после последнего дня?
— Гипотезы: (а) clamp (сейчас); (б) wrap на начало след. строки (как в формах); (в) Tab выходит из сетки на след. контрол виджета.
— Реф: Timetta — Tab слева-направо с переносом на след. строку.

**E4.3.** **Esc** — сейчас отменяет ТОЛЬКО ввод в ячейке. Нужен ли единый Esc: закрыть любой открытый поповер/меню (cheatsheet Esc не закрывает — IMPECCABLE_AUDIT #9), снять active?
— Гипотезы: (а) Esc закрывает текущий слой (input → меню → снять выделение), каскадом; (б) Esc только отмена ввода (сейчас); (в) Esc везде закрывает верхний слой (унифицировать поповеры).
— Реф: impeccable interaction — ESC закрывает overlay (нативно у dialog/popover, у нас вручную); единообразие выхода.

**E4.4.** **Home/End/PageUp/PageDown** — нужны ли? (Home → первый день строки, End → последний/Итого, PageDown → след. неделя).
— Гипотезы: (а) не нужны; (б) Home/End по строке; (в) PageUp/Down = смена недели; (г) Ctrl+Home = первая ячейка таблицы.
— Реф: Excel/Sheets — Home/End/PageUp/Down стандартны; для плотной сетки полезны.

### Группа E4.B — быстрый ввод

**E4.5.** **Shift+Enter** заявлен в cheatsheet как «часы на все будни строки», но в `use-keyboard.keyAction` НЕ обработан (Enter всегда = move-down, shift не различается для Enter). Это **расхождение cheatsheet vs реализация** — Shift+Enter нужен и его доделать, или убрать из подсказки?
— Гипотезы: (а) реализовать Shift+Enter = заполнить будни строки текущим значением; (б) убрать из cheatsheet (вводит в заблуждение); (в) заменить на другую комбинацию.
— Реф: расхождение док-vs-код; impeccable — подсказка не должна обещать несуществующее.

**E4.6.** Нужны ли **быстрые клавиши ввода нормы / повтора**: напр. `N` = норма дня в активную ячейку, `Ctrl+D` = повторить значение слева/сверху?
— Гипотезы: (а) не нужно (печать цифры достаточно); (б) `Ctrl+D`/`Ctrl+R` заполнить вниз/вправо (Excel); (в) клавиша «норма дня».
— Реф: Excel Ctrl+D/Ctrl+R; экономит ввод при повторяющихся часах.

**E4.7.** **Delete/Backspace = удалить запись** (seed `0` → commit 0). Подтверждаем, что обнуление = удаление записи (а не «0 часов» как валидное значение)? И нужно ли undo на это?
— Гипотезы: (а) Del = удалить (сейчас, 0 ⇒ нет записи); (б) Del = 0 (значение), отдельный жест на удаление; (в) Del + undo-тост.
— Реф: Timetta — очистка ячейки удаляет аллокацию; impeccable — undo для деструктивного.

### Группа E4.C — глобальные шорткаты

**E4.8.** Нужны ли **глобальные шорткаты виджета**: `?` открыть подсказку (сейчас только клик по «?»), `←/→` или `[`/`]` смена недели, `1/2/3` переключение День/Неделя/Проект, `Ctrl+Enter` отправить на согласование?
— Гипотезы: (а) добавить набор (?, смена недели, режимы, submit); (б) только `?` и submit; (в) ничего глобального (всё мышью).
— Реф: Timetta/Harvest — `?` открывает шорткаты; стрелки/скобки для периода. Ограничение: глобальный keydown ловим на контейнере виджета (есть `onContainerKeyDown` в week-grid), не на window.

**E4.9.** Конфликт: **стрелки** уже заняты навигацией по ячейкам. Для смены недели нужна др. комбинация (`[`/`]`, `Ctrl+←/→`, `PageUp/Down`)?
— Гипотезы: (а) `[` / `]`; (б) `Ctrl/Alt+←/→`; (в) `PageUp/PageDown`.
— Реф: распространено `[`/`]` или Ctrl+стрелки для смены периода, чтобы не конфликтовать с навигацией.

**E4.10.** Где **ловить глобальные клавиши** при отсутствии window-слушателей? Контейнер виджета (`onKeyDown` на корневом div) ловит только когда фокус внутри. Это приемлемо, или нужны клавиши, работающие даже без фокуса в сетке?
— Гипотезы: (а) контейнерный keydown достаточно (фокус обычно в виджете); (б) сделать корневой div focusable (tabIndex=0) и автофокусить при входе; (в) ограничиться клавишами уровня ячейки.
— Реф: Remote DOM — нет `window.addEventListener('keydown')`; только React-обработчики в дереве виджета.

### Группа E4.D — обнаруживаемость

**E4.11.** Как пользователь **узнаёт, что есть горячие клавиши**? Сейчас только кнопка «?» в тулбаре (мышь). Добавить ли подсказку «Нажмите ? для горячих клавиш» / hint у поля?
— Гипотезы: (а) hint-текст «? — горячие клавиши» в пустом состоянии/при первом входе; (б) сделать `?` рабочей клавишей (открывает cheatsheet); (в) оставить только кнопку.
— Реф: impeccable onboard — «keyboard shortcut hints (⌘K shown on search box)»; обнаруживаемость через point-of-use.

**E4.12.** Стоит ли **показывать шорткат рядом с действием** (в меню ⋯ / в тулбаре): напр. «Заполнить будни (Shift+Enter)», «Отправить (Ctrl+Enter)»?
— Гипотезы: (а) да, дописать комбинации в подписи пунктов меню/кнопок; (б) только в cheatsheet; (в) не показывать.
— Реф: impeccable — шорткаты у действий повышают adoption; стандарт desktop-приложений.

**E4.13.** Cheatsheet сейчас закрывается **только кликом-вне**, не по Esc (IMPECCABLE_AUDIT #9). Добавить Esc-close для единообразия со всеми поповерами?
— Гипотезы: (а) да, Esc закрывает (унификация); (б) и Esc, и клик-вне (стандарт поповера); (в) оставить как есть.
— Реф: impeccable interaction — overlay закрывается по Esc; сейчас autocomplete/filter-chip закрываются по Esc, cheatsheet — нет (непоследовательно).

**E4.14.** Нужна ли **полная справка по клавишам в одном месте** (cheatsheet) синхронизированная с реальной реализацией (`use-keyboard`)? Сейчас уже расходятся (Shift+Enter в подсказке, нет в коде — E4.5). Сделать ли cheatsheet генерируемым из единого источника (SSOT клавиш)?
— Гипотезы: (а) SSOT-список клавиш → и cheatsheet, и обработчик из одного массива; (б) ручная синхронизация (риск дрейфа); (в) тесты на соответствие cheatsheet↔keyAction.
— Реф: SSOT-дисциплина проекта (как category-meta/tag-meta); предотвращает расхождение док-vs-код.

- **title-as-tooltip** (76 title vs 24 aria-label, IMPECCABLE_AUDIT): подсказки недоступны на touch/клавиатуре — E1.13.
- **Cheatsheet не закрывается по Esc** (IMPECCABLE_AUDIT #9) — E4.13.
- **Нет хранилища «seen»-флагов** в Remote DOM (нет localStorage) — блокирует классический онбординг, нужен серверный канал (E3.5).
- **prefers-reduced-motion недоступен** через host — нужен собственный «спокойный режим» или минимум анимаций (E2.10).

**Ограничение-инвариант для всех ответов:** Remote DOM — нет drag, нет надёжного hover/host-DOM, нет window-keydown, нет localStorage, нет нативных dialog/popover. Любое решение проектируется под это.
