## G1. Поиск

**Зона кода:** `grid/autocomplete.tsx`, `grid/filter-chip.tsx`, `grid/filters-bar.tsx`, `grid/use-filters.ts`, `grid/add-row.tsx`, `shared/use-drill.ts`, `shared/breadcrumbs.tsx`.

**Текущее состояние (факты из кода):**
- **Глобального поиска (Cmd+K / Command Menu) НЕТ** в time-приложении. В платформе Twenty CommandMenu есть, но в виджеты time не проброшен.
- **Поиск внутри фильтр-дропдаунов ЕСТЬ** (`filter-chip.tsx`): плейсхолдер «Поиск…», substring-match `label.toLowerCase().includes(q)`, case-insensitive, **без debounce** (срабатывает сразу).
- **Автокомплит Проект/Вид работ ЕСТЬ** (`autocomplete.tsx` в `add-row.tsx`): substring-match, недавние (recentIds) поднимаются под заголовком «Недавние», навигация ↑↓/Enter/Tab/Esc, высота меню ограничена 260px (MAX_MENU_HEIGHT), **явного лимита числа вариантов нет**, debounce нет. Заглушки «Ничего не найдено» / «Нет доступных вариантов».
- **Быстрый переход через drill-down ЕСТЬ** (`use-drill.ts` + `breadcrumbs.tsx`): стек уровней в памяти (без URL), возврат кликом по крошке. Только для отчётов/планирования, не по записям.
- **Сохранённых фильтров НЕТ** (в research отмечены как нужные — Timetta «Сохранённый фильтр», Kimai Bookmarks).

### Группа G1.1 — Глобальный поиск

**G1.1.** Нужен ли вообще глобальный поиск (Cmd+K / отдельная строка) в time-приложении, или хватает фильтров+автокомплита?
— (а) не нужен, фильтров достаточно; (б) нужен лёгкий — только проекты+сотрудники; (в) полноценный Cmd+K с переходом к записи/проекту/отчёту; (г) нужен, но как кнопка-строка в шапке, не хоткей (нет глобальных слушателей клавиш в Worker).
— реф: Twenty — Command Menu Cmd+K; Timetta/Kimai — глобального нет, на каждом списке локальный «Поиск…».

**G1.2.** Если глобальный поиск нужен — что он ищет?
— (а) только проекты; (б) проекты + сотрудники + отделы; (в) + записи времени (по комментарию/виду работ); (г) + отчёты/страницы навигации.
— реф: Timetta — поиск ограничен сущностями списка; Kimai — поиск в пределах раздела.

**G1.3.** Хоткей Cmd+K технически невозможен внутри Worker (нет глобальных слушателей). Где разместить точку входа поиска?
— (а) видимая строка «Поиск…» в шапке таймшита; (б) иконка-лупа, раскрывается в инпут; (в) отказаться от хоткея, только клик.
— реф: Kimai — поле «Поиск…» всегда видимо над списком.

### Группа G1.2 — Поиск по таймшиту / проектам / сотрудникам

**G1.4.** Нужен ли поиск по содержимому таймшита (найти запись по комментарию / виду работ / проекту в текущей сетке)?
— (а) нет, сетка маленькая; (б) да, подсветка совпадений в строках; (в) да, через фильтр Проект/Вид работ (уже есть), отдельного не надо.
— реф: Kimai — поиск по записям внутри раздела Timesheet.

**G1.5.** Поиск сотрудников (руководитель) — как искать при большом штате?
— (а) автокомплит в фильтре Сотрудник (как сейчас, substring); (б) + группировка по отделу в дропдауне; (в) + поиск по табельному номеру/email, не только ФИО.
— реф: Kimai — фильтр User с поиском; Timetta — выбор сотрудника в списке.

**G1.6.** Поиск проектов — нужен ли поиск по коду проекта / клиенту, а не только по названию?
— (а) только название (как сейчас); (б) + код проекта; (в) + клиент/заказчик; (г) + «мои проекты» приоритетом.
— реф: Kimai — поле Проект с автоподстановкой Клиента.

**G1.7.** Должен ли поиск/фильтр сотрудника учитывать только активных (не уволенных) по умолчанию?
— (а) только активные, уволенные по галке; (б) все вперемешку; (в) активные сверху, уволенные внизу серым.
— реф: Timetta — фильтр по статусу ресурса.

### Группа G1.3 — Поиск в фильтр-дропдаунах (автокомплит)

**G1.8.** Поиск в чипе сейчас substring (включая середину слова). Достаточно ли, или нужен приоритет совпадений «с начала»?
— (а) substring как сейчас; (б) сначала «начинается с», потом «содержит»; (в) fuzzy-поиск (опечатки).
— реф: Timetta/Kimai — обычно prefix+contains.

**G1.9.** Нужен ли debounce при вводе в поиск (сейчас фильтр мгновенный)?
— (а) мгновенный ок (данные локальные); (б) debounce 150–300мс если список большой; (в) зависит — для серверного поиска да, для локального нет.
— реф: типовой UX — debounce при сетевом поиске.

**G1.10.** При большом числе значений (сотни проектов) — нужен ли лимит вывода + «показать ещё» / счётчик «найдено N»?
— (а) показывать всё (скролл в 260px); (б) лимит 50 + «ещё N»; (в) счётчик «найдено N из M».
— реф: Kimai — пагинация/лимит в больших списках.

**G1.11.** Недавние («Недавние» сверху) работают в автокомплите add-row. Нужны ли недавние/частые и в фильтр-чипах?
— (а) нет, в фильтрах не нужно; (б) да, недавно выбранные сверху; (в) «избранные» проекты закреплять.
— реф: Kimai — Recent activities.

**G1.12.** Множественный выбор в чипе (Set). Нужен ли «Выбрать все» / «Снять все» / инверсия внутри дропдауна?
— (а) только ручной выбор; (б) + «Выбрать все найденные»; (в) + «Снять все».
— реф: Kimai — мультиселект Project/Activity/Tags.

### Группа G1.4 — Сохранённые фильтры и быстрый переход

**G1.13.** Сохранённых фильтров нет, в research отмечены нужными. Внедрять?
— (а) не нужно; (б) да, именованные пресеты per-пользователь; (в) + общие пресеты от руководителя для команды.
— реф: Timetta «Сохранённый фильтр» (dropdown на гриде); Kimai Bookmarks.

**G1.14.** Если сохранённые фильтры — где хранить и как показывать?
— (а) дропдаун «Фильтр ▾» рядом с FiltersBar; (б) чипы-пресеты строкой; (в) дефолтный фильтр на открытии.
— реф: Timetta — dropdown «Сохранённый фильтр» на каждом списке.

**G1.15.** Drill-down (breadcrumbs) — стек в памяти, без URL. Нужно ли сохранять путь в URL/состоянии (чтобы поделиться/обновить страницу)?
— (а) память ок; (б) в URL для шаринга и F5; (в) только последний срез запоминать per-сессия.
— реф: Timetta — OLAP drill с сохранением среза.

**G1.16.** Быстрый переход «к проекту»/«к сотруднику» из строки таймшита/отчёта (клик на имя → карточка/фильтр)?
— (а) не нужно; (б) клик на проект → фильтр по проекту; (в) клик → drill в детализацию; (г) клик → карточка проекта/сотрудника.
— реф: Timetta — клик по строке открывает деталь.

**G1.17.** Нужна ли история поиска / последние запросы при повторном открытии поиска?
— (а) нет; (б) да, 5 последних; (в) только если есть глобальный поиск.
— реф: типовой UX command-palette.

**G1.18.** Пустой результат поиска — какое поведение помимо «Ничего не найдено»?
— (а) только текст (как сейчас); (б) + кнопка «Сбросить фильтр»; (в) + подсказка «создать проект/строку».
— реф: Kimai — пустое состояние с действием.

**G1.19.** Подсветка совпадения (выделение введённой подстроки в найденных пунктах) — нужна?
— (а) нет; (б) да, жирным/цветом; (в) только в автокомплите add-row.
— реф: типовой автокомплит.

**G1.20.** Должен ли поиск в фильтрах искать сразу по нескольким полям (например проект ищем и по названию, и по коду в одном инпуте)?
— (а) одно поле — одно значение; (б) мульти-поле в одном инпуте; (в) переключатель «искать в: название/код».
— реф: Kimai — поиск по нескольким атрибутам.

## G2. Массовые операции

**Зона кода:** `grid/row-menu.tsx`, `grid/toolbar.tsx`, `grid/copy-menu.tsx`, `grid/use-timesheet-actions.ts`, `grid/use-keyboard.ts`, `reports/approval-bar.tsx`, `logic-functions/approval.logic.ts`.

**Текущее состояние (факты из кода):**
- **Выделения строк/записей чекбоксами НЕТ.** Мультиселекта строк нет. Фильтры скрывают/показывают, но не выделяют.
- **batch-edit (смена проекта/статуса/вида работ пакетом) НЕТ.** Правка только построчно.
- **batch-approve ЕСТЬ** (`approval.logic.ts`, `approval-bar.tsx`): approve/reject **всех SUBMITTED сразу** (не выборочно). Reject требует комментарий. submit — все DRAFT периода → SUBMITTED. Только руководитель; нельзя согласовывать свои записи (CISO-003).
- **«Применить ко всем» / fill ЕСТЬ:** «Заполнить будни (8ч)» построчно (`bulkFill`), «Заполнить 8×5» по всей неделе (`fillStandardWeek`), «Копировать прошлую неделю» (структура / структура+часы), «Очистить строку» / «Очистить неделю». Все, кроме очистки, **не перетирают** заполненные и согласованные ячейки.
- **Массовое удаление:** только целая строка (`clearRow`) или вся неделя (`clearWeek`, с confirm). Согласованные (APPROVED) не удаляются (CISO-012).
- **Выбор диапазона ячеек (Excel-style) НЕТ.** Есть Shift+Enter (значение ячейки → все будни строки), Tab/Arrows-навигация.

### Группа G2.1 — Выделение строк/записей

**G2.1.** Нужны ли чекбоксы выделения строк в таймшите/отчёте для пакетных действий?
— (а) не нужны; (б) да, inline-чекбоксы в первой колонке (поповеры/порталы в Worker недоступны — селекция должна быть inline); (в) да, но только в режиме «согласование».
— реф: Kimai — чекбоксы строк для bulk-actions.

**G2.2.** Где выделение нужнее — в моём таймшите, в согласовании руководителя, в отчётах?
— (а) только согласование; (б) согласование + отчёты; (в) везде.
— реф: Timetta — выделение в списках на согласование.

**G2.3.** Диапазон строк Shift+Click технически возможен (локальный обработчик в виджете). Внедрять или хватит «выбрать все»?
— (а) только «выбрать все/найденные»; (б) + Shift+Click диапазон; (в) + Ctrl+Click отдельные.
— реф: Kimai — выбор диапазона строк.

**G2.4.** Нужна ли панель действий над выделением (sticky-бар «Выбрано N: [действие] [действие]»)?
— (а) не нужна; (б) да, появляется при ≥1 выделенной; (в) да, фиксируется внизу виджета.
— реф: Kimai — bulk-action toolbar при выделении.

### Группа G2.2 — Batch-edit

**G2.5.** Нужна ли массовая смена проекта у выделенных записей (перенести часы в другой проект)?
— (а) нет; (б) да, у выделенных строк; (в) да, через RowMenu «Перенести в проект…».
— реф: Timetta — перенос записей между проектами.

**G2.6.** Массовая смена вида работ / категории у выделенных?
— (а) нет; (б) да, вид работ; (в) да, вид работ + категория.
— реф: Kimai — bulk-edit Activity.

**G2.7.** Массовая смена статуса (например вернуть из SUBMITTED в DRAFT пачкой)?
— (а) нет, статус только через approve-flow; (б) да, «отозвать с согласования» пачкой; (в) да, любой переход для руководителя.
— реф: Timetta — массовый возврат на доработку.

**G2.8.** Массовое проставление тегов выделенным записям?
— (а) нет тегов в batch; (б) да, добавить тег; (в) добавить/снять тег.
— реф: Kimai — bulk Tags.

**G2.9.** Batch-edit — модалка не помещается в Worker-виджет (нет порталов к body). Как редактировать пачку?
— (а) inline-поповер у sticky-бара (в пределах виджета); (б) отдельная страница/режим редактирования; (в) пошагово в самой строке-шаблоне.
— реф: ограничение `UI_PLAYBOOK §0`.

### Группа G2.3 — Применить ко всем / fill

**G2.10.** «Заполнить 8×5» по всей неделе и «Заполнить будни» построчно — оба нужны, или дублируют друг друга?
— (а) оба; (б) убрать построчный (есть глобальный); (в) объединить с выбором «эта строка / вся неделя».
— реф: Timetta — autofill на уровне строки и недели.

**G2.11.** Нужна ли норма заполнения из настроек (не жёстко 8ч), учёт графика сотрудника/праздников?
— (а) фикс 8ч (как сейчас); (б) из настроек сотрудника (FTE%); (в) + производственный календарь (праздники → 0).
— реф: Timetta — заполнение по расписанию (schedule), учёт FTE.

**G2.12.** «Применить значение ячейки ко всем будням строки» (Shift+Enter) — нужно ли расширить на «ко всей неделе всех строк» или «к выбранным дням»?
— (а) как сейчас (будни строки); (б) + выбор дней; (в) + ко всем строкам.
— реф: Kimai QuickEntry — заполнение по сетке.

**G2.13.** Копирование прошлой недели — нужен ли выбор «из какой недели» (не только предыдущей)?
— (а) только предыдущая; (б) + выбор любой прошлой недели; (в) + «копировать из шаблона».
— реф: Timetta — copyLines из выбранного периода / из ресурс-плана.

**G2.14.** Создание строк из ресурс-плана (как Timetta createLinesFromResourcePlan) — внедрять?
— (а) нет; (б) да, структура из плана; (в) структура + плановые часы.
— реф: Timetta — createLinesFromResourcePlan(WithHours).

### Группа G2.4 — Batch-approve (расширение существующего)

**G2.15.** Сейчас approve/reject применяется ко ВСЕМ SUBMITTED сразу. Нужно ли выборочное (галочками)?
— (а) всё сразу ок; (б) да, выбрать конкретные записи/сотрудников; (в) + по строкам внутри сотрудника.
— реф: Timetta — выборочное согласование позиций.

**G2.16.** Согласование по сотрудникам пачкой (выбрать несколько сотрудников → approve их периоды)?
— (а) нет, по одному; (б) да, мультиселект сотрудников; (в) «согласовать весь отдел».
— реф: Timetta — массовое согласование по команде.

**G2.17.** Частичное отклонение (reject отдельных строк, остальное approve) одним проходом?
— (а) нет, всё или ничего; (б) да, смешанное действие; (в) reject строки → возврат всему периоду.
— реф: Timetta — построчное reject с комментарием.

**G2.18.** Reject требует общий комментарий. Нужен ли комментарий per-запись/строку?
— (а) общий ок; (б) + опционально per-строка; (в) обязательный per-строка.
— реф: Timetta — комментарий к конкретной позиции.

### Группа G2.5 — Массовое удаление / диапазон ячеек

**G2.19.** Массовое удаление выделенных записей (не целой строки/недели) — нужно?
— (а) нет, хватает clearRow/clearWeek; (б) да, удалить выделенные ячейки; (в) да, удалить выделенные строки.
— реф: Timetta removeLines; Kimai bulk-delete.

**G2.20.** «Очистить неделю» — опасное. Нужны ли доп. предохранители помимо confirm?
— (а) confirm как сейчас; (б) + показать сколько записей/часов удалится; (в) + undo после удаления.
— реф: типовой UX destructive-action с превью+undo.

**G2.21.** Undo для массовых операций (fill / copy / clear / batch-edit) — нужен?
— (а) нет; (б) да, один шаг назад (toast «Отменить»); (в) полноценная история действий.
— реф: Kimai — нет undo; типовой UX — toast-undo.

**G2.22.** Выбор диапазона ячеек как в Excel (A1:C5, заполнить/очистить выделенное) — внедрять при ограничениях Worker?
— (а) нет, слишком сложно без нативного DOM; (б) да, через клавиатуру (Shift+Arrows); (в) да, клик-перетаскивание (drag в Worker недоступен — только клавиатура/клик-якоря).
— реф: Excel/Google Sheets; ограничение `UI_PLAYBOOK §0` (нет нативного drag).

**G2.23.** При массовых операциях согласованные (APPROVED) ячейки защищены. Как сообщать пользователю, что часть пропущена?
— (а) тихо пропускать (как сейчас); (б) toast «N согласованных не изменено»; (в) подсветка пропущенных.
— реф: CISO-012; типовой UX — явный фидбэк.

- **PDF-генератора НЕТ.** Нет jspdf/react-pdf/pdfkit. `@react-pdf/renderer` упомянут в `research/credos-it-catalog/TECH_STACK.md`, но в коде не используется. В `research/credos-it-catalog/SCOPE.md` «экспорт в PDF/Word» — невыполненный чекбокс.
- **CSV-экспорт ЕСТЬ, но без скачивания файла:** detail-отчёт и табель-Т-13-сетка генерируют CSV-строку (UTF-8 **BOM**, разделитель **`;`** под RU-Excel/1С), которая показывается в textarea для ручного Ctrl+A→Ctrl+C. `Blob`/`URL.createObjectURL`/`anchor.download` в Worker недоступны.
- **Табель Т-13 как сетка ЕСТЬ** (`timesheet-grid`): сотрудник×день за месяц, буквенные коды Я/ОТ/Б/ДО/К/НН, код приоритетнее часов при withCodes. Но это CSV-данные, не печатная форма.
- **PII-риск (CISO-010):** код «Б» (больничный) — медицинские ПДн (152-ФЗ), экспорт требует маскировки для не-HR и role-guard.

## G4. Timezone / локаль / форматы

**Зона кода:** `grid/format.ts`, `grid/use-week.ts`, `reports/use-period.ts`, `reports/month-label.ts`, `constants/labels.ts`, `objects/credos-time-entry.object.ts`, `logic-functions/reports-detail.ts`.

**Текущее состояние (факты из кода):**
- **Часы вывод — ТОЧКА** (`fmtHours`): `8`, `2.5`, `0.25`. HH:MM на вывод нет.
- **Часы ввод — гибко** (`parseHours`): `1.5`/`1,5` (точка/запятая), `1:30`, `1ч30м`/`1ч`/`30м`/`90м`, `1h30`/`1h`. Шаг 0.25, диапазон 0..24.
- **Дата записи — `FieldType.DATE_TIME`** (`credos-time-entry.object.ts`), дефолт `now`. Все расчёты в **UTC** (`Date.UTC`, `getUTC*`, `toISOString`) — «чтобы не зависеть от таймзоны песочницы».
- **Первый день недели — понедельник** (`use-week.ts`, `dow=(getUTCDay()+6)%7`), ISO 8601 (Пн–Вс).
- **Даты RU вручную** (`month-label.ts`): `MONTHS_SHORT=['янв','фев',...]`, неделя «13–19 января 2026» / «30 декабря – 5 января 2026», месяцы в родительном падеже. Дни `['Пн','Вт',...]`. `Intl`/`toLocaleDateString` НЕ используется.
- **CSV** — разделитель `;`, UTF-8 **BOM**, часы как число с **точкой**.
- **Номер недели ISO НЕ вычисляется** (навигация сдвигом 7 дней, без №).
- **Нормы:** 40 ч/нед, 8 ч/день (`labels.ts`).

### Группа G4.1 — Часовой пояс записей

**G4.1.** Записи считаются в UTC. Дата записи — это дата по UTC или по локали сотрудника? Риск: запись «вечер 30-го» по МСК может попасть на 30/31 в зависимости от трактовки.
— (а) дата = календарный день локали сотрудника (хранить date-only, не datetime); (б) UTC как есть; (в) фиксированная таймзона организации (МСК).
— реф: Timetta — учёт в дате, без времени суток; Kimai — таймзона пользователя.

**G4.2.** Поле — `DATE_TIME` с дефолтом `now`. Нужно ли время суток вообще, или таймшит оперирует только датой+часами (длительность)?
— (а) только дата + длительность (Duration-only); (б) дата + время начала/конца; (в) и то и то по настройке.
— реф: Kimai — три режима (Duration-only / Begin-End / Punch); Timetta — Duration по дате.

**G4.3.** Нужна ли поддержка сотрудников в разных таймзонах (распределённая команда)?
— (а) все в одной ТЗ организации; (б) ТЗ per-сотрудник; (в) не актуально (один офис).
— реф: Kimai — таймзона на пользователя.

**G4.4.** Граница недели/месяца считается по UTC. Совпадает ли это с ожиданием бухгалтерии (месяц по календарю РФ)?
— (а) считать границы по локали/МСК, не UTC; (б) UTC ок; (в) проверить на записях у границы месяца.
— реф: 1С:ЗУП — месяц по календарю РФ.

### Группа G4.2 — Форматы дат и чисел RU

**G4.5.** Десятичный разделитель часов: вывод сейчас ТОЧКА («2.5»). RU-стандарт — запятая. Менять вывод на запятую?
— (а) точка везде (как сейчас); (б) запятая в UI, точка в CSV; (в) запятая и в UI, и в CSV (но CSV `;`-разделённый → запятая безопасна для Excel-RU).
— реф: RU-локаль — десятичная запятая; Timetta — Decimal-формат.

**G4.6.** Ввод принимает и точку, и запятую — оставить обе?
— (а) да, обе (как сейчас); (б) только запятая (RU); (в) обе + показывать нормализованный результат.
— реф: гибкий ввод — UX-плюс.

**G4.7.** Формат даты в отчётах/CSV — какой нужен для 1С/бухгалтерии?
— (а) `ДД.ММ.ГГГГ`; (б) ISO `ГГГГ-ММ-ДД` (как сейчас в CSV?); (в) обе — UI человекочитаемо, CSV ISO.
— реф: 1С — `ДД.ММ.ГГГГ`; обмен — ISO.

**G4.8.** Месяцы захардкожены массивом (без `Intl`). Нужна ли локализация под другие языки (англ. для иностранных сотрудников)?
— (а) только RU; (б) RU+EN переключатель; (в) `Intl` с локалью пользователя.
— реф: Kimai — мультиязычный UI.

**G4.9.** Подпись недели «13–19 января 2026» (родительный падеж). Нужен ли № недели ISO рядом?
— (а) только диапазон дат; (б) + «Неделя 3»; (в) только номер недели.
— реф: ISO 8601 week number; Timetta — диапазон дат.

### Группа G4.3 — Первый день недели и номер недели

**G4.10.** Первый день недели — понедельник (зашит). Нужна ли настройка (вс — для некоторых)?
— (а) всегда Пн (RU-стандарт); (б) настройка Пн/Вс; (в) Пн жёстко, не усложнять.
— реф: ISO 8601 — Пн; US — Вс.

**G4.11.** Номер недели ISO не вычисляется. Нужен ли он (в шапке, в отчётах, в выборе периода)?
— (а) не нужен; (б) показывать № недели; (в) + навигация по № недели / datepicker с неделями.
— реф: ISO 8601 week; Timetta — выбор недели.

**G4.12.** Выходные/праздники — учитываются ли производственным календарём РФ (перенос выходных, праздничные дни)?
— (а) только Сб/Вс выходные (как сейчас); (б) + госпраздники РФ; (в) + переносы (постановления Правительства).
— реф: производственный календарь РФ; Timetta — календарь нерабочих дней.

### Группа G4.4 — Форматы часов: «1:30» / «1ч30м»

**G4.13.** Ввод поддерживает HH:MM и «1ч30м», но вывод всегда десятичный. Нужен ли HH:MM на вывод/отображение?
— (а) только десятичный (как сейчас); (б) переключатель Decimal/HH:MM (как Timetta); (в) HH:MM везде.
— реф: Timetta — выбор формата Decimal / HH:MM; Kimai — длительность в часах.

**G4.14.** Если вводят «1:30» — показывать как «1:30» или нормализовать к «1.5»?
— (а) нормализовать к десятичному (как сейчас); (б) сохранять введённый формат; (в) зависит от настройки формата.
— реф: Timetta — единый формат отображения по настройке.

**G4.15.** Квантование к 0.25 (15 мин). Подходит ли шаг, или нужен 0.5 / 0.1 / произвольный?
— (а) 0.25 (15 мин) ок; (б) настраиваемый шаг; (в) 0.5 (полчаса).
— реф: Timetta/Kimai — округление по настройке.

**G4.16.** Диапазон 0..24 ч/день. Нужны ли переработки >24 (двойные смены/ошибки) — блокировать или предупреждать?
— (а) жёстко ≤24; (б) предупреждение >12 (есть OVERTIME_WARN), >24 блок; (в) разрешить с подтверждением.
— реф: REQ-0019 overtimeWarn; Timetta — предупреждение об аномалии.

**G4.17.** При вводе «90м» → «1.5». Нужно ли показывать пользователю, как распознан ввод (превью «= 1.5 ч»)?
— (а) нет, сразу значение; (б) превью при вводе; (в) tooltip с распознанным форматом.
— реф: UX гибкого ввода.

### Группа G4.5 — CSV / экспорт-локаль

**G4.18.** CSV: `;` + BOM + точка в числах. Точка в числе при `;`-разделителе для Excel-RU безопасна, но 1С может ждать запятую. Проверить под целевую систему?
— (а) оставить точку; (б) запятая в числах (т.к. разделитель `;`); (в) настройка под 1С vs Excel.
— реф: 1С:ЗУП — десятичная запятая; Excel-RU — зависит от локали.

**G4.19.** Кодировка/разделитель CSV — фиксированы. Нужны ли варианты (UTF-8/Windows-1251, `;`/`,`/Tab) под разные приёмники?
— (а) фикс UTF-8 BOM `;` (как сейчас); (б) выбор кодировки; (в) выбор разделителя.
— реф: 1С исторически Windows-1251; Excel-RU — UTF-8 BOM.

**G4.20.** Формат даты в CSV — какой ждёт бухгалтерия для импорта?
— (а) ISO `ГГГГ-ММ-ДД`; (б) `ДД.ММ.ГГГГ`; (в) согласовать с конкретной 1С-конфигурацией.
— реф: 1С-обмен; уточнить у заказчика целевую систему.

**G4.21.** Имена месяцев в CSV/отчётах — текстом («январь») или числом? Для машинной обработки лучше число.
— (а) число/ISO в CSV, текст в UI; (б) текст везде; (в) число везде.
— реф: машинный обмен — числовые поля.
