## Выводы (консенсус 5 конкурентов)

### Ось 1 — Source of truth: ЧЕЛОВЕК (bottom-up) — ЕДИНОГЛАСНО
Все 5: план первичен НА ЧЕЛОВЕКЕ, отдел/команда = производный агрегат снизу-вверх. Timetta прямо: «на подразделения планировать нельзя». **→ наш bottom-up (PLANNING_EMPLOYEE_LEVEL §3.2, person-plan-backend Dev2) ПОДТВЕРЖДЁН 5/5 + 2 аналитика. Отдел = контейнер/остаток, не самостоятельное число.**

### Ось 2 — «Уже занято»: heatmap/полоса утилизации по порогам — ЕДИНОГЛАСНО
Все показывают занятость цветом (free→full→over) + остаток. **RemDOM-friendly (рендер, не интеракция).** → наша heatmap доски (Загрузка%/Баланс) = тот же паттерн. Availability Bar (Resource Guru: сегменты free/booked/overtime) — образец для ячейки сотрудника. Toggle часы/% (Float) — кнопка.

### Ось 3 — Режим на проект: ИЗБЫТОЧНО
Resource Guru/Runn обходятся БЕЗ per-project режима (глобально + Views). Только Forecast имеет (allocations vs tasks). **→ нам per-project режим не нужен как «отдел/человек», уровень — по строке (2-й аналитик). planMethod (EVEN/MANUAL) — это способ РАСКИДА (ортогонально), оставить на проекте.**

### Ось 4 — Память контекста: БРАТЬ (Float/Resource Guru сильно)
Resource Guru: Views (сорт+группировка+фильтр) persist даже после logout, per-user. Float: saved views, фильтры follow между экранами. **→ запоминать срез/период/группировку в user-prefs (дёшево, ценно). Связь G1.13/C3.8 (сохранённые виды).**

### Ось 5 — Единый экран + Group by: ЛУЧШИЙ паттерн (Resource Guru)
Resource Guru = ОДИН Schedule + переключатель «Group by» (Project/Client/Job/...) — тот же грид перегруппирован. Runn/Forecast = раздельные экраны (хуже). **→ для нас ОДИН экран доски + срез Отделы/Люди/Проекты (переключатель, не разные экраны). Без drag критично — segmented, не таскать. У нас уже срезы Отделы/Люди — расширить, не плодить экраны.**

### Ось 6 — Анти-двойной-счёт: один источник + взаимоисключающие режимы — КЛЮЧЕВОЕ
- Forecast: «при project-level allocations задачи НЕ учитываются» — режимы ВЗАИМОИСКЛЮЧАЮЩИЕ, не суммируются.
- Resource Guru: одна ёмкость человека, бронь в одном месте; при группировке по проекту чужие брони **faded** (видны, не считаются повторно).
- Float/Runn: один источник = срез той же записи, двойного счёта нет by design.
**→ наш SSOT-MAX (PLANNING_EMPLOYEE_LEVEL §8.3: Demand=max(план,бронь), детальный вычитается из остатка) ПОДТВЕРЖДЁН. + UX-приём: чужие брони/слоты показывать faded при группировке (Resource Guru).**

## Рекомендация архитектору для СПЕКИ планирования

1. **Source of truth = человек (slot/booking на emp).** Отдел = Σ людей + остаток (есть, person-plan-backend). НЕ вводить план отдела как независимое число (только остаток).
2. **Один экран доски + срез Отделы/Люди/Проекты** (переключатель, Resource Guru-модель). НЕ раздельные экраны. Уровень ввода плана = по строке (2-й аналитик), не отдельный режим.
3. **«Booked» = heatmap/полоса утилизации** по порогам (free/full/over) в ячейке сотрудника. Toggle ч/%. RemDOM-ок.
4. **Память контекста:** срез/период/группировка в user-prefs (persist).
5. **Анти-двойной-счёт:** Demand=max(план,бронь) (есть) + чужие слоты/брони faded при группировке по проекту/отделу.
6. **planMethod (EVEN/MANUAL)** — способ раскида, ортогонален source-of-truth, оставить на проекте.
7. **P1-фикс префилла** (project-plan-panel фильтр !employeeId, итер.125) — ДО унификации панелей.
8. **Пороги heatmap** — свериться (наши ±5/15 vs Timetta ±10/20 vs Forecast under/full/over) — вопрос заказчику (целевая утилизация).

**Унификация панелей** (project-plan-panel + employee-plan-panel) — ПОСЛЕ спеки: единый компонент таблица-план с уровнем по строке, общий префилл с фильтром по уровню, общий heatmap-рендер.

## Применимость / сохранены как референсы

Все 4 (Float/Runn/Forecast/Resource Guru) ПОДХОДЯТ как вспомогательные референсы — сохранены в `research/competitors/{float,runn,forecast,resource-guru}/`. Главная ценность каждого:
- **Resource Guru** — единый экран + Group by + Views-persist + Availability Bar + faded-брони (лучший UX-образец для нас).
- **Float** — person-first, util toggle ч/%, saved views, remaining capacity.
- **Forecast** — взаимоисключающие режимы учёта (анти-двойной-счёт), soft/hard allocations.
- **Runn** — heatmap availability (зел→красный), tentative-сценарии (вкл/выкл неподтверждённого).
- **Timetta** (главный, локально) — bottom-up, generic→человек, 5 методов раскида, RBAC-scope.

**Kimai — НЕ референс по планированию** (capacity-планирования нет вообще).
