## 1. СОГЛАСОВАНИЕ — модель таймшита Timetta (усиливает A4.*, B1.17, F3.*)

**Факт из `raw-timesheet-full.json` — структура TimeSheet:**
```
TimeSheet { dueDate, dateFrom, dateTo, name, approved (timestamp|null),
  userId, departmentId, legalEntityId, templateId, stateId, stateEntered,
  billableDuration, nonBillableDuration, editAllowed, deleteAllowed,
  schedule, template, user, state, timeSheetLines, timeOffRequests, timeAllocations }
```

Ключевые уточнения к ответам:

- **Таймшит = запись на сотрудника×неделю** (name="Мария Осташкова 13.04.2026-19.04.2026", dateFrom=Пн, dateTo=Вс). → подтверждает наш недельный период и submit периодом целиком (**A4.1** ✓).
- **`editAllowed` / `deleteAllowed`** — сервер отдаёт вычисленные права прямо в записи. → **КРИТИЧНО для CISO-005 / A4.18/A4.19 / W3B**: Timetta резолвит права на СЕРВЕРЕ и отдаёт флагом, клиент их не вычисляет. Это ровно та модель, которую я рекомендовал в итер.103 (recall/revoke через server-identity, не client `workspaceMemberRef`). **Референс-подтверждение: брать editAllowed/deleteAllowed с сервера, а не доверять клиентскому actor.**
- **`approved`** = timestamp (не bool) → фиксирует момент согласования (наш approvedAt ✓, **A4** ).
- **`timeSheetLines` ОТДЕЛЬНО от `timeOffRequests`** → отсутствия (отпуск/больничный) — НЕ строки часов, отдельная сущность. Прямое подтверждение **W3A.22** (комменты только к часам, отсутствия отдельно) и **A4.7** (недобор считать минус подтверждённые отсутствия). ✓
- **`billableDuration`/`nonBillableDuration`** — у Timetta есть, у нас НЕ вводим (**NB no-billable**). Подтверждает осознанность отказа.

**Статусы (из `raw-odata-States.json`):** TimeSheet lifecycle = `Draft` → `Submitted` (На согласовании) → `Approved` (Согласовано); есть также `ReadyToApprove` (Готова к согласованию), `Cancelled`, `Snapshot`. → подтверждает **A4.20(а)** «4 статусов достаточно»: ядро Timetta = Draft/Submitted/Approved (+отклонение через возврат). Наш DRAFT/SUBMITTED/APPROVED/REJECTED покрывает.

## 2. СРОК ОТПРАВКИ (dueDate) — точная механика (усиливает F4.5, F4.19, B4.9)

**Факт (`ux-i18n-ru.json` + `TimesheetTemplates`):**
- `dueDateLag` = **«лаг в РАБОЧИХ днях, добавляемый к дате окончания периода таймшита для вычисления срока отправки на согласование»**. В базовом шаблоне `dueDateLag: 2`.
- Проверка на данных: период Пн 30.03 – Вс 05.04 → `dueDate` = 07.04 (Вт) = dateTo + 2 рабочих дня.
- Термин Timetta: **«Нормативная дата отправки на согласование»**.

Уточнение к **F4.5** (когда напоминать): не абстрактное «за N дней», а **конкретная модель — dueDate = конец периода + `dueDateLag` рабочих дней (дефолт 2 → вторник)**. Реализовать `dueDateLag` настройкой (наш аналог в Settings). → **F4.19** обратный отсчёт = до dueDate, не до конца периода.

Дополнительно (`ux-i18n`): **«количество рабочих дней до окончания таймшита, за которое его можно отправлять на согласование»** — Timetta имеет ОКНО отправки (нельзя слать слишком рано). Кандидат в Settings (новый, к **B4** / REQ-0019).

## 3. PRE-SUBMIT ВАЛИДАЦИЯ — правила Timetta (усиливает A4.6, UC-VAL-03, A4.2)

**Факт (`ux-i18n-ru.json`, ValidationRule-семантика):**
- **«Укажите минимальное количество непустых строк таймшита для отправки на согласование»** → Timetta: правило `min non-empty lines`. Наш UC-VAL-03 (недобор недели) — родственное; Timetta параметризует МИНИМУМ строк, не только часов. Кандидат-расширение **A4.6**.
- **«Удалять пустые строки во время отправки на согласование»** (`clearEmptyLinesAutomatically`) → авто-очистка пустых строк при submit. Кандидат к **A1.11 / G2** (вместо ручного «Очистить»).
- **«Пропускать согласование, если таймшит пустой (0 строк)»** (`emptyAutoApproval`).
- **«Пропускать согласование, если на все рабочие дни есть согласованные отсутствия или рабочих дней нет»** (`timeOffAutoApproval`) → **прямое подтверждение A4.7**: недобор/согласование учитывает подтверждённые отсутствия; полностью покрытая отсутствиями неделя не требует submit.

Вывод: наша мягкая проверка недобора (UC-VAL-03) — в духе Timetta, но Timetta даёт БОЛЬШЕ управляемых правил (min строк, авто-очистка, auto-approval пустых/отсутствий). Завести как опции валидации (**A4.6 расширение**, в REQ-0019/Settings).

## 4. КОЛОНКИ ТАЙМШИТА — TimesheetTemplate (усиливает F1.11, B2.8, W3A)

**Факт (`raw-odata-TimesheetTemplates-expand.json`, «Базовый шаблон»):**
```
showRole:false · showActivity:true · showClient:true · showProjectCostCenter:false
· showTariff:true · showBillCode:false · isDefault:true · dueDateLag:2
· createAutomatically:false · clearEmptyLinesAutomatically:false
· timeOffAutoApproval:false · emptyAutoApproval:false
```

Уточнение к **F1.11** (выбор колонок): Timetta делает это через **TimesheetTemplate** с булевыми тогглами на КАЖДУЮ колонку (showActivity/showClient/showRole/showTariff/showBillCode/showProjectCostCenter). → наш column-picker должен быть НАБОРОМ тогглов в шаблоне/настройке, а не произвольным. Конкретный список колонок-кандидатов: Активность(=Вид работ), Клиент, Роль, Тариф(NB), Код биллинга(NB), Центр затрат. Из них нам применимы: **Вид работ, Клиент, Роль; Тариф/BillCode/CostCenter — NB, пропускаем**.

→ **B2.8** (колонки реестра «Трудозатраты»): набор Timetta-«Деталей» = Дата·Клиент>Проект>Работа·Вид работ·Тариф·Часы·Комментарий — наш набор без Тарифа корректен.

`createAutomatically` (авто-создание таймшита на период) — кандидат к онбордингу (**E3.10** предзаполнение).

## 5. КОДЫ Т-13 / ОТСУТСТВИЯ — TimeOffTypes Timetta (усиливает C4.9, G3.4, A3.25)

**Факт (`raw-odata-TimeOffTypes-full.json`):**
| Timetta name | code | paid |
|---|---|---|
| Ежегодный отпуск | **П** | да |
| Отпуск за свой счёт | **БО** | нет |
| Декретный отпуск | **Д** | да |
| Учебный отпуск | **У** | да |
| Больничный | **Б** | да |

Уточнения:
- Timetta использует **«П»** для ежегодного отпуска (мы — **ОТ**); **«БО»** за свой счёт (мы — **ДО**). Расхождение кодов — **сверить с бланком Т-13 РФ**: официальные коды Т-13 = ОТ (ежегодный), ДО (за свой счёт), Б (больничный), У (учебный), Р (отпуск по беременности). Timetta-«П/БО» — их внутренние, НЕ канон Т-13. → **наш набор Я/ОТ/Б/ДО/К/НН ближе к официальному Т-13, оставляем** (C4.9). Но добавить **У (учебный)** и **Д/Р (декрет)** в маппинг — Timetta подтверждает их наличие в РФ-практике (**G3.4 расширение**).
- **`paid` флаг** на типе отсутствия → нужен для расчёта (оплачиваемое/нет). Кандидат к нашей absence-модели (REQ-0006).
- **`timeOffRequests` отдельны от строк часов** (см. §1) → подтверждает **C4.11** (табель по проекту не содержит отсутствий — они вне проектных строк) и **W3A.22**.

- Наши 4 вкладки (Сводка/Тренд/Проекты/Незаполненные) покрывают «Часы проектов», «Часы сотрудников», «Незаполненные». **«Утилизация по месяцам»** как отдельный пресет — кандидат (A3.1).

## 7. ПЛАНИРОВАНИЕ — поля проекта Timetta (усиливает W3B, PLANNING_MODELS, Planning-Records)

**Факт (`raw-odata-Projects-deep.json` — поля дат/плана проекта):**
```
startDate, endDate,
plannedStartWorkDate, plannedEndWorkDate,
estimatedStartWorkDate, estimatedEndWorkDate,
isAutoPlanning, calculationBaseUpdated, outdated
```

Уточнения — важно для модели хранения плана:
- Timetta РАЗДЕЛЯЕТ **plomanned** (плановые) и **estimated** (оценочные) даты работ + базовые startDate/endDate. → у нас один `startDate/endDate`. Подтверждает трёхосевой инвариант (бронь≠план≠факт) и то, что «план-оценка» — отдельный слой.
- **`isAutoPlanning`** (булев флаг авто-планирования) → **прямой референс к моему `planMethod` (EVEN/MANUAL)** из Planning-Records/W3B.6. Timetta уже имеет флаг «авто vs ручное планирование» на проекте. **Подтверждает решение: завести `planMethod`/`isAutoPlanning`-аналог.**
- **`outdated` / `calculationBaseUpdated`** → Timetta помечает план «устаревшим» при изменении базы расчёта. Кандидат: при сдвиге дат/объёма помечать раскид «требует пересмотра» (родственно **W3B.4** «алерт о расхождении»).

→ Усиливает **Planning-Records #3** (источник истины planMethod) и **W3B.6** (хранить метод флагом — как Timetta `isAutoPlanning`).
