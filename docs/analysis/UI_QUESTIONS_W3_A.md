## Контекст: что УЖЕ реализовано (WI-01, решение C1)

Это **не повтор волны-1**. Волна-1 спрашивала «куда деть контролы из ячейки» — заказчик ответил, решение C1 реализовано. Здесь — следующий слой развилок, которые всплывают **внутри** уже выбранной модели.

Реализовано (по факту кода):
- `hour-cell.tsx` — ЧИСТЫЙ ввод: клик / печать цифры → сразу `<input>`. Никаких плавающих кнопок в ячейке. Ячейка несёт только статус (число, замок, цвет-переработка, точка-коммент). Размер по факту кода: **58×32px** (день-эталон 72×38).
- `row-menu.tsx` (⋯, слева у строки) — пункты: «Дублировать строку» → «Заполнить будни нормой» → «Комментарий к записи…» → «Обнулить часы» → «Убрать строку» (danger). Разделители перед «Заполнить» и «Обнулить». Подписи-hint у каждого. При `rowLocked` остаётся только «Дублировать».
- Чип-нормы в ячейке **убран** (норма ставится печатью цифры или пунктом меню).
- `bulk-fill` сведён к 2 действиям (строка-меню «Заполнить будни нормой» + тулбар).
- `copy-guard` для APPROVED.
- Клавиатура (`use-keyboard.ts`): стрелки/Tab/Enter — навигация, печать цифры → seed-ввод, Esc — откат. **Shift+Enter** на активной ячейке = bulk-fill значения на будни строки (`week-grid.tsx`).
- Комментарий — отдельная панель в `row-menu` (по дню с часами, инлайн-поле, blur/Enter сохраняет, Esc откат).

Несоответствие, замеченное при разборе: решения F-блока (`TIMESHEET_CELL_CONTROLS.md`) приняли «высота 40, ширина дня 64», но в коде до сих пор **58×32**. Часть вопросов ниже это разруливает.

## Группа 1. Клавиатура нового «клик = ввод»

**W3A.1** Сейчас при печати цифры на активной (не редактируемой) ячейке seed-символ становится первым символом ввода (`seed === '0' ? '' : seed`). Если в ячейке уже есть 8 и пользователь печатает «5» — текущая реализация **затирает** (начинает с «5», старое 8 не сохраняется в draft). Это ожидаемо?
— Гипотезы: (а) печать всегда **заменяет** старое значение (Excel-style, как сейчас) — быстрый перебивочный ввод; (б) печать **дописывает** к старому (8 → 85); (в) печать на непустой ячейке сначала выделяет, второй символ заменяет.
— Реф: Timetta/Kimai — печать на ячейке с числом заменяет целиком (overwrite), как сейчас. Подтвердить — оставляем (а).

**W3A.2** Esc сейчас откатывает draft к сохранённому значению и закрывает input, фокус остаётся на ячейке (она снова `active`). Esc два раза подряд — нужно ли, чтобы второй Esc снимал выделение ячейки целиком (как «выход из сетки»)?
— Гипотезы: (а) Esc только откат ввода, выделение остаётся (как сейчас); (б) первый Esc — откат, второй — снять активность; (в) Esc всегда оставляет активность.
— Реф: Timetta — Esc отменяет правку, ячейка остаётся выбранной. Kimai так же. Оставить (а).

**W3A.3** Tab сейчас идёт строго вправо по дням (`dCol: ±1`), на краю **упирается** (clampCell, не переносит на след. строку). Нужен ли перенос: Tab на воскресенье → понедельник следующей строки?
— Гипотезы: (а) оставить упор на краю (как сейчас) — предсказуемо, не «улетаешь»; (б) перенос на первый день следующей строки (Excel/Timetta «таб-проход всей сетки»); (в) Tab на «Итого» не заходит, но переносит на след. строку.
— Реф: Timetta — Tab проходит дни и переносит на следующую строку (wrap). Kimai — упирается. Развилка реальная, влияет на скорость заполнения недели.

**W3A.4** Enter сейчас = «подтвердить и уйти на строку ВНИЗ» (`dRow: 1`). На последней строке Enter упирается. Нужно ли, чтобы Enter на последней строке **создавал новую строку** (фокус в форму AddRow)?
— Гипотезы: (а) упор (как сейчас); (б) Enter на последней строке → фокус в AddRow (поле «код проекта»); (в) Enter на последней строке → ничего, но Tab переносит.
— Реф: Kimai — Enter на последней строке открывает quick-add. Timetta — упор. Ускоритель для «забил неделю, добавляю следующий проект».

**W3A.5** **Конфликт Shift+Enter.** Сейчас Shift+Enter = bulk-fill значения ячейки на будни строки (`week-grid.tsx`). Но привычный табличный Shift+Enter = «подтвердить и уйти ВВЕРХ» (обратный Enter). Это конфликт. Как развести?
— Гипотезы: (а) оставить Shift+Enter = bulk-fill, навигацию вверх делать только стрелкой ↑ (потеря привычки, но bulk-fill ценен с клавиатуры); (б) Shift+Enter = вверх (вернуть табличную норму), bulk-fill повесить на другую комбинацию (напр. Ctrl/Cmd+Enter или Alt+→); (в) убрать клавиатурный bulk-fill совсем — он есть в меню ⋯ «Заполнить будни нормой», дубль не нужен.
— Реф: Timetta/Kimai — Shift+Enter = вверх (стандарт), bulk-fill только через меню/кнопку. Рекомендация аналитика: **(б) или (в)** — Shift+Enter как «вверх» интуитивнее, а bulk-fill уже доступен в меню.

**W3A.6** Куда уходит фокус ПОСЛЕ сохранения ввода по Enter? Сейчас: commit → `setEditing(false)` → `onKey(Enter)` → ячейка ниже становится active (но НЕ в режиме edit — просто выделена). Нужно ли, чтобы следующая ячейка сразу открывалась на ввод, или достаточно «выделена, печатай дальше»?
— Гипотезы: (а) выделена, не в edit (как сейчас) — печать цифры откроет ввод; (б) сразу edit-режим в следующей ячейке (autoFocus input) — поток «Enter-печать-Enter-печать» без лишнего шага.
— Реф: Timetta — после Enter ячейка ниже выделена, печать продолжает (как (а)). Kimai — аналогично. Оставить (а), подтвердить.

**W3A.7** Delete/Backspace на активной ячейке сейчас = `seed: '0'` → обнуляет (через ввод пустого draft → parseHours → 0). Это сразу коммитит 0 или открывает пустой input для подтверждения?
— Гипотезы: (а) Delete мгновенно обнуляет без подтверждения (как сейчас по факту seed='0'→draft=''→commit при blur/Enter); (б) Delete открывает пустой input, нужен Enter для фиксации; (в) Delete на заполненной — мгновенно, требует подтверждения только при >N часов.
— Реф: Timetta/Kimai — Delete очищает ячейку сразу. Уточнить, нет ли риска случайного стирания при навигации.

**W3A.8** Home/End внутри строки сетки (решение D2.7 приняло «добавить Home/End — начало/конец строки»). Это **не реализовано** в `use-keyboard.ts` (там только стрелки/Tab/Enter/печать). Подтверждаем приоритет добавления, и Home = первый день (Пн) или первый РЕДАКТИРУЕМЫЙ день (пропуская locked)?
— Гипотезы: (а) Home=Пн, End=Вс (геометрия, просто); (б) Home/End=первая/последняя редактируемая (пропуск locked); (в) +Ctrl+Home/End = первая/последняя ячейка всей сетки.
— Реф: Excel-стандарт — Home/End по строке, Ctrl+Home/End по таблице.

## Группа 2. Меню строки ⋯ — порядок, иконки, lock, деструктив

**W3A.9** Текущий порядок пунктов: Дублировать → [разд] → Заполнить будни нормой → Комментарий → [разд] → Обнулить часы → Убрать строку. «Дублировать» первым — это самое частое действие?
— Гипотезы: (а) самое частое — «Заполнить будни нормой» (ежедневный ввод), его наверх; (б) оставить «Дублировать» первым (создание новой строки — стартовое действие); (в) порядок по частоте: Заполнить → Комментарий → Дублировать → [разд] → Обнулить → Убрать.
— Реф: Kimai — в контекст-меню строки сверху «Fill week», потом Duplicate. Timetta — «Заполнить по расписанию» первым. Аналитик склоняется к (в).

**W3A.10** Сейчас пункты меню **без иконок** (только текст + hint-подпись). Нужны ли иконки слева для быстрого скана?
— Гипотезы: (а) без иконок — чисто, hint-подписи уже несут смысл (как сейчас); (б) добавить иконки (⧉ дублировать, ▦ заполнить, ✎ коммент, ⌫ обнулить, 🗑 убрать) — быстрее находить глазами; (в) иконки только у деструктивных (обнулить/убрать) — акцент на риске.
— Реф: Timetta/Kimai — иконки в контекст-меню есть. Но текст с подписями (наш вариант) читабельнее под Remote DOM. Развилка вкуса заказчика.

**W3A.11** Деструктивные пункты («Обнулить часы», «Убрать строку») сейчас выполняются **сразу по клику** (`run(onClearRow)` закрывает меню и делает). Решение D4.11/D4.12 приняло confirm со счётчиком («Удалить 12 записей? Нельзя отменить»). Это **не реализовано** в `row-menu.tsx`. Подтверждаем добавление confirm — для обоих или только «Убрать строку»?
— Гипотезы: (а) confirm для обоих (обнулить + убрать) со счётчиком записей; (б) confirm только «Убрать строку» (необратимо), «Обнулить» — мгновенно + undo-toast; (в) оба мгновенно + undo-toast «Отменить» 5 сек (быстрее, прощает ошибки).
— Реф: Kimai — confirm-диалог при удалении. Timetta — undo-toast. Аналитик: **(б)** — баланс скорости и защиты.

**W3A.12** Для lock-строки (`rowLocked`) сейчас показывается ТОЛЬКО «Дублировать строку». Нужно ли там же показывать «Комментарий к записи…» в read-only (просмотр существующих комментов согласованной недели)?
— Гипотезы: (а) только Дублировать (как сейчас) — согласованное не трогаем; (б) + «Посмотреть комментарии» (read-only список, без правки); (в) + «Отозвать согласование» (если у юзера права manager — связь с A4.25).
— Реф: Timetta — у согласованной строки доступны «копировать» и «открыть детали» (read-only). Аналитик: (б) полезно для аудита.

**W3A.13** Разделители сейчас группируют: [Дублировать] | [Заполнить, Коммент] | [Обнулить, Убрать]. Логика групп — «создать / заполнить-править / удалить». Эта группировка читается верно, или «Комментарий» логичнее в отдельной группе (это не «заполнение»)?
— Гипотезы: (а) оставить как есть; (б) Комментарий — в свою группу (4 группы); (в) Дублировать+Заполнить в одну (созидательные), Комментарий отдельно, Обнулить+Убрать (деструктив).
— Реф: визуальная группировка — impeccable: 2-3 кластера оптимально, 4+ дробит. Аналитик: (а), не переусложнять.

## Группа 3. Быстрый ввод нормы БЕЗ чипа

**W3A.14** Чип-нормы убран; норма ставится печатью цифры или меню «Заполнить будни нормой». Но норма дня = 8 (или 7 в короткий день) — печатать «8» каждый день в каждую ячейку утомительно. Достаточно ли печати+меню, или нужен ещё один быстрый affordance для ОДНОЙ ячейки?
— Гипотезы: (а) достаточно: печать «8»+Enter быстро, а массовое — «Заполнить будни нормой»; (б) добавить dbl-click по пустой ячейке = поставить норму дня (один жест, без печати); (в) добавить пресет-панель снизу при фокусе (Kimai-style C3: чипы «8 / 4 / норма»).
— Реф: Kimai — preset-чипы под ячейкой при фокусе (вариант C3 из `TIMESHEET_CELL_CONTROLS §F3`). Timetta — нет ячеечного пресета, только «по расписанию». Развилка: dbl-click дёшев и не конфликтует с click=edit (первый клик уже открыл input — нужно решить, что значит dbl).

**W3A.15** Если добавляем dbl-click=норма (W3A.14б): первый клик уже открывает input. Тогда dbl-click — это «клик по уже открытому input»? Технически в Remote DOM это ненадёжно. Альтернатива — норма по горячей клавише на выбранной ячейке?
— Гипотезы: (а) не dbl-click, а клавиша: на активной ячейке «N» или Space = вставить норму дня; (б) dbl-click всё же (принять риск Remote DOM); (в) отказаться — печать достаточно.
— Реф: нет прямого аналога; Excel — нет «нормы», но Ctrl+D протягивает сверху. Аналитик: (а) клавиша надёжнее dbl-click под Remote DOM.

**W3A.16** Как часто реально нужна норма в ОДНУ ячейку (а не во всю строку)? Если редко — affordance не нужен; если часто (короткие/неполные недели) — нужен. Какой типичный сценарий заказчика?
— Гипотезы: (а) почти всегда заполняют всю неделю разом → хватает «Заполнить будни нормой», точечная норма не нужна; (б) часто правят отдельные дни (отгул, неполный день) → нужен быстрый точечный ввод; (в) смешанно.
— Реф: Timetta-сценарий — массовое «по расписанию» + ручная корректировка отдельных дней. Ответ определяет нужность W3A.14/15.

**W3A.17** Сейчас норма дня (короткий день/праздник) должна браться из `useDailyNorm` (производственный календарь, решения A1.12/A1.15). Печать «8» вручную игнорирует календарь — пользователь может вбить 8 в день, где норма 7. Нужна ли подсказка/индикация «норма этого дня = 7» при вводе?
— Гипотезы: (а) нет — что вбил, то и засчитано (доверие пользователю); (б) показывать норму дня бледным плейсхолдером в пустой ячейке («7»); (в) подсветка ячейки если введённое ≠ норма дня (мягкий сигнал отклонения, связь с A1.28/D-факт-vs-план).
— Реф: Timetta — норма дня видна в шапке дня, отклонение подсвечивается. Аналитик: (б) плейсхолдер-норма дёшев и информативен.

## Группа 4. Дубль «с часами?»

**W3A.18** Решение A1.23 приняло «спрашивать при дублировании: с часами?». Сейчас `onDuplicate` в `row-menu` = просто prefill проекта в AddRow (БЕЗ часов, БЕЗ вопроса). Где задавать вопрос «с часами?» — поповер прямо из меню, или в форме AddRow после префилла?
— Гипотезы: (а) сабменю в ⋯: «Дублировать ▸ Только проект+вид / С часами недели»; (б) чекбокс «скопировать часы» в форме AddRow после клика «Дублировать»; (в) два отдельных пункта меню: «Дублировать строку» и «Дублировать со часами».
— Реф: Kimai — Duplicate копирует запись целиком (со часами) одним действием. Timetta — «копировать строки и часы» отдельной опцией. Аналитик: (в) — два явных пункта понятнее сабменю под Remote DOM.

**W3A.19** Дефолт при дублировании — с часами или без?
— Гипотезы: (а) дефолт БЕЗ часов (как сейчас) — дублируют чтобы добавить ДРУГОЙ вид работ на тот же проект (часы будут другие); (б) дефолт СО часами (Kimai) — чаще копируют похожую неделю; (в) запоминать последний выбор пользователя.
— Реф: Kimai дефолт = со часами; Timetta = выбор. Зависит от сценария: hint текущего пункта «тот же проект, новый вид работ» намекает что дубль = для нового вида → дефолт (а). Подтвердить у заказчика.

**W3A.20** Если дублируем СО часами в строку с тем же проект+вид работ — получится **дубль-конфликт** (две строки на одну аллокацию день×проект×вид). Как обрабатывать?
— Гипотезы: (а) запретить дубль если проект+вид совпадут (дубль только для нового вида работ); (б) разрешить, суммировать при сохранении; (в) разрешить, показать предупреждение «такая строка уже есть».
— Реф: Timetta — не даёт две строки на одну аллокацию (мерджит). Kimai — допускает несколько записей в день. Важно для целостности данных — уточнить модель.

## Группа 5. Комментарий из меню

**W3A.21** Комментарий привязан к **дню с часами** (запись день×проект×вид = `description`). Панель показывает по одному полю на каждый день строки с часами. Это верный уровень, или заказчику нужен ещё «комментарий ко всей строке/неделе» (общий)?
— Гипотезы: (а) только по дню (как сейчас, = аллокация) — соответствует модели данных; (б) + общий коммент строки (отдельное поле «о работе в целом»); (в) только один общий на строку (упростить).
— Реф: Timetta/Kimai — коммент на уровне записи дня (как у нас). Подтвердить (а).

**W3A.22** Панель комментариев показывает дни ТОЛЬКО с часами (`commentTargets` фильтрует `hours > 0`). Если пользователь хочет прокомментировать день БЕЗ часов (напр. «был на больничном») — он не сможет (нет записи). Это ок?
— Гипотезы: (а) ок — нет часов = нет записи = нечего комментировать (как сейчас); (б) разрешить коммент на пустой день (создаст запись с 0 ч + текст) — для пометок отсутствия; (в) отсутствия — отдельный механизм (коды Т-13 Я/ОТ/Б), не комментарий.
— Реф: Timetta — отсутствия отдельной сущностью, не комментом. Аналитик: (а)+(в) — комменты только к часам, отсутствия отдельно (связь с Т-13 кодами C4.9/C4.10).

**W3A.23** Превью существующего комментария: сейчас в ячейке только точка-индикатор (5px), а текст виден лишь в `title` (hover-тултип) и при открытии панели. Достаточно ли, или нужно превью текста в пункте меню «Комментарий к записи…» (напр. «Комментарий (3)» со счётчиком)?
— Гипотезы: (а) точка + панель достаточно (как сейчас); (б) счётчик в пункте меню «Комментарий к записи (2)» — видно что комменты есть не открывая; (в) превью первых слов в hint пункта.
— Реф: Timetta — иконка 💬 + счётчик. Аналитик: (б) счётчик дёшев и информативен.

**W3A.24** Сохранение коммента сейчас по blur ИЛИ Enter, Esc — откат к исходному, Enter ещё и **закрывает всю панель** (`onDone` → close). Если у строки 3 дня с комментами, и я правлю первый и жму Enter — панель закроется, не дав перейти ко второму. Это мешает?

— Реф: формы с несколькими полями — Enter обычно = след. поле или submit всей формы, не закрытие. Аналитик: (б) удобнее для многодневных правок.

## Группа 6. Размеры, узкий экран, перенос

**W3A.25** Решения F-блока приняли «высота 40px, ширина дня 64px», но код всё ещё **58×32px** (день-эталон 72×38). Заказчик просил крупнее. Финальные размеры ячейки недели?

**W3A.26** При ширине дня 64px и 7 днях + «Итого» 66px + «Проект» (эластичный) + «Вид работ» 176px — на узком виджете строка не влезет. Решение D3.3 приняло «sticky левые 2 колонки + h-scroll по дням». Подтверждаем h-scroll именно по дням (левые колонки зафиксированы)?
— Гипотезы: (а) sticky Проект+Вид работ, скроллятся дни (D3.3); (б) sticky только Проект, Вид работ скроллится с днями; (в) на узком — схлопывать «Вид работ» в иконку/тултип, дни не скроллить.
— Реф: Timetta/Kimai — sticky левая колонка(и) + h-scroll периода. Подтвердить (а).

**W3A.27** Высота строки фиксированная (32, цель 40). Если в «Проект» длинное название — сейчас ellipsis (одна строка). Нужен ли перенос названия проекта на 2 строки (растёт высота строки) или жёсткий ellipsis+tooltip?
— Гипотезы: (а) жёсткий ellipsis + tooltip по focus (D3.4, фикс-высота, ровная сетка); (б) перенос на 2 строки (видно полностью, но строки разной высоты — рвёт сетку); (в) ellipsis, но название проекта в 2 уровня (код жирным + клиент мелким под ним).
— Реф: Timetta — ellipsis + tooltip, фикс-высота. Аналитик: (а), сетка должна быть ровной (tabular).

**W3A.28** Много строк (15-20 проектов у сотрудника): сейчас вертикальный скролл всего блока (`overflowY: auto`). Шапка дней (`WeekHeader`) и футер-итоги (`FooterTotals`) уезжают при скролле? Нужен ли sticky-хедер дней при вертикальном скролле?
— Гипотезы: (а) sticky шапка дней + sticky футер «Итого» при вертикальном скролле (видно дни и сумму всегда); (б) только sticky шапка; (в) оставить как есть (всё скроллится).
— Реф: Timetta/Kimai — шапка периода sticky при скролле строк. Решение D3.6 приняло sticky хедер — подтвердить распространение на эту сетку.

**W3A.29** Ширина «Вид работ» сейчас фикс 176px. При крупных днях (64px) и узком экране — это много. Сжимать «Вид работ» или «Проект» первым?
— Гипотезы: (а) «Проект» эластичный (minmax 0,1.4fr) сжимается первым, «Вид работ» держит 176px (как сейчас); (б) «Вид работ» → minmax(0, 176px) тоже сжимается; (в) на узком объединить «Проект / Вид работ» в одну колонку (2 строки текста).
— Реф: Timetta — проект+задача в одной колонке на узком. Аналитик: (а) сейчас норм, (в) — для очень узкого как fallback (связь с GRID_TEMPLATE_SINGLE, который уже есть).

**W3A.30** Шрифт в ячейке сейчас 13px (день 16px). При высоте 40px — поднять кегль до 14-15px для читаемости (заказчик просил крупнее)?
— Гипотезы: (а) 14px (принято в F-блоке для недели); (б) 15px (ближе к дню 16); (в) 13px оставить, крупнее только ячейка.
