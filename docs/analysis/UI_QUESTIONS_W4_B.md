## Тема 1. Роли — UX (кто что видит и может)

### W4B.1 — «Сотрудник» как явная роль: что у него в навигации?
RBAC_MODEL.md планирует создать `employee.role.ts`. Сейчас рядовой = function-role + отсутствие manager-роли.
**Вопрос:** какие разделы рядовой сотрудник видит в левом меню? Только «Моё время» (своя сетка) + «Мои отчёты», или ему доступна доска загрузки (capacity) и общие отчёты в read-only?
- **Гип. A:** рядовой видит только «Моё время» + личную сводку по себе; доска/общие отчёты скрыты целиком.
- **Гип. B:** рядовой видит доску/отчёты, но обезличенно (КОД вместо ФИО) и без действий (read-only).
- **Гип. C:** видит всё, что и руководитель, кроме кнопок действий (минимальный объём работы — текущее состояние де-факто).
- **Реф:** Timetta разделяет области «Моя работа» (Пользователь) и «Команда» (Управление командой) — рядовой не видит чужие таймшиты вообще (`settings-permissions-default-permissions-sets`).

### W4B.2 — Роль «РП» (руководитель проекта) ≠ «Руководитель отдела»
Сейчас одна роль `Руководитель` (isManager). Заказчик упоминает РП.
**Вопрос:** нужна ли отдельная роль «Руководитель проекта» со scope «свои проекты» (а не «свой отдел»)? Или РП = тот же isManager?
- **Гип. A:** одна роль isManager достаточно; РП и начальник отдела — это про scope (проект vs отдел), а не про разные роли.
- **Гип. B:** две разные роли: РП видит/правит план и согласует по своим проектам (любой отдел), начальник отдела — по своему отделу (любой проект).
- **Гип. C:** РП — атрибут на проекте (поле «менеджер проекта»), не роль; права выводятся из «я менеджер этого проекта».
- **Реф:** Timetta = роли «Управление проектами» (Мои проекты) и «Управление командой» (Мои подчинённые) — это ДВА независимых набора прав, человек может иметь оба.

### W4B.3 — Админ vs Руководитель: граница настроек
RBAC_MODEL.md: Владелец пишет глобальный `credosTimeSettings`, Руководитель — только `approvalRequired`/`capacityFactor` своего отдела.
**Вопрос:** какие настройки REQ-0019 (тоггл ФИО, напоминания, weekStartsOn, fillThreshold) меняет только Админ, а какие — руководитель для своего отдела?
- **Гип. A:** все глобальные настройки (включая revealEmployeeNames) — только Админ; руководитель не видит раздел «Настройки» вообще.
- **Гип. B:** руководитель видит раздел, но поля глобальные (ФИО-тоггл) — disabled с подсказкой «только администратор».
- **Гип. C:** часть настроек per-отдел (порог заполнения, approvalRequired) руководитель меняет для своего отдела; глобальные — Админ.

### W4B.4 — Влияние CISO-005 (server-identity) на UX переключения
До CISO-005 isManager на фронте = client-resolved, кнопки можно вернуть DevTools'ом.
**Вопрос:** после server-identity нужно ли менять UX (например, при попытке запрещённого действия показывать toast «нет прав» вместо тихого скрытия кнопки)?
- **Гип. A:** оставить тихое скрытие (кнопки нет — нет вопросов), сервер просто отклоняет аномалии 403.
- **Гип. B:** показывать явное состояние «доступно только руководителю» (disabled + tooltip), чтобы UX был предсказуем и обучал ролям.
- **Гип. C:** комбинировать: скрывать необратимые (approve), но disabled+tooltip для read-режимов (отчёты byEmployee).

### W4B.5 — Переключение контекста / «смотреть как»
Сейчас impersonation/«view as» нет (`week-grid.tsx:131` viewEmployeeId для руководителя — это фильтр, не смена личности).
**Вопрос:** нужен ли руководителю режим «открыть таймшит сотрудника» (просмотр/правка чужой недели) и «вернуться к своему»?
- **Гип. A:** не нужно; руководитель работает в отчётах/доске, чужой таймшит не открывает поячеечно.
- **Гип. B:** нужен просмотр чужого таймшита (read-only) для проверки перед согласованием — без правки.
- **Гип. C:** нужна и правка чужого (заполнить за отсутствующего) — но с аудит-записью «кто за кого».
- **Реф:** Timetta «Управление командой» открывает таймшиты подчинённых; правка за сотрудника — отдельное право.

### W4B.6 — «Мой отдел» vs «все отделы» — тоггл или авто-scope?
После CISO-005 руководитель ограничен своим отделом + дочерними.
**Вопрос:** видит ли руководитель переключатель «Мой отдел / Все» (если у него права на несколько), или scope жёстко прибит к его отделу без выбора?
- **Гип. A:** жёсткий авто-scope: руководитель видит только свой отдел + дочерние, выбора нет.
- **Гип. B:** дефолт «Мой отдел», но есть дроп «Все отделы» для тех, у кого есть права (топ-менеджмент).
- **Гип. C:** scope = дерево отделов с чекбоксами (выбрать поддерево), как фильтр в отчётах.

### W4B.7 — Иерархия подчинённости в модели
Связи «сотрудник → его руководитель» в модели сейчас нет (scope = через `department`).
**Вопрос:** scope руководителя считается по дереву отделов или по прямому подчинению (manager-of)?
- **Гип. A:** по дереву отделов (руководитель отдела видит свой отдел + дочерние) — REQ-0011 уже даёт иерархию отделов.
- **Гип. B:** по прямому подчинению (поле `managerRef` у сотрудника) — точнее для матричных структур.
- **Гип. C:** гибрид: отдел для большинства, ручные исключения (со-руководитель) через доп. назначения.

## Тема 2. ПДн — UX (имена, КОД, согласие, маскирование)

### W4B.8 — Кто видит тоггл «Показывать ФИО» и кто его включает?
Тоггл в `settings/general-section.tsx` раздел «Безопасность», глобальный singleton.
**Вопрос:** тоггл revealEmployeeNames видит/переключает только Админ, или и руководитель?
- **Гип. A:** только Админ (это политика 152-ФЗ уровня воркспейса).
- **Гип. B:** руководитель видит состояние (read-only), включает Админ.
- **Гип. C:** per-отдел: руководитель может раскрыть ФИО внутри своего отдела, между отделами — всегда КОД.

### W4B.9 — Что видит рядовой сотрудник: ФИО или КОД?
CISO-007 ACCEPTED → revealNames=true допустимо для всех (все = сотрудники Кредос-С). Но тоггл по дефолту false.
**Вопрос:** при revealNames=true рядовой сотрудник тоже видит ФИО всех коллег в общих отчётах, или ФИО раскрывается только руководителям, а рядовому всегда КОД?
- **Гип. A:** тоггл глобальный → при true ФИО видят все авторизованные (текущее поведение сервера).
- **Гип. B:** ФИО только руководителям независимо от тоггла; рядовому — КОД даже при reveal=true.
- **Гип. C:** рядовой видит ФИО только своего отдела (коллеги), чужие отделы — КОД.
- **Реф:** OLAP_PII_SECURITY.md: измерение `employee` gated за isManager (`redactedPII:true`) — конфликтует с глобальным тогглом, надо согласовать.

### W4B.10 — Формат КОДа: понятен ли пользователю?
`employeeCode` = `Сотрудник·<deptCode>·<4 hex>`, напр. `Сотрудник·РАЗР·A3F1`.
**Вопрос:** этот КОД достаточно читаем для руководителя, чтобы опознать человека (при reveal=false), или нужен другой ярлык (инициалы, табельный №, должность+отдел)?
- **Гип. A:** КОД ок как технический различитель; опознание не требуется при reveal=false (смысл маскирования).
- **Гип. B:** заменить на инициалы + отдел (`И.П. · РАЗР`) — баланс приватности и узнаваемости.
- **Гип. C:** использовать табельный номер из 1С/Битрикс — устойчивая связь с кадровым учётом.

### W4B.11 — Маскирование на доске загрузки (capacity)
`employee-row.tsx:50` показывает `employee.name` plain — маскирование на доске НЕ реализовано на фронте (в отличие от отчётов).
**Вопрос:** должна ли доска загрузки тоже подчиняться revealNames (показывать КОД при false), или доска — всегда ФИО для руководителя?
- **Гип. A:** доска = инструмент руководителя → всегда ФИО (планировать по КОДу неудобно).
- **Гип. B:** доска подчиняется тому же тогглу, что и отчёты (единая политика ПДн).
- **Гип. C:** доска показывает ФИО своего отдела, КОД — чужих (если межотдельные проекты).

### W4B.12 — Маскирование в экспорте CSV
CISO-010: detail-CSV без role-guard, SICK-коды (медПДн) в табеле. Экспорт у `export-csv.tsx`, `timesheet-grid-rest.ts` (параметр withCodes).
**Вопрос:** кто может выгружать CSV и что в нём маскируется? Рядовой вообще может экспортировать?
- **Гип. A:** экспорт — только руководитель/Админ; рядовой кнопки не видит.
- **Гип. B:** экспорт доступен всем, но рядовому — обезличенный (КОД, без SICK-детализации).
- **Гип. C:** медицинские коды (Б/больничный) маскируются для всех кроме HR/Админ; обычный табель — по роли.
- **Реф:** KIMAI — экспорт-флаг блокирует, но не маскирует; Timetta — отдельные гранулы на export/print.

### W4B.13 — Согласие 152-ФЗ / правовое основание в UI
152FZ-003: нет зафиксированного правового основания; CISO-008: `note` может содержать медПДн.
**Вопрос:** нужен ли в UI элемент про 152-ФЗ — баннер согласия при первом входе, ссылка на ЛНА «Положение об учёте рабочего времени», подсказка-предупреждение в полях свободного текста?
- **Гип. A:** ничего в UI; основание оформляется кадрами офлайн (ЛНА под подпись), приложение не показывает.
- **Гип. B:** подсказки в полях `description`/`note`: «Не вводите медданные и ПДн третьих лиц».
- **Гип. C:** баннер-ознакомление при первом входе + ссылка на политику обработки ПДн в подвале.

### W4B.14 — Поле `note` отсутствия и медПДн
CISO-008: тип «больничный» провоцирует ввод диагноза → спецкатегория 152-ФЗ.
**Вопрос:** для отсутствий типа «больничный» нужно ли запрещать/скрывать свободный комментарий, или достаточно подсказки?
- **Гип. A:** для больничного убрать поле note вовсе (только даты + тип).
- **Гип. B:** поле есть, но с явной валидацией/подсказкой «без диагноза».
- **Гип. C:** note виден только автору и HR-роли (field-level), руководитель видит факт отсутствия без текста.

## Тема 3. Доступ к действиям (кто планирует / согласует / правит / удаляет)

### W4B.15 — Правка чужой записи: разрешена ли руководителю в UI?
RBAC_MODEL.md: руководитель R/W чужих своего отдела через REST; но в grid правка идёт по своему таймшиту.
**Вопрос:** должен ли руководитель прямо в сетке править/добавлять часы за подчинённого, или его правка чужого — только через отдельный экран/импорт?
- **Гип. A:** руководитель не правит чужие часы вручную; только согласует/отклоняет.
- **Гип. B:** руководитель открывает таймшит подчинённого и правит (см. W4B.5), с пометкой авторства.
- **Гип. C:** правка чужого — только Админ; руководитель read-only на чужие записи.

### W4B.16 — Удаление: ownership-guard (UC-TS-07)
CISO-005: delete берёт ownership из client-supplied ref (IDOR). После фикса — guard «только свои».
**Вопрос:** кто может удалять записи и что показывать при запрете? Soft-delete (восстановимо, `default-role.ts:39`) — виден ли «корзина»/восстановление?
- **Гип. A:** удаляет только автор свои DRAFT-записи; APPROVED — никто (только revoke→delete).
- **Гип. B:** руководитель может удалить чужую DRAFT своего отдела; есть «восстановить» (soft-delete) у Админа.
- **Гип. C:** удаление скрыто полностью, вместо него «обнулить часы» (текущий row-menu) — записи не исчезают.

### W4B.17 — Recall / revoke согласования
CISO-011: APPROVED не блокирует прямой PATCH. WI-10 recall не специфицирован.
**Вопрос:** может ли сотрудник отозвать свою SUBMITTED-запись? Может ли руководитель снять APPROVED обратно в DRAFT?
- **Гип. A:** сотрудник отзывает только SUBMITTED (не APPROVED); APPROVED размораживает только руководитель.
- **Гип. B:** после APPROVED — заморожено для всех, изменения только через Админа с аудит-записью.
- **Гип. C:** revoke как явное действие с обязательным комментарием (история согласований).
- **Реф:** `approval.logic.ts:225` уже есть «сотрудник отзывает только свои» для recall submit.

### W4B.18 — Read-only состояния: что серое и почему
После APPROVED ячейка locked (`row-menu.tsx` disabled), фон серый, меню скрыто.
**Вопрос:** достаточно ли визуально понятно, ПОЧЕМУ ячейка серая (согласовано)? Нужна ли иконка-замок + tooltip «согласовано <кем> <когда>»?
- **Гип. A:** серого фона достаточно; детали — в бейдже статуса периода (approval-bar).
- **Гип. B:** добавить замок + tooltip с автором/датой согласования на заблокированной ячейке.
- **Гип. C:** показывать на наведении «Согласовано. Чтобы изменить — запросите отзыв».

### W4B.19 — Согласование своей записи (SoD в UX)
C2: actor ≠ автор (`approval.logic.ts:159`), `skippedOwn` счётчик.
**Вопрос:** когда руководитель batch-согласует период, где есть и его собственные записи — как показать, что свои пропущены? Скрыть свои из очереди или показать с пометкой «нельзя — ваша»?
- **Гип. A:** свои записи не попадают в очередь согласования вовсе (фильтруются из списка).
- **Гип. B:** показываются, но чекбокс disabled + подсказка «ваши записи согласует другой руководитель».
- **Гип. C:** после batch — toast «N согласовано, M ваших пропущено (требуют другого согласующего)».

### W4B.20 — Кто согласует записи самого руководителя?
SoD запрещает самосогласование, но цепочки согласования нет.
**Вопрос:** чьи записи согласует руководитель отдела (свои)? Вышестоящий, второй руководитель, Админ?

- **Гип. C:** Админ/назначенный со-согласующий.

REQ-0013: проект касается 2-3 отделов (join `credosTimeProjectDepartment` с долями).
**Вопрос:** РП межотдельного проекта видит часы ВСЕХ участников проекта (всех отделов), а начальник отдела — только долю своего отдела в этом проекте?
- **Гип. A:** РП видит весь проект целиком (все отделы); начальник отдела — только своих сотрудников в проекте.
- **Гип. B:** РП видит весь проект, но ПДн чужих отделов маскированы (КОД); начальник отдела — ФИО своих.
- **Гип. C:** оба видят агрегаты по проекту, но drill-down до сотрудника — только в рамках своего scope.
- **Реф:** Timetta «Управление проектами» = Мои проекты (весь проект), «Управление командой» = Мои подчинённые (срез людей) — пересечение даёт нужную ячейку.

### W4B.23 — Чьи проекты видит рядовой при вводе часов
Рядовой выбирает проект/этап/тип в сетке.
**Вопрос:** рядовой видит ВСЕ проекты воркспейса в выпадашке или только проекты, где числится его отдел / где он в команде?
- **Гип. A:** все активные проекты (текущее — справочник открыт на чтение всем).
- **Гип. B:** только проекты своего отдела (REQ-0013 даёт привязку).
- **Гип. C:** только проекты, где он в команде (требует модели «команда проекта»).
- **Реф:** Timetta подставляет проекты по членству в команде проекта (`time-tracking-faq-select-role-in-timesheet`).

### W4B.24 — Контекст роли при вводе времени (Timetta-паттерн)
Сейчас запись = сотрудник + проект + тип работ; «в какой роли работал» не фиксируется.
**Вопрос:** нужно ли при вводе часов указывать роль/функцию (РП, аналитик, тестер) для последующей детализации OLAP, или тип работ это уже покрывает?
- **Гип. A:** тип работ (workType) достаточен; отдельная роль избыточна.
- **Гип. B:** добавить поле «роль в проекте» (подставляется из команды проекта), опционально.
- **Гип. C:** роль обязательна для межотдельных проектов (чтобы развести доли отделов по функциям).
- **Реф:** Timetta требует роль в таймшите (проектная/основная/доп), Админ может отключить поле.

### W4B.25 — Drill-down byEmployee и аудит доступа
OLAP_PII_SECURITY.md: измерение `employee` gated isManager. 152FZ-006: нет журнала «кто смотрел чьи данные».
**Вопрос:** когда руководитель проваливается в отчёте до конкретного сотрудника (byEmployee, чужой отдел через межотдельный проект) — нужен ли аудит-лог и видимое уведомление «доступ зафиксирован»?
- **Гип. A:** аудит не нужен на dev; на прод — тихий серверный лог без UI-уведомления.
- **Гип. B:** серверный аудит + ничего в UI (прозрачно для руководителя).
- **Гип. C:** при drill-down в чужой отдел — баннер «вы просматриваете данные вне вашего отдела, доступ логируется».
