## B1. Личный кабинет «Мои трудозатраты»

Факты: две вкладки `Segmented` («Мои часы» / «Мои периоды», `my-time-dashboard.tsx:65-70`). KPI — 4 карточки (Факт/Норма/Недогруз/Утилизация, `my-hours.tsx:108-113`). Баннер недозаполнения + дайджест руководителю (`reminder-banner.tsx`). Прямой кнопки перехода в таймшит нет — только текст «Внесите трудозатраты на странице «Записи»» (`reminder-banner.tsx:89`).

### Группа 1.1. Структура и навигация ЛК

**B1.1.** Двух вкладок («Мои часы» / «Мои периоды») достаточно, или нужна третья — например «Согласование» (для руководителя) прямо в ЛК?
— Гипотезы: (а) две вкладки, согласование — отдельный пункт меню (как сейчас); (б) добавить таб «Команда»/«Согласование» в ЛК для руководителя; (в) объединить «Мои часы» и «Мои периоды» в один скролл.
— Реф: Timetta — отдельные разделы «Таймшиты» (`shared.groups.timesheets`) и «Согласование» (`shared.groups.approval`) внутри личного экрана учёта.

**B1.2.** Нужна ли в ЛК прямая кнопка-CTA «Открыть таймшит / Заполнить неделю», ведущая сразу на нужную неделю в сетке? Сейчас — только текстовая подсказка, без ссылки/кнопки.
— Гипотезы: (а) кнопка в баннере недозаполнения → открывает «Записи» на проблемной неделе; (б) кнопка на каждой строке «Мои периоды» → открыть эту неделю; (в) оставить текстом без перехода.
— Реф: Timetta/Kimai — из списка периодов клик по периоду открывает его таймшит.

**B1.3.** Клик по строке недели в «Мои периоды» должен открывать таймшит этой недели?
— Гипотезы: (а) да, строка кликабельна → таймшит недели; (б) строка раскрывает детали (записи) inline; (в) некликабельна (как сейчас).
— Реф: Timetta «Таймшиты» — список периодов, элемент ведёт в период.

**B1.4.** Какой период по умолчанию показывают «Мои часы» — текущая неделя, текущий месяц, или последний выбранный? Есть ли переключатель периода в ЛК?
— Гипотезы: (а) текущая неделя; (б) текущий месяц; (в) переключатель неделя/месяц/квартал как в отчётах.
— Реф: Timetta — навигация `‹ дата — дата ›` + «Выбор периода» во всех списках.

### Группа 1.2. KPI-карточки (Факт / Норма / Недогруз / Утилизация)

**B1.5.** Четыре KPI достаточно, или нужны ещё (например «Согласовано ч», «На согласовании ч», «Биллируемые ч», «Переработка»)?
— Гипотезы: (а) 4 как есть; (б) +статус согласования; (в) +billable/утилизация по клиентским.
— Реф: Kimai — billable-часы и бюджет; Timetta — утилизация/рентабельность.

**B1.6.** «Недогруз» выводится как `−X` (недобор) / `+X` (перегруз) / `—` (нет нормы). Знак `+` для перегруза — это норм или путает (плюс читается как «хорошо»)?
— Гипотезы: (а) оставить ±; (б) «перегруз X» / «недобор X» словами; (в) цветовое кодирование вместо знака.
— Реф: общий UX — раздельные подписи понятнее знаков.

**B1.7.** «Утилизация» = доля клиентских часов, %. Нужна ли подпись/тултип с формулой и порогом «нормы» (например 75%)?
— Гипотезы: (а) только число; (б) число + тултип-формула; (в) число + цвет по порогу (зел/жёлт/красн).
— Реф: Timetta — утилизация как ключевая метрика, обычно с целевым значением.

**B1.8.** Карточки KPI должны быть кликабельными (drill-down в отчёт/записи) или это чисто индикаторы?
— Гипотезы: (а) индикаторы; (б) клик ведёт в «Отчёты» с пред-фильтром; (в) клик в «Записи» периода.
— Реф: Timetta «По периодам» (`byPeriods`) — агрегаты с переходом в детали.

### Группа 1.3. Баннер недозаполнения

**B1.9.** Сейчас баннер показывается при ЛЮБОМ `under > 0` (порога нет). Нужен ли порог (например показывать, только если недобор ≥ 4 ч или прошёл день напоминания)?
— Гипотезы: (а) любой недобор; (б) порог в часах; (в) показывать только начиная с reminderDayOfWeek; (г) только за закрытые/прошедшие дни недели.
— Реф: Timetta — напоминание привязано к dueDate (вторник след. недели), не к каждому часу.

**B1.10.** Баннер показывает только ТЕКУЩУЮ неделю. Нужно ли агрегировать прошлые незаполненные недели («У вас 2 незакрытые недели»)?
— Гипотезы: (а) только текущая; (б) текущая + N прошлых; (в) одна строка-сводка по всем долгам.

**B1.11.** Поведение баннера при заполненной неделе: прятать совсем (как сейчас) или показывать зелёный «Неделя заполнена ✓»?
— Гипотезы: (а) скрыть; (б) короткий success-стейт; (в) success + кнопка «Отправить на согласование».
— Реф: общий UX дашбордов — позитивный пустой стейт уместен.

**B1.12.** Текст баннера ведёт на «Записи». Если появится CTA-переход (B1.2) — баннер должен открывать конкретную неделю, а не общий список?

### Группа 1.4. Дайджест руководителю

**B1.13.** Дайджест показывает только тех, кто НЕ заполнил текущую неделю. Нужны ли ещё статусы — «на согласовании у меня N», «отклонено мной N»?
— Гипотезы: (а) только незаполнившие; (б) +ожидают согласования; (в) полноценный mini-approval прямо в дайджесте.
— Реф: Timetta «Согласование» — входящие на утверждение как отдельный список.

**B1.14.** При `revealEmployeeNames=false` дайджест показывает «Сотрудник · {deptCode}». Это приемлемо для руководителя (он и так знает свою команду) — или руководителю ФИО надо показывать всегда, а скрывать только в общих отчётах?
— Гипотезы: (а) единое правило (как сейчас, скрыто для всех); (б) руководитель видит ФИО своей команды, остальное скрыто; (в) роль-based раскрытие.
— Реф: CISO-007 / 152-ФЗ — минимизация ПДн; но руководитель — законный обработчик своей команды.

**B1.15.** Нужна ли в дайджесте кнопка «Напомнить» (пинг сотруднику) или «Открыть его таймшит»?
— Гипотезы: (а) только список; (б) кнопка-напоминание; (в) переход в таймшит сотрудника (если есть права).

**B1.16.** Дайджест — это сворачиваемый блок внутри баннера ЛК. Уместно ли это, или дайджест руководителя должен быть отдельной карточкой/вкладкой?
— Гипотезы: (а) внутри баннера (как сейчас); (б) отдельная KPI-карточка «Команда»; (в) отдельная вкладка ЛК.

### Группа 1.5. Статусы периодов

**B1.17.** Статус недели агрегируется по приоритету REJECTED > DRAFT > APPROVED > SUBMITTED (`period-status.ts`). При частичном согласовании (часть записей APPROVED, часть SUBMITTED) показывается SUBMITTED. Это понятно пользователю или нужен составной бейдж «3/5 согласовано»?
— Гипотезы: (а) один агрегированный статус; (б) бейдж-дробь «согласовано N из M»; (в) прогресс-полоска.
— Реф: Timetta статусы Черновик→На согласовании→Согласовано→Отклонено — обычно цельный статус периода.

**B1.18.** При REJECTED показывается первый rejectComment, даже если отклонённых записей несколько. Нужно ли показывать все причины / счётчик?
— Гипотезы: (а) первый коммент; (б) «N отклонено» + раскрытие списка; (в) самый поздний коммент.

**B1.19.** Окно истории «Мои периоды» = 182 дня (`WINDOW_DAYS=182`). Этого достаточно, или нужна пагинация/фильтр по году?
— Гипотезы: (а) ~26 недель хватает; (б) фильтр период/год; (в) бесконечный скролл.

## B2. Карточки сущностей и реестры внутри них

Факты: 3 развитые карточки (Проект — 10 вкладок; Сотрудник — 3; Отдел — 3). Реестры внутри — нативные FIELDS-relation (инлайн, авто-фильтр по родителю) + RECORD_TABLE (полный реестр, без авто-фильтра). Сводка/Бюджет/Команда проекта — front-component'ы. Заголовки карточек: проект=`code`, сотрудник=`lastName`, отдел=`name` (авто-поле, проблема). Все CRUD-действия нативны (Twenty).

### Группа 2.1. Состав и порядок вкладок

**B2.1.** Карточка проекта имеет 10 вкладок (Сводка, Обзор, Трудозатраты, Этапы, Связи с 1С, Бюджет, Команда, Отделы, Брони, Документы). Это не много? Нужна ли группировка/скрытие редких?
— Гипотезы: (а) 10 как есть; (б) объединить Сводка+Бюджет+Команда в одну «Обзор»; (в) убрать редкие (Связи с 1С / Брони) за «…ещё».
— Реф: Timetta карточка проекта — Детали/Активность/Вложения/Задания/Состояния/История (≈6), метрики в «Деталях».

**B2.2.** Порядок вкладок проекта: первой идёт «Сводка» (front-component), второй «Обзор» (реквизиты). Какая должна открываться по умолчанию — дашборд-сводка или реквизиты?
— Гипотезы: (а) Сводка первой (как сейчас); (б) Обзор первой (реквизиты важнее); (в) запоминать последнюю.
— Реф: Timetta — первой обычно «Детали» (реквизиты+агрегаты).

**B2.3.** «Сводка» и «Обзор» проекта — два разных таба. Не дублируют ли они смысл? Может, объединить реквизиты + ключевые метрики в один экран?
— Гипотезы: (а) держать раздельно; (б) объединить; (в) Сводка = виджеты-карточки сверху + реквизиты ниже на одной вкладке.

**B2.4.** Карточка сотрудника НЕ имеет вкладки «Обзор» — реквизиты (lastName, headedDepartments) вшиты в таб «Отделы». Нужна ли отдельная «Обзор» с ФИО/должностью/email/код?
— Гипотезы: (а) добавить «Обзор» сотрудника; (б) оставить в «Отделы»; (в) шапка карточки с реквизитами + табы только реестры.
— Реф: Timetta карточка работника — отдельный профиль + назначения.

**B2.5.** Карточка отдела: «Обзор» (руководитель/вышестоящий/код), «Сотрудники», «Проекты». Нужны ли «Загрузка/ёмкость» (матрица REQ-0016) и «Записи отдела» как вкладки? Сейчас их нет.
— Гипотезы: (а) добавить вкладку «Загрузка» (front-component матрица); (б) смотреть через «Сотрудники»→карточка; (в) отдельный отчёт вне карточки.
— Реф: Timetta — загрузка/ёмкость подразделения как отдельное представление.

### Группа 2.2. Реестры внутри карточек (inline vs RECORD_TABLE)

**B2.6.** Многие вкладки содержат ДВА представления одной связи: инлайн-таблица FIELDS (авто-фильтр по родителю) + RECORD_TABLE (полный реестр объекта, без авто-фильтра). Не путает ли пользователя «две таблицы записей»?
— Гипотезы: (а) оставить обе (инлайн=связанные, реестр=все); (б) только инлайн (отфильтрованную); (в) только реестр с предзаданным фильтром.
— Реф: Timetta — внутри карточки видны только связанные записи; «все» — в общем списке.

**B2.7.** RECORD_TABLE в карточке показывает ВЕСЬ реестр объекта без фильтра по родителю (нативное ограничение Twenty). Это ожидаемо или нужно технически добиться предфильтра?
— Гипотезы: (а) принять как есть + подпись «весь реестр»; (б) front-component с RestApiClient + предфильтр; (в) убрать RECORD_TABLE, оставить инлайн.

**B2.8.** Какие колонки нужны в инлайн-реестре «Трудозатраты» внутри карточки проекта/сотрудника? Сейчас: Состав работ, Дата, Часы, Проект, Вид работ, Сотрудник, Статус.
— Гипотезы: (а) текущий набор; (б) сократить (в карточке проекта «Проект» избыточен); (в) +Этап/+Комментарий/+billable.
— Реф: Timetta «Детали» строки: Дата · Клиент>Проект>Работа · Вид работ · Тариф · Часы · Комментарий.

**B2.9.** В карточке проекта колонка «Проект», а в карточке сотрудника колонка «Сотрудник» — избыточны (контекст и так известен). Скрывать контекстную колонку?
— Гипотезы: (а) скрывать колонку-родителя в инлайн-реестре; (б) оставить для единообразия; (в) разные view под каждый контекст.

**B2.10.** Реестры FTE-назначений и долей отделов используют join-объекты (`credosTimeEmployeeDepartment`, `credosTimeProjectDepartment`). Клик ведёт в карточку join-записи, а оттуда в сотрудника/отдел. Это лишний «прыжок»? Нужен прямой переход?
— Гипотезы: (а) принять двойной переход (нативно); (б) показывать join-поля inline без перехода; (в) front-component с прямой ссылкой.

### Группа 2.3. Поля «Обзор»

**B2.11.** «Обзор» проекта = 11 полей (code, externalCode, company, department, manager, category, status, startDate, endDate, plannedEffort, description). Все нужны на первом экране, или часть — во второстепенную секцию?

— Реф: Timetta — реквизиты сгруппированы (основное / финансы / команда).

**B2.12.** Заголовок карточки проекта = `code`, у проекта нет TEXT-поля `name`. Нужно ли добавить «название проекта» и сделать его заголовком (а код — подзаголовком)?
— Гипотезы: (а) добавить `name`, заголовок=name; (б) оставить code (компании удобнее по коду); (в) заголовок «code — name».
— Реф: Timetta `projectItemName` = «Наименование проекта» отдельно от кода.

**B2.13.** Заголовок карточки отдела = `name` (авто-поле, без TEXT labelIdentifier) — заголовок фактически пустой/служебный. Добавить явное TEXT-поле «Название отдела»?
— Гипотезы: (а) добавить TEXT `name`; (б) использовать `code` как заголовок; (в) «code — name».

**B2.14.** Карточка отдела «Обзор» — всего 3 поля. Нужны ли ещё: численность (Σ FTE), активные проекты, коэф. ёмкости (из настроек)?
— Гипотезы: (а) 3 поля; (б) +вычисляемые метрики front-component'ом; (в) метрики в отдельной вкладке «Сводка».

### Группа 2.4. Front-component вкладки (Сводка / Бюджет / Команда)

**B2.15.** Сводка/Бюджет/Команда проекта — кастомные front-component'ы (REST API). Какие именно метрики критичны в «Сводке»? Сейчас: статус/категория/бюджет/команда/этапы/период/активность.
— Гипотезы: (а) текущий набор; (б) +план vs факт деньги; (в) +утилизация/рентабельность.
— Реф: Timetta «Детали» — План vs Факт, прогресс бюджета (время+деньги), разбивка по сотрудникам/работам/периодам.

**B2.17.** «Команда» = сотрудники, кто списывал время + Σ часов. Нужны ли плановые роли/брони (а не только факт), как состав команды проекта?
— Гипотезы: (а) только фактически списавшие; (б) +забронированные/назначенные; (в) две секции «план» и «факт».
