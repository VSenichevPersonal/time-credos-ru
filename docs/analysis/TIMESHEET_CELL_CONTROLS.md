## A. КОРНЕВОЙ РАЗБОР СИМПТОМОВ

### Симптом 1 (корневой UX-баг): «контролы появляются при наведении, но нажать их не получается»

Контролы в `hour-cell.tsx`: чип-норма «8» (строки 166-202), кнопка ⇥ «заполнить будни» (226-264), кнопка ✎ комментарий через `CellComment` (268-270).

**Условие показа всех трёх — `active`, а НЕ `hover`:**
```
{active && !locked && value === 0 && ... ? <button норма> ...}      // чип
{active && !locked && value > 0 && onFill && (<button ⇥>)}          // заполнить
{active && !locked && value > 0 && onCommitDescription && <CellComment/>}  // ✎
```
`active` = координатная активность из `use-keyboard.ts` (`nav.isActive(row,col)`), т.е. ячейка «выбрана» клавиатурой/кликом. Заказчик называет это «при наведении мышки», но фактически контролы привязаны к **выбранной** ячейке, не к hover. (CSS `:hover` тут не используется — в Remote DOM псевдоклассы ненадёжны, всё на JS-`onMouseEnter`.)

**Почему нажать почти невозможно — фатальный конфликт обработчиков ячейки:**
```
// hour-cell.tsx, контейнер ячейки:
onClick={() => { onActivate(); if (locked) return; setDraft(...); setEditing(true); }}
onMouseDown={onActivate}
```
- `onMouseDown` → `onActivate()` → ячейка становится `active`. Контролы появляются.
- Тот же клик немедленно даёт `onClick` → `setEditing(true)` → ячейка **перерисовывается в `<input>`** (ветка `if (editing) return <input/>`, строки 92-128). Кнопки исчезают из DOM в том же тике.

**Итог:** один клик по ячейке = «выделить + сразу войти в редактирование». Кнопки (норма/⇥/✎) рендерятся только в состоянии `active && !editing`, которое при клике мышью существует доли мс. Чтобы кнопку увидеть и нажать, надо:
1. сделать ячейку active **без** клика по ней (т.е. клавиатурой-стрелками прийти на неё), и
2. не нажимать саму ячейку, а попасть точно в кнопку.

Кнопки используют `onMouseDown={(e) => e.preventDefault()}` (чтобы не терять фокус) + `onClick` со `stopPropagation`. В теории `mousedown` на кнопке должен сработать до входа в edit. Но:
- хит-зоны крошечные: чип высотой 18px, ⇥ и ✎ — 18×18px в углах ячейки **32px высотой** (строка 83 `height: 32`). При ширине колонки **58px** (`week-header.tsx`: `repeat(7, 58px)`) две угловые кнопки 18px + число почти перекрывают друг друга.
- порядок событий mousedown-на-ячейке vs mousedown-на-кнопке в Remote DOM не гарантирован; промах мимо 18px = клик по ячейке = вход в edit = кнопка пропала.
- из-за `firstEmptyCell` автофокуса (`week-grid.tsx`) активна обычно ПУСТАЯ ячейка → виден только чип нормы; ⇥ и ✎ (нужны при `value > 0`) почти никогда не показываются мышью, т.к. клик по заполненной ячейке мгновенно уводит в edit.

**Вывод по симптому 1:** контролы достижимы практически только с клавиатуры (прийти стрелкой, не кликая), а мышью — нет. Это не «недокрутили hover», это **архитектурный конфликт: клик одновременно = выбрать и = редактировать**, поэтому промежуточное состояние «выбрана, кнопки видны» нельзя задержать мышью.

### Симптом 2: ячейки мелкие

- Неделя (`hour-cell.tsx`): `height: 32`, ширина колонки `58px`, `fontSize: 13`, `padding 0 8px`. В эти 58×32 надо вместить: число + чип/⇥/✎ + точку-маркер коммента + замок. Физически тесно → кнопки наезжают, хит-зоны меньше 24px (минимум доступности).
- День (`day-row.tsx`): ячейка крупнее — `width: 72, height: 38, fontSize: 16`. Здесь контролов в ячейке нет (коммент вынесен в строку отдельным полем). День ощущается удобнее именно из-за размера и отсутствия конфликта.

**Вывод:** 58×32 физически мал для модели «контролы внутри ячейки». Либо расширять ячейку (ширина колонок дней), либо убрать контролы из ячейки.

### Симптом 3: «Заполнить 8×5 работает странно если 2+ строк»

Здесь **ДВА разных механизма с похожими названиями** — корень путаницы:

1. **Чип/⇥ в ячейке** (`onFill` → `weekly-grid.bulkFill` → `actions.bulkFill`, `use-timesheet-actions.ts:218`):
   заполняет будни **ТОЛЬКО ОДНОЙ строки** — той, чья ячейка активна, — **значением этой ячейки** (не обязательно 8), в **пустые** будни (заполненные не трогает).
2. **Кнопка тулбара «Заполнить 8×5»** (`toolbar.tsx:91` → `onFillStandardWeek` → `actions.fillStandardWeek` → `calcFillStandardWeek`, `use-timesheet-actions.ts:95`):
   заполняет **ВСЕ строки недели** фиксированной нормой **8 ч** (`DAILY_NORM_HOURS`) в пустые будни.
3. **Меню строки ⋯ → «Заполнить будни (8 ч)»** (`row-menu.tsx` → `onFillWeekdays` → `bulkFill(rowKey, 8)`):
   одна строка, фиксированные 8 ч.

**Что воспринимается как баг на 2+ строк:** пользователь жмёт ⇥ в ячейке, ожидая «заполнить 8×5» по аналогии с кнопкой, а ⇥ заливает **только активную строку** её собственным значением. При 1 строке разницы не видно; при 2+ — «почему заполнилась только одна?» / «почему не 8, а то что было в ячейке?». Три разных действия с почти одинаковыми словами «заполнить будни» дают три разных результата (какая строка? какое значение?). Логика каждого по отдельности корректна — баг в **неоднозначности модели**, а не в коде расчёта.

Дополнительно: `calcFillStandardWeek` корректно проходит по `rowList` (все строки), пропускает выходные/заполненное/locked. Тут именно две строки заполняются обе. А `bulkFill` (чип ⇥) — по одной. Заказчик, вероятно, тестирует чип ⇥ и видит «заполнилась не та / не все».

## B. УТОЧНЯЮЩИЕ ВОПРОСЫ ЗАКАЗЧИКУ (28 шт.)

Формат: вопрос — гипотезы — как у Timetta/Kimai.

### Группа 1. Модель взаимодействия с ячейкой (клик/выделение/редактирование)

**В1.** Клик по ячейке — должен **выделять** её (показать контролы, не входя в ввод) или **сразу открывать ввод**?
— Гипотезы: (а) 1 клик = выделить, 2-й клик/Enter/печать = ввод; (б) 1 клик = сразу ввод (как сейчас).
— Реф: Timetta/Kimai QuickEntry — клик = сразу inline-ввод в ячейку (число печатается), контролы не внутри ячейки.

**В2.** Нужно ли вообще промежуточное состояние «ячейка выбрана, но не редактируется»? Сейчас именно оно ломает контролы.
— Гипотезы: (а) убрать его, действия вынести из ячейки; (б) оставить, но входить в ввод только по двойному клику/Enter.

**В3.** Поведение Enter: сейчас Enter = «подтвердить и уйти на строку вниз». Это ожидаемо?
— Гипотезы: (а) Enter = вниз (как сейчас, Excel-style); (б) Enter = вправо; (в) Enter = просто подтвердить, фокус на месте.
— Реф: табличный стандарт — Tab вправо, Enter вниз.

**В4.** Tab — переходит на следующую ячейку дня (как сейчас) или должен прыгать на следующую строку после последнего дня?
— Реф: Timetta — Tab слева-направо по дням, перенос на след. строку.

**В5.** Какой ввод важнее по приоритету — мышь или клавиатура? Сейчас де-факто клавиатура (контролы мышью недостижимы), а заказчик работает мышью.
— Это определит всю переделку.

### Группа 2. Быстрый ввод нормы (чип «8»)

**В6.** Нужен ли чип-кнопка «8» прямо в пустой ячейке (1 клик ставит норму дня)?
— Гипотезы: (а) да, оставить, но сделать надёжно нажимаемым; (б) убрать — норму ставит кнопка «Заполнить 8×5» / меню строки; (в) заменить на «печать 8 + Enter».
— Реф: в Timetta/Kimai чипа-в-ячейке нет; норма ставится через копирование прошлой недели или автозаполнение из расписания.

**В7.** Норма дня берётся из производственного календаря (`useDailyNorm`, короткий день/праздник). Чип должен показывать именно её (6/7/8 ч), а не всегда 8?
— Сейчас чип показывает `norm` дня. Кнопка тулбара — всегда 8. Несогласованность.

**В8.** Где удобнее ставить норму одним действием: в ячейке (чип), в меню строки (⋯), в шапке дня (клик по «8 ч» под датой → залить день), или только глобальной кнопкой?

### Группа 3. Размещение контролов (hover / всегда-видимые / меню)

**В9.** Контролы ячейки (норма/заполнить/коммент) — где им жить?
— Гипотезы: (а) убрать из ячейки → всё в меню строки ⋯ + ввод числа; (б) оставить в ячейке, но крупнее и без конфликта с edit; (в) показывать в отдельной панели под выбранной ячейкой.
— Реф: Timetta — действия в тулбаре и контекст-меню строки/ячейки, НЕ плавающие кнопки в ячейке.

**В10.** Допустимо ли, что меню строки ⋯ становится единственным местом для «заполнить будни / очистить / дублировать / коммент», а ячейка отвечает только за ввод числа?
— Это самый надёжный вариант под Remote DOM (нет flaky-hover, крупные пункты с подписями).

**В11.** Нужно ли контекстное меню по правому клику на ячейке (Timetta «добавить строку до/после», «очистить»)? Или только кнопка ⋯?
— Замечание: правый клик в Remote DOM может быть недоступен/перехвачен host-ом — нужно проверить.

**В12.** Сейчас при выборе пустой ячейки виден чип «8», а при выборе заполненной — ⇥ и ✎. Это разное наполнение по состоянию — ок или путает?

### Группа 4. Комментарий к записи

**В13.** Комментарий привязан к чему: к **ячейке** (день+проект+вид работ — как сейчас, `description` на записи), к **строке** (проект+вид работ на неделю), или к **записи дня**?
— Реф: Timetta/Kimai — комментарий на уровне аллокации (ячейки/записи дня). Текущая реализация совпадает.

**В14.** Как открывать комментарий: иконка ✎ в ячейке (сейчас, недостижима мышью), пункт меню строки, или поле в режиме День (там работает)?

**В15.** Нужен ли индикатор «у ячейки есть комментарий» (сейчас — точка-маркер 5px слева сверху)? Достаточно ли заметен?

### Группа 5. Дублирование

**В16.** «Дублировать строку» (меню ⋯) сейчас = префилл проекта в форму добавления, вид работ выбираешь заново, часы вводишь заново. Это ожидаемо, или дублировать надо **вместе с часами**?

**В17.** Нужен ли «копировать прошлую неделю со часами» как основной ускоритель (он есть в `CopyMenu`)? Заказчик им пользуется?

### Группа 6. Bulk-fill — главный источник путаницы

**В18.** Что заказчик понимает под «Заполнить 8×5»? Сейчас есть ТРИ разных действия (см. раздел A.3). Нужно свести к одному понятному?

**В20.** Заполнять заполнять **только пустые** будни или **перезаписывать** уже введённые?

## C. РЕКОМЕНДАЦИЯ АНАЛИТИКА (как должно быть)

Опирается на: референсы (Timetta/Kimai inline-ввод + действия в меню/тулбаре), impeccable (хит-зоны ≥24px, один понятный путь действия, нет скрытых flaky-контролов), ограничения Remote DOM (нет drag, нет надёжного hover/host-DOM).

### C1. Модель «клик = редактирование», БЕЗ промежуточного состояния «выбрана-с-кнопками»

Главная причина бага — попытка усидеть на двух стульях: ячейка одновременно «выбирается» (показывает кнопки) и «редактируется» (становится input). Решение — **развести**:

- **Клик по ячейке = сразу inline-ввод** (как Timetta/Kimai). Никаких плавающих кнопок внутри ячейки.
- **Контролы строки убрать из ячейки** в меню строки `⋯` (оно уже есть, `row-menu.tsx`, с крупными пунктами и подписями — надёжно под Remote DOM).
- **Норму дня** ставить: печать «8»+Enter, ИЛИ пункт меню строки «Заполнить будни», ИЛИ глобальная «Заполнить 8×5». Чип-в-ячейке — удалить как ненадёжный (или, если заказчик настаивает — см. C2-альт).
- **Комментарий** — пункт меню строки «Комментарий к дню…» (по выбранной ячейке) ИЛИ как в режиме День (отдельное поле). Иконку ✎ в ячейке убрать.

Это убирает конфликт обработчиков целиком: у ячейки остаётся один `onClick` → ввод. Контролы становятся достижимы мышью (крупное меню), клавиатура продолжает работать (печать цифры → ввод).

### C2. Альтернатива (если заказчик хочет контролы видимыми в сетке)

Если важно «нажимать прямо в сетке, не открывая меню»:
- **Увеличить ячейку** до ~72×40 (см. В24) и показывать контролы по `active` **без входа в edit по простому клику**: 1 клик = выбрать (кнопки видны и нажимаемы), ввод = двойной клик / Enter / печать. Хит-зоны кнопок ≥24px.
- Минус: расход ширины (7×72 + колонки), потеря места под «Проект/Вид работ». Менее чисто, чем C1.

Рекомендую **C1** как основной (надёжнее под Remote DOM, ближе к референсам), C2 — fallback по требованию заказчика.

### C3. Размер ячеек

- Поднять высоту ряда недели с 32 до ~38-40 (выровнять с Днём), `fontSize` 13→14.
- Ширину колонок дней 58→64-68 при условии, что 9 колонок влезают без горизонтального скролла; иначе оставить 58, но при модели C1 тесная ячейка уже не проблема (в ней только число).

### C4. Фикс bulk-fill — устранить неоднозначность

- **Свести к двум явным, по-разному названным действиям:**
  1. «**Заполнить будни этой строки**» (меню строки ⋯) — норма дня, только пустые. Убрать чип ⇥ из ячейки ИЛИ переименовать так, чтобы было очевидно «эта строка».
  2. «**Заполнить 8×5 (вся неделя)**» (тулбар) — все строки, норма дня (а не хардкод 8), только пустые.
- **Унифицировать значение:** везде использовать норму дня из `useDailyNorm` (короткий день/праздник), а не константу 8. Сейчас тулбар льёт 8, чип — `norm`. Сделать SSOT.
- **Чип ⇥ «значение активной ячейки»** — либо убрать (источник «странного» поведения на 2+ строк), либо явно подписать «продлить N ч на будни строки».

## F. АЛЬТЕРНАТИВНЫЙ РАЗБОР — аналитик-продукт (дополнение к арх-анализу)

**Дата:** 2026-06-21  
**Метод:** сравнение Timetta/Kimai + синтез 30 ответов заказчика → конкретные решения.

---

### F1. Как устроены референсы (конкретно)

#### Timetta (SaaS-таймшит, ru-рынок)

| Аспект | Timetta |
|---|---|
| Клик по ячейке | Сразу inline-ввод, `<input type="number">` на месте ячейки |
| Промежуточное состояние «выбрана» | **Нет** — click=edit, без фазы «кнопки видны» |
| Контролы строки | Плавающий ряд кнопок появляется при **hover на строку** (не на ячейку) |
| Норма дня | Авто из расписания (schedule-based), клик «Расставить по расписанию» = глобальный |
| Комментарий | Иконка 💬 при hover строки → tooltip-форма (не в ячейке) |
| Bulk-fill | «По расписанию» (глобально) / «Копировать прошлую неделю» |
| Ввод форматов | `8`, `8.5`, `8,5`, `8:30` — все принимает; `1ч30м` — нет |
| Статус строки | Pill в левой колонке (черновик / отправлен / согласован) |
| Dot-маркер коммента | Есть — маленький dot в правом нижнем углу ячейки |
| Мобайл | Отдельный упрощённый вид; web-грид — только desktop |

**Ключевой вывод Timetta:** контролы на строку (hover по строке), ячейка — только число. Полное разделение ввода и управления.

#### Kimai (open-source, EU-рынок)

| Аспект | Kimai |
|---|---|
| Клик по ячейке | Сразу inline-ввод |
| Preset-кнопки | Preset chips ПО КОНФИГУ пользователя (не фикс «8»), показываются при фокусе ячейки в отдельной всплывающей панели СНИЗУ |
| Bulk-fill | «Fill week» в контекст-меню строки; «Fill all» — кнопка тулбара |
| Комментарий | Отдельный клик «+» в ячейке → modal (не inline) |
| Форматы | `8`, `8.5`, `8:30`; **не принимает** `1ч30м` |
| Duplicate | Копирует строку **со всеми часами** (отличие от текущей нашей логики) |
| Рабочие субботы | Учитывает production calendar (конфиг); `isWeekend` НЕ хардкод |

**Ключевой вывод Kimai:** preset-панель выезжает СНИЗУ при фокусе — не конфликтует с click=edit, т.к. живёт вне ячейки.

#### Harvest / Toggl Track (глобальный SaaS)

Оба используют **клик = сразу ввод** + действия только в меню строки/тулбаре. Нет hover-кнопок в ячейке. Ячейка — минималистичный input.

**Вывод по всем трём:** hover-кнопки внутри ячейки — паттерн, которого у лидеров нет. Архитектор прав: C1 = индустриальный стандарт.

### F2. Синтез 30 ответов заказчика → конкретные решения

Заказчик ответил на 30 вопросов. Ниже — решения по группам:

#### Модель взаимодействия (В1-В5)
**Решено:**
- 1 клик = **сразу ввод** (подтверждено заказчиком — вариант «б»)
- Промежуточное состояние «выбрана-с-кнопками» — **убрать**
- Enter=вниз, Tab=вправо — **оставить**
- Esc=отмена — **оставить**
- Приоритет — **мышь** (значит Remote DOM hover реально нужен → C1 без hover в ячейке — правильное решение)

#### Чип нормы (В6-В8)
**Решено:**
- Чип «8» в ячейке — **оставить**, но механика меняется: показывать при **фокусе ячейки** (т.е. уже в режиме edit, как Kimai-preset), не как overlay до input
- Реализация: когда input активен и value=0 → показывать chip-панель СНИЗУ (вне ячейки, не конфликтует с input)
- Значение чипа = **norma из settings** (useDailyNorm), не хардкод 8
- Второй чип «↩ повтор» (последнее значение строки) — опционально (Kimai pattern)

#### Контролы (В9-В12)
**Решено:**
- Hover-кнопки в ячейке — **убрать** (C1 основной)
- Все row-level действия → меню строки **⋯**
- Правый клик — **не делать** (Remote DOM risk)
- Контролы ячейки разные по состоянию (empty/filled) — упразднить деление, ячейка = только число

#### Комментарий (В13-В15)
**Решено:**
- Комментарий к **ячейке** (день × проект = credosTimeEntry.description) — подтверждено
- Открывать: **пункт меню строки** «Комментарий к [Пн]…» или поле при вводе (inline под input)
- Dot-индикатор — **оставить**, сделать чуть крупнее (6-7px, corner)

#### Дублирование (В16-В17)
**Решено:**
- Duplicate = **со всеми часами** (как Kimai). Текущая логика prefill-without-hours → переделать
- «Копировать прошлую неделю» (CopyMenu) — **оставить**, заказчик пользуется

#### Bulk-fill (В18-В23)
**Решено:** (детали в F3)
- Свести к двум явным действиям с разными именами
- Везде норма из `useDailyNorm` (убрать хардкод DAILY_NORM_HOURS=8)
- Чип ⇥ «значение ячейки» — **убрать**; вместо него пункт в ⋯ «Заполнить будни нормой»
- Только **пустые** — подтверждено (не перезаписывать)
- Рабочие субботы — **учитывать** (есть у заказчика); `isWeekend` → заменить на `isNonWorkingDay(date, calendar)`

#### Размеры (В24-В25)
**Решено:**
- Высота ряда 32 → **40px** (touch/mouse friendly, ≥ Material min)
- Ширина дней 58 → **64px** при C1 (нет кнопок в ячейке → 64px достаточно, не 72)
- «Проект/Вид работ» сужается → **ellipsis + tooltip** при hover (текст при hover строки, не ячейки)

#### Статусы, клавиатура, мобайл (В26-В28)
**Решено:**
- APPROVED read-only + замок — подтверждено; copy-from APPROVED **разрешено**
- Статус строки — **показывать** pill слева (черновик/согл./отклонён)
- **Десктоп-приоритет**; планшет опционально позже
- Индикатор факт vs план — **да**, простая подсветка ячейки (уже есть частично)

### F3. Третий вариант C3 (Kimai-style preset panel)

Архитектор описал C1 и C2. Есть вариант C3, который совмещает плюсы обоих:

**C3: Клик = ввод + preset-панель снизу при фокусе**

```
Состояние: ячейка сфокусирована (input активен, value=0):

┌────────────────────┐
│ [Пн] input:  |     │   ← input в ячейке (64×40)
└────────────────────┘
┌──────────────────────┐
│  [8 ч] [4 ч] [↩ 6] │  ← preset-панель СНИЗУ (вне ячейки, absolute/portal)
└──────────────────────┘
   клик на пресет → вставляет значение в input → blur = save
```

**Плюсы C3:**
- Нет конфликта click=edit vs кнопки: панель появляется ПОСЛЕ того как input открыт
- Кнопки крупные (≥32px), не конкурируют за место в ячейке
- Паттерн Kimai — проверен в production
- Remote DOM: панель можно рендерить как sibling к grid, не internal к ячейке

**Минусы C3:**
- Усложнение архитектуры (портал/позиционирование в Remote DOM — нет getBoundingClientRect)
- В Remote DOM absolute positioning сложнее (нет host-DOM координат)

**Вывод:** C3 концептуально лучший UX, но Remote DOM делает его реализацию сложнее. Приоритет: **C1 в первой волне**, C3 — если заказчик настоит на чипах.
