# UI-вопросы волны 4-A: ДАННЫЕ · ИНТЕГРАЦИИ · ОТЧЁТЫ · СВЯЗИ КАРТОЧЕК

**Формат каждого:** `**W4A.x** вопрос — гипотезы (а)(б)(в) — реф: Timetta/Kimai/1С`
**Цель волны:** углубление по 4 темам, слабо покрытым в волнах 1–3 (W1/2 = таймшит/планирование/поиск/массовые; W3 = ячейки + планирование). Здесь — выгрузка в учётные системы, импорт справочников, аналитика для руководства, связность данных и аудит.

## Контекст: что УЖЕ реализовано (НЕ переспрашиваем)

- **Экспорт detail-отчёта:** CSV, 7 колонок `[Дата, Сотрудник, Отдел, Проект, Вид работ, Часы, Статус]` (`reports-detail.ts:111`, тип `DetailRow` :14-22). Разделитель `;`, BOM (опц.), UTF-8, `\r\n`, экранирование RFC 4180 (`detailToCsv` :131, `escapeCsv` :115). **XML/XLSX нет.**
- **Табельная сетка** (`reports-timesheet-grid.ts`): коды Т-13 — `Я` (явка), `ОТ` (отпуск VACATION), `Б` (больничный SICK), `ДО` (без сохранения UNPAID), `К` (командировка, future), `НН` (прочее/неизвестно). Часы > 0 → `Я` приоритетнее отсутствия (:191). При фильтре по проекту коды отсутствий скрыты (:161).
- **Метрики reports-calc:** `fact`, `client`, `norm` (база×headcount×capacityFactor − отсутствия), `util=client/fact`, `under=norm−fact`, разбивка `byCategory`. Динамика — `computeTimeseries` по месяцам (YYYY-MM). **Денежных метрик НЕТ** (`[[no-billable-concept]]` :11). План/факт по проектам — `projects-plan-fact.ts` (`planned/fact/remaining/overrun/pct`).
- **Связь с 1С:** объект `credosTimeBillingLink` — junction проект↔документ 1С (M:N задел). Поля: `externalSystem` (дефолт `'1С'`), `docType`, `externalId`, `number`, `date`, `amount`. **Синхронизации/API нет** — комментарий «Синхронизация позже» (:17).
- **Внешние коды справочников:** только `Project.externalCode` (внешний код Директум/1С для сверки, nullable). У employee/work-type/department/stage поля externalId НЕТ.
- **Связи:** `Project.company` → стандартный Company (Twenty CRM); `Project.stages` → Stage; `Project.department` + multi-dept через `ProjectDepartment` (доли в часах `plannedEffortShare`). **Проект↔договор (Opportunity/Deal) НЕ реализовано.**
- **employee↔department:** primary `Employee.department` + история через `EmployeeDepartment` (`ftePercent` 0–100, `startDate`, `endDate`). Руководитель — `Department.head` → Employee.
- **Аудит:** стандартные Twenty `createdAt/updatedAt/createdBy/updatedBy` (встроенные, в UI не выведены явно). Кастом в TimeEntry — `approvedBy`, `approvedAt`, `rejectComment`.
- **Settings (singleton):** нормы, валидация, планирование, согласование, напоминания, `revealEmployeeNames` (ПДн 152-ФЗ).

## Группа 1. Выгрузка в 1С / ЗУП (Т-13, табель, маппинг)

**W4A.1** Как заказчик СЕЙЧАС переносит данные о часах в учётную систему — есть ли вообще выгрузка из текущего процесса, и куда (1С:ЗУП / 1С:Бухгалтерия / 1С:ERP / Excel-табель руками)? — (а) ничего нет, табель ведут отдельно в 1С вручную; (б) выгрузка в Excel → импорт в 1С; (в) прямой обмен через типовой формат 1С (XML EnterpriseData); (г) интеграции нет и не планируется (1С отдельно). — реф: 1С — типовой обмен идёт через XML EnterpriseData или загрузку табеля Т-13.

**W4A.2** Что именно нужно выгружать в 1С — табель учёта рабочего времени (Т-13, по сотрудникам/дням/кодам явки) или распределение часов по проектам (для управленческого учёта/себестоимости)? — (а) только Т-13 (кадры/расчёт зарплаты); (б) только проектные часы (управленка); (в) оба отдельными файлами; (г) проектные часы + договоры (для актов). — реф: Timetta — отдельно «табель» (Т-13) и «отчёт по проектам»; в 1С это разные приёмники.

**W4A.3** Если нужен Т-13 — текущая табельная сетка уже даёт коды `Я/ОТ/Б/ДО/К/НН`. Достаточно ли этого набора, или 1С:ЗУП заказчика использует расширенный справочник видов времени (РВ, С — сверхурочные, Н — ночные, РП — работа в выходной)? — (а) хватает 6 кодов; (б) нужны ещё РВ/С/Н; (в) нужен полный справочник «Виды использования рабочего времени» из ЗУП заказчика (прислать). — реф: 1С:ЗУП — буквенные/цифровые коды Т-13 (В=выходной, РВ=работа в выходной, С=сверхурочные, Н=ночные).

**W4A.4** В Т-13 для дней без явки и без отсутствия (выходные, праздники) — нужно ли проставлять код `В` (выходной) из производственного календаря, или пустую ячейку? Сейчас выходные просто пустые. — (а) пусто ок; (б) проставлять `В` по `workdayCalendar`; (в) `В` для выходных + отдельно праздники. — реф: классический Т-13 — выходные помечаются `В`.

**W4A.5** Формат файла выгрузки в 1С — какой приоритетнее: CSV (как сейчас detail), XLSX (готовый бланк Т-13 с шапкой/итогами), или XML EnterpriseData (машинный обмен)? — (а) CSV хватит; (б) нужен XLSX-бланк Т-13 (печатная форма); (в) XML для автозагрузки; (г) сначала CSV, XML позже. — реф: 1С принимает CSV-загрузкой, но «родной» обмен = XML EnterpriseData; печать табеля = XLSX/PDF.

**W4A.6** Маппинг сотрудника при выгрузке — по какому ключу 1С опознаёт человека: ФИО, табельный номер, ИНН, или внешний GUID? Сейчас у Employee нет поля «табельный номер» / externalId. — (а) ФИО (рискованно при тёзках); (б) добавить `personnelNumber` (табельный №); (в) добавить `externalId` (GUID 1С); (г) оба. — реф: 1С:ЗУП — ключ сотрудника = табельный номер или GUID, ФИО ненадёжно.

**W4A.7** Маппинг проекта/договора при выгрузке управленческих часов — по `Project.externalCode` (есть, для Директум), или нужен отдельный код именно для 1С (статья затрат / номенклатурная группа / договор)? — (а) externalCode переиспользуем; (б) нужно отдельное поле `accountingCode` (статья/номенклатурная группа 1С); (в) маппинг через `billingLink` на договор 1С. — реф: 1С — себестоимость разносится по номенклатурным группам/статьям, не по «коду проекта».

**W4A.8** Период выгрузки в 1С — фиксированный календарный месяц (как Т-13), или произвольный диапазон / по статусу согласования (только APPROVED)? — (а) только месяц; (б) месяц + только согласованные записи; (в) произвольный период; (г) месяц для Т-13, период для управленки. — реф: Т-13 — строго календарный месяц; управленческий разнос может быть периодом.

**W4A.9** Должна ли выгрузка в 1С учитывать ТОЛЬКО согласованные (APPROVED) записи, чтобы в учёт не попал черновик/отклонённое? Сейчас detail-CSV выгружает все статусы. — (а) только APPROVED; (б) все, но с колонкой статуса (как сейчас); (в) настраиваемо галочкой при выгрузке. — реф: Timetta — в биллинг/выгрузку идут только утверждённые часы.

**W4A.10** Нужна ли ДВУСТОРОННЯЯ связь с 1С (объект `billingLink` это заготовка под M:N проект↔документ) — подтягивать обратно из 1С номер/дату/сумму акта на карточку проекта, или выгрузка односторонняя? — (а) только выгрузка (односторонне); (б) подтягивать акты/счета обратно вручную (ввод в billingLink); (в) автосинк из 1С по API (фаза позже). — реф: billingLink уже спроектирован под хранение docType/number/date/amount.

## Группа 2. Импорт справочников (проекты / сотрудники / виды работ)

**W4A.11** Как заказчик планирует ПЕРВИЧНО загрузить справочники (десятки проектов, штат сотрудников, виды работ) — руками по одному, импортом из Excel, или выгрузкой из 1С/Директум? — (а) ручной ввод (мало записей); (б) массовый импорт Excel/CSV; (в) выгрузка из 1С:ЗУП (сотрудники) + Директум (проекты); (г) штатный импорт Twenty (CSV-import есть в платформе). — реф: Twenty — встроенный CSV-import объектов; 1С/Директум — выгрузка справочников в XML/Excel.

**W4A.12** Сотрудники — источник истины это 1С:ЗУП (кадры) или Twenty workspace (пользователи)? Сейчас Employee имеет `workspaceMemberRef` (связь с пользователем). Кто заводит нового сотрудника? — (а) кадры в 1С → импорт в time; (б) админ заводит вручную в time; (в) автосоздание Employee из WorkspaceMember при появлении пользователя; (г) гибрид. — реф: Timetta — сотрудники синкаются из HR-системы; Twenty — WorkspaceMember = аккаунт.

**W4A.13** Нужна ли регулярная синхронизация штата (новые приёмы / увольнения / переводы между отделами), или импорт разовый при внедрении? Сейчас `EmployeeDepartment` хранит историю с датами — кто её ведёт? — (а) разовый импорт, дальше руками; (б) периодический импорт-обновление из 1С; (в) переводы вводит руководитель/админ в time. — реф: 1С:ЗУП — кадровые приказы (приём/перевод/увольнение) — источник истории отделов.

**W4A.14** Компании (клиенты) для `Project.company` — берутся из стандартного CRM-объекта Company Twenty. Они уже ведутся в CRM, или для time-модуля их надо заводить/импортировать отдельно? — (а) Company уже в CRM (переиспользуем); (б) импортировать клиентов из 1С/Excel; (в) синк Company из внешней CRM/1С. — реф: Twenty — Company стандартный объект, может уже содержать клиентов.

**W4A.15** Виды работ (`workType`) — это глобальный справочник или у каждого отдела свой (поле `department` nullable = глобальный)? Как заказчик хочет их вести — централизованно админом или отделы сами? — (а) глобальный единый список; (б) per-отдел (каждый РП заводит свои); (в) глобальные + отдельские дополнения; (г) импорт из существующего классификатора работ. — реф: Kimai — activity бывают глобальные и привязанные к проекту; Timetta — типы работ глобальный справочник.

**W4A.16** При импорте/обновлении справочников — как разрешать дубли и обновления (matching по коду)? Например повторный импорт проектов: обновлять по `code`/`externalCode` или создавать новые? — (а) upsert по `code`; (б) upsert по `externalCode`; (в) всегда создавать (дедуп руками); (г) пропускать существующие. — реф: 1С/Twenty-import — matching обычно по внешнему ключу (код/GUID).

**W4A.17** Нужен ли массовый импорт ПЛАНОВЫХ данных (план по проектам/отделам, бронь) из Excel, или планирование только в UI? — (а) только UI; (б) импорт плана из Excel при старте проекта; (в) импорт из MS Project / Директум-плана. — реф: Timetta — ресурсный план вводится в UI, но есть импорт из проектных систем.

## Группа 3. Отчёты глубже (для РП и руководства)

**W4A.18** Денежных метрик в отчётах сейчас НЕТ (`[[no-billable-concept]]`). Нужна ли руководству рентабельность в РУБЛЯХ (ставка сотрудника/часа × часы, сравнение с бюджетом договора), или достаточно «рентабельности в часах» (util, план/факт)? — (а) только часы (как сейчас); (б) нужны деньги: ставка×часы → себестоимость; (в) нужны деньги: биллинг (выручка по клиентским часам) и маржа; (г) деньги позже, фаза 2. — реф: Timetta — billable rate, revenue, cost, margin — ядро отчётности; Kimai — hourly rate на проект/пользователя.

**W4A.19** Если деньги нужны — где хранить ставку: на сотруднике (cost rate), на проекте/договоре (billing rate), на виде работ, или комбинация? Сейчас таких полей нет. — (а) cost rate на сотруднике; (б) billing rate на проекте/договоре; (в) обе ставки (себестоимость + выручка); (г) матрица сотрудник×проект. — реф: Timetta — раздельные cost rate (сотрудник) и bill rate (проект/роль); Kimai — rate на user/project/activity с приоритетом.

**W4A.20** Какие СВОДНЫЕ отчёты нужны руководству сверх текущих (detail + agg + timesheet-grid)? — (а) утилизация по отделам/сотрудникам за период; (б) план/факт по проектам с отклонением; (в) динамика команд (тренд загрузки по месяцам, есть `computeTimeseries`); (г) сравнение периодов (этот месяц vs прошлый, факт vs прошлый год). — реф: Timetta — utilization, project profitability, time variance; Kimai — отчёты по user/project/customer.

**W4A.21** Сравнение план/факт по периодам — нужно ли видеть отклонение НЕ только итогом, но и помесячно (план месяца vs факт месяца, освоение бюджета по timeline)? — (а) только итог проекта (как `projects-plan-fact`); (б) помесячная раскладка план/факт; (в) burn-down (остаток бюджета по времени). — реф: Timetta — project burn / план-факт по периодам.

**W4A.22** Нужен ли ДАШБОРД (сводный экран с KPI-плитками + графики), или хватает табличных отчётов с фильтрами? Сейчас дашборд-кода нет. — (а) хватает таблиц; (б) нужен дашборд руководителю (util %, недогруз, топ-проекты); (в) дашборд РП (его проекты: план/факт/команда); (г) персональный дашборд сотрудника (мои часы/норма/статус). — реф: Timetta — дашборды по ролям; Kimai — виджеты на главной.

**W4A.23** Экспорт АГРЕГИРОВАННЫХ отчётов (не только detail) — нужен ли экспорт сводных таблиц (util по отделам, план/факт) в CSV/XLSX для руководства, или агрегаты смотрят только на экране? — (а) только на экране; (б) экспорт агрегатов в CSV; (в) экспорт в XLSX с форматированием; (г) экспорт графиков/дашборда в PDF. — реф: Timetta/Kimai — экспорт любого отчёта в Excel/CSV/PDF.

**W4A.24** Утилизация сейчас `util = client / fact` (доля клиентских часов в отработанных). Руководству привычнее `util = client / norm` (доля клиентских от нормы рабочего времени) — что считать «утилизацией»? — (а) client/fact (текущее); (б) client/norm (от ёмкости); (в) обе метрики рядом; (г) настраиваемо. — реф: Timetta — utilization = billable / capacity (от нормо-часов), не от факта.

**W4A.25** Нужны ли отчёты по ОТСУТСТВИЯМ отдельно (сводка отпусков/больничных по сотрудникам/отделам за период) для кадров/руководителя? Данные есть в absence. — (а) не нужно (это кадры в 1С); (б) сводка отсутствий по периоду; (в) остатки отпусков (нужен ввод начислений — сложно); (г) только в табеле Т-13. — реф: Timetta — absence report; обычно остатки отпусков ведёт HR-система.

## Группа 4. Связи карточек и аудит данных

**W4A.26** Проект↔Договор НЕ реализовано (нет связи с Opportunity/Deal). Нужна ли связь проекта с договором/сделкой для актов и биллинга, или договор живёт только в 1С (через `billingLink`)? — (а) не нужно (1С отдельно); (б) связать с Opportunity Twenty (сделка/договор в CRM); (в) отдельный объект «Договор» в time; (г) хватает `billingLink` (ссылка на док 1С). — реф: Timetta — проект ↔ контракт ↔ клиент — обязательная цепочка для биллинга.

**W4A.27** Цепочка проект→компания→договор: при выгрузке акта/отчёта клиенту — какой уровень группировки нужен (по проекту, по договору, по клиенту с несколькими проектами)? — (а) по проекту; (б) по клиенту (все проекты компании); (в) по договору (если появится связь); (г) все три уровня в отчёте. — реф: Timetta — отчёт клиенту группируется по контракту/проекту.

**W4A.28** История «сотрудник↔отдел» (`EmployeeDepartment` с датами+FTE) — отчёты сейчас используют primary `Employee.department` или историю с датами? При переводе сотрудника в середине месяца — на какой отдел падают его часы? — (а) текущий primary отдел (история игнорируется); (б) отдел на дату записи (по истории); (в) пропорционально периодам. — реф: 1С — табель/затраты разносятся по подразделению НА ДАТУ, с учётом переводов.

**W4A.29** Аудит изменений: стандартные Twenty `createdBy/updatedBy/createdAt/updatedAt` есть, но в UI не выведены. Нужен ли руководителю/админу видимый журнал «кто и когда менял запись часов» (особенно после согласования)? — (а) не нужно; (б) показывать createdBy/updatedAt в детали записи; (в) полный журнал изменений (audit log) для согласованных записей; (г) только факт правки после approve (флаг «изменено после согласования»). — реф: Timetta — lock согласованного периода + журнал; 152-ФЗ — журналирование действий с ПДн.

**W4A.30** Защита ПДн (`revealEmployeeNames`, CISO-007): при ВЫГРУЗКЕ в 1С/CSV для расчёта зарплаты ФИО обязательно. Как развести «маскировка имён в UI/отчётах» и «полные ФИО в выгрузке для 1С»? — (а) выгрузка всегда с полными ФИО (по роли с правом); (б) выгрузка по табельному номеру (без ФИО); (в) маскировка не распространяется на экспорт для уполномоченной роли; (г) две выгрузки (обезличенная управленка / именная для ЗУП). — реф: 152-ФЗ — минимизация ПДн в UI, но расчёт зарплаты требует идентификации; разграничение по ролям.

**W4A.31** При удалении справочной записи (проект/сотрудник/отдел) со связанными часами — что должно происходить с историческими записями времени? Сейчас onDelete у связей разный (CASCADE/SET_NULL). Нужно ли запрещать удаление при наличии часов (архивирование вместо delete)? — (а) запрет удаления при связанных записях; (б) мягкое архивирование (флаг `isArchived`, скрыт из выбора, история цела); (в) текущее поведение БД ок. — реф: Timetta/Kimai — справочники не удаляются, а деактивируются (visible=false), чтобы не рвать историю.

### Ключевые блокеры / развилки

1. **W4A.1 + W4A.2** — КАК заказчик выгружает в учёт СЕЙЧАС и ЧТО нужно (Т-13 vs проектные часы) — определяет весь скоуп интеграции 1С. Спросить конфиг ЗУП.
2. **W4A.3 + W4A.6 + W4A.7** — справочник кодов явки и ключи маппинга (табельный №, статья затрат) — без этого выгрузка в 1С нерабочая; нужны поля, которых сейчас НЕТ.
3. **W4A.18 + W4A.19** — деньги (ставки/маржа) go/no-go — крупнейшее расширение модели; сейчас осознанно `[[no-billable-concept]]`. Риск переусложнения (см. memory «не переусложнять»).
4. **W4A.28** — отчёты по истории отдела на дату записи vs primary — скрытый источник ошибок при переводах в середине периода.
5. **W4A.30** — коллизия 152-ФЗ-маскировки и выгрузки именного табеля для ЗУП — нужно ролевое разграничение экспорта.
6. **W4A.31** — архивирование справочников вместо удаления — затрагивает целостность истории (реф: оба конкурента деактивируют, не удаляют).
