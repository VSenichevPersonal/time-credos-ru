## 0. TL;DR — рекомендованная схема диалога

Заказчик хочет: при «Планировать» выбирать **способ** (Равномерно / Вручную по месяцам), для равномерного — диапазон **«С … ПО»**, и чтобы раскид шёл по **рабочим** дням (фикс бага из `PLANNING_MODELS.md`).

**Рекомендация — заменить inline-поля строки на inline-ПАНЕЛЬ плана (поповер на `useState`)**, открываемую кнопкой «✎ План» в строке проекта в режиме «Планировать». Панель — пошаговая, но все шаги на одном экране (не wizard-переходы):

```
┌─ План: «Внедрение СЭД» ───────────────────────── [×] ─┐
│ 1. Способ:  (•) Равномерно   ( ) Вручную по месяцам   │
│ 2. Период:  С [01.07.2026]  ПО [13.09.2026]           │
│ 3. Объём:   [ 500 ] часов        ▸ единица: часы ▾     │
│                                                        │
│ ── Превью раскида (по рабочим дням) ──────────────────│
│  июл  ████████ 184 ч                                  │
│  авг  ████████ 184 ч                                  │
│  сен  ███      132 ч   Σ = 500 ч ✓                     │
│  ⚠ авг: 184 ч > ёмкость 160 ч (овербукинг +24)        │
│                                                        │
│              [ Отмена ]   [ Сохранить план ]          │
└────────────────────────────────────────────────────────┘
```

**MVP (фаза 1):** способ-селектор (Равномерно дефолт) + поля «С/ПО» + раскид по рабочим дням + живое превью + предупреждение овербукинга. **Фаза 2:** «Вручную по месяцам» (таблица период×часы). **Фаза 3:** единица FTE%/ч-в-неделю, гранулярность проект×сотрудник.

Технически реализуемо в Remote DOM: панель = `useState`-поповер (паттерн `cell-comment.tsx`/`row-menu.tsx`), `<input type="date">` УЖЕ работает в песочнице (`project-plan-row.tsx:155`), превью считается в воркере функцией-двойником `plannedHoursInPeriod` до сохранения — без сети.

## 2. 18 вопросов UX/UI + рекомендованный ответ

> Формат: **Q** вопрос · **A** рекомендация · *почему (реф/песочница)*.

### Блок 1. Открытие и контейнер диалога

**Q1. Где/как открывается ввод плана: inline-поля в строке (как сейчас) или отдельная панель?**
**A1.** Inline-**панель** (поповер) по кнопке «✎ План» в строке проекта; строка остаётся компактной, панель раскрывается под/над строкой. Inline-поля «часы+дата» оставить как быстрый путь для тривиального случая — но способ/диапазон/превью требуют места, которого в строке нет.
*Песочница: панель = `useState`-поповер, паттерн `cell-comment.tsx`. Модалка-как-host запрещена → именно inline-поповер. Timetta вводит план в отдельной карточке брони — подтверждает «карточка, не строка».*

**Q2. Панель — пошаговый wizard (экран→экран) или один экран со всеми полями?**
**A2.** **Один экран**, шаги — визуальные секции (1 Способ → 2 Период → 3 Объём → Превью) сверху вниз. Wizard в Remote DOM поповере = лишние состояния и потеря контекста доски.
*impeccable: не дробить простую форму на шаги; когнитивная нагрузка ниже на одном экране при ≤4 полях. Превью требует видеть все поля сразу.*

**Q3. Где якорится панель, чтобы не перекрыть доску и не уехать за вьюпорт?**
**A3.** Под строкой проекта (push-down, сдвигает строки ниже — как раскрытие drill в `dept-row`), а не absolute-overlay. Ширина = ширина области полей строки. Закрытие: «×», Esc, клик «Сохранить/Отмена».
*Песочница: overlay с точным позиционированием в воркере хрупок; push-down — нативный поток, надёжен. Esc-закрытие — паттерн `project-plan-row.tsx:83`.*

### Блок 2. Способ распределения

**Q4. Как показать выбор способа и какой дефолт?**
**A4.** Радио-группа (`role=radiogroup`) из 2 пунктов: **(•) Равномерно** (дефолт) · **( ) Вручную по месяцам**. Дефолт «Равномерно» — покрывает 80% кейсов и совместим с текущим поведением.
*Реф: Timetta `booking-methods.md` — равномерное распределение один из базовых; для MVP берём 2 из 5. `PLANNING_MODELS.md` рек. №2 — «равномерно + вручную закрывают 80%».*

**Q5. Сразу давать все 5 методов Timetta (фронт/бэк/вся-ёмкость/%-расписания)?**
**A5.** **Нет в MVP.** Только Равномерно + Вручную. Фронт-/бэк-загрузку и «вся ёмкость» — фаза 3, как доп. radio-пункты в той же группе (расширяется без перестройки UI).
*Реф: `PLANNING_MODELS.md` B3/B4 — гипотезы без подтверждённого спроса. keep-it-simple: не gold-platить.*

**Q6. Показывать ли у проекта на доске бейдж текущего способа («равномерно»/«вручную»)?**
**A6.** Да, маленький бейдж в строке проекта рядом с часами (напр. «≈ равномерно» / «✎ вручную»). Прозрачность «откуда часы».
*Реф: Timetta — метод виден в карточке брони. impeccable: статус системы должен быть виден.*

### Блок 3. Период «С … ПО»

**Q7. Какие поля диапазона и какой контрол?**
**A7.** Два `<input type="date">`: **«С»** и **«ПО»**. «С» дефолт = `project.startDate` или сегодня; «ПО» = `project.endDate`. Сейчас «С» не редактируется — это и есть запрос заказчика.
*Песочница: `type="date"` уже работает (`project-plan-row.tsx:155`). Заказчик прямо просил «С (дата) … ПО (дата)».*

**Q8. Валидация диапазона?**
**A8.** «ПО» ≥ «С»; иначе блок «Сохранить» + подпись «дата ПО раньше С». Пустая «ПО» → см. Q14. Способ раскида не сохраняем при невалидном диапазоне (не молча).
*Песочница: блокировка кнопки + текст ошибки (не `confirm`). Паттерн `parseEffort` уже отбрасывает невалид.*

**Q9. «С» = старт раскида и для прошедших периодов?**
**A9.** MVP: «С» = буквальная дата начала раскида (как ввёл руководитель), прошлое тоже получает план. «От сегодня»/точку прогноза — отложить (фаза 3, чекбокс «не планировать прошлое»).
*Реф: Timetta «точка прогноза» — мощно, но это отдельная фича (`forecasting-principles.md`). `PLANNING_MODELS.md` E1.*

### Блок 4. Объём и единица

**Q10. В чём вводим объём и нужен ли выбор единицы в MVP?**
**A10.** MVP — **часы (итого на диапазон)**, как сейчас; селектор единицы показать как `▾`-дропдаун (disabled-заглушка «часы»), чтобы UI был готов к расширению. Фаза 3 — добавить «ч/неделя» и «FTE %».
*Реф: Timetta 4 единицы (`resource-plan.md`). Но MVP не строить раньше спроса. impeccable: не показывать неактивные опции как активные — заглушку пометить «скоро».*

**Q11. Как вводить FTE%/ч-в-неделю, когда добавим?**
**A11.** Тот же объём-инпут, меняется суффикс и пересчёт в превью: «50 %» → раскид = 50% рабочих часов каждого периода; «20 ч/нед» → 20 × число недель. Хранить нормализовано в часах (`plannedEffort`) + поле `planUnit` для отображения.
*Реф: Timetta «процент от расписания», FTE. Инвариант Σ=plannedEffort сохраняется.*

### Блок 5. «Вручную по месяцам» (фаза 2)

**Q12. Контрол ручного ввода — таблица период×часы, в каком масштабе?**
**A12.** Таблица **месяц × часы** (строка на каждый месяц диапазона С…ПО), инпут часов в каждой. Масштаб «месяц» — дефолт (крупно, помещается в поповер); «недели» — опция при коротких проектах.
*Реф: Timetta масштаб день/нед/месяц (`resource-plan.md`). Месяцы компактнее для поповера. `PLANNING_MODELS.md` C2.*

**Q13. При смене диапазона/способа что с ручными значениями?**
**A13.** При переходе Равномерно→Вручную: предзаполнить таблицу равномерным раскидом (не пустую). При сдвиге дат проекта ручные значения **сохранять** (не пересчитывать), показать алерт «диапазон изменился, проверьте раскид». Строка-сверка «Σ введено = N / план M».
*Реф: Timetta ручной метод — значения остаются при сдвиге плашки. `PLANNING_MODELS.md` C3.*

### Блок 6. Живое превью

**Q14. Что показывает превью и когда обновляется?**
**A14.** Под полями — **мини-бар по периодам горизонта** (недели или месяцы как на доске) с числом часов в каждом, считается по **рабочим дням** на лету (в воркере, без сети) при каждом изменении поля. Внизу — строка-сверка «Σ = plannedEffort ✓». Это прямой ответ на «сколько ляжет в неделю».
*Реф: Timetta «общая сумма по строке за все периоды» (`resource-plan.md`). impeccable: немедленная обратная связь. Превью = функция-двойник `plannedHoursInPeriod`, но делитель — Σ рабочих часов (фикс бага из `PLANNING_MODELS.md` рек.№1, переиспользует `workHoursBetween`).*

**Q15. Превью показывать в рабочих или календарных днях?**
**A15.** **Рабочих** (без выходных/праздников РФ из `credosTimeWorkdayCalendar`). Это и есть фикс бага: ёмкость доски считается по рабочим (`deptCapacity`), план — по календарным → не бьются. Превью сразу демонстрирует совпадение план↔ёмкость.
*Реф: Timetta делит по рабочему расписанию. `PLANNING_MODELS.md` B2/рек.№1 — корень непонимания заказчика.*

### Блок 7. Овербукинг, проект без endDate, гранулярность

**Q16. Предупреждать ли об овербукинге прямо в превью?**
**A16.** Да, мягко: период, где раскид > ёмкости отдела, подсветить (⚠ + «184 ч > 160 ч») в превью, но **НЕ блокировать** сохранение. Доска и так показывает конфликт (`composeCell`, `conflict`).
*Реф: Timetta — овербукинг не блокируется, показывается. `PLANNING_MODELS.md` H1. impeccable: предупреждать, не запрещать.*

**Q17. Проект без даты «ПО» — обязать или разрешить?**
**A17.** MVP: для способа «Равномерно» **обязать «ПО»** (иначе раскид = 0, проект невидим — текущая скрытая потеря Demand). Если «ПО» пусто — подпись «укажите дату ПО, иначе план не попадёт на доску», кнопка «Сохранить» неактивна. Авто-конец (фронт-метод Timetta) — фаза 3.
*Реф: `calc-load.ts:195` — без endDate раскид=0. `PLANNING_MODELS.md` E3. Обязать проще, чем фронт-метод; честнее для пользователя.*

**Q18. Гранулярность плана в диалоге: проект / проект×отдел / проект×сотрудник?**
**A18.** MVP — **проект целиком** (как сейчас; доли отделов REQ-0013 раскладываются автоматически по `sharesByProject`). План×сотрудник — фаза 3 (сейчас нагрузка делится `1/headcount`, `calc-load.ts:436`). В диалоге MVP не усложнять выбором уровня.
*Реф: Timetta планирует поимённо, но это отдельный экран «ресурсный план». `PLANNING_MODELS.md` D2 — без подтверждённого спроса. keep-it-simple.*

### Доп. вопросы (роли, отмена, происхождение часов)

## 4. Поэтапный план реализации

### Фаза 1 — MVP (закрывает прямой запрос заказчика)
1. **Фикс раскида по рабочим дням** в `calc-load.ts`: делитель `plannedHoursInPeriod` = Σ рабочих часов диапазона (через `workHoursBetween` + `hoursByDay`), а не `endDate−startDate+1`. Инвариант Σ=plannedEffort сохранить. (рек.№1 `PLANNING_MODELS.md`).
2. **Inline-панель плана**: новый компонент `project-plan-panel.tsx` (поповер на `useState`, паттерн `cell-comment.tsx`), триггер «✎ План» в строке проекта вместо/рядом с inline-полями.
3. **Поля**: радио способа (Равномерно дефолт; «Вручную» — задизейблен в MVP с пометкой «скоро»), «С» (`type=date`, теперь редактируемая), «ПО» (`type=date`), объём (часы, как `parseEffort`).
4. **Живое превью**: функция-двойник раскида по рабочим дням считает часы по периодам горизонта на лету; мини-бар + строка-сверка Σ; подсветка овербукинга (сравнение с `deptCapacity`).
5. **Сохранение**: `usePlanEdit.save` (PATCH `plannedEffort/startDate/endDate`) — уже готов. Валидация ПО≥С, ПО обязательна.
6. **Бейдж способа** «≈равновномерно» в строке.

*Объект проекта менять НЕ обязательно для фазы 1: «С»=startDate, «ПО»=endDate, способ=всегда равномерно. Поле способа понадобится только с фазой 2.*
