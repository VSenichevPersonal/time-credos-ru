**Дата:** 2026-06-21
**Автор:** аналитик AITEAM
**Повод:** заказчик не понимает модель планирования («500 ч до 13.09» → как система это раскидывает) и хочет разобрать альтернативы.

---

## 0. TL;DR

- **Сейчас** план задаётся как «итого часов на срок» (`plannedEffort` + `startDate` + `endDate`) и **размазывается РАВНОМЕРНО по календарным дням** (не по рабочим!) от старта до конца. Других режимов нет. Ручной правки распределения нет.
- **Главная неточность нашей формулы:** делитель = число **календарных** дней (`endDate − startDate + 1`), а НЕ рабочих. Праздники, выходные, отпуска, неполная ставка в раскид плана **не входят**. Часы «капают» и в субботу, и 1 января.
- **Timetta** (главный референс): план задаётся в 4 единицах (часы / % от расписания / FTE / дни), 3 масштабах (день/неделя/месяц) и **5 методах распределения** (ручной, фронт-загрузка, вся ёмкость, равномерно, % расписания). Наша «равномерно» — лишь 1 из 5, и то по календарным, а не рабочим дням.
- **Kimai**: планирования-раскида нет вообще — есть только **бюджет-потолок** (время + деньги) с прогресс-баром на уровнях Клиент/Проект/Работа. Это про контроль лимита, не про распределение во времени.
- Нужно решение заказчика по ~35 вопросам (раздел 4): единица плана, метод распределения, гранулярность, ручная правка, старт раскида, учёт нерабочих дней, связь план/бронь/факт, визуализация.

### 1.2 ТОЧНАЯ формула раскида (`calc-load.ts:189-205`)

Единственная функция распределения — `plannedHoursInPeriod()`. Её используют ВСЕ слои: проекты, доли отделов, планы отдела без проекта, брони.

```
plannedHoursInPeriod(plannedEffort, startDate, endDate, period):
  totalDays = utcDay(endDate) − utcDay(startDate) + 1        // КАЛЕНДАРНЫЕ дни, не рабочие
  overlap   = min(endDate, period.to) − max(startDate, period.from) + 1
  return plannedEffort × overlap / totalDays
```

**Что это значит на примере «500 ч, старт 14.07, до 13.09»:**
- `totalDays` = 62 календарных дня (включая все выходные/праздники).
- На один календарный день приходится `500 / 62 ≈ 8.06 ч` — и в будни, и в выходные одинаково.
- В колонку «неделя» попадает `8.06 × 7 ≈ 56.4 ч`; в неполную стартовую/конечную неделю — пропорционально пересечению.
- Итог по горизонту = 500 ч (инвариант сохраняется), но внутри размазано **календарно ровно**.

### 1.3 Важные следствия и «дыры» текущей модели

1. **Делитель — календарные дни, не рабочие.** Праздники РФ, выходные, отпуска, неполная ставка НЕ уменьшают плановую нагрузку в раскиде. (Контраст: ЁМКОСТЬ доски — `deptCapacity`/`employeeLoadCells` — наоборот, считается по рабочим часам календаря и вычитает отсутствия. То есть план и ёмкость считаются по РАЗНЫМ правилам → визуально load может «не биться» с интуицией.)
2. **Нет `endDate` → раскида нет вообще.** `if (!plannedEffort || !startDate || !endDate) return 0` — проект без даты окончания в Demand на доске = 0 часов (попадает в список «без плана», `deptProjectLoads.unplanned`).
3. **Единственный метод — «равномерно».** Нет фронт-/бэк-загрузки, нет ручного ввода по периодам, нет выравнивания по ёмкости.
4. **Старт раскида = жёстко `startDate`.** Не «от сегодня», не «от точки прогноза». Прошедшие периоды тоже получают плановые часы.
5. **Гранулярность плана — проект целиком ИЛИ доля отдела** (REQ-0013, `plannedEffortShare`). На уровне СОТРУДНИКА плана нет: `employeeLoadCells` делит нагрузку отдела поровну `1/headcount` (см. `calc-load.ts:425-461`, комментарий «allocation по людям в модели нет»).
6. **Ручная правка распределения отсутствует.** `usePlanEdit` умеет PATCH-ить только `plannedEffort` и даты целиком → пересчёт раскида авто. Нельзя задать «в июле 300, в августе 200».
7. **Бронь использует тот же равномерный раскид** (`bookingHoursInPeriod`), хотя по смыслу (Timetta) у брони должно быть 5 методов.
8. **Период колонок** — недели или месяцы (`buildPeriods`), якорь `anchor` (обычно «сегодня»), горизонт `count`.

## 2. Референсы: как раскидывают план «у взрослых»

### 2.1 Timetta (главный легальный референс — `research/timetta/docs/`)

**Ресурсный план** (`projects-components-projects-resource-plan.md`) — план оценивается в разрезе **член команды × работа × календарный период**. Переключатели:

- **Единица планирования:** часы / **% от расписания** / **FTE** / **дни**.
- **Масштаб:** по дням / неделям / месяцам (`booking-scale.md`). При смене масштаба значения распределяются **пропорционально рабочему времени по расписанию пользователя** (не календарно ровно!).
- **Режимы представления:** Прогноз / Оценка.
- Авто-заполнение из структуры работ (для «автоматического типа планирования»).

**5 методов бронирования** (`resources-components-booking-methods.md`) — прямой ответ на вопрос заказчика «какие бывают модели раскида»:

| Метод Timetta | Что делает | Овербукинг |
|---|---|---|
| **Ручное планирование** | менеджер вбивает ёмкость в каждый слот руками; при сдвиге плашки введённые значения остаются | возможен |
| **Фронтальное планирование часов** | бронирует ВСЮ доступную ёмкость с даты старта, пока не наберётся N часов; дата окончания вычисляется авто (front-loaded по доступности) | НЕ приводит |
| **Вся доступная ёмкость** | на каждый слот периода — все свободные часы ресурса | НЕ приводит |
| **Равномерное распределение** | N часов поровну между **рабочими днями** периода (≈ наша модель, но по рабочим, не календарным) | возможен |
| **Процент от расписания** | X% от рабочего расписания на каждый рабочий день | возможен |

Ключевые принципы:
- **Soft/Hard бронь** — отдельная ось (резерв), НЕ план-оценка и НЕ факт; soft не потребляет ёмкость, hard потребляет; овербукинг не блокируется, показывается конфликтом. (У нас это уже реализовано — `composeCell`.)
- **Точка прогноза** (`forecasting-principles.md`): факт до точки + оценка после; точка = сегодня / начало недели / начало месяца, **сдвигается во времени**. У нас раскид всегда от `startDate`, точки прогноза нет.
- **Ёмкость** = FTE × Расписание × Календарь; отпуска/праздники уменьшают.

### 2.2 Kimai (`research/timetta-kimai-timesheet-views.md`, `research/kimai/`)

- **Планирования-распределения НЕТ.** Есть только **`BudgetTrait`**: денежный `budget` + временной `timeBudget` на уровнях **Клиент / Проект / Работа**, с прогресс-барами и % выполнения.
- Это **потолок-лимит** («не более 500 ч»), а не модель раскида во времени. Полезно как отдельная фича «бюджет/лимит проекта», ортогональная распределению.
- Наш `budgetRemaining` (plannedEffort − factHours) — уже примитивный аналог Kimai-бюджета по времени.

### 2.3 Общие PSA / resource-management модели (MS Project, Float, Resource Guru, Forecast)

| Модель | Суть | Когда нужна |
|---|---|---|
| **Even spread (равномерно)** | часы / число дней. **Наша текущая.** | простая оценка, «прикинуть нагрузку» |
| **Manual per-period** | менеджер сам вводит часы по неделям/месяцам | реальные неравномерные проекты (этапы) |
| **Front-loaded / back-loaded** | больше часов в начале / в конце | внедрения (старт-тяжёлые), сдача (финиш-тяжёлые) |
| **Milestone / by-stage** | план привязан к этапам/вехам, у каждой свои даты и часы | проекты с фазами (у нас есть `project-stages`!) |
| **% allocation / FTE** | «Иванов на 50% с июля по сентябрь» | мышление «людьми», а не «часами» |
| **Capacity leveling** | автораспределение с учётом свободной ёмкости (как Timetta «фронтальное»/«вся ёмкость») | избежать овербукинга автоматически |

## 4. Вопросы заказчику (сгруппированы; вопрос + гипотезы + реф)

> Цель: понять ЖЕЛАЕМУЮ модель. Каждый вопрос — с вариантами-гипотезами, чтобы заказчику было проще выбрать.

### A. Единица плана (в чём вводим)

**A1.** В чём руководителю удобнее вводить план: «итого часов на срок» (как сейчас) / «часов в неделю» / «% занятости (50%)» / «человеко-дни»?
*Гип.: часы итого (привычно) · часы/нед (прозрачнее нагрузка) · % (мышление людьми). Реф: Timetta — 4 единицы на выбор.*

**A2.** Нужна ли единица «FTE / % от ставки» («Иванов на 0.5 ставки с июля»)? Или только часы?
*Гип.: да, для людей · нет, только часы проекта. Реф: Timetta FTE/% от расписания.*

**A3.** Нужен ли «бюджет-потолок» отдельно от плана-раскида (нельзя списать больше N часов/денег)?
*Гип.: да (Kimai BudgetTrait) · хватает `budgetRemaining` · не нужно. Реф: Kimai budget + timeBudget.*

**A4.** Деньги планируем или только часы? (сейчас ограничение [[no-billable-concept]] — ставок нет).
*Гип.: только часы · нужны деньги/ставки позже. Реф: Kimai — деньги; Timetta — rate matrices.*

### B. Метод распределения (как раскидываем по времени)

**B1.** Подтверждаете, что дефолт — «равномерно по сроку»? Или нужен другой основной метод?
*Гип.: равномерно ок · чаще неравномерно (этапы) · фронт-загрузка. Реф: Timetta — равномерно лишь 1 из 5.*

**B2.** Делить часы по **рабочим** дням (без выходных/праздников) или по **календарным** (как сейчас)?
*Гип.: рабочие (правильнее, исчезнут «часы в субботу») · календарные (проще). Реф: Timetta делит по рабочему расписанию. ВАЖНО: сейчас баг-подобное поведение — календарно.*

**B3.** Нужна ли «фронт-загрузка» (больше часов в начале) и «бэк-загрузка» (к сдаче)?
*Гип.: да, для внедрений · да, для сдачи · нет. Реф: Timetta «Фронтальное планирование».*

**B4.** Нужен ли метод «занять всю свободную ёмкость» (автозаполнение до 100%)?
*Гип.: да (быстро закрыть ресурс) · нет (риск овербукинга). Реф: Timetta «Вся доступная ёмкость».*

**B5.** Нужен ли авто-метод, который НЕ создаёт овербукинг (раскид с учётом свободной ёмкости)?
*Гип.: да, важно · нет, овербукинг ок (видно конфликт). Реф: Timetta фронт-метод; PSA capacity leveling.*

### C. Ручной ввод и правка распределения

**C1.** Нужно ли руководителю править раскид вручную по периодам («в июле 300, в августе 200»)?
*Гип.: да, обязательно · нет, авто хватит. Реф: Timetta «Ручное планирование».*

**C2.** Если правит вручную — в каком масштабе вводит: по неделям / месяцам / дням?
*Гип.: месяцы (крупно) · недели (точнее) · дни (детально). Реф: Timetta масштаб день/неделя/месяц.*

**C3.** После ручной правки: при сдвиге дат проекта значения должны **остаться** или **пересчитаться**?
*Гип.: остаться (как ввёл) · пересчитаться авто. Реф: Timetta ручной метод — значения сохраняются при сдвиге плашки.*

**C4.** Можно ли смешивать: часть периодов вручную, остальное авто-равномерно по остатку?
*Гип.: да (гибко) · нет (либо-либо). Реф: гибридов в Timetta нет — выбор метода целиком.*

**C5.** Нужны ли «отмена/повтор» (undo/redo) при ручном планировании?
*Гип.: да · не критично. Реф: Timetta ресурсный план — undo/redo есть.*

### D. Гранулярность плана

**D1.** На каком уровне планируем часы: проект целиком / проект×отдел (доли, REQ-0013) / проект×сотрудник / по этапу?
*Гип.: отдел (как сейчас) · сотрудник (точнее) · этап. Реф: Timetta — член команды × работа.*

**D2.** Нужен ли план на уровне **сотрудника** (сейчас нет — нагрузка отдела делится поровну `1/headcount`)?
*Гип.: да, важно кто именно · нет, отдела хватает. Реф: Timetta планирует поимённо.*

**D3.** Нужно ли привязывать план к **этапам/вехам** проекта (у нас есть `project-stages`)? У каждого этапа свои даты+часы?
*Гип.: да (фазы внедрения) · нет (проект целиком). Реф: PSA milestone-планирование.*

**D4.** Доли отделов (`plannedEffortShare`): задаёт ли РП их вручную, или авто-делить поровну?
*Гип.: вручную · поровну авто · по headcount. Реф: Timetta — раскладка по подразделениям команды.*

### E. Старт и горизонт распределения

**E1.** С какой даты начинать раскид: от `startDate` проекта (сейчас) / от сегодня / от выбранной даты?
*Гип.: startDate · сегодня (прошлое не планируем). Реф: Timetta «точка прогноза» = сегодня/нед/мес.*

**E2.** Прошедшие периоды должны получать плановые часы или обнуляться (там уже факт)?
*Гип.: обнулять, показывать факт · оставлять план. Реф: Timetta факт-до-точки + оценка-после.*

**E3.** Проект **без даты окончания**: что делать (сейчас план = 0, не виден на доске)?
*Гип.: считать «открытым» с авто-концом (фронт-метод) · требовать endDate · считать бессрочно по часам/нед. Реф: Timetta фронт-метод вычисляет конец сам.*

**E4.** Горизонт доски (недели/месяцы вперёд) — фиксированный или настраиваемый пользователем?
*Гип.: настраиваемый · фикс (напр. 12 нед / 6 мес). Реф: Timetta — навигация по периодам.*

### F. Нерабочее время, ставка, отсутствия

**F1.** Раскид плана должен «огибать» праздники/выходные (0 часов в эти дни)?
*Гип.: да (иначе нереалистично) · нет (упрощённо). Реф: Timetta — по расписанию. Сейчас НЕ огибает.*

**F2.** Отпуска/больничные сотрудников должны уменьшать план или только ёмкость?
*Гип.: только ёмкость (план остаётся, виден конфликт) · и план тоже (перераспределять). Реф: Timetta — ёмкость падает, план — нет.*

**F3.** Неполная ставка (FTE < 100%) учитывается в раскиде плана?
*Гип.: да · нет (только в ёмкости/headcount). Реф: Timetta FTE × расписание.*

**F4.** Индивидуальные графики (4/8 ч/день, удалёнка) — нужны в планировании?
*Гип.: да · общий календарь хватает. Реф: Timetta — индивидуальные расписания.*

### G. План vs Бронь vs Факт, перепланирование

**G1.** Связь план ↔ бронь: бронь — это уточнение плана или отдельная ось (как сейчас, Timetta)?
*Гип.: отдельная (резерв) · бронь = детализация плана. Реф: Timetta «оценка ≠ резерв».*

**G2.** При отставании (факт < план к дате) — перераспределять остаток на будущее автоматически?
*Гип.: да (catch-up) · нет (РП решает) · только алерт. Реф: PSA reforecast; Timetta точка прогноза.*

**G3.** Нужен ли «прогноз = факт до сегодня + остаток плана вперёд» (а не чистый план)?
*Гип.: да (реалистичнее) · нет (план как есть). Реф: Timetta forecasting-principles.*

## 5. Топ-рекомендации (из референсов)

1. **Срочно решить B2** (рабочие vs календарные дни) — сейчас часы «капают» в выходные/праздники, это и есть корень непонимания заказчика. Переход на рабочие дни — малое изменение в `plannedHoursInPeriod` (делитель = Σ рабочих часов диапазона, переиспользовать `workHoursBetween`, который УЖЕ есть в `calc-load.ts`).
2. **Добавить выбор метода раскида** (C1/B1): минимум «равномерно» (по рабочим) + «вручную по периодам». Это закрывает 80% запроса. Полный набор 5 методов Timetta — следующая фаза.
3. **Решить судьбу проектов без `endDate`** (E3) — сейчас они невидимы на доске (план = 0), это скрытая потеря Demand.
4. **План на уровне сотрудника** (D2) — если заказчик мыслит людьми; сейчас только отдел÷headcount.
5. **Бюджет-потолок** (A3) по образцу Kimai — ортогонален раскиду, дёшев, ценен для контроля.
