## 1. ДВИЖОК ВАЛИДАЦИИ ТАЙМШИТА (Timetta) — 10 типов правил

**Где у нас:** validateEntry (день: hours<0/>max ERROR, >overtime WARNING) + validateWeekUnderfill (недобор). **Узко.**
**Timetta (`settings-time-accounting-timesheet-validation-rules.md`):** конфигурируемый движок, 2 уровня (Ошибка-блок / Предупреждение-разрешить, кастомный текст), применяется при submit. **10 типов:**

| Тип правила | Наш статус |
|---|---|
| Дата отправки (допустимые дни до конца периода) | нет — связан с dueDateLag (F4.5) |
| Мин. число заполненных строк | нет (W5A — «min lines») |
| Лимит часов/день | ЕСТЬ (maxHoursPerDay) |
| **Обязательность полей** (вид работ/роль/центр затрат/коды/тариф/комментарий/кастом) | частично |
| **Отклонение от расписания в %** (по дням или за период, напр. 25%/125%) | нет (есть только недобор) |
| Отклонение от расписания в часах (2ч/2ч) | нет |
| **Превышение плановых трудозатрат** (лимит списания vs план проекта) | нет |
| Запрет пустых строк | нет |
| Актуальный центр затрат | NB |
| Выбор тарифа | NB |

**Ценность арху:** наша валидация — частный случай. Стоит спроектировать как **конфигурируемый набор правил на шаблоне** (rule type + level + text + threshold), а не хардкод. MVP-кандидаты сверх текущего: «мин. строк», «запрет пустых строк», «отклонение от нормы в %/ч», «превышение плана». Закрывает A4.6/A4.9/UC-VAL-03 системно.

## 2. МОДЕЛЬ DETAIL-ОТЧЁТА (Timetta) — полный список 40+ полей

**Где у нас:** detail-CSV = 7 колонок; UC-RPT-10 обещал «40+ полей» — было неясно какие. **Теперь точно** (`analytics-report-types-timesheets-detailed-report.md`).

**Группы полей Timetta «Таймшиты детально»:**
- **Орг-иерархия:** Организация + менеджер + его подразделение; Программа + менеджер; Проект + ЮрЛицо + менеджер + его подр.
- **Работа:** Работа (ProjectTaskId), Этап (FirstLevelTaskId), Роль (RoleName).
- **Сотрудник:** Пользователь, Подразделение, ЮрЛицо, Локация, Уровень, Грейд, **Руководитель (SupervisorId)**.
- **Таймшит:** TimesheetId, ЮрЛицо таймшита, Центр затрат.
- **Отсутствие:** Заявка (TimeOffRequestId), Тип отсутствия.
- **Финансы (NB у нас):** Счёт, Выручка T&M, Тариф+ставка, Себестоимость (+ норм./БВ/ВП варианты).
- **Часы:** **HoursTotal · HoursNormalizedTotal · HoursBillable · HoursNonBillable · HoursProjects · HoursTimeOffs.**

**Ценность арху:**
1. **Применимые нам поля (без NB-финансов):** дата, орг-иерархия (проект→менеджер→подр.), работа+этап+роль, сотрудник+подр+уровень+**руководитель**, тип отсутствия, разбивка часов (всего/проекты/отсутствия). → расширить computeDetail до этого набора (UC-RPT-10/W4A.18/W6B.1) — пресетами «краткий/полный».
2. **«Нормализованные часы» (Normalized)** — Timetta отдельно считает нормализованные часы (приведение к норме). Концепт для util/FTE-корректировки (связь W5B.13 план÷FTE).
3. **SupervisorId как поле записи** — Timetta тянет руководителя в каждую строку отчёта (для группировки «по руководителю»). У нас scope по дереву отделов (W4B.7) — поле supervisor дало бы прямую иерархию.

## 3. WORKFLOW-ДВИЖОК СОГЛАСОВАНИЯ (Timetta) — подтверждает и расширяет наш approval

**Где у нас:** submit/approve/reject/recall/revoke (WI-10), SoD, isManager. **Жёсткая 1-уровневая цепочка.**
**Timetta (`supervisor-approval-process.md` + `mass-operations.md` + `change-approved-timesheet.md`):**

| Паттерн Timetta | Наш статус / действие |
|---|---|
| **Многоуровневый маршрут** (несколько согласующих, конфигурируемый route, роли ИЛИ конкретные люди) | у нас 1 уровень → roadmap (A4.17 иерархия) |
| **Reject любым → возврат автору на доработку**, процесс прерывается | ЕСТЬ (REJECTED→правка→resubmit, W5A.6) ✓ |
| **Re-approval skip:** уже согласовал + строки не изменились → пропуск этапа | = идемпотентность W5A.7 (повторное approve no-op) ✓ подтверждает решение |
| **Автор пропускает свой этап** согласования (если он в маршруте) | = SoD skippedOwn (W5A.9) ✓ |
| **HR/админ принудительно** approve ИЛИ вернуть в Draft, минуя процесс | = revoke admin-override (W5A.25, W4B.20). Timetta: «право на принудительное изменение состояния» — гранула |
| **Очередь «Входящие» в «Моя работа»** + email + in-app уведомления согласующим | A4.11 (очередь), F4 (уведомления) ✓ |
| **История согласования на вкладке «Активность»** | = audit revokedBy/resolvedBy (W5A.11/12, W4A.29) — Timetta хранит в Activity-логе ✓ подтверждает нужность |
| **Mass: нельзя смешивать состояния в батче** | = W5A.8 (mixed batch) — Timetta ЗАПРЕЩАЕТ разные состояния в одной операции ✓ |
| **Mass: confirm с числом выбранных** + правила валидации блокируют недопустимый переход | W6A.18 (confirm+count) + A4.6 (validation блок) ✓ |
| **Mass-операции: Согласовать / Редактировать(→Черновик) / Переоткрыть** | «Переоткрыть» = наш revoke; «Редактировать» = recall→draft ✓ |

**Ценность арху:**
- **Подтверждение нашего дизайна:** идемпотентность approve, SoD-skip автора, admin-override revoke, запрет mixed-state batch — всё совпадает с Timetta-каноном. Решения W5A валидны.
- **Чего нет и стоит заложить:** **workflow как конфигурируемый маршрут** (роль/человек × уровни) вместо хардкода isManager. Это закрывает A4.17 (кто согласует руководителя) и W4B.2 (РП vs начальник отдела) системно. Сейчас 1 уровень — но модель состояний под многоуровневость заложить (поле «маршрут/шаг»).
- **«Принудительное изменение состояния» = отдельная гранула прав** (не просто isManager). Связь CISO RBAC (W4B.3).

## 4. ТИПЫ ПЕРИОДОВ ТАЙМШИТА (Timetta) — 5 типов + per-department

**Где у нас:** недельный (Пн-Вс), хардкод; approvalPeriod WEEK/MONTH в Settings.
**Timetta (`settings-time-accounting-timesheet-periods.md`):** 5 типов, **scope = подразделение** (разные отделы — разные периоды):

1. **Недельный** (дефолт).
2. **Недельный по конец месяца** — неделя на стыке месяцев РАЗБИВАЕТСЯ на два таймшита (Пн-Чт авг + Пт-Вс сен). **Решает нашу проблему недель, пересекающих месяц** (W5C.7 граница месяца, Т-13 месячный учёт).
3. **Месячный.**
4. **Полумесячный** (1-14 / 15-конец).
5. **Произвольный** (для закрытия «окон» при смене типа периода).

**Ценность арху:**
- **«Недельный по конец месяца»** — прямое решение конфликта «недельный таймшит vs месячный Т-13/1С» (W4A.8, W5C.7). Неделя не пересекает месяц → каждый таймшит в одном месяце → чистая месячная агрегация для 1С. **Рекомендую этот тип как дефолт для РФ** (месячный учёт зарплаты).
- **Per-department scope** периода — отвечает на W5B.26 (смена approvalPeriod): период привязан к подразделению, не глобально.
- **«Произвольный» для миграции** — паттерн закрытия «окна» при смене типа (W5B.26 «что со старыми согласованиями»).

## 5. KIMAI — exported-флаг (аудиторский след)

**Kimai (`BEST_PRACTICES.md` §8):** после экспорта `timesheet.exported=true` → запись нельзя редактировать.
**Ценность арху:** = наш W5C/G3.16 (lock после выгрузки в 1С). Подтверждает: ввести флаг `exported` на записи → read-only после выгрузки кадрам/1С. Слой защиты СВЕРХ APPROVED-lock (согласовано ≠ выгружено). Связь с CASCADE-риском (W5C.23) — exported-записи тем более нельзя терять каскадом.

## 6. KIMAI — meta-fields (key-value без миграций)

**Kimai (`BEST_PRACTICES.md` §2):** произвольные key-value поля на сущностях без schema-миграций.
**Ценность арху:** для будущих кастом-полей (центр затрат, табельный №, accountingCode из W4A.6/7) без новых миграций каждый раз. Twenty SDK даёт customFields — проверить, покрывает ли паттерн (избежать twenty-sdk-apply-граблей при частых field-add).

## Сводка: что арх может взять СЕЙЧАС

**Подтверждения (наши решения = Timetta-канон, реализовывать смело):**
- Идемпотентный approve, SoD-skip автора, admin-override revoke, запрет mixed-state batch, история в audit-логе (§3) — W5A валиден.
- Недельный период (§4), факт по всем статусам, отсутствия отдельным слоем (UI_ANSWERS_TIMETTA_ENRICHMENT).

**Новое к проектированию (архитектурные заделы):**
1. **§1 Движок валидации** — конфигурируемые правила на шаблоне (2 уровня + threshold + текст) вместо хардкода. MVP+: мин.строк, запрет пустых, отклонение %/ч, превышение плана.
2. **§2 Detail-отчёт** — расширить до Timetta-модели полей (без NB-финансов): орг-иерархия + работа/этап/роль + сотрудник/уровень/**руководитель** + тип отсутствия + разбивка часов. Пресеты краткий/полный. + концепт «нормализованные часы».
3. **§3 Workflow-маршрут** — заложить модель многоуровневого согласования (роль/человек × шаги) под будущее A4.17/W4B.2; «принудительное изменение» = отдельная гранула RBAC.
4. **§4 Тип периода «недельный по конец месяца»** — дефолт для РФ, решает недели-через-месяц для Т-13/1С; per-department scope.
5. **§5 exported-флаг** — read-only после выгрузки в 1С (слой сверх APPROVED), усиливает защиту от CASCADE-потери.
6. **§6 meta/customFields** — для accountingCode/табельного № без частых миграций.

**Связь с P0/HIGH:** §3 подтверждает W5A-фиксы (идемпотентность/mixed-batch/audit); §5 усиливает аргумент против CASCADE (W5C.23); §1 системно закрывает A4.6/UC-VAL-03; §2 закрывает UC-RPT-10 «40+ полей»; §4 решает W5C.7/W4A.8 границу месяца.
