# Маркетинг ↔ Продажи в B2B: что добавить в marketing-вкладку проекта

Аналитик AITEAM · 2026-06-22 · READ-ONLY (рекомендации, не код)
Объект: `apps/time/src/objects/credos-time-project.object.ts`
Вью: `apps/time/src/views/credos-time-project-card-marketing.view.ts`
Опции NDA: `apps/time/src/constants/select-options.ts` (`NDA_LEVEL_OPTIONS`, 290–323)

Контекст: Кредо-С — B2B ИБ/ИТ-интегратор, ~72 чел. НЕ enterprise-advocacy-платформа.
Цель — минимальный набор пометок на проекте, чтобы продажи и маркетинг знали:
кого/что можно показывать публично, в КП и в тендерах, и кого можно дать в референс.
Биллируемость НЕ вводим (решение проекта, `no-billable-concept`).

---

## 1. Что УЖЕ есть в marketing-вкладке

| Поле | Тип | Назначение |
|---|---|---|
| `ndaLevel` | SELECT (NONE / CLIENT_ONLY / CLIENT_SECRET) | Градация секретности: NONE — можно о клиенте и о работах; CLIENT_ONLY — можно о клиенте, нельзя о работах; CLIENT_SECRET — нельзя о клиенте. Дефолт NONE. |
| `canPublishOnSite` | BOOLEAN | Разрешение на публикацию на сайте. Дефолт false. |
| `isPublished` | BOOLEAN | Факт публикации. Дефолт false. |
| `publishedUrl` | TEXT | Ссылка на публикацию. |

Также вне вкладки полезны продажам/маркетингу: `company` (Клиент → стандартный Company CRM),
`category` (категория работ), `status`, `description`, `manager`.

Что закрыто: публичный сайт + базовая NDA-градация.
Чего НЕТ: разрешения тоньше сайта (КП/тендер/презентация/логотип), кейс-пайплайн,
референс-готовность для продаж, отзыв клиента, отрасль клиента, результат проекта,
срок/факт согласования с клиентом, защита клиента от пере-использования в референсах.

## 2. Разведка — что трекают лидеры B2B (best practices)

Источники: HubSpot/Salesforce alignment, ReferenceEdge (Salesforce-native), Influitive,
UserEvidence, Gartner B2B marketing-sales alignment, ABM-паттерны.

| Практика лидеров | Что конкретно трекают | Релевантно Кредо-С? |
|---|---|---|
| **Reference-ready флаг** | Можно ли показывать аккаунт в питчах/звонках продаж | ДА — продажам нужен быстрый фильтр «кого показать» |
| **Anti-fatigue / не беспокоить чаще N** | Счётчик использований референса, дата последнего использования | ЧАСТИЧНО — 72 чел, хватит «дата последнего референса», без счётчика-автоматики |
| **Matching по industry + size + use-case** | Отрасль/вертикаль клиента, размер сделки, тип задачи — для подбора похожего референса под открытую сделку (ABM) | ДА — отрасль ценна для ИБ (банки/госсектор/ритейл), но size-категорию можно отложить |
| **Logo rights** | Право на логотип ОТДЕЛЬНО от названия (сайт/презентации/КП/тендеры) | ДА — у интегратора логотип в тендерной заявке = частый кейс |
| **Case study pipeline** | Статус кейса (план/в работе/опубликован), формат (статья/видео/PR/награда), согласование клиентом | ЧАСТИЧНО — статус кейса полезен, но формат-перечень можно сузить |
| **Testimonial / отзыв** | Есть/нет, цитата, ЛПР-автор | ЧАСТИЧНО — флаг «есть отзыв» + цитата; автор берётся из CRM-контакта |
| **Sales-enablement permission** | Можно ли упоминать в КП/тендерах/питчах ОТДЕЛЬНО от публичного сайта | ДА — критично: NDA часто разрешает «закрыто в тендере», но не на сайте |
| **Результат/метрика для кейса** | Что достигли (нарратив для маркетинга) | ДА — но как свободный текст, не структурированные метрики |
| **Срок/дата/кто согласовал** | Дата согласования публикации, срок действия, кто у клиента согласовал | ЧАСТИЧНО — дата согласования + кто согласовал; «срок действия» обычно бессрочно у интегратора → P3 |

Главный вывод разведки: у ИБ-интегратора с частыми НДА **ключевой разрыв — не сайт,
а sales-enablement-градации тоньше сайта** (КП/тендер/презентация/логотип). Лидеры это
решают набором булевых разрешений + reference-ready флагом. Для 72 чел не нужна
advocacy-платформа с геймификацией и счётчиками — нужны пометки на карточке проекта.

## 3. РЕКОМЕНДАЦИЯ — поля ДОБАВИТЬ (приоритизировано)

Принцип: минимум полей, выводимых из NDA-градации; каждое булево разрешение —
это «можно показывать в канале X», и оно НЕ должно противоречить `ndaLevel`
(валидация — задача Dev/арх, см. вопросы §4).

### P1 — добавить в первую волну (MVP)

| Поле | Тип | Зачем | Связь с NDA |
|---|---|---|---|
| `canUseInProposals` | BOOLEAN (дефолт false) | Можно упоминать проект в КП/тендерах/питчах (закрытый канал, продажам). Главный разрыв: часто разрешено в тендере, но НЕ на сайте. | Допустимо при `CLIENT_ONLY`; при `CLIENT_SECRET` — нельзя называть клиента (валидировать). |
| `canUseLogo` | BOOLEAN (дефолт false) | Право на ЛОГОТИП клиента (сайт/презентации/КП/тендеры) — отдельно от названия. Частый кейс интегратора. | Только если разрешено называть клиента (не `CLIENT_SECRET`). |
| `referenceReady` | BOOLEAN (дефолт false) | Флаг для продаж: этот проект можно показывать в питчах / дать клиента в референс-звонок. Самый востребованный фильтр sales. | Зависит от устной/письменной договорённости, но обычно требует не-`CLIENT_SECRET`. |
| `clientIndustry` | SELECT или TEXT | Отрасль/вертикаль клиента (банк/госсектор/ритейл/промышленность/…) — для ABM-подбора похожего референса под сделку. | Не конфиденциально само по себе; полезно даже при `CLIENT_SECRET` (без имени клиента). |

Обоснование P1: эти 4 поля закрывают самый частый рабочий запрос продаж интегратора
(«кого и где можно показать») и почти ничего не стоят (3 булевых + 1 select).
`canUseInProposals` и `canUseLogo` — прямое расширение NDA-логики, которой пока нет
между «сайт» и «совсем нельзя».

### P2 — вторая волна (когда P1 приживётся)

| Поле | Тип | Зачем | Связь с NDA |
|---|---|---|---|
| `caseStudyStatus` | SELECT (NONE / PLANNED / IN_PROGRESS / PUBLISHED) | Кейс-пайплайн: видно, какие проекты в работе у маркетинга. | Требует `canPublishOnSite`/`canUseInProposals` для движения к PUBLISHED. |
| `hasTestimonial` + `testimonialQuote` | BOOLEAN + TEXT | Отметка наличия отзыва + цитата ЛПР для КП/сайта. Автор — из контакта CRM. | Цитата публикуется по тем же правилам, что и проект. |
| `marketingResult` | TEXT | Результат/метрика проекта для маркетинг-нарратива («снизили инциденты на N%», «прошли аттестацию по 152-ФЗ»). | Содержание результата может быть под `CLIENT_ONLY` (нельзя о работах) — заполнять осторожно. |
| `publishApprovedAt` + `publishApprovedBy` | DATE + TEXT | Дата согласования публикации с клиентом + кто согласовал (имя/контакт у клиента). Защита: доказуемость согласия. | Триггер для `canPublishOnSite`/`canUseLogo` — фактическое разрешение. |

### P3 — отложить / только если попросят

| Поле | Тип | Почему откладываем |
|---|---|---|
| `lastReferencedAt` | DATE | Anti-fatigue («не беспокоить чаще N»). Для 72 чел и редких референс-звонков избыточно; вернуть, если продажи начнут жаловаться на пере-использование клиентов. |
| `dealSizeCategory` | SELECT (S/M/L) | Категория размера сделки для matching. Полезно для ABM, но добавляет ручной ввод; отложить до зрелости. |
| `caseStudyFormat` | SELECT (статья/видео/PR/награда) | Формат кейса. Нужен только когда `caseStudyStatus` реально используется маркетингом. |
| `publishExpiresAt` | DATE | Срок действия разрешения. У интегратора согласие обычно бессрочное → редко нужно. |

### MVP-набор (рекомендуемый к реализации сейчас)

Только **P1**: `canUseInProposals`, `canUseLogo`, `referenceReady`, `clientIndustry`.
Это превращает вкладку из «публикация на сайте» в полноценную matrix-разрешений
sales↔marketing, не раздувая объект (16 → 20 полей; объект близок к лимиту размера —
обратные relation-поля уже вынесены в `src/fields/`, см. комментарий в объекте 296–298).

## 4. Вопросы заказчику (развилки)

1. **`canUseInProposals` — один флаг или раздельно КП и тендеры?** Рекомендую один
   (закрытый канал = одно разрешение). Разделять только если по части клиентов «в КП можно,
   в гос-тендере нельзя».
2. **`clientIndustry` — SELECT (фикс-список отраслей) или свободный TEXT?** SELECT лучше для
   ABM-фильтрации, но нужен согласованный перечень вертикалей Кредо-С (банки, госсектор,
   промышленность, ритейл, телеком, …). Дайте список — оформим как опции (SSOT в select-options).
3. **Кто проставляет эти флаги — PM проекта, маркетинг или продажи?** Влияет на роли/права.
   По аналогии с `canPublishOnSite` — вероятно PM/маркетинг.
4. **Нужна ли жёсткая валидация «разрешение vs NDA»** (запрет ставить `canUseLogo` при
   `CLIENT_SECRET`), или достаточно предупреждения/доверия пользователю? Влияет на объём
   работ Dev (валидатор/logic-function vs просто поля).
5. **`referenceReady` — нужен ли контакт-референс у клиента** (кого продажам звать), или
   достаточно флага + контакта из стандартного CRM Company? Рекомендую второе (не дублировать CRM).

### Сводка для arch

**Топ-рекомендация:** превратить marketing-вкладку из «публикация на сайте» в matrix
sales-enablement-разрешений. Главный неперекрытый разрыв у ИБ-интегратора с частыми НДА —
это разрешения тоньше публичного сайта: упоминание в КП/тендерах и право на логотип
отдельно от названия, плюс reference-ready флаг для продаж и отрасль клиента для ABM-подбора
похожего референса. Это 4 дешёвых поля поверх существующего `ndaLevel`, без advocacy-платформы
и без биллируемости. Кейс-пайплайн, отзывы, дата согласования — P2; anti-fatigue и size — P3.

**P1-поля:**
- `canUseInProposals` — BOOLEAN — можно в КП/тендерах/питчах (закрытый канал, отдельно от сайта)
- `canUseLogo` — BOOLEAN — право на логотип клиента отдельно от названия
- `referenceReady` — BOOLEAN — можно показывать в питчах / дать в референс продажам
- `clientIndustry` — SELECT/TEXT — отрасль клиента для ABM-подбора
