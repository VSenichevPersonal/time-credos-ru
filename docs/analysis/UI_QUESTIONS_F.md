## F1. ПЛОТНОСТЬ ДАННЫХ (compact/comfortable, размеры, вместимость, реестры)

### Группа F1.A — Нужен ли регулятор плотности
**F1.1** Нужен ли переключатель плотности (compact / comfortable / по-умолчанию) вообще, или фиксированный текущий размер устраивает?
— Гипотезы: (а) глобальный 2-режима (compact/standard), сохраняется в настройках пользователя; (б) 3 режима (compact/cozy/comfortable, как Gmail); (в) не нужен — один размированный вид; (г) только для таблиц-реестров, не для таймшита.
— реф: Timetta TimesheetTemplate (выбор колонок, не плотности) — настраивают что показать, не размер; impeccable spatial — плотность = функция роли (product-register плотнее).

**F1.2** Если переключатель есть — он глобальный (на весь модуль) или per-экран (таймшит отдельно, реестры отдельно, доска отдельно)?
— Гипотезы: (а) глобальный один; (б) per-экран запоминается; (в) глобальный + override на «тяжёлых» реестрах.

**F1.3** Где хранить выбор плотности (Remote DOM, нет надёжного localStorage)?
— Гипотезы: (а) в `credosTimeSettings` per-user; (б) в общем singleton (один на всех — плохо для разных ролей); (в) в URL/view-state (теряется при перезагрузке, как drill).
— реф: ADR — drill теряется при reload (известный пробел); плотность важнее, лучше per-user в settings.

### Группа F1.B — Что именно сжимать
**F1.4** Что меняет «compact» в таймшите: только высоту строки (32→26px), или ещё font (13→12px) и padding ячейки (`0 8px`→`0 5px`)?
— Гипотезы: (а) только высота строки; (б) высота + padding; (в) высота + padding + font; (г) высота + padding, font НЕ трогаем (читаемость).
— реф: impeccable typography — не опускать body ниже 12px; цифры таймшита — tabular, 12px предел.

**F1.5** В compact-режиме скрывать ли вторичные элементы строки (точка-категории 7px gap, кнопка ✎-комментарий, бейджи), оставляя их по hover/фокусу?
— Гипотезы: (а) скрывать вторичное, показывать по фокусу строки; (б) ужимать всё пропорционально без скрытия; (в) скрывать только ✎ и бейджи, точку-категории оставить.
— реф: Remote DOM — нет надёжного hover; «по фокусу» надёжнее hover.

**F1.6** Высота строки/ячейки — это про вместимость или про touch-target? Если будут планшеты, compact конфликтует с 44px хит-зоной — что приоритетнее?
— Гипотезы: (а) compact только desktop, на touch принудительно comfortable; (б) compact доступен всегда (десктоп — мышь); (в) touch-режим — отдельная ветка (см. раздел D3.14).
— реф: impeccable interaction — 44px touch min; конфликт с плотностью.

### Группа F1.C — Сколько строк влезает / виртуализация
**F1.7** Целевая вместимость без скролла на типовом экране (1440×900 в виджете): сколько строк таймшита / строк реестра нужно видеть? Это драйвер выбора плотности?
— Гипотезы: (а) ~15 строк таймшита comfortable / ~22 compact — цель; (б) не считаем, плотность по вкусу; (в) цель «неделя одного сотрудника без скролла».

**F1.8** Реестры (сотрудники/проекты/справочники) могут быть длинными — нужна ли виртуализация строк или sticky-заголовок при compact, или объёмы малы?
— Гипотезы: (а) sticky-header + пагинация (как REST 60/стр); (б) виртуализация при >100 строк; (в) объёмы малы (десятки), ничего не нужно.
— реф: REST курсор 60/стр уже есть; Kimai — пагинация таблиц.

### Группа F1.D — Плотность таблиц-реестров и доски
**F1.9** Плотность реестров (project-team, project-summary, settings-таблицы) — наследует глобальный режим или у реестров своя «всегда плотная» норма?
— Гипотезы: (а) наследует; (б) реестры всегда compact (это «база данных»); (в) своя настройка.

**F1.10** Доска планирования (capacity) и отчёты (reports) — применять плотность к ним? Там ячейки уже фикс (COL_W 56 / COL_W_GAP 68).
— Гипотезы: (а) да, compact сужает колонки-периоды; (б) нет — доска/heatmap нечитаема при сужении; (в) только высота строк доски, ширину колонок не трогаем.

**F1.11** Нужна ли настройка «показывать/скрывать колонки» в реестрах и отчётах (как Timetta TimesheetTemplate showActivity/showRole/...), отдельно от плотности?
— Гипотезы: (а) да, чек-лист колонок per-view; (б) фиксированный набор колонок; (в) только в detail-отчёте (там 40+ полей / 6 групп).
— реф: Timetta TimesheetTemplate — showActivity/showClient/showTariff и т.д.; у нас detail-отчёт уже 40+ полей.

**F1.12** Чередование строк (rowAlt #fcfcfd) и зебра — оставлять при compact или убрать (плотнее = граница достаточна)?
— Гипотезы: (а) оставить зебру всегда; (б) убрать зебру в compact (только нижняя граница); (в) зебра только в реестрах, не в таймшите.

**F1.13** Размер шрифта цифр-часов — отдельная настройка от плотности (бухгалтеру важна читаемость цифр)?
— Гипотезы: (а) нет, едет с плотностью; (б) отдельный тумблер «крупные цифры»; (в) цифры всегда ≥13px независимо от плотности.

## F3. UNDO/REDO + АВТОСЕЙВ-КОНФЛИКТЫ

### Группа F3.A — Undo/redo
**F3.1** Нужен ли Undo (Ctrl+Z) ввода/изменения/удаления в таймшите (сейчас НЕТ)? Это снимает страх ошибки при автосейве.
— Гипотезы: (а) да, Undo последнего действия (ввод/очистка ячейки/удаление строки); (б) полный стек undo/redo; (в) не нужен (есть reload с сервера); (г) только undo удаления (самое болезненное).
— реф: ANALYST_FINDINGS UC12 — Ctrl+Z как «дёшево-вперёд»; Timetta/Kimai — undo в research НЕ упомянут.

**F3.2** Если Undo есть — глубина: только последнее действие или N шагов? И живёт ли он после reload (Remote DOM теряет состояние)?
— Гипотезы: (а) только последнее, в памяти сессии; (б) 10 шагов в памяти; (в) бесконечно до перезагрузки.

**F3.3** Где показывать «Отменить»: тост после действия («Запись удалена · Отменить»), хоткей, или оба?
— Гипотезы: (а) toast с «Отменить» после удаления (5-6с); (б) только Ctrl+Z; (в) toast + Ctrl+Z; (г) кнопка в тулбаре.
— реф: impeccable — «undo toast» лучше «вы уверены» для обратимых действий.

### Группа F3.B — Подтверждения
**F3.4** Удаление/очистка СТРОКИ сейчас без подтверждения (только danger-цвет в меню), а «Очистить неделю» — с модалкой. Унифицировать?
— Гипотезы: (а) удаление строки → toast-undo (без модалки); (б) удаление строки → confirm-модалка (как неделя); (в) confirm только если в строке есть часы; (г) оставить как есть.
— реф: impeccable — confirm для необратимого, undo для обратимого; удаление строки с данными = ближе к необратимому.

**F3.5** «Очистить неделю» удаляет только несогласованные. Текст модалки это объясняет достаточно, или показать счётчик «будет удалено N записей»?
— Гипотезы: (а) добавить счётчик удаляемых; (б) текущий текст ок; (в) счётчик + список затронутых дней.

**F3.6** Нужно ли подтверждение/предупреждение при уходе со страницы с НЕсохранённым черновиком ячейки (открыт inline-редактор, не было blur)?
— Гипотезы: (а) автокоммит при уходе (как blur); (б) предупреждение «есть несохранённое»; (в) отбросить незакоммиченное молча (текущее поведение).

### Группа F3.C — Автосейв и индикатор
**F3.7** Автосейв сейчас по blur/Enter/Tab без debounce. Достаточно, или добавить debounce-автосейв во время набора (на случай если юзер не делает blur)?
— Гипотезы: (а) оставить blur/Enter (явный коммит); (б) + debounce 1.5-2с при наборе; (в) + автосейв по таймеру всей формы.
— реф: B4.14 (открытый вопрос autosave vs Save/Cancel); Kimai/Timetta — чаще явное сохранение строки.

**F3.8** save-indicator (idle→saving→saved 2с→error) сейчас в тулбаре грида. Достаточно заметен, или дублировать у ячейки (точечный «✓» на сохранённой)?
— Гипотезы: (а) только тулбар-индикатор; (б) + микро-«✓» на ячейке после сохранения; (в) + строка целиком подсвечивается на миг.
— реф: impeccable motion — подтверждение должно быть рядом с действием; реф D1.4 (единый save-indicator).

**F3.9** При ошибке сохранения сейчас: ERROR-тост (не гаснет) + reload с сервера (откат к серверному значению). Это правильное поведение или терять ввод плохо?
— Гипотезы: (а) ок — откат + явная ошибка; (б) НЕ откатывать, оставить ввод + «Повторить» в тосте; (в) откат + сохранить введённое в «черновик для повтора».
— реф: B4.15/B4.16 (открытые) — откат молча vs error-toast vs блок коммита.

**F3.10** Нужна ли явная пара кнопок Save/Cancel на строке/форме как АЛЬТЕРНАТИВА автосейву для осторожных пользователей (бухгалтер)?
— Гипотезы: (а) только автосейв (быстрее); (б) автосейв + опция «требовать явное сохранение» в настройках; (в) Save/Cancel везде.

### Группа F3.D — Конфликты параллельной правки
**F3.11** Конфликт-detection при параллельной правке (два менеджера / сотрудник+менеджер правят одну неделю) сейчас НЕТ (нет версий/etag). Нужен ли?
— Гипотезы: (а) да, optimistic-concurrency (версия записи, «изменено другим — перезагрузите»); (б) last-write-wins достаточно (редкий кейс); (в) только мягкое «данные обновлены» при расхождении после сохранения.
— реф: APPROVED уже lock (CISO-012); конфликт реален для DRAFT/SUBMITTED.

**F3.12** Если конфликт ловим — как разрешать: показать обе версии, перезатереть, или заблокировать с reload?
— Гипотезы: (а) «изменено другим, перезагрузите» + reload (просто); (б) merge-вид с выбором; (в) блок строки «редактирует <кто>».

**F3.13** Нужна ли «живая» индикация, что запись/неделю кто-то правит прямо сейчас (presence), или это избыточно для масштаба?
— Гипотезы: (а) нет, масштаб мал; (б) лёгкий маркер «редактируется» при коллизии сохранения; (в) полноценный presence (overkill).

**F3.14** Черновик vs сохранённое: сейчас «черновик» = серверный статус DRAFT, локального несохранённого черновика формы нет (теряется при reload до blur). Нужен ли локальный draft-буфер на случай вылета?
— Гипотезы: (а) нет, blur коммитит сразу; (б) буфер незакоммиченного в settings/память на восстановление; (в) автокоммит по таймеру вместо буфера.

**F3.15** Восстановление после сбоя/перезагрузки: что должно вернуться (drill теряется — известный пробел)? Нужно ли восстанавливать выбранную неделю/фильтры/незакоммиченную ячейку?
— Гипотезы: (а) восстанавливать период/фильтры (view-state в settings); (б) ничего (текущее); (в) только период недели.

## F4. УВЕДОМЛЕНИЯ / АЛЕРТЫ

### Группа F4.A — Архитектура доставки (ключевой блокер)
**F4.1** In-app уведомления нативно НЕ доставляются (SDK песочницы: нет sendEmail/sendNotification, cron-payload не документирован). Реализован только ДЕТЕКТ + баннер. Какой канал доставки нужен заказчику в принципе?
— Гипотезы: (а) email-дайджест (через внешний relay/connection); (б) Telegram/MAX-бот; (в) только in-app (баннер+центр, без push); (г) интеграция с уведомлениями платформы Twenty (если появится API).
— реф: reminders.logic.ts комментарий — доставка push/email follow-up «когда появится канал»; research/timetta — звуки уведомлений (10 mp3), но API не документирован.

**F4.2** Если канала нет — устраивает ли «pull-модель» (пользователь сам открывает «Мои часы» и видит баннер), или нужен именно «push» (приходит без открытия)?
— Гипотезы: (а) pull достаточно для MVP (баннер+отчёт); (б) нужен push хотя бы email раз/неделю; (в) pull сейчас, push когда появится канал.

**F4.3** Нужен ли единый «центр уведомлений» (колокольчик/список) в UI, или хватает контекстных баннеров+тостов?
— Гипотезы: (а) центр уведомлений (иконка-колокол + лента + счётчик непрочитанных); (б) только контекстные баннеры/тосты (текущее); (в) центр только для руководителя (дайджесты/эскалации).
— реф: impeccable — центр нужен когда уведомлений много/разнотипных; сейчас тип один (недозаполнение).

### Группа F4.B — Напоминания заполнить таймшит
**F4.4** Сейчас напоминание = янтарная плашка над «Мои часы» (REQ-0019). Текст/триггер устраивает или нужно усилить (когда показывать, как настойчиво)?
— Гипотезы: (а) плашка достаточно; (б) + бейдж-счётчик на пункте меню «Записи»; (в) + модалка в reminderDayOfWeek (настойчивее).

**F4.5** Когда напоминать: фикс. день недели (reminderDayOfWeek в settings) или адаптивно (за день до дедлайна периода, как Timetta dueDate)?
— Гипотезы: (а) фикс. день (текущее, простое); (б) за N дней до закрытия периода; (в) оба — настройка в settings.
— реф: Timetta — напоминание привязано к dueDate периода (вторник след. недели).

**F4.6** Порог напоминания (threshold недозаполнения) — настраивается в settings (0=строго). Показывать ли его пользователю и давать ли менеджеру менять?
— Гипотезы: (а) скрытая системная настройка; (б) менеджер настраивает в UI настроек; (в) персональный «не напоминать если добрал ≥X%».

**F4.7** Напоминать ли о НЕотправленных на согласование (статус DRAFT, но часы внесены), а не только о недозаполненных по часам?
— Гипотезы: (а) да, два повода: «не заполнил» + «не отправил»; (б) только недозаполнение; (в) «не отправил» — отдельный мягкий намёк.
— реф: Timetta — период надо «заполнить И отправить»; у нас детект только по часам (norm−fact).

### Группа F4.C — Дайджесты руководителю и эскалации
**F4.8** Дайджест руководителю = сворачиваемый список в баннере + отчёт missing-view. Нужна ли сводка-цифра/частота (раз/неделю), или достаточно on-demand?
— Гипотезы: (а) on-demand (открыл — увидел, текущее); (б) еженедельный email-дайджест руководителю; (в) бейдж-счётчик «N не заполнили» в меню отчётов.

**F4.9** Эскалации (НЕ реализованы) нужны? Например: не заполнил неделю → через X дней уведомить руководителя/выше.
— Гипотезы: (а) да, 2-уровневая эскалация (сотрудник → руководитель → след. уровень); (б) нет, дайджеста достаточно; (в) эскалация только в отчёте (помечать «просрочка N дней»).
— реф: missing-timesheets.ts комментарий — эскалации это follow-up.

**F4.10** Кому адресован дайджест: прямому руководителю отдела, или конфигурируемо (кто получает по какому отделу)?
— Гипотезы: (а) руководитель своего отдела (isManager); (б) конфигурируемые получатели; (в) все менеджеры видят весь срез.

**F4.11** ФИО в дайджесте раскрываются только при revealEmployeeNames=true (CISO-007), иначе «Сотрудник · OV». Это устраивает для эскалаций (нужно ведь знать кого пинать)?
— Гипотезы: (а) для эскалации руководителю ФИО раскрывать всегда (он и так знает команду); (б) держать CISO-007 строго (код отдела); (в) ФИО для прямого руководителя, код — для верхних уровней.
— реф: конфликт ПДн отмечен в INDEX (Т-13 требует ФИО vs CISO-007 скрывает).

### Группа F4.D — Тосты успех/ошибка
**F4.12** Тосты сейчас ТОЛЬКО в grid (validation-toast: ERROR не гаснет, WARNING 6с, правый-низ). Распространить единый тост-механизм на весь модуль (сохранение справочника, экспорт, согласование)?
— Гипотезы: (а) да, единый ToastHost на весь модуль; (б) тосты только в grid; (в) единый, но успех — тихий (короткий), ошибка — настойчивый.
— реф: D1.4 единый save-indicator; impeccable — успех тихо, ошибка заметно.

**F4.13** Нужны ли тосты УСПЕХА (сейчас успех молчит, есть только save-indicator)? Например «Согласовано», «Экспорт готов», «Тег создан».
— Гипотезы: (а) тост успеха для дискретных действий (согласование/экспорт/создание), не для каждого ввода ячейки; (б) успех всегда молча (индикатор достаточно); (в) успех только для «тяжёлых» действий.

**F4.14** Позиция/поведение тостов: правый-низ (как сейчас), стек, авто-гашение. ERROR не гаснет — это правильно, или давать «×» + автогашение длиннее?
— Гипотезы: (а) оставить (ERROR держится до закрытия, WARNING 6с); (б) ERROR тоже автогаснет (12с) с «×»; (в) единые правила: успех 3с, warn 6с, error до закрытия.
— реф: impeccable motion — error persistent, success transient.

**F4.15** aria-live сейчас: ERROR=assertive, WARNING=polite. Распространить корректную a11y-озвучку на все будущие тосты/баннеры?
— Гипотезы: (а) да, единые правила aria-live по severity; (б) не приоритет; (в) только для ошибок.

### Группа F4.E — Прочее
**F4.16** Звук уведомления (Timetta имеет 10 mp3) — нужен или продукт-инструмент молчит?
— Гипотезы: (а) без звука (рабочий инструмент); (б) опц. тихий звук для критичных (отклонение согласования); (в) настройка вкл/выкл.
— реф: research/timetta/RECON.md — bright-notification, soft-chime и др.

**F4.17** Уведомления о СОБЫТИЯХ согласования (запись отклонена с комментарием / акт создан) — это уже алёрты пользователю. Через какой канал их доводить?
— Гипотезы: (а) баннер на «Мои часы» + статус-цвет REJECTED (текущая база); (б) + тост при входе; (в) + центр уведомлений; (г) + email.
— реф: approval flow уже шлёт «уведомления» внутри (commit 8197a51) — уточнить что именно.

**F4.18** Нужен ли пользователю контроль над уведомлениями (что получать: напоминания / отклонения / дайджесты), или всё включено системно?
— Гипотезы: (а) персональные preferences (чек-лист типов); (б) всё системно вкл (мало типов); (в) только глобальный вкл/выкл напоминаний (как сейчас reminderEnabled).

**F4.19** Дедлайн/срок периода как часть уведомления: показывать ли «до закрытия периода N дней» (Timetta dueDate)?
— Гипотезы: (а) да, обратный отсчёт в баннере/шапке «Мои часы»; (б) нет; (в) только в reminderDayOfWeek-напоминании.
— реф: Timetta dueDate (вторник след. недели) — обратный отсчёт.

**F4.20** Где «живёт» уведомление о новом действии для согласующего (менеджеру пришли записи на согласование)? Сейчас он сам идёт в очередь согласования.
— Гипотезы: (а) бейдж-счётчик «N на согласовании» в меню; (б) дайджест-баннер менеджеру; (в) pull (сам открывает очередь, текущее); (г) email при поступлении.
