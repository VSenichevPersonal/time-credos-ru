# План по сотруднику + мэтчинг на отдел — анализ + рекомендация

**Дата:** 2026-06-21
**Автор:** Аналитик AITEAM (итерация 118)
**Повод (заказчик):** планирование ресурса сейчас ТОЛЬКО по отделу (загрузка сотрудника = план отдела ÷ численность поровну). Нужно: планировать ПО СОТРУДНИКУ + чтобы агрегировалось на отдел.
**Опора:** разведка Timetta (resource-plan/booking/project-team) + Kimai (планирования НЕТ). Связано: PLANNING_RECORDS_CONSISTENCY, PLANNING_MODELS, W3B.25, W5B.13, REQ-0010/0013.

## 0. TL;DR

- **Timetta планирует СНИЗУ-ВВЕРХ:** аллокация персональная (`ресурс × работа × период`), отдел — производная сумма (измерение в отчёте). «На подразделения планировать НЕЛЬЗЯ» (project-team.md явно). Top-down нет.
- **Kimai:** capacity/resource-планирования НЕТ вообще. Референса нет — ориентир только Timetta.
- **У нас сейчас наоборот:** план на отдел (`plan-slot {project,[department],month}`), сотрудник = `deptLoad ÷ headcount` (поровну, без персоны). Это «top-down деление», противоположно Timetta.
- **Рекомендация:** перейти на bottom-up — добавить `employee` в plan-slot, отдел = Σ персональных слотов + нераспределённый остаток. Наш аналог Timetta-«generic→именной» = слот без employee (отдел) → слот с employee (персона).

## 1. Что есть сейчас (по коду)

| Сущность | Поля | Уровень |
|---|---|---|
| `credos-time-plan-slot` (WI-47) | `project, department?(nullable,SET_NULL), periodMonth(YYYY-MM), plannedHours` | проект×[отдел]×месяц |
| `credos-time-employee-department` | `ftePercent(0-100), startDate, endDate` | сотрудник↔отдел (история+доля) |
| `credos-time-booking` | `employee, project/company, hours, SOFT/HARD, startDate, endDate` | персональная бронь |
| `employeeLoadCells` (calc-load.ts:611) | `planLoad = deptLoad × (1/headcount)` | **деление поровну, без персоны** |

**Корень проблемы:** `employeeLoadCells` делит загрузку отдела ПОРОВНУ на численность (`share = 1/headcount`). Персональной аллокации нет. Полуставочник получает ту же долю плана, что полный → завышенная util (баг W5B.13).

**Уже готова половина инфраструктуры:** plan-slot имеет `department` nullable — структура расширяема персональным измерением. employee-department даёт принадлежность+FTE по датам.

## 2. Модель Timetta (референс)

**Аллокация = `(resource, projectTask, period) → plannedHours`** — персональная, до человека и работы.
- `ProjectTeamMember {projectId, resourceId, roleId, competenceId, primaryTariffId, isActive}` — членство в команде (доля/период — НЕ здесь, а в ячейках плана).
- `ResourcePlan` — ячейки часов: «Оценка» (EstimatedHours, текущая) + «План» (PlannedHours, master) + «Факт» (ActualHours из таймшитов).
- **Rollup СНИЗУ-ВВЕРХ:** отдел/проект/роль — аналитики ОТЧЁТА (Σ группировкой), не носители плана.
- **Generic→именной:** `IsGenericResource` (безымянный профиль роль+уровень+пул) на старте → замена на конкретного человека, часы переезжают.
- **Booking ≠ ResourcePlan:** разные сущности. RP = оценка трудозатрат PM-ом (детально до работ, допускает generic). Booking = долгосрочная загрузка человека на проект (Soft/Hard, без детализации). `ResourceGap = Demand − Capacity` из брони/расписания.
- **Масштаб день/нед/месяц**, спуск пропорционально рабочему расписанию сотрудника.

## 3. Рекомендация: bottom-up на нашей модели

### 3.1 Расширить plan-slot персональным измерением
Добавить опциональное `employee` (nullable, SET_NULL) к `plan-slot {project, department?, employee?, periodMonth, plannedHours}`:
- **`employee` задан** → персональный слот (новый режим, = Timetta RP-ячейка `resource×period`).
- **`employee` пуст, `department` задан** → отдельский слот (текущий режим = Timetta generic-уровень).
- **оба пусты** → план проекта целиком (как сейчас EVEN).

Это прямой аналог Timetta-пары «generic (отдел/роль) → именной». Дедуп-ключ расширяется: `(project, department?, employee?, periodMonth)`.

### 3.2 Rollup вместо деления (ядро фикса)
Заменить `employeeLoadCells`:
```
БЫЛО:  planLoad = deptLoad × (1/headcount)         // поровну
СТАЛО: planLoad[emp,period] =
         Σ персональных слотов сотрудника (employee=emp)
       + доля нераспределённого остатка отдела
```
где **нераспределённый остаток отдела** = `deptPlan[period] − Σ(всех персональных слотов сотрудников отдела)`. Остаток делится поровну (как сейчас) ИЛИ по FTE (предпочтительно, закрывает W5B.13) на тех, у кого нет персонального слота.

### 3.3 Мэтчинг отдел↔сотрудники (наш аналог ResourceGap)
Для пары (отдел, период):
- `Σ персональных слотов` vs `deptPlan` →
  - **остаток > 0** → не распределено (показать «N ч не назначено» + опц. деление поровну/по FTE остатка).
  - **остаток < 0** → переаллокация (персональные слоты превысили план отдела) → warning.
  - **остаток = 0** → инвариант «Σ по сотрудникам = загрузка отдела» соблюдён.
- Принадлежность сотрудника отделу в периоде — из `employee-department {ftePercent, startDate, endDate}` (наше расширение сверх Timetta single-dept — оставить, ценно).

### 3.4 Что НЕ делать (keep-it-simple)
- **Booking оставить отдельной осью** (Timetta разделяет осознанно). Не сливать с plan-slot. Booking = «кто занят» (ёмкость), slot = «оценка трудозатрат».
- **Generic-as-entity + роль-в-членстве** (ProjectTeamMember) — НЕ тащить на первой итерации. Generic эмулируется слотом без employee. Роль — позже (W4B.24 ответ: workType достаточен на MVP).
- **ResourcePlan-версии** (Оценка/План/master) — не нужны MVP. Один слой слотов.
- **Detail до работ/этапов** (Timetta projectTask) — у нас slot до месяца, не до работы. Достаточно.

## 6. Открытые вопросы (для арха/заказчика)

1. **Направление:** подтверждаем bottom-up (план на человека → Σ отдел) vs гибрид (план отдела + детализация на людей с остатком)? Рекомендую bottom-up с остатком (покрывает оба).
2. **Остаток нераспределённого** делить поровну (как сейчас) или по FTE (точнее)? Рекомендую FTE (W5B.13).
3. **Кто планирует по людям:** РП проекта (его команда) vs руководитель отдела (его люди)? Связь W4B.2 (РП vs начальник отдела).
4. **Переаллокация (Σ слотов > план отдела):** warning или блок? Рекомендую warning (мягко, как овербукинг).
5. **Гранулярность slot:** месяц (как сейчас WI-47) или нужна неделя для персонального? Рекомендую месяц (консистентно).
6. **Связь с booking:** персональный слот (оценка) и hard-бронь (ёмкость) на одного человека+проект — суммировать (двойной счёт?) или max? Связь W6C.25. Рекомендую: slot=план-оценка ось, booking=резерв ось, в Demand берётся один источник (booking если есть, иначе slot) во избежание двойного счёта.

## 7. КОНСИСТЕНТНОСТЬ план-сотрудник ↔ план-отдел: полный SSOT + динамика

**Требование заказчика:** план сотрудника и план отдела (в котором он/они состоят) должны БИТЬСЯ везде — БД, расчёт, UX/UI, отчёты — без конфликтов. Полный SSOT, данные динамические (не дублированные).

### 7.1 Принцип SSOT: что ХРАНИМ, что ВЫЧИСЛЯЕМ

**Хранимый слой (единственный источник истины), persisted:**
- `plan-slot {project, department?, employee?, periodMonth, plannedHours}` — ручной план (MANUAL).
- `project {plannedEffort, startDate, endDate, planMethod}` — агрегат для EVEN-раскида.
- `dept-plan {department, periodMonth, plannedHours}` — план отдела БЕЗ проекта (REQ-0012).
- `employee-department {ftePercent, startDate, endDate}` — принадлежность+доля.
- `booking {employee, project, hours, SOFT/HARD}` — резерв (отдельная ось).
- `time-entry` — факт.

**Вычисляемый слой (НИКОГДА не хранить дублем, считать на лету):**
- `deptLoad[period]`, `employeeLoad[period]`, `util`, `gap`, `ratio`, остаток нераспределённого — ВСЁ derived из хранимого через ОДНУ функцию `calc-load`.
- Исключение: `factHours/budgetRemaining` на проекте — хранимые derived, но идемпотентный rollup (Σ из источника) уже есть; план НЕ материализуем.

**Правило:** для одной величины (загрузка отдела в периоде) — РОВНО ОДИН путь вычисления. Сейчас риск: доска и отчёты считают разными формулами (W5B.21 сетка фикс-8/40 vs отчёт календарь). → свести к единой `plannedHoursInPeriod`/`employeeLoadCells` для ВСЕХ потребителей (доска, панель-превью, отчёты, KPI). Это уже частично так (WI-05 единая формула) — закрепить как инвариант.

### 7.2 Источник истины при planMethod (конфликт EVEN vs MANUAL vs персона)

Иерархия разрешения загрузки проекта в периоде (детерминированная, без двойного счёта):
```
1. project.planMethod == MANUAL и есть слоты проекта за период
     → загрузка = Σ слотов (НЕ формула из plannedEffort)
2. иначе (EVEN)
     → загрузка = plannedHoursInPeriod(plannedEffort, dates, рабочие дни)
```
Внутри MANUAL — разрешение по гранулярности слота (от частного к общему):
```
employee-слот (project,emp,month)  — персональный, высший приоритет
  ⊂ department-слот (project,dept,month) — отдельский остаток
    ⊂ project-слот (project,month)        — общий остаток
```
**Инвариант неконфликтности:** для (project, period) загрузка считается ОДИН раз по самому детальному заданному уровню; верхние уровни дают ТОЛЬКО остаток (не суммируются с детальными повторно). Формула:
```
deptLoad(dept,period) = Σ_emp∈dept empSlot(project,emp,period)
                      + max(0, deptSlot(project,dept,period) − Σ_emp empSlot)   // остаток отдела
employeeLoad(emp,period) = Σ_project empSlot(project,emp,period)
                         + доля(emp) от нераспределённого остатка отдела
```
Двойного счёта нет: персональные слоты вычитаются из отдельского остатка.

### 7.3 Инварианты консистентности (проверять в БД + UI + тестах)

| # | Инвариант | Где проверять | При нарушении |
|---|---|---|---|
| I1 | `Σ empSlot(dept) ≤ deptPlan` (или = при полной аллокации) | calc-load + панель | warning «переаллокация +N ч» (не блок) |
| I2 | `Σ_dept ftePercent(emp) ≤ 100%` на дату | форма назначения (W5B.15) | warning >100% |
| I3 | emp-слот существует ⟹ у emp активное `employee-department` в этом периоде | upsert /s/plan-slots + триггер | reject слот-сироту / пометить «вне отдела» |
| I4 | загрузка отдела на доске == Σ загрузок сотрудников отдела (срез «Люди») | единая calc-load для обоих срезов | расхождение = баг формулы |
| I5 | план-превью в панели == загрузка на доске после сохранения | общая функция `plannedHoursInPeriod` | расхождение = два пути (запрет) |
| I6 | EVEN-проект НЕ имеет слотов; MANUAL-проект загрузка только из слотов | guard в calc-load (ветка planMethod) | смешение = двойной счёт |
| I7 | при смене отдела сотрудника (employee-department endDate) исторические слоты остаются на старом отделе на даты до перевода | rollup по дате периода, не по текущему отделу (W4A.28) | часы «прыгают» между отделами |

### 7.4 Динамичность (реактивность данных)

- **Изменение хранимого → мгновенный пересчёт derived** на следующем чтении (на лету в `useMemo`/server-calc). Снапшота нет — кроме согласованных периодов (W5B.2: заморозка нормы/headcount закрытых периодов — иначе прошлые планы «плывут»).
- **Смена FTE/назначения** (employee-department) → пересчёт и ёмкости, и распределения остатка. По дате периода (I7), не по текущему состоянию.
- **Добавление emp-слота** → автоматически уменьшает отдельский остаток (I1), доска/отчёты/панель синхронны (единая формула I4/I5).
- **Конкурентная правка** слота двумя (РП+руководитель) — LWW или optimistic (как W5A.7/W5C.3) на /s/plan-slots; дедуп-ключ `(project,dept?,emp?,month)` защищает от дублей.

### 7.5 Консистентность в UX/UI (чтобы «билось» визуально)

- **Один компонент-источник чисел:** панель «Планировать», доска, отчёты читают ОДИН calc-load результат. Не считать в компонентах локально.
- **Срез «Отделы» и срез «Люди» доски** — две проекции ОДНОЙ модели: раскрыл отдел → видишь сотрудников, Σ которых = строка отдела (I4 визуально). Если не бьётся — формула разошлась.
- **Панель отдела** показывает: план отдела · Σ назначено сотрудникам · остаток (I1) — пользователь видит мэтчинг явно.
- **Индикатор рассинхрона:** если `Σ empSlot ≠ deptPlan` — бейдж «N ч не распределено» / «переаллокация +N ч» (не молчать).
- **При смене метода/дат** — пересчёт превью на лету (W3B.4 алерт о расхождении).

### 7.6 Что добавить к этапности (§5) для SSOT

- **Фаза 1** +: единая calc-функция для доски/панели/отчётов (свести W5B.21 расхождение); guard planMethod (I6); rollup по дате периода (I7/W4A.28).
- **Фаза 2** +: инварианты I1-I5 как unit-тесты (защита от регрессии формул); индикатор остатка/переаллокации в UI.
- **БД-уровень:** дедуп-индекс слота `(project,dept?,emp?,month)`; FK employee/department SET_NULL (слот переживает удаление отдела/сотрудника как история); валидация I3 в upsert.

**Главное по SSOT:** хранить ТОЛЬКО атомы (слоты/назначения/брони/факт), всё агрегированное (отдел/сотрудник/util/gap) — вычислять одной функцией на лету. Конфликт план-сотрудник↔план-отдел исключается правилом «детальный уровень вычитается из остатка верхнего» (I6, §7.2) — они не два независимых числа, а один с детализацией.

## 8. Стыковка ТРЁХ осей плановой нагрузки: plan-slot ↔ dept-plan ↔ booking (SSOT)

**Факт из кода (`calc-load.composeCell:551`):** Demand сейчас СУММИРУЕТ три источника:
```
load (Demand) = planLoad + hardBooking
  planLoad = Σ projectDeptHoursInPeriod  (план ПРОЕКТА: slot MANUAL | EVEN-раскид)
           + Σ deptPlanHoursInPeriod     (план ОТДЕЛА без проекта, REQ-0012)
  hardBooking = Σ персональных HARD-броней сотрудника
  softBooking — отдельный слой (НЕ в Demand)
```

### 8.1 Что есть и в каком статусе (важно — заказчик: «нерабочий endpoint»)

| Ось | Сущность | Endpoint | Статус |
|---|---|---|---|
| План проекта (ручной по месяцам) | `plan-slot` | **`/s/plan-slots` (read/upsert) — РАБОТАЕТ** (WI-47, plan-slots.logic.ts) | бэк готов; **MANUAL-режим в панели = заглушка «скоро»** → потому кажется «нерабочим» |
| План проекта (равномерный) | `project.plannedEffort+dates` | через capacity-rest PATCH | РАБОТАЕТ (EVEN дефолт) |
| План отдела без проекта | `dept-plan` («Плановые загрузки» в сайдбаре) | нативный CRUD Twenty | объект есть, в Demand считается; **в UX путается с «детализацией плана»** (Planning-Records #10) |
| Бронь ресурса | `booking` | нативный CRUD | РАБОТАЕТ, в Demand (HARD) |

**ВЫВОД по «нерабочему endpoint»:** `/s/plan-slots` (=«Плановое распределение по месяцам») технически ГОТОВ, но НЕ подключён в UI — в панели «Планировать» режим «Вручную по месяцам» = disabled-заглушка (фаза-2, см. W3-B §0). Поэтому слоты не создаются из UI → реестр «Плановое распределение» пуст → выглядит нерабочим. **Активация = протянуть MANUAL-радио + таблицу месяц×часы в project-plan-panel → вызвать plan-slots-rest (контракт уже описан в plan-slots.logic.ts).** Это и есть недостающее звено + сюда же ляжет персональное измерение (§3.1).

### 8.2 РИСК ДВОЙНОГО СЧЁТА (главная угроза консистентности)

Три оси СУММИРУЮТСЯ в Demand → если один ресурс×проект×период покрыт ДВУМЯ осями, ёмкость потребляется дважды:
- **plan-slot(project) + hard-booking(тот же emp, тот же project)** → ×2 (W6C.25). Самый частый конфликт: РП забронировал человека (booking) И расписал план проекта (slot) на тот же месяц.
- **dept-plan(отдел) + project-slot(проект того же отдела)** → концептуально разное (план БЕЗ проекта vs план проекта), но если РП завёл «плановую загрузку отдела» дублируя проектный план — ×2.
- **emp-slot + dept-slot того же проекта** → решено правилом §7.2 (детальный вычитается из остатка). Распространить на ВСЕ оси.

### 8.3 SSOT-правило стыковки (детерминированное, без двойного счёта)

**Единица истины Demand на (ресурс/отдел, период) = МАКСИМАЛЬНО детальный заданный источник, верхние дают только остаток, бронь — отдельная ось с правилом слияния:**

```
1. ПЛАН (оценка трудозатрат) — одна величина, разрешается по детальности:
   empSlot(project,emp) ⊂ deptSlot(project,dept) ⊂ projectSlot(project) ⊂ EVEN(plannedEffort)
   + deptPlan (REQ-0012, отдельный «проект-less» план отдела) — СУММИРУЕТСЯ (это иная работа, не дубль проекта)

2. БРОНЬ (резерв ресурса) — отдельная ось (Timetta-канон REQ-0004 «бронь ≠ план»):
   HARD-бронь потребляет ёмкость. НО при наличии и плана, и брони на emp×project×период:
   Demand_вклад = MAX(planОценка, hardBooking)   — НЕ сумма
   (план и бронь описывают ОДНУ занятость двумя способами: оценка PM-а vs закреплённый резерв;
    берём больший как фактическую потребность в ёмкости, не складываем)
```

**Обоснование MAX, не сумма:** план-оценка и бронь — два взгляда на одну занятость человека на проекте. Складывать = считать, что он занят и «по плану», и «по брони» одновременно (физически тот же человек). MAX = «потребность = максимум из заявленного». Timetta держит их РАЗНЫМИ сущностями именно чтобы не складывать вслепую; ResourceGap считается из брони ИЛИ оценки по настройке, не из суммы.

**Альтернатива (на решение арха/заказчика):** если бронь и план — действительно независимые потребности (бронь под будущее + план под текущее), тогда сумма корректна. → **вопрос заказчику (см. ниже).**

### 8.4 Инварианты стыковки трёх осей (дополнение к I1-I7)

|---|---|---|
| I8 | emp×project×период покрыт планом И бронью → Demand = MAX (или сумма по решению), НЕ оба слепо | calc-load composeCell |

### 8.5 Что показать в UX чтобы «билось»

- **Тултип/разбивка ячейки доски:** «Demand N ч = план M + бронь K (max)» / «проект А 20 + проект Б 15 + бронь 10» — происхождение часов явно (I1/§7.5).
- **При создании dept-plan** («Плановая загрузка отдела»): подсказка «Это план БЕЗ проекта (резерв/пресейл/непроектная работа). Для плана проекта используйте «Планировать» на проекте» — снять путаницу Planning-Records #10.
