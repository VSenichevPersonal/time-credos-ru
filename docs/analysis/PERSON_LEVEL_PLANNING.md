## 0. TL;DR

- **План на человека НЕ существует.** Доска «Люди» (`employeeLoadCells`, `calc-load.ts:611-657`) НЕ хранит персональный план — она **делит загрузку отдела поровну** `planLoad = deptLoad × (1/headcount)`. Это **отображение-производная**, а не данные. В коде прямо: «allocation по людям в модели нет — равномерное распределение».
- **Единственная персональная ось — `credosTimeBooking`** (employee × project/company × период × hours, SOFT/HARD). Но это **РЕЗЕРВ ёмкости** (бронь), намеренно отделённый от план-оценки и факта (инвариант Timetta). Бронь отвечает на «зарезервировать Иванова под Альфу», а НЕ на «у Иванова в июле план 60 ч по Альфе».
- **Плановые слоты** (`credosTimePlanSlot`) — это `project × [department] × periodMonth → plannedHours`. **Поля `employee` НЕТ** (подтверждено: `universal-identifiers.ts:814-824` — только period/hours/project/department).
- **Конкуренты** (Timetta, Float, Runn, Kantata): person-level — это **ресурсный план = член команды × работа × период**, единицы часы / % расписания / FTE, мягкая (soft) и жёсткая (hard) аллокация, овербукинг показывается (не блок), назначает РП/ресурс-менеджер. Top-down (отдел→человек) и bottom-up оба поддержаны.
- **Топ-рекомендация:** НЕ плодить новый объект и НЕ переиспользовать booking как план. **Расширить `credosTimePlanSlot` опциональным additive-полем `employee` (RELATION, nullable, SET_NULL).** Ключ слота становится `project × [department] × [employee] × periodMonth`. Это даёт person-level плановые часы тем же помесячным механизмом, что уже работает (panel + `/s/plan-slots` + `slotsHoursInPeriod`), сохраняет dept-уровень (слот без employee = как сейчас), и НЕ трогает booking (резерв остаётся отдельной осью).
- **Вопросов к заказчику: 26** (раздел 4).

## 1. Что есть СЕЙЧАС (факты из кода)

### 1.1 Карта сущностей планирования

| Сущность | Файл | Ось/ключ | employee? | Семантика |
|---|---|---|---|---|
| Проект | `objects/credos-time-project.object.ts` | проект | нет | `plannedEffort`+даты+`planMethod` (EVEN/MANUAL) |
| **Плановый слот** | `objects/credos-time-plan-slot.object.ts` | **project × [department] × periodMonth** | **НЕТ** | ручной помесячный раскид (MANUAL), `plannedHours` |
| Доля отдела в проекте | `objects/credos-time-project-department.object.ts` | project × department | нет | `plannedEffortShare` (часы доли отдела) |
| План отдела без проекта | `objects/credos-time-dept-plan.object.ts` | department | нет | резерв/пресейл/прочее, `plannedEffort`+даты |
| **Бронь ресурса** | `objects/credos-time-booking.object.ts` | **employee × project/company × период** | **ДА** | РЕЗЕРВ ёмкости (SOFT/HARD), `hours`, отдельно от плана и факта |
| Назначение в отдел (FTE) | `objects/credos-time-employee-department.object.ts` | employee × department | да | `ftePercent`+даты → headcount = Σ FTE |

**Вывод по гранулярности плана:** глубже отдела план НЕ опускается. Самый детальный уровень — `plannedEffortShare` (часы отдела в проекте) или слот `project×department×month`. Дальше — деление поровну.

### 1.2 Доска «Люди» = производная, НЕ данные

`employeeLoadCells` (`calc-load.ts:611-657`):

```
factor   = коэф. отдела сотрудника
share    = dept.headcount > 0 ? 1 / dept.headcount : 0
capacity = period.workHours × factor − absHours(employee)   // личная ёмкость — РЕАЛЬНАЯ
deptLoad = Σ projectDeptHoursInPeriod(проекты отдела) + Σ deptPlanHoursInPeriod(планы отдела)
planLoad = deptLoad × share                                  // ← РАВНОМЕРНОЕ ДЕЛЕНИЕ
booking  = empBookingHours(employee)                         // личные брони НЕ делятся
load     = planLoad + booking.hard                           // composeCell
```

Что это значит:
- **Личная ёмкость человека — настоящая** (его календарь × коэф. отдела − его отсутствия).
- **Личный план человека — фейковый**: это `1/N` доли отдела. Если в отделе 5 человек и проект грузит отдел на 100 ч, у каждого «план» = 20 ч — независимо от того, кто реально работает на проекте.
- **Личные брони — настоящие персональные** (employee × project, не делятся). Это единственное «честное» личное число на доске «Люди».

То есть доска «Люди» **уже визуально существует**, но 90% её содержимого (plan-часть) — арифметика от отдела, а не план человека.

### 1.3 Как booking соотносится с планом

Booking — **резерв**, по инварианту Timetta (`booking.object.ts:33-51`) намеренно НЕ связан ни с план-оценкой, ни с фактом:
- SOFT — потребность (пресейл/РП), НЕ потребляет ёмкость → пунктирный слой.
- HARD — подтверждена, потребляет ёмкость → входит в `load`.
- Овербукинг не блокируется, показывается конфликтом (`composeCell`, `conflict = load > capacity`).
- Раскид `hours` — той же `plannedHoursInPeriod` (равномерно по рабочим дням периода брони).

**Чего booking НЕ делает:** он не привязан к помесячным плановым часам проекта и не участвует в Σ-сверке с `plannedEffort`. Бронь «Иванов → Альфа, июль, 80 ч» — это «держим Иванова», а не «план работ Иванова по Альфе на июль». В Timetta это разные слои (assignment/resource-plan vs booking), но у нас **booking — единственное, что вообще привязывает часы к человеку**.

### 1.4 Чего НЕТ (gap для person-level плана)

1. Плановых часов на конкретного человека в проекте/периоде (project × employee × month → hours).
2. Распределения объёма проекта между людьми (кто сколько делает) — есть только доли ОТДЕЛОВ.
3. «Личного плана» в честном смысле — доска «Люди» показывает `1/N`.
4. Аллокации в FTE%/доле на человека (есть только FTE на отдел в `employee-department`).
5. Наследования «план отдела → планы людей внутри отдела».

## 2. Разведка конкурентов: person-level resource planning

### 2.1 Timetta (главный легальный референс)

Ресурсный план = **член команды × работа × календарный период** (см. [[PLANNING_MODELS]] §2.1, research/timetta/docs/). Ключевое для person-level:
- **Единица:** часы / % от расписания / **FTE** / дни — переключаются.
- **Масштаб:** день / неделя / месяц; при смене масштаба значения распределяются **пропорционально рабочему расписанию пользователя** (не календарно).
- **5 методов раскида** (ручной, фронт-загрузка, вся ёмкость, равномерно по рабочим дням, % расписания).
- **Назначение конкретного сотрудника на работу проекта** — это и есть атом person-level. Авто-заполнение из структуры работ для «автоматического типа планирования» (isAutoPlanning-аналог).
- **Soft/Hard бронь** — отдельная ось резерва (у нас уже есть в booking).
- Кто назначает: РП формирует ресурсный план; ресурс-менеджер подтверждает брони (soft→hard).
- **Top-down + bottom-up:** план оценивается на члене команды, агрегируется вверх до проекта/отдела.

### 2.2 Float

- **Единица аллокации:** **часы/день** (стандарт) ИЛИ **% capacity** (100% = полная занятость в диапазоне, минус праздники/отпуска).
- Назначение person→project на период; утилизация = аллокация/ёмкость; overtime = сверх ёмкости.
- **Tentative vs Confirmed** аллокации (= soft/hard).
- Гранулярность — до дня, по человеку.

### 2.3 Runn

- Аллокация person→project, но **менее гранулярно**, чем Float: проекты/фазы/вехи, без точных задач/дневного календаря.
- Soft booking — отдельно от confirmed (см. runn.io/blog/soft-booking-resources).

### 2.4 Kantata (ex-Mavenlink), Saviom, Forecast

- **Kantata:** FTE Allocated Hard / FTE Allocated Soft — раздельный учёт hard/soft в FTE; allocations = person × project × период в часах или FTE.
- Общий паттерн PSA: аллокация хранится как **запись (resource assignment)** = person × project × период × объём(часы|FTE|%), с типом soft/hard; овербукинг показывается; разрешение конфликтов — ручное (менеджер двигает) либо capacity leveling (авто).

### 2.5 Что из этого важно для НАС

| Аспект person-level | Timetta | Float | Runn/Kantata | У нас сейчас | Кандидат MVP |
|---|---|---|---|---|---|
| Атом плана | член команды × работа × период | person × project × день | person × project × фаза | НЕТ (1/N отдела) | **person × project × месяц** (слот+employee) |
| Единица | часы/%/FTE/дни | часы/день, % | часы/FTE | часы | **часы** (как слот), %/FTE — follow-up |
| Soft/Hard | бронь (отд. ось) | tentative/confirmed | soft/confirmed | booking SOFT/HARD | оставить в booking |
| Кто назначает | РП + ресурс-мен. | планировщик | РП | — | РП проекта |
| Овербукинг | показ, не блок | показ | показ | показ (composeCell) | оставить |
| Top-down/bottom-up | оба | bottom-up | top-down | top-down (отдел) | оба (слот без emp = отдел) |

## 3. Booking vs план человека — дубль или нет?

**Не дубль, но граница тонкая.** Это главный риск.

| | Booking (резерв) | Person-план (предлагаемый) |
|---|---|---|
| Вопрос | «держим ли мы человека?» | «сколько человек спланирован сделать?» |
| Привязка к plannedEffort проекта | нет (своя ось) | да (Σ слотов человека = доля проекта) |
| Σ-сверка | нет | да (часть Σ-сверки слотов проекта) |
| Жизненный цикл | пресейл → договор (soft→hard) | план работ по выигранному проекту |
| Учёт в Demand | hard → load, soft → пунктир | как plan slot → load |

**Риск:** если ввести person-план, на человека станет ДВА «плановых» числа — бронь и план. Заказчику надо явно ответить (вопросы Q9, Q22, Q23): это разные слои (как Timetta) или person-план поглощает booking? **Рекомендация (см. §5): оставить раздельно** (Timetta-инвариант), но в UI на доске «Люди» показывать оба слоя визуально различимо.

## 4. Волна вопросов заказчику (W7) — 26 вопросов

Формат: вопрос → гипотезы → реф.

### A. Модель данных и гранулярность

**Q1. Атом person-плана — это `project × employee × месяц`?**
- (а) Да, помесячно (= расширить plan-slot полем employee). [Timetta месяц-масштаб; наш MANUAL уже помесячный]
- (б) По неделям (мельче). [Float дневной; сложнее, новый объект]
- (в) Без разбивки по периодам: «Иванов на Альфе 200 ч за весь срок». [проще, но теряем раскид]

**Q2. Единица плана человека — часы, %FTE или доля?**
- (а) Часы (как сейчас слот/бронь). [простейшее, additive]
- (б) % от его расписания. [Float %, Timetta % расписания]
- (в) FTE (0.5 ставки). [Timetta/Kantata FTE; но FTE у нас уже = занятость в отделе]
- Гипотеза-MVP: **часы** (минимальное изменение), %/FTE — follow-up.

**Q3. План человека независим или это детализация доли отдела (`plannedEffortShare`)?**
- (а) Детализация: Σ людей отдела = доля отдела (вложенный раскид отдел→люди). [согласованность, Timetta агрегация]
- (б) Независимая ось: план человека сам по себе, отдел = Σ людей (bottom-up). [Float bottom-up]
- (в) Обе: слот без employee = отдел, слот с employee = человек, складываются по правилам.

**Q4. Σ-сверка: план людей должен сходиться с `plannedEffort` проекта?**
- (а) Мягкая индикация рассинхрона (как `slotsVsPlannedEffort`, не блок). [наш текущий паттерн]
- (б) Жёсткая (нельзя превысить). [строже, риск трения]
- (в) Без сверки.

**Q5. Человек может быть в плане без отдела / в нескольких отделах проекта?**
- (а) Слот человека несёт и departmentId, и employeeId. [учёт мульти-отдела REQ-0013]
- (б) Только employee, отдел берётся из его назначения. [проще]

### B. Связь отдел ↔ человек

**Q6. Отдел = Σ людей (после ввода person-плана) или остаётся независимым?**
- (а) Если у проекта есть person-слоты — load отдела = Σ людей; иначе fallback на долю/plannedEffort. [как MANUAL уже вытесняет EVEN]
- (б) Отдел и люди — параллельные слои, не суммируются автоматически.

**Q7. Что делать с текущим `1/N`-делением на доске «Люди»?**
- (а) Заменить: если есть person-слот — показываем его; иначе fallback `1/N`. [плавная миграция]
- (б) Убрать `1/N` совсем (только явный план). [честно, но пустая доска до ввода планов]

**Q8. Наследование «план отдела → люди»: автораскид доли отдела по людям?**
- (а) Нет, вручную (РП сам назначает людей). [keep-it-simple]
- (б) Кнопка «разложить поровну/по FTE» как стартовая заготовка. [Timetta авто-заполнение]

### C. Booking vs план (критично)

**Q9. Person-план и booking — разные слои или один?**
- (а) Разные (Timetta-инвариант): план = трудозатраты, booking = резерв. [сохраняет архитектуру]
- (б) Person-план заменяет booking. [меньше сущностей, но ломает пресейл-сценарий компании-без-проекта]
- (в) Booking остаётся для пресейла (company, без проекта), person-план — для выигранных проектов.

**Q10. Если разные — оба входят в Demand человека? Как не задвоить?**
- (а) План в load, HARD-бронь тоже в load, но это РАЗНЫЕ проекты/периоды (не пересекаются по смыслу). [Timetta]
- (б) Если есть person-план по проекту — бронь этого же проекта не суммируется (план приоритетнее).

**Q11. Soft/Hard у person-плана нужны?**
- (а) Нет, план всегда «твёрдый», soft живёт в booking. [разделение ответственности]
- (б) Да, статус слота (черновик/утверждён). [ближе к Timetta «прогноз/оценка»]

### D. Кто назначает и права

**Q12. Кто создаёт person-план?**
- (а) РП проекта (`project.manager`). [Timetta РП]
- (б) Руководитель отдела (раскладывает свою долю). [top-down]
- (в) Оба в своих зонах.

- (б) Может предложить корректировку (заявка). [сложнее]
- (в) Не видит вообще (только РП/руководитель). [как сейчас]

### E. Период и раскид

**Q14. Период слота человека — месяц (как сейчас slot) или гибче?**

**Q16. Ёмкость человека для конфликта — личный календарь × коэф?**
- (а) Да, `employeeLoadCells` уже считает реальную личную ёмкость. [готово]

## 5. Рекомендация реализации (MVP)

### 5.1 Решение по модели данных: расширить `credosTimePlanSlot` полем `employee`

**Выбор из трёх вариантов:**

| Вариант | Плюсы | Минусы | Вердикт |
|---|---|---|---|
| **A. + поле `employee` в plan-slot** | переиспользует panel+REST+`slotsHoursInPeriod`+Σ-сверку; additive (nullable); слот без emp = текущий dept-уровень; один механизм | надо расширить ключ дедупа и `/s/plan-slots` | **✅ ВЫБРАН** |
| B. Переиспользовать booking | booking уже person×project×период | booking — РЕЗЕРВ, не план (ломает Timetta-инвариант); нет помесячной Σ-сверки; задвоение смыслов | ❌ |
| C. Новый объект `credosTimePersonPlan` | чистая модель | дублирует 90% plan-slot; новый объект+view+nav+REST+calc; против keep-it-simple | ❌ |

**Почему A:** `plan-slot` уже = `project × [department] × periodMonth`. Добавление `[employee]` делает ключ `project × [department] × [employee] × periodMonth` — естественное углубление гранулярности на один уровень. Слот без employee работает ровно как сейчас (dept-уровень), слот с employee — person-уровень. Booking остаётся отдельной осью резерва (инвариант Timetta сохранён). Это минимальное аддитивное изменение, дающее заказчику ровно то, что он просит.

### 5.2 План для Dev2 (бэк)

1. **Объект `credos-time-plan-slot.object.ts` — добавить поле** (additive, [[twenty-sdk-apply-gotchas]]: новый UUID v4, имя `employee` — не резерв, отдельный sync без drop):
   - `employee`: RELATION → `credosTimeEmployee`, `isNullable: true`, `onDelete: SET_NULL`, `joinColumnName: 'employeeId'` (зеркало `department` в этом же объекте).
   - Обратное поле `Employee.planSlots` (новый field в `src/fields/`, ONE_TO_MANY) + новый field-id в `universal-identifiers.ts`.
   - **Гранулярность по умолчанию не меняется**: дефолт employee=null → текущее поведение.
2. **Дедуп/ключ:** расширить ключ upsert в `/s/plan-slots` route до `project × department × employee × periodMonth`. read-by-key учитывает employeeId (null-safe). `plannedHours<=0` → удаление слота (как сейчас).
3. **Контракт REST** (`plan-slots-rest.ts`, тип `PlanSlotInput`): добавить `employeeId?: string | null` в read-ответ и upsert-вход (зеркало `departmentId`).
4. **Логика `calc-load.ts`** — новая функция `projectEmployeeHoursInPeriod(project, employeeId, period, spread, slotsByEmployee)`:
   - В `employeeLoadCells`: ЕСЛИ у проекта есть person-слоты данного сотрудника → `planLoad = Σ слотов человека` (по аналогии с `useManual`/`slotsHoursInPeriod`); ИНАЧЕ fallback на текущее `deptLoad × (1/headcount)`.
   - `buildSlotsByEmployee(slots)` — карта `employeeId → PlanSlot[]` (зеркало `buildSlotsByProject`).
   - Σ-сверка: расширить `slotsVsPlannedEffort` опционально по employee (мягкая, не блок).
5. **Без биллируемости** ([[no-billable-concept]]) — только часы, никаких ставок.
6. **Отчёты (опц., по ответу Q22):** plan-fact по человеку = Σ person-слотов vs Σ его time-entry.

### 5.3 План для Dev1 (фронт)

1. **Панель «✎ План» проекта (`project-plan-panel.tsx`), режим MANUAL:** к помесячным строкам добавить выбор **сотрудника** (рядом с уже существующим `slotDept`/`slotEmployee` по аналогии). Минимальный вариант (Q18б): в строке месяца — селект сотрудника + часы. Расширенный (Q18а): матрица человек×месяц.
   - Состояние `slotEmployee: Record<string, string|null>` зеркалит `slotDept`.
   - Σ-сверка слотов уже есть (`reconcileSlots`) — учесть разбивку по людям.
2. **Доска «Люди» (`employee-row.tsx`):** план человека — заливка ячейки (как сейчас), бронь — `BookingMarker` (уже есть). Если есть person-слот — показываем его, иначе fallback `1/N` (Q7а). Tooltip: «план N ч (по слотам) / ёмкость M ч».
3. **Drill-down (опц., Q17в):** раскрытие проекта внутри отдела → строки людей с их часами.
4. **Список «кого назначать» (Q20):** сотрудники отделов проекта (по долям/departmentId).
5. **impeccable:** различимость слоёв план/бронь, читаемость матрицы, пустые состояния («план по людям не задан → показываем долю отдела»).
