## Группа 1. Частично-согласованная неделя

**W5A.1** → **(а)** lock построчный, DRAFT/REJECTED-строки той же недели редактируемы. Подтверждаю текущее. Ref=T: lock на уровне строки таймшита, не периода. **Но** добавить визуальный индикатор недели «частично согласовано» (W5A.3).

**W5A.2** → **(а)** добавление новой строки в неделю с APPROVED допустимо — новая = DRAFT, согласуется отдельным submit. Ref=T: дозаполнение после approve = новый submit. (Совпадает с incremental submit W5A.4.)

**W5A.3** → **(б)** ввести «3/4 согласовано» в бейдже периода (счётчики, A4.21). Сейчас «3 approved + 1 draft» = DRAFT — вводит в заблуждение. Считать по остатку незакрытых. Ref=T прогресс-индикатор.

**W5A.4** → **(в)** повторный submit «досылает» только DRAFT + предупреждение «часть периода согласована, отправляется N черновиков». Не блокировать (incremental). Ref=K incremental submit.

## Группа 2. Approve/Reject гранулярность

**W5A.5** → **(а)+(б)** UI шлёт ВСЕ submittedIds сотрудника (frontend-инвариант, есть) + **backend defense: валидировать что ids = полный submit сотрудника** (не доверять клиенту, класс CISO-005). A4.13. Ref=T reject целиком.

**W5A.6** → **(а)+(б)** правка REJECTED-ячейки → DRAFT (естественный переход в time-entry-api) **И** submit поднимает `status[in]:DRAFT,REJECTED` (страховка, если сотрудник не согласен с reject и шлёт без правки). **ДЫРА подтверждена — чинить.** Ref=T: исправленный отклонённый отправляется тем же submit.

**W5A.7** → **(б)** optimistic-lock: если статус сменился между read/write — skip, вернуть «уже обработано N». **Гонка реальна (read-then-write без CAS).** Эталон Timetta: повторное approve = no-op, single-resolver. **CISO/корректность — чинить.**

**W5A.8** → **(б)** добавить счётчик «пропущено N (уже решены)» в ответ + toast. Сейчас статусные skip молча теряются. A4.15. Ref=K bulk рапортует skipped.

**W5A.9** → **(а)** toast «согласовано N, пропущено M (ваши записи — согласует вышестоящий)». skippedOwn возвращается. A4.16. Ref=T SoD-фильтрация.

## Группа 3. rejectComment / аудит-поля

**W5A.10** → **(в) MVP: причина видна до первой правки, потом гаснет** + **(б) roadmap: лог reject-причин**. Сейчас затирание при любом не-REJECTED. Для аудита повторных отклонений нужна история (Ref=T: комментарий ревизии в истории таймшита) — но это отдельный журнал, не MVP. MVP — не затирать до явной правки.

**W5A.11** → **(б) добавить `revokedBy`/`revokedAt`** отдельными полями. Сейчас revoke обнуляет approvedBy/At → **кто отозвал нигде = потеря аудита**. CISO-критично. Ref=T: Reopen фиксируется с автором. **Чинить.**

**W5A.12** → **(б) разделить `resolvedBy`/`resolvedAt` (resolver, approve+reject) от `approvedBy` (approve-only)** ИЛИ минимум **(в)** в UI подписывать «отклонил: …» для REJECTED. Сейчас approvedBy на REJECTED = отклонивший (двойная семантика). Рекомендую (б) — чисто; связать с revokedBy (W5A.11) в общий resolver-аудит. Ref=T отдельные поля.

## Группа 4. recall — границы/каскады

**W5A.13** → **(а)** recall разрешён всё время в SUBMITTED, флаг «открыто руководителем» НЕ нужен. keep-it-simple. Ref=T: recall до решения, «в работе» не блокирует.

**W5A.14** → **(в)** частичный recall разрешён осознанно (отозвать 2 из 5 строк для правки одной). backend не запрещает подмножество — ок. Не ломает A4.1 (submit целиком ≠ recall целиком).

**W5A.15** → **(в)** факт = ВСЕ записи независимо от статуса (текущий computeFactHours). recall/revoke factHours НЕ меняют. Ref=T: **факт по фактическим часам, не по статусу согласования** (прямое подтверждение разведки). Подтверждаю (а)+(в).

## Группа 5. «Акт» (A4.27) и каскады

**W5A.16** → **(в) Акт = снимок** на момент создания, revoke на него не влияет. Ref=T/Директум: акт = снимок, ревизия часов требует НОВОГО акта. (Дополнительно W5A.18.)

**W5A.17** → **(в) только APPROVED + предупреждение** «N часов не вошли (не согласованы)». Акт = согласованное, но прозрачность по несогласованному остатку. Ref=Директум: акт по согласованным трудозатратам.

**W5A.18** → **(а) запретить дубль, если Акт за период/клиент уже есть** (один акт = период×клиент). Пересоздание — через явную отмену старого. Ref=Директум/1С: нумерация без дублей.

**W5A.19** → **(б) разрешить, акт по APPROVED-части** (см. W5A.17в) — НЕ блокировать до полного согласования периода. Гибче: акт можно по согласованному, остальное — следующим актом. Ref=Директум.

## Группа 6. submit — выборка/норма

**W5A.20** → **(а) MVP: нет DRAFT → кнопки нет.** **(б) roadmap:** Timetta допускает submit нулевого таймшита (подтверждение отсутствия работ) — но это требует явного «подтвердить нулевой период», не MVP. Ref=T нулевой submit.

**W5A.21** → **(б) терминальный визуальный статус «не требует согласования»** для записей проектов без approvalRequired (не «черновик»!). Сейчас навсегда DRAFT = вводит в заблуждение. Факт учитывается, процесса нет. Ref=K: вне approval-flow = авто-принято. **Чинить визуал.**

**W5A.22** → **(а) period = неделя (Пн–Вс), submit недельный.** **Разведка Timetta ПОДТВЕРЖДАЕТ: TimeSheet недельный** (dateFrom=Пн..dateTo=Вс, не месяц — корректирую прежнее предположение про месяц). A4.1. ✓

## Группа 7. lock APPROVED — границы

**W5A.23** → **(б)+(в)** lock APPROVED должен запрещать и смену проекта/даты (это «перенос» согласованного часа); перенос = **revoke + пересоздание** (явный flow). Сейчас lock только на hours/desc самой записи, статусные переходы в обход — ок (revoke легитимен). Ref=T: согласованная строка read-only до Reopen.

**W5A.24** → **(в) НЕ вводить REVOKED-статус**, различать через `approvedAt is null` + `revokedBy/revokedAt` (из W5A.11). revoke = чистый возврат в SUBMITTED. Ref=T: **Reopen возвращает в Pending без отдельного статуса** (прямое подтверждение). A4.20 «4 статуса» сохраняется. revokedBy даёт аудит-отличие без нового статуса.

**W5A.25** → **(б) запретить hard-delete проекта с APPROVED-записями/Актом; архивировать (soft-delete).** Ref=T: **проект с трудозатратами архивируется, не удаляется**. Связь с C1.7 (архивация vs удаление). **CISO/аудит — потеря согласованной истории недопустима.**

## Группа 8. агрегат-статус / уведомления

**W5A.26** → **(б) «частично отклонено: 1 из 4»** вместо чистого REJECTED при частичном reject. Сейчас одна REJECTED красит весь период — но баннер причины (A4.23) полезен. Компромисс: бейдж «частично отклонено N/M» + баннер причины по отклонённым. (Согласуется с W5A.3.)

**W5A.27** → **(а)+(в)** уведомление сотруднику при revoke «согласование отозвано, требуется проверка» + **поле причины для revoke** (по аналогии с reject-comment — почему отозвали). A4.29. Ref=T: Reopen шлёт нотификацию владельцу. *(Доставка — F4.1 блокер канала, пока in-app/бейдж.)*

**W5A.28** → **(б) уведомить руководителя «сотрудник отозвал отправку»**, если запись была в его очереди. Очередь перестраивается + явный сигнал. Ref=T recall убирает из очереди (у них без алерта, но нам полезно для прозрачности).

**W5A.29** → **(в) collect errors, не бросать на середине** + идемпотентный повтор (уже APPROVED пропустятся). Полная транзакционность (б) избыточна для REST-батча; per-item результат (Ref=K) практичнее. **Сейчас throw на середине = частичное применение без отката — чинить на collect-errors.**

**W5A.30** → **(б) submit включает REJECTED** (`status[in]:DRAFT,REJECTED`) — переотправка исправленного тем же submit. Устраняет UX-конфликт «бейдж отклонено, но submit шлёт не все». Ref=T: исправленный отклонённый отправляется повторно тем же submit. **(= W5A.6, единый фикс.)**
