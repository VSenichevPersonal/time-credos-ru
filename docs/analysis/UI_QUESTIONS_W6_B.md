## 2. Уровни drill по каждой оси (что → что)

**W6B.5. «Естественная» следующая ось при drill (`PREFERRED_NEXT`): отдел→проект, проект→сотрудник, сотрудник→проект, группа→вид, категория→проект. Это правильные дефолты для вашей работы?**

(б) отдел должен сначала проваливаться в сотрудников (кто в отделе), потом в проекты;
(в) сотрудник→вид работ важнее, чем сотрудник→проект (что человек делает).
— реф: `drill-axis.ts:25` PREFERRED_NEXT.

**W6B.6. Drill бесконечен по доступным осям (любая комбинация cross-filter), пока не кончатся оси. Нужен ли предел глубины или «терминальный» уровень (например, после сотрудника — стоп)?**
(а) без предела — гибкий OLAP, пользователь сам решает;
(б) предел 3 уровня (как у большинства BI) — глубже теряется смысл;
(в) терминальный уровень = лист записей (detail), drill дальше осей упирается в записи.
— реф: `reports-calc.ts:656` availableDims = все кроме groupBy и фильтров; глубина не ограничена.

**W6B.7. Drill-до-отдельных-записей (лист detail) из таблицы среза — нужен? Сейчас detail доступен ТОЛЬКО через CSV-экспорт, на экране лист записей не показывается.**
(а) да — клик по конечной строке открывает список записей (Дата/Кто/Часы/Коммент) на экране;
(б) нет — detail только для выгрузки в CSV, на экране агрегаты достаточно;
(в) показывать записи в раскрывающейся панели под строкой (inline).
— реф: `reports-detail.computeDetail` есть, UI листа нет; экспорт сужается до 3 осей (dept/project/employee).

**W6B.8. При drill экспорт CSV сужается ТОЛЬКО по осям dept/projectId/employeeId. Если путь drill содержит category/workType — эти фильтры в CSV НЕ применяются (выгрузка шире экрана). Это баг для пользователя или приемлемо?**

— реф: `reports-dashboard.tsx:203-208` exportFilters только dept/project/employee.
