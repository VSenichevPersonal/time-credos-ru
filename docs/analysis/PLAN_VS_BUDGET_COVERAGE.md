## 0. TL;DR (ответы заказчику)

1. **Стыковка ЕСТЬ, но только в одном месте и только в одном режиме.** `reconcileSlots` (plan-preview.ts:359) сверяет Σ слотов с `plannedEffort` — но ТОЛЬКО внутри MANUAL-панели одного проекта при ручном вводе. На доске, в строке проекта, в отчётах, для EVEN-режима, для пресейла — стыковки НЕТ.
2. **«Сколько распланировано» как видимое число — НЕТ.** Нигде на доске/в отчёте не показано «распланировано N из M ч». Есть `budgetRemaining = plannedEffort − factHours` (план vs ФАКТ), но это про ОСВОЕНИЕ, а не про «всё ли распланировано».
3. **«Всё ли распланировано» — НЕТ индикатора остатка** (`plannedEffort − Σslots`) и **нет warning переаллокации** (Σslots > plannedEffort) на уровне строки/доски.
4. **🔴 КРИТИЧНО — SSOT-конфликт.** В MANUAL-режиме панель ПЕРЕЗАПИСЫВАЕТ `plannedEffort = Σслотов` (`project-plan-panel.tsx:212` → `plannedEffort: manualRecon.sum`). Это **СТИРАЕТ смысл `plannedEffort` как бюджета-оценки**: после сохранения «бюджет» автоматически = «распланировано», остаток всегда 0, недо-/переаллокацию посчитать НЕ от чего. Сверка `reconcileSlots` после сохранения всегда «✓» — она бессмысленна вне открытой панели.
5. **Пресейл (company-booking) — нет плановой трудоёмкости вообще.** `credos-time-booking` имеет только `hours` (ёмкость на период) и `winChance` (%). Нет `expectedEffort`/estimate → для пресейла «сколько распланировано / всё ли» считать НЕ от чего. Связи с CRM Opportunity (сделкой) НЕТ (в коде «opportunity» только в комментариях).

## 1. ТЕКУЩЕЕ: что есть / чего нет

### 1.1 `plannedEffort` — что это сейчас (двусмысленно)

`credos-time-project.object.ts:116` — `plannedEffort` (label «Плановые часы», FLOAT). **Заявленный смысл** (по комментариям/полям): плановая трудоёмкость = БЮДЖЕТ часов проекта. Подтверждение: `budgetRemaining = plannedEffort − factHours` (object.ts:150-159) трактует его как бюджет для освоения.

**Но фактически смысл «плавает» по режиму:**
- **EVEN** (дефолт): `plannedEffort` = бюджет-объём, вводится руками, раскидывается равномерно по рабочим дням. Здесь это ЧИСТЫЙ бюджет.
- **MANUAL**: панель пишет `plannedEffort = Σслотов` (`project-plan-panel.tsx:200-216`). Бюджет ЗАТИРАЕТСЯ суммой распределения. **Бюджет ≠ распланировано перестаёт существовать как разделимые величины.**

> Это и есть главный флаг для арха: одно поле тащит ДВА разных смысла (бюджет-оценка vs авто-Σ-раскида), которые в MANUAL схлопываются. Нельзя строить индикатор coverage поверх поля, которое само себя перезаписывает.

### 1.2 Три РАЗНЫХ числа (ось «внутри плана» — не путать)

| Число | Поле / расчёт | Смысл | Статус |
|---|---|---|---|
| **Бюджет** (Y) | `project.plannedEffort` (в EVEN) | плановая трудоёмкость = оценка-бюджет | ЕСТЬ, но в MANUAL затирается (§1.1) |
| **Распланировано** (X, allocated) | Σ slots (MANUAL) / Σ EVEN-раскид | сколько РАСПРЕДЕЛЕНО по периодам/людям | расчёт есть (`sumSlotHours`/EVEN), как ВИДИМОЕ число — нет |
| **Факт** | `project.factHours` | списанные часы | ЕСТЬ |
| **Остаток освоения** | `budgetRemaining = план − факт` | сколько бюджета НЕ ОСВОЕНО | ЕСТЬ |
| **Остаток распределения** | `plannedEffort − Σslots` | ВСЁ ли распланировано | **НЕТ** |
| **Переаллокация** | `Σslots − plannedEffort > 0` | распределено БОЛЬШЕ бюджета | **НЕТ** |

**Ключевая путаница:** `budgetRemaining` (есть) = план − **ФАКТ** = освоение. Это НЕ ответ на «всё ли распланировано». Пример: бюджет 500, распланировано 300, факт 0 → 200 ч НЕ распределено (дыра спроса), но `budgetRemaining` показывает 500 — будто всё в порядке.

### 1.3 `reconcileSlots` — единственная существующая сверка

`plan-preview.ts:359-367`: `reconcileSlots(slots, target) → {sum, target, ok}`, допуск 1 ч. Используется в `project-plan-panel.tsx:135-138` (MANUAL) — живая Σ-сверка «Σ раскид = N ч · план M ч · ✓/≠».

**Ограничения:**
- Только в MANUAL-режиме (EVEN не порождает слотов, сверять нечего → EVEN считается «100% распланирован» по определению).
- Только при ОТКРЫТОЙ панели одного проекта.
- `target` = `effort` (текущий ввод в поле «Объём»), а при сохранении `plannedEffort` становится = `sum` → **после сохранения сверка всегда ✓** (бюджет затёрт, §1.1). Сверка живёт только ВНУТРИ сессии ввода.
- Аналог для персональных слотов (Σлюди vs план отдела) — нет в этом хелпере (см. PERSON_PLAN_CONSISTENCY_AUDIT W8.7).

### 1.4 Чип «людям N ч»

`personalSlotsByMonth` (plan-preview.ts:381) + чип в панели (`project-plan-panel.tsx:318-338`): read-only «людям N ч» = Σ персональных слотов проекта. Показывает, сколько уже роздано людям — НО это НЕ сверка с бюджетом (нет «из M ч»), и тоже только в открытой панели.

### 1.5 Доска / строка проекта / отчёты — где НЕ показывают coverage

| Место | Показывает план? | Показывает «распланировано X/Y»? | Остаток / over? |
|---|---|---|---|
| Строка проекта на доске (`project-plan-row`) | загрузку по периодам | **нет** | **нет** |
| Панель плана (MANUAL, открыта) | reconcileSlots Σ vs объём | частично (Σ vs объём, в сессии) | ≠-флаг есть, нет числа остатка |
| Панель плана (EVEN) | превью раскид + Σ-сверка vs объём | нет (EVEN=100% по определению) | overbooking vs ёмкости (другое) |
| Отчёт «Проекты план/факт» (`projects-plan-fact.ts:138`) | `planned=plannedEffort` | **нет** (`remaining=план−ФАКТ`) | overrun vs факта (не vs распланировано) |
| Отчёты OLAP/timeseries (`reports-calc.ts`) | `plannedEffort` скаляр | **нет**, слоты не используются (B1) | нет |

**Вывод:** «распланировано X из Y», остаток, недо-/переаллокация **не показаны нигде, где их ждёт пользователь** (строка проекта, доска, отчёт). Единственная сверка — эфемерна (открытая MANUAL-панель) и обнуляется при сохранении.

### 1.6 Пресейл и связь со сделкой

- `credos-time-booking` (company-бронь = пресейл без проекта): поля `hours` (ёмкость на период), `winChance` (% конверсии SOFT→HARD, object.ts:141), `bookingType` (SOFT/HARD). **НЕТ поля плановой трудоёмкости** (estimate/expectedEffort) → нет target для coverage пресейла.
- `winChance` есть, но **нигде не используется для взвешивания** (`winChance × hours`) — это пока «голое» поле-метка.
- **Связи с CRM Opportunity (сделкой) НЕТ.** Booking ссылается на standard `Company`, не на Opportunity. Поиск «opportunity/presale» по коду даёт только комментарии и типы — реальной relation нет. Откуда брать оценку пресейла из CRM — не реализовано.

## 2. Разведка: «бюджет vs распланировано» у конкурентов

| Продукт | Бюджет (Y) | Распланировано (X) | Индикатор «X из Y» / остаток | Пере/недо-аллокация | Пресейл / сделка |
|---|---|---|---|---|---|
| **Float** | Project budget (часы или $) | Σ scheduled (allocated) hours | Бар «budget remaining» = по SCHEDULED часам; over-budget % | Над бюджетом → **бар красный**, «Over budget %» | — (фокус delivery) |
| **Runn** | Project budget (hours/fees) | Σ assignments (planned) | «X of Y planned», остаток бюджета | over-allocation подсветка | tentative-проекты (вероятностный pipeline) |
| **Forecast** | Baseline / budget | allocated (demand) | budget burn vs allocated vs actual | over/under heatmap | — |
| **Timetta** | EstimatedHours / оценка проекта | resource-plan allocated | Σ ресурс-плана vs оценка | overbooking не блок (показ) | **Win Chance** на проекте/сделке → weighted forecast |
| **Harvest** | Project budget (hours/fees) | planned/logged | budget vs spent бар | over-budget alert | — |

**Индустриальный стандарт (повторяется):**
1. **Три разделённые величины: Бюджет → Распланировано (scheduled/allocated) → Факт.** Все три видны раздельно, бюджет НЕ перезаписывается распределением (наш конфликт §1.1 — против стандарта).
2. **«Остаток бюджета» считается по SCHEDULED (распланировано), а не только по факту** (Float явно: remaining budget = scheduled-based). У нас `budgetRemaining` = только факт — половина картины.
3. **Over-budget = визуальный warning (красный бар / %), НЕ блок** — согласуется с нашей философией overbooking-не-блок.
4. **Win Chance / probability** используется для weighted-forecast (вероятность × часы) — у нас поле есть (`winChance`), применения нет.

Источники: [Float — Project budgets](https://support.float.com/en/articles/2732830-project-budgets), [Float — Single Project Report](https://support.float.com/en/articles/8811004-single-project-report), [Runn vs Float](https://www.runn.io/blog/runn-vs-float), [Timetta — Project management](https://timetta.com/features/project-management).

## 3. РЕКОМЕНДАЦИЯ (MVP, additive, keep-it-simple, без биллируемости)

### 3.1 🔴 СНАЧАЛА развести `plannedEffort` (бюджет) и «распланировано» (Σ) — иначе coverage невозможен

Корень: в MANUAL `plannedEffort` затирается Σслотов (§1.1). Пока это так — остаток всегда 0, индикатор бессмыслен. **Развилка для арха/заказчика (W11.12 — критично):**
- **(а) `plannedEffort` = БЮДЖЕТ-оценка (вводится руками, НЕ перезаписывается).** «Распланировано» = derived Σслотов (MANUAL) / =plannedEffort (EVEN). Coverage = `plannedEffort − Σslots`. **Рекомендую** — соответствует стандарту §2, даёт настоящий остаток/over.
  - Правка минимальна: в `project-plan-panel.save` (MANUAL) НЕ писать `plannedEffort: manualRecon.sum`, а оставлять введённый `effort` (бюджет). Σслотов остаётся в слотах как «распланировано».
- (б) `plannedEffort` = авто-Σ (как сейчас) — тогда coverage между ними не существует, остаётся только «бюджет vs факт». Отказ от запроса заказчика про «всё ли распланировано».

> Без выбора (а) пункты 3.2–3.3 нереализуемы корректно. Это блокер-вопрос.

### 3.2 Индикатор «распланировано X / Y ч (Z%)» + остаток (строка проекта + панель + отчёт)

При условии 3.1(а). SSOT: **Y = `project.plannedEffort` (бюджет), X = Σ slots распланировано** (для EVEN X=Y по определению).

Показывать триаду на:
- **строке проекта** (компактно): `Y ч · распланировано X ч · остаток (Y−X)`;
- **панели плана** (Σ-блок — уже частично есть через reconcileSlots, добить число остатка);
- **отчёте «Проекты план/факт»** (добавить колонку «Распланировано» рядом с planned/fact).

Флаг «всё ли»:
- остаток ≈ 0 (±1 ч допуск, как `reconcileSlots`) → ✓ «распланировано полностью»;
- остаток > 0 → «N ч не распределено» (жёлтый) — дыра спроса;
- остаток < 0 → «переаллокация +N ч» (терракот) — warning, НЕ блок (§2 п.3).

Переиспользовать: `sumSlotHours`/`reconcileSlots` (plan-preview.ts) уже считают X и сверку — вынести из MANUAL-панели в общий расчёт строки/отчёта.

### 3.3 Три величины раздельно (не путать с budgetRemaining)

Рядом, но РАЗДЕЛЬНО (как Float/Forecast burn): **Бюджет → Распланировано → Факт**. `budgetRemaining` (план−факт) оставить как «освоение», НЕ выдавать за «всё ли распланировано». Опционально — мини-бар burn на проекте.

### 3.4 Пресейл (опц., P2)

- (а) добавить `expectedEffort` (ожидаемая трудоёмкость, FLOAT, ADDITIVE) на `credos-time-booking` → target для coverage пресейла «забронировано X из ожидаемых Y»;
- (б) применить `winChance`: взвешенный резерв = `winChance × expectedEffort` (weighted forecast, как Timetta Win Chance) — для pipeline-картины;
- (в) связь booking ↔ CRM Opportunity (брать estimate из сделки) — отдельный roadmap-вопрос (W11), сейчас relation нет.
- MVP: пресейл БЕЗ coverage (только Σ часов брони), `expectedEffort` — следующая волна.

### 3.5 Что НЕ делать (keep-it-simple)

- Не вводить новый объект «бюджет» — `plannedEffort` достаточно (после развода смысла 3.1).
- Не материализовать «распланировано» — derived Σслотов.
- Не блокировать переаллокацию — только warning (стандарт §2 п.3, как overbooking доски).
- Не вводить bill-rate/деньги ([[no-billable-concept]]) — всё в часах.

## 5. Резюме для арха

**Топ-находка (🔴 критично):** `plannedEffort` несёт ДВА смысла, которые в MANUAL схлопываются — панель сохраняет `plannedEffort = Σслотов` (`project-plan-panel.tsx:212`), **стирая бюджет-оценку**. Из-за этого «остаток распределения» всегда 0, а единственная сверка `reconcileSlots` (plan-preview.ts:359) живёт лишь внутри открытой панели и обнуляется при сохранении. Coverage-индикатор «распланировано X из Y» невозможен, пока бюджет и Σ не разведены.

**Стыковка с бюджетом по факту:** есть ТОЛЬКО `reconcileSlots` (MANUAL-панель, эфемерно) + read-only чип «людям N ч». На доске/в строке проекта/в отчётах «распланировано X/Y», остаток, недо-/переаллокация — **НЕТ**. `budgetRemaining` = план−ФАКТ (освоение), а не «всё ли распланировано». Пресейл (booking) плановой трудоёмкости не имеет, связи с CRM Opportunity нет, `winChance` не применяется.

**Рекомендация (MVP, additive):** (1) развести смысл `plannedEffort`=бюджет (не перезаписывать Σслотов) — блокер; (2) индикатор «распланировано X / Y ч (Z%)» + остаток/✓ на строке проекта + панели + отчёте (переиспользовать `sumSlotHours`/`reconcileSlots`); (3) переаллокация = warning, не блок; (4) пресейл `expectedEffort`+`winChance` — P2. Без новых экранов и без биллируемости.
