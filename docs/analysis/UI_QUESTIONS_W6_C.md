## 0. Что УЖЕ реализовано и УЖЕ спрошено (НЕ переспрашиваем)

**Реализация (из кода — фиксирую, чтобы углублять края, а не дублировать):**
- **Метрики доски** (`cap-tokens.ts`): 4 переключаемые — `free` (свободно, ч), `pct` (загрузка %=ratio), `plan` (load=Demand, ч), `gap` (дефицит/профицит, ч+%). `fact` упоминается как ширина колонки, но в `LoadCell` нет — факт на доске НЕ выводится. Формулы: `capacity = workHours×headcount×capacityFactor − absence`; `load = ΣprojectDeptHours + ΣdeptPlanHours + hardBooking`; `free = capacity − load`; `gap = load − capacity` (`calc-load.ts:194-201, 474-495`).
- **Resource Gap пороги — финальны:** `GAP_OK=0.05` (±5% баланс), `GAP_NEAR=0.15` (±15% «сильно»). Профицит — синий 0.09/0.24; дефицит — янтарь→терракот `#fbe4dd`. Иконки ●/▲/▼. Подпись «+12ч 8%» / «−40ч 25%» (`cap-tokens.ts:48-82`).
- **Овербукинг:** `conflict = capacity>0 && load>capacity` (100% порог), inset-обводка терракот + иконка ▲ (`calc-load.ts:458`, `cap-tokens.ts:142-143`). В панели — мягкий баннер «N периодов перегруза», НЕ блокирует сохранение (`project-plan-panel.tsx:271-288`).
- **Booking soft/hard:** `bookingType` SELECT enum {SOFT,HARD} (`credos-time-booking.object.ts`). HARD входит в load (потребляет), рисуется ▪+ч; SOFT не входит, пунктир, показ при `tentativeBookingEnabled` (`booking-marker.tsx`, `calc-load.ts:378-411`).
- **Период:** week (≤горизонта) и month; week-шапка двухстрочная (месяц-бэнд + дни), month — одна строка «янв 26» (`period-header.tsx:81-144`). В превью авто-выбор: ≤70 дней → недели, иначе месяцы (`plan-preview.ts:99-102`).
- **Drill 3 уровня:** отдел → проекты/планы-без-проекта → доли-по-отделам (только если ≥2 отдела) (`board-rows.tsx`, `project-detail.tsx`, `planned-project-row.tsx:60-61`).

Валидация: С обязательна, ПО опц. (открытый план до horizonEnd), «ПО раньше С» → ошибка (`plan-preview.ts:300-307`).
- **Проект без endDate:** раскид до `horizonEnd` доски; без horizonEnd → вклад 0 → в unplanned (`calc-load.ts:222-261`).
- **Доли отдела:** `plannedEffortShare` (часы, НЕ %), Σ долей ≈ plannedEffort (мягко), уровень-3 рисует часы (не метрику), подсветка текущего отдела (`credos-time-project-department.object.ts`, `planned-project-row.tsx:129-224`).

## 1. Панель «Планировать» — компоновка, валидация, формат превью

**W6C.1** Превью авто-переключает неделя/месяц на отметке 70 дней (`plan-preview.ts:99-102`) — это незаметный для пользователя скачок гранулярности. Показывать ли явную подпись режима?
- (а) подпись над баром: «раскид по неделям» / «по месяцам»;
- (б) ручной переключатель неделя/месяц как на доске (mode-switcher);
- (в) оставить авто молча — реф: T (booking-scale неделя/месяц явные кнопки), impeccable (не менять представление без сигнала).

**W6C.2** Порог авто-переключения = ровно 70 дней. Это «10 недель» по сути. Удобен ли порог?
- (а) оставить 70;
- (б) 62 (≈2 мес — короткие проекты в неделях, длинные в месяцах);
- (в) привязать к числу колонок (если >12 баров → месяцы) — реф: код `previewGranularity`.

**W6C.3** Σ-сверка допускает расхождение ≤1ч и при >1ч рисует «≠» красным, но НЕ блокирует сохранение (`project-plan-panel.tsx:122,378`). Это норма?
- (а) ≤1ч ок, >1ч — только визуальный warning (текущее);
- (б) показывать дельту явно: «Σ=98ч · план=100ч (−2ч)»;
- (в) при большой дельте (>5%) предлагать «подогнать план под Σ» — реф: T (Σ-сверка таблицы раскида).

**W6C.4** Поле «Объём» парсит запятую→точку, отрицательные/NaN→null (`project-plan-panel.tsx:36-42`). Что показывать при null/пусто в превью?
- (а) превью пустое, кнопка активна (план без объёма — даты только);
- (б) превью пустое + подсказка «укажите объём»;
- (в) блокировать сохранение без объёма — реф: код (`canSave` сейчас не проверяет effort).

**W6C.5** Открытый план (ПО пусто) тянется до `horizonEnd` доски (`plan-preview.ts:198`). Превью показывает раскид до края горизонта — но горизонт двигается. Понятно ли пользователю, что план «обрезан горизонтом», а не закончится?
- (а) текущая подсказка «↗ открытый план — раскид до конца горизонта доски»;
- (б) добавить в превью маркер края «…далее за горизонтом»;
- (в) при открытом плане показывать средн. ч/период вместо хвостовых баров — реф: W3B.19 (открытый план).

**W6C.6** Футер с кнопками закреплён снизу (вне скролла), превью скроллится (`project-plan-panel.tsx:386`). При длинном превью (год по неделям ≈52 бара) — сколько баров показывать?
- (а) все со скроллом (текущее);
- (б) свернуть >12 в «показать все» с агрегацией по месяцам;
- (в) фикс-высота с виртуализацией — реф: impeccable (плотность списка).

**W6C.7** Бар превью: ширина = hours/maxHours×100%, цвет accent/warnSolid при over (`project-plan-panel.tsx:322,335`). Бар нормирован на МАКС период, не на ёмкость. Это путает «насколько загружу»?
- (а) нормировать на ёмкость периода (бар=util, over видно как >100%);
- (б) две полосы: план (accent) на фоне ёмкости (серый);
- (в) оставить нормировку на max — реф: W3B.16 (содержимое бара), T (load vs capacity overlay).

## 2. Способ «Вручную по месяцам» — точная UX слот-объекта (фаза-2)

**W6C.8** Слот-объект `credosTimePlanAllocation` (PLANNING_RECORDS_CONSISTENCY §5) при ручном вводе — гранулярность РЕДАКТИРОВАНИЯ строки. Radio назван «Вручную по **месяцам**», но превью EVEN на коротких проектах в неделях. Конфликт?
- (а) ручной ввод всегда по месяцам (даже короткий проект);
- (б) ручной по той же гранулярности, что и превью (неделя/месяц авто);
- (в) ручной даёт выбор шкалы перед вводом — реф: W3B.1 (гранулярность ручного).

**W6C.9** При ручном вводе по месяцам ячейка ввода — что именно вводит пользователь?
- (а) часы за месяц напрямую (целое/float);
- (б) FTE/доля ставки за месяц (пересчёт в часы по норме);
- (в) % от общего объёма — реф: W3B.2 (абсолют/%), но здесь — точный формат ВВОДА в ячейку.

**W6C.10** Σ-сверка ручного ввода: введено ΣМ ≠ общий Объём поля. Кто главный?
- (а) Объём = read-only итог Σ ячеек (вводишь по месяцам — Объём складывается);
- (б) Объём задаётся, остаток автораспределяется на пустые месяцы;
- (в) две независимые величины + warning при расхождении — реф: W3B.3, T (план = Σ слотов).

**W6C.11** Правка одного месяца в ручном режиме: что с остальными?
- (а) правка месяца НЕ трогает другие (точечная);
- (б) правка месяца перераспределяет остаток по нетронутым (locked/unlocked);
- (в) правка ломает Σ → подсветка до ручной балансировки — реф: T (lock period rows).

**W6C.12** Переход EVEN→MANUAL: первый вход в ручной — ячейки пусты или предзаполнены равномерным раскидом?
- (а) предзаполнить текущим EVEN-раскидом (правишь точечно);
- (б) пусто (вводишь с нуля);
- (в) предзаполнить, но пометить «авто, не тронуто» — реф: impeccable (не терять контекст при смене режима).

**W6C.13** Источник истины при `planMethod=MANUAL`: слоты или формула? При сдвиге дат С/ПО уже введённых вручную слотов?
- (а) слоты — истина, сдвиг дат НЕ двигает слоты (предупредить о выходе за диапазон);
- (б) сдвиг пропорционально перетягивает слоты;
- (в) сдвиг очищает ручные слоты с подтверждением — реф: W3B.4, PLANNING_RECORDS_CONSISTENCY §5.

**W6C.14** Ключ слота — календарный (`2026-07`) vs offset от старта (W3B.7 не решён). Для «Вручную по месяцам» что естественнее?

## 3. Метрики доски — точные формулы и уровни вывода

**W6C.15** Метрика `plan` показывает `load` = Demand = план+hardBooking (`cap-tokens.ts:162`). Название «План» вводит в заблуждение — туда входят hard-брони. Переименовать?
- (а) «Спрос/Demand» (точно по содержимому);
- (б) «План+бронь»;
- (в) оставить «План», но в легенде уточнить состав — реф: код, INDEX (англицизм «Gap» уже отмечен).

**W6C.16** `fact` (факт) НЕ выводится на доске капасити (нет в `LoadCell`), хотя ширина колонки `fact` заложена. Нужен ли факт на доске планирования?
- (а) нет — доска про будущее (план/ёмкость), факт в отчётах;
- (б) да, 5-я метрика «Факт» (Σ time-entry за период);
- (в) гибрид: текущий+прошлые периоды показывают факт, будущие — план — реф: T (план/факт overlay), Kimai.

**W6C.17** На уровне Компании (Summary) показывается сводка всех отделов (`calc-load.ts:558-580`). Какая метрика по умолчанию на верхнем уровне?
- (а) та же, что выбрана глобально (free/pct/plan/gap);
- (б) всегда gap на компании (стратегический «где дефицит/профицит»);
- (в) pct (общая загрузка компании) — реф: T (company-level utilization).

**W6C.18** Σ-средняя по строке = avgRatio только по периодам с ёмкостью (`cap-tokens.ts:115-119`). Период с capacity=0 (весь отдел в отпуске) исключён из среднего. Корректно?
- (а) да, иначе деление на 0 искажает (текущее);
- (б) показывать (n из N периодов) у среднего;
- (в) средняя по часам (Σload/Σcap), не среднее-отношений — реф: код, W5B (округление).

**W6C.19** В срезе «Люди» загрузка сотрудника = доля план-часов отдела поровну по headcount (`calc-load.ts:520`), HARD-брони — личные (не делятся). Точное правило деления при дробном FTE сотрудника?
- (а) поровну 1/headcount независимо от FTE (текущее);
- (б) пропорционально FTE сотрудника (0.5 ставки → половина доли);
- (в) пропорционально + личные брони сверх — реф: W5B.13-18 (дробный FTE), но здесь — деление план-доли в «Людях».

**W6C.20** FirstFreePeriod = первый период с ratio<0.9 (90%), показ «свободен с XYZ» зелёным (`calc-load.ts:543-553`, `dept-row.tsx:92-96`). Порог 90% — финал для ответа продажам «когда обещать старт»?
- (а) 0.9 ок;

- (в) два порога: «свободен» <70% и «есть запас» <90% — реф: T (resource availability).

## 4. Booking soft→hard — переход, индикация, овербукинг

**W6C.21** Перехода SOFT→HARD в панели «Планировать» НЕТ — только ручная правка `bookingType` в карточке брони (INDEX: «SOFT→HARD только ручной правкой»). Где должно быть «подтвердить бронь»?

- (в) пакетное подтверждение всех soft проекта при старте — реф: T (booking confirm), W5B (SOFT-бронь).

**W6C.22** SOFT-брони показываются только при `tentativeBookingEnabled` (`calc-load.ts:391`). При выключенном флаге soft невидимы, но считаются в фоне. Это путает («куда делась бронь»)?
- (а) флаг скрывает визуал, расчёт не трогает (текущее);
- (б) при выключенном флаге soft вообще не создаётся;
- (в) всегда показывать soft, флаг лишь меняет «учитывать в gap или нет» — реф: код, W3B (soft в demand).

**W6C.23** Овербукинг-обводка = inset-тень 1.5px терракот + иконка ▲ 8px (`cap-tokens.ts:142-143`). Без подписи «перегруз». Достаточно ли для распознавания?
- (а) обводка+иконка достаточно (минимализм);
- (б) добавить tooltip «перегруз: load N ч > ёмкость M ч»;
- (в) подпись «+Nч» красным в углу ячейки — реф: impeccable (a11y, не только цвет — иконка есть), W3B.21.

**W6C.24** Овербукинг срабатывает строго при `load>capacity` (>100%). Мягкий порог (90%) предложен в W3B.20. На ДОСКЕ (не в панели) — нужен ли промежуточный «почти полный» сигнал отдельно от gap-янтаря?

**W6C.25** HARD-бронь входит в load и потребляет ёмкость наравне с планом проекта (`calc-load.ts:457`). Если на сотруднике HARD-бронь + план того же проекта — двойной счёт?
