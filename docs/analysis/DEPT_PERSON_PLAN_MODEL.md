## 0. TL;DR (ответы заказчику в одну строку)

1. **«Данные будут заполнены при открытии плана отдела?»** — Сейчас ДА, но НЕПРАВИЛЬНО (баг): dept-панель подтягивает и персональные слоты тоже, мешает их в один месяц (last-write-per-month) → видишь чужое/искажённое число. Должно: dept-панель показывает ТОЛЬКО нераспределённый остаток отдела, персональные слоты — read-only агрегатом «Σ назначено людям», не в полях ввода.
2. **«Режим планирования для каждого проекта?»** — Да, одно поле `project.planMethod` (EVEN|MANUAL). Это НЕ «dept-mode vs person-mode» — отдел и человек живут в одной MANUAL-модели как разные уровни детализации, переключателя «планировать отделом ИЛИ людьми» нет и НЕ нужно (так у всех конкурентов).
3. **«Запоминается последний выбранный?»** — Да, `planMethod` сохраняется на проект и панель открывается в нём (`methodOf`, project-plan-panel.tsx:52). Уровень (отдел/человек) определяется не флагом, а тем, на чьей строке нажал «План».
4. **«Консистентность?»** — Обеспечивается правилом §7.2 SSOT «детальный уровень вычитается из остатка верхнего» (rollup в `employeeLoadCells`/`deptLoadCells`). КЛЮЧЕВОЙ ФИКС — починить чтение слотов в dept-панели (один баг), не трогая модель.

## 1. Текущее поведение (факты по коду) + проблемы

### 1.1 Две панели планирования

| Панель | Файл | Что фильтрует при чтении слотов | Уровень |
|---|---|---|---|
| **Отдел** (строка проекта) | `project-plan-panel.tsx:142` | `fetchPlanSlots(project.id)` — **БЕЗ фильтра employeeId** | проект×[отдел]×месяц |
| **Человек** (строка сотрудника) | `employee-plan-panel.tsx:112` | `slots.filter(s => s.employeeId === employee.id)` — **корректно** | проект×emp×месяц |

### 1.2 БАГ протечки персональных слотов в dept-панель (подтверждён по коду)

`project-plan-panel.tsx:143-150`:
```ts
fetchPlanSlots(project.id).then((slots) => {
  const hoursMap: Record<string, string> = {};
  for (const s of slots) {
    hoursMap[s.periodMonth] = String(s.plannedHours);   // ← ключ ТОЛЬКО месяц
    deptMap[s.periodMonth] = s.departmentId ?? null;
  }
```
- `fetchPlanSlots(project.id)` возвращает ВСЕ слоты проекта: и отдельские (`employeeId=null`), и персональные (`employeeId≠null`).
- `hoursMap` индексируется **только по `periodMonth`** → для месяца, где есть и dept-слот, и person-слот(ы), в поле ввода попадает **последний по итерации** (last-write-per-month). Слоты разных людей одного месяца тоже схлопываются.
- **Сценарий заказчика:** «сперва план по людям → потом открыл план отдела» → в полях dept-панели видны ЧУЖИЕ персональные часы (часы Иванова в поле месяца отдела). При сохранении dept-панель перезапишет/создаст dept-слот с этими числами → искажение + потенциальный двойной счёт на доске (dept-остаток + персональные).
- Симметрично: `employee-plan-panel` фильтрует правильно (строка 112) → обратный сценарий («сперва отдел → потом человек») чистый, но человек НЕ видит, что отдел уже спланирован сверху.

### 1.3 planMethod — одно поле на проект, НЕ режим «отдел/человек»

- `project.planMethod ∈ {EVEN, MANUAL}` — последнее сохранённое, round-trip через `methodOf()` (project-plan-panel.tsx:52). Запоминается ✓.
- Это способ РАСКИДА (равномерно vs руками по месяцам), **не** ось «планируем отделом или людьми». Персональные слоты создаются из отдельной person-панели независимо от `planMethod`.
- **Проблема:** нет единой «ментальной модели режима» для пользователя. Он видит радио EVEN/MANUAL в dept-панели, и отдельную person-панель — связь между ними в UI никак не выражена. Отсюда вопрос заказчика «есть ли режим планирования».

### 1.4 Rollup-модель (корректна концептуально, см. §7.2 PLANNING_EMPLOYEE_LEVEL)

`calc-load.ts:709 employeeLoadCells`:
```
employeeLoad = Σ персональных слотов emp + доля(FTE) нераспределённого остатка отдела
остаток = max(0, deptLoad − Σ всех персональных слотов отдела)   (строка 759)
```
Модель «person = детализация dept-тотала, вычитается из остатка» — анти-двойной-счёт встроен в РАСЧЁТ. **Но в UI (панелях ВВОДА) этот принцип не отражён** — отсюда баг §1.2: панель отдела не знает, что часть месяца уже «съедена» персональными слотами.

### 1.5 Сводка проблем консистентности/UX

| # | Проблема | Источник | Тяжесть |
|---|---|---|---|
| P1 | Персональные слоты протекают в поля dept-панели (last-write-per-month) | project-plan-panel.tsx:142-150 | **высокая** (искажение + риск 2× счёта) |
| P2 | dept-панель не показывает «Σ назначено людям» / остаток | нет в UI | средняя (нет мэтчинга визуально) |
| P3 | person-панель не показывает «отдел уже спланирован сверху» | нет в UI | низкая (информативность) |
| P4 | Нет единой ментальной модели «уровень = отдел/человек» | архитектура UI | средняя (путаница заказчика) |

## 2. Разведка конкурентов: как устроено dept ↔ person

Источники: офиц. доки Timetta ([resource-plan](https://timetta.com/ru/docs/projects/components/projects/resource-plan), [project-team](https://timetta.com/ru/docs/projects/components/projects/project-team)); help-центры Float/Forecast/Runn/Saviom/Ganttic.

| Продукт | Модель | generic→named | Режим: глоб/переключ | При открытии team-плана (есть строки людей) | Anti-double-count | SSOT |
|---|---|---|---|---|---|---|
| **Timetta** | гибрид, крен top-down (план на роль/пул → бронь людей) | да: универс.ресурс **замещается** named, часы переезжают | per-project (тип авто/ручной) | агрегат по строкам команды; представления Оценка/Прогноз | замена, не сумма; централиз. бронь | строки ресурсов; команда = агрегат |
| **Float** | bottom-up; team = Σ людей | да: Placeholder (cap=0) → **Swap person** | глоб. Schedule, статусы per-project | **агрегат (сумма)**, правка на строках людей | placeholder cap=0; total вычисляемый | строки людей (Schedule = SSOT) |
| **Forecast** | гибрид | да: placeholder demand → Assign → **удаляется** | per-project, 3 режима (combined = **max**) | агрегат supply/demand heatmap, не редактируемый | replacement; combined=max не sum | строки named (hard) |
| **Runn** | bottom-up (assignment-centric) | да: placeholder → **Transfer** на человека | views (Project/People), не тумблер | **read-only агрегат** (Σ людей) | один ассайнмент: placeholder ИЛИ человек | строки людей |
| **Saviom** | гибрид | да: `ZZ_*` placeholder → request → **replaced** | per-booking (hard/soft/tentative) | агрегат + gap | замена; early-warning | строки людей |
| **Ganttic** | чистый bottom-up | частично (Unassigned Tasks drag) | per-view над одними данными | **агрегат read-only** у заголовка группы | плановый уровень РОВНО ОДИН | строки ресурсов |

### 2.1 Индустриальный стандарт (повторяется у лидеров)

1. **SSOT = строки по людям; уровень отдела/команды — производный агрегат, НЕ самостоятельно вводимая величина.** «План команды» нигде не редактируется как тотал — суммируется снизу.
2. **generic→named через ЗАМЕНУ (replace/swap/transfer), а не параллельное сосуществование** — главный механизм anti-double-count.
3. **Placeholder = спрос (demand) по роли; named = ёмкость (capacity).** Два разных смысла.
4. **Top-down существует только как слой спроса, а не жёсткая квота «Σлюди ≤ план-команды».** Ни один продукт публично не вводит такой cap.
5. **Режим — per-project / per-view (разные проекции одних данных), а не глобальный тумблер «отдел или люди».**

### 2.2 Где мы относительно стандарта

- ✅ Rollup-расчёт уже bottom-up-совместим (person вычитается из остатка отдела) — соответствует п.1.
- ✅ FTE-headcount уже снизу (REQ-0011) — соответствует SSOT-строкам людей.
- ⚠️ **Расходимся в UI:** dept-панель РЕДАКТИРУЕТ отдельские слоты как самостоятельную величину И протекают персональные (п.1 нарушен в UX). Лидеры дают team-уровень read-only-агрегатом.
- ❌ Нет «generic→named replace»-жеста (п.2) — но это Фаза-2/3 (см. PLANNING_EMPLOYEE_LEVEL §5), не блокер.

## 3. РЕКОМЕНДУЕМАЯ СПЕКА (детерминированно, SSOT, keep-it-simple)

### 3.1 Одна ментальная модель: «уровень детализации», НЕ режим-переключатель

**НЕ вводить** тумблер «планировать отделом ИЛИ людьми». Вместо этого — единая модель «отдел = контейнер, человек = детализация внутри него» (как все конкуренты, п.1/п.5):

```
План проекта на месяц = Σ персональных слотов людей  +  нераспределённый остаток отдела
                        └─ задаётся в person-панели ─┘  └─ задаётся в dept-панели ─┘
```
- **`planMethod` (EVEN|MANUAL)** остаётся как есть — это способ раскида общего объёма, ортогонален уровню. Запоминается на проект (уже работает). EVEN не порождает слотов.
- **Уровень (отдел/человек)** определяется строкой, на которой нажал «План» — НЕ отдельным флагом. Так не нужен ни новый объект, ни переключатель.

### 3.2 Правило «person → dept»: что показывать в dept-панели

**Когда у проекта есть персональные слоты, dept-панель НЕ должна их редактировать.** Детерминированно:

```
Для месяца M в dept-панели:
  personSum(M)  = Σ plannedHours слотов где employeeId≠null И departmentId=отдел И month=M
  deptSlot(M)   = plannedHours слота где employeeId=null  И departmentId=отдел И month=M  (остаток)
  ИТОГО план отдела за M = personSum(M) + deptSlot(M)

Поле ВВОДА в dept-панели = deptSlot(M)  (ТОЛЬКО остаток, НЕ pers] sum)
Read-only строка        = «назначено людям: personSum(M) ч»  (если >0)
```

**Фикс P1 (ключевой):** в `project-plan-panel.tsx` при префилле читать ТОЛЬКО отдельские слоты:
```ts
const deptOnly = slots.filter(s => !s.employeeId);   // ← добавить фильтр
for (const s of deptOnly) hoursMap[s.periodMonth] = String(s.plannedHours);
// персональные — отдельной картой для read-only показа personSum
```

### 3.3 Правило «Σлюди vs план-отдела» (мэтчинг)

Следуем стандарту п.4 — **жёсткой квоты НЕ вводим** (`[[keep-it-simple]]`, лидеры не используют). Мягкая индикация (= I1 из PLANNING_EMPLOYEE_LEVEL §7.3, не дублирую расчёт):

| Ситуация | Поведение dept-панели |
|---|---|
| `personSum < планОтдела` | остаток > 0 → поле «остаток отдела» = разница; подпись «N ч не распределено по людям» |
| `personSum = планОтдела` | остаток 0 → поле остатка 0/пусто; «✓ полностью распределено» |
| `personSum > планОтдела` (если задан общий объём) | **warning** «переаллокация +N ч» (НЕ блок, как овербукинг доски) |

### 3.4 UX-правила панелей (точные)

**dept-панель (project-plan-panel), когда есть персональные слоты:**
1. Поля ввода по месяцам = **только отдельский остаток** (фикс P1, §3.2).
2. Над/под полем месяца — read-only чип «👤 людям: N ч» когда `personSum(M) > 0`.
3. Σ-блок панели: «план отдела = Σ остаток + Σ людям; не распределено / переаллокация» (мэтчинг §3.3).
4. НЕ редиректить и НЕ блокировать — отдел остаётся редактируемым (его остаток), персональное только показываем.

**person-панель (employee-plan-panel) — почти готова:**
5. Оставить фильтр по employeeId (уже есть, строка 112) ✓.
6. Добавить (P3, опц.) read-only подсказку «отдел спланирован сверху на M ч» — чтобы человек понимал контекст. Низкий приоритет.

**Анти-двойной-ввод / анти-двойной-счёт:**
7. Ввод: одно число — в одном месте. Персональные часы вводятся ТОЛЬКО в person-панели; dept-панель их показывает read-only. Отдельский остаток — только в dept-панели. Пересечения полей ввода нет.
8. Счёт: уже обеспечен rollup'ом (`deptLoad − Σpersonal`, calc-load.ts:759). Фикс P1 убирает рассинхрон между тем, что ввели, и тем, что посчитали.

### 3.5 Чтобы пользователь не путался (1 ясная модель)

- **Срез «Отделы» и срез «Люди»** доски — две проекции ОДНОЙ модели (раскрыл отдел → видишь людей, Σ которых = строка отдела, I4 §7.5). Это и есть «ментальная модель» — не два плана, а один с zoom.
- **Формулировка в UI:** в dept-панели заголовок «План отдела» + подпись «складывается из плана по людям и нераспределённого остатка». Person-панель: «План сотрудника (часть плана отдела)».
- **planMethod** объяснить тултипом как «способ раскида объёма», чтобы не путать с уровнем.

### 3.6 Что НЕ делать (keep-it-simple)

- Не вводить отдельный объект «режим планирования проекта» — `planMethod` достаточно.
- Не вводить generic-as-entity / роль-в-членстве сейчас (Фаза-3, PLANNING_EMPLOYEE_LEVEL §3.4).
- Не вводить жёсткую квоту «Σлюди ≤ план-отдела» (п.4 стандарта).
- Не материализовать агрегаты (всё derived, §7.1).

## 4. Вопросы заказчику (требуют решения)

**Q1. Модель «отдел = остаток + люди» vs «отдел и люди — независимые числа»?**
Гипотезы: (а) отдел = контейнер, его поле = только нераспределённый остаток, персональное read-only [рекомендую, стандарт п.1/п.5]; (б) два независимых поля, отдел НЕ вычитает персональное (риск 2× счёта); (в) при появлении первого person-слота dept-уровень для месяца становится read-only целиком.
Реф: Float/Runn/Ganttic — team read-only агрегат; наш rollup уже под (а).

**Q2. person → dept: что в dept-панели когда есть персональные слоты?**
Гипотезы: (а) поле = остаток, чип «людям N ч» рядом [рекомендую]; (б) поле disabled с подписью «распределено по людям, правьте в срезе Люди»; (в) показывать агрегат `остаток+люди`, но правка пишет только в остаток.
Реф: §3.2.

**Q3. Σлюди > план-отдела (переаллокация) — warning или блок?**
Гипотезы: (а) warning, сохранение разрешено [рекомендую, как овербукинг доски]; (б) hard-блок; (в) авто-поднять план отдела до Σлюди.
Реф: п.4 стандарта (квоты нет); I1 §7.3.

**Q4. Нужен ли видимый «режим» в UI или достаточно неявного (уровень = строка)?**
Гипотезы: (а) неявно — план открывается там, где нажал, planMethod запоминается [рекомендую, keep-it-simple]; (б) явный сегмент-контрол «Отдел / Люди» в одной панели; (в) подсказка-онбординг про модель без переключателя.
Реф: конкуренты используют views/zoom, не тумблер (п.5).

**Q5. person-панель: показывать ли «отдел спланирован сверху»?**
Гипотезы: (а) да, read-only подсказка [рекомендую, информативность]; (б) нет (минимум); (в) показывать остаток отдела доступный для распределения.
Реф: P3, §3.4.6.

**Q6. Кто планирует по людям vs по отделу — РП проекта или руководитель отдела?**
Гипотезы: (а) РП — и людей, и отдел (его проект) [рекомендую MVP]; (б) РП — отдел-остаток, руководитель отдела — своих людей; (в) оба, конкурентная правка по дедуп-ключу.
Реф: W4B.2; §7.4 (LWW/optimistic). Дубль вопроса №3 PLANNING_EMPLOYEE_LEVEL — синхронизировать ответ.

**Q7. generic→named «replace»-жест (часы переезжают со слота отдела на человека) — сейчас или Фаза-2?**
Гипотезы: (а) Фаза-2, сейчас ручной ввод в двух панелях [рекомендую]; (б) сразу drag/replace как Timetta/Float; (в) не нужен.
Реф: п.2 стандарта; PLANNING_EMPLOYEE_LEVEL §5 Фаза-2.

## 5. Резюме

**Рекомендуемая модель (1 абзац):** Держать ОДНУ модель «отдел = контейнер плана, человек = детализация внутри него» (индустриальный стандарт всех 6 конкурентов: SSOT на строках людей, уровень отдела — производный агрегат/остаток, не самостоятельно вводимое число). Никакого переключателя «режим отдел/человек» — уровень определяется строкой, на которой нажат «План»; `planMethod` (EVEN|MANUAL) остаётся ортогональным «способом раскида» и уже запоминается на проект. Консистентность обеспечена существующим rollup'ом «детальный уровень вычитается из остатка верхнего» (calc-load.ts:759) — менять расчёт НЕ надо, надо привести UI ввода в соответствие с ним.

**Ключевой фикс (P1):** в `project-plan-panel.tsx:143-150` префилл слотов читает ВСЕ слоты проекта и индексирует только по месяцу → персональные часы протекают в поля dept-панели (last-write-per-month). Добавить `slots.filter(s => !s.employeeId)` для полей ввода + отдельно агрегировать персональные в read-only чип «людям N ч».
