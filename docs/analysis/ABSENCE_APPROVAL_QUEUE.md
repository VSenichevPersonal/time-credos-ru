# Отсутствия (REQ-0006) + очередь согласования — анализ + рекомендация

**Дата:** 2026-06-21
**Автор:** Аналитик AITEAM (итерация 124, проактивно)
**Повод:** команда закрыла approval-движок (recall/revoke/CAS/аудит) + person-plan. Отсутствия (REQ-0006) — смежный непокрытый домен: влияет на ёмкость (которую person-plan только трогал) и Т-13. Очередь согласования — UX-вопрос.
**Опора:** разведка Timetta (time-off + supervisor-approval). Kimai: absence-модели НЕТ (только deprecated HOLIDAY-флаг + платный плагин вне разведки) → референс ТОЛЬКО Timetta.

## 1. Модель заявки на отсутствие (Timetta)

**Отдельная сущность** «Заявка на отсутствие» со СВОИМ lifecycle (не таймшит), но ТОТ ЖЕ approval-движок:
- Поля: `absenceType` (классификатор: код 1-2 симв, наименование, **`paid`-флаг**, `unit`={рабочий день|календарный день|час}), `startDate/endDate`, `description`, `attachment` (заявление/больничный), `applicant` (руководитель может за подчинённого).
- Lifecycle: `Черновик → Согласовано`, кастомизируется воркфлоу. Согласующие: Автор/Руководитель автора/со-менеджеры/группы/юзеры. ВСЕ «за» → согласовано; один «отклонил» → возврат. **Тот же движок, что таймшит.**
- Запрет пересекающихся периодов.
- **Баланс отсутствий** (отдельный компонент): записи «Начисление»/«Использование». Согласовано→авто «Использование»; возврат→удаление. Политики начисления: фиксированные (нед/мес/год×N) + из переработок (часы согласованных таймшитов).

## 2. Влияние на ёмкость / норму / Т-13

- **Ёмкость:** согласованные заявки УМЕНЬШАЮТ доступную ёмкость (ResourceGap=Demand−Capacity). **У нас уже `absenceHoursByEmpInPeriod` вычитает** — модель совпадает.
- **Табель:** отсутствие = ОТДЕЛЬНАЯ строка таймшита, СУММИРУЕТСЯ с явкой (не «или-или»): проектные часы + код отсутствия в свои дни. У Timetta Т-13-кодов нет (буквенный код типа в ячейке) — наш маппинг `absenceType→графа Т-13` = корректное расширение.
- **Утилизация:** отсутствия исключаемы из формулы (настраиваемо) — учесть в отчётах.

## 3. Очередь согласования / «Входящие»

- **Выделенный экран:** «Моя работа → Входящие» — туда падают И таймшиты, И заявки на согласование от меня.
- Согласующий видит: состояние/автор/воркфлоу; действия «Согласовать / Отклонить (с комментарием)»; история — вкладка «Активность».
- Фильтры/представления (сохранённые, Список/Доска по состояниям). Массовые: чекбоксы→«Изменить состояние»→Согласовать/Переоткрыть; диалог с count; **нельзя миксовать состояния**; валидация блокирует недопустимые.
- Уведомления: email + внутрисистемные при попадании в очередь. Re-approval skip если уже согласовал.
- Автор: раздел «Отправленные» (статус+кто согласует). Админ/HR: принудительно (lifecycle, минуя workflow).

## 4. Рекомендация для нас (REQ-0006)

### Фаза 1 (переиспользуя существующий approval-движок — дёшево)
1. **Объект `AbsenceRequest`:** `applicant(→Person), absenceType(SELECT UPPER_SNAKE), startDate/endDate, description, attachment, status(DRAFT/SUBMITTED/APPROVED/REJECTED)`. **Тот же lifecycle + approval-движок, что таймшит** (recall/revoke, SoD, CAS-батчи, аудит resolvedBy/revokedBy) — огромный reuse, не писать заново.
2. **Очередь = View `credos-absence-approval`** (фильтр SUBMITTED), аналог `credos-time-approval`. Отдельного inbox-экрана НЕ делаем (принято для таймшита).
3. **Маппинг `absenceType→графа Т-13`** (уже есть) + ёмкость вычитает по APPROVED-заявкам (переключить `absenceHoursByEmpInPeriod` с кодов в табеле на заявки).
4. **Запрет пересекающихся периодов** (валидация перед SUBMIT).
5. **`paid`-флаг** на absenceType (Timetta) — для Т-13 (оплачиваемое/нет) и будущего баланса.

### Фаза 2 [отложить]
- **Баланс отсутствий** (`AbsenceBalance` + cron-начисления, политики fixed+переработки) — сложно, не блокирует MVP.
- Дробные дни (¾/½/¼) и unit=час.
- **Объединённый «Входящие»** (таймшит+заявки в одной очереди) — RemDOM не даёт удобного host-DOM, держим раздельные View.
- Массовое согласование заявок (после стабилизации одиночного).
- Исключение отсутствий из util (отчёты).

## 5. SSOT-консистентность отсутствий (связь с person-plan/ёмкостью)

- **SSOT = APPROVED AbsenceRequest** (атом). Ёмкость, норма, Т-13-код — derived из него, не дублировать.
- **Сейчас расхождение:** Т-13-код берётся из absenceType в табеле, ёмкость — из `absenceHoursByEmpInPeriod`. После REQ-0006 — ОДИН источник (APPROVED-заявка) → и ёмкость, и Т-13, и util из него. Убирает рассинхрон.
- **Инвариант:** отсутствие в периоде → ёмкость сотрудника падает → его доля плана (rollup person-plan §3.2) пересчитывается. Уже учтено (`absenceHoursByEmpInPeriod` в employeeLoadCells). После REQ-0006 источник = заявки.
- **Связь с approval:** заявка-отсутствие и таймшит — РАЗНЫЕ сущности, но ОДНА очередь логически (Timetta «Входящие»). У нас — раздельные View, согласующий ходит в обе. Уведомления (F4) — единый механизм на оба.
