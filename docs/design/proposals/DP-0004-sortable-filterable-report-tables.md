# DP-0004 — Сортируемые/фильтруемые таблицы отчётов (ресерч + план)

**Автор:** Dev 1 · **Дата:** 2026-06-21 · **Статус:** RESEARCH / PROPOSED (далее, не сейчас)
**Запрос заказчика:** в «Отчётах» — кликабельные заголовки таблиц: **сортировка + фильтрация**. «Проведи большой ресерч — как в Twenty, есть ли готовые компоненты/практики, или как написать.»

## Ресерч: что даёт Twenty

| Вариант | Что это | Применимо нам? |
|---|---|---|
| **RECORD_TABLE-виджет** (`type:'RECORD_TABLE'` в page-layout) | Платформенная таблица объекта с полным sort/filter/group/поиск (как RecordIndex). Рендерит ПЛАТФОРМА. | ❌ **Только для реальных объектов/view.** Отчёты — вычисляемые агрегаты `/s/reports` (не хранимые записи) → нет объекта → нельзя. |
| **Чарт-виджеты** (Pie/Bar/Line/AggregateChart) в page-layout | Платформенные графики по view-агрегации. | ❌ Не таблица; агрегируют по объекту, не по нашему /s/reports. |
| **Front-component** (наш дашборд) | Песочница React (Remote DOM), без host-DOM. SDK **НЕ экспортирует** datagrid/table/sort-хук. | ✅ Наш случай — пишем сами. |

**Вывод:** готового sort/filter-datagrid для кастомных агрегат-таблиц в песочнице **нет**. RECORD_TABLE — мимо (только объекты). Пишем своё лёгкое — это **проверенный** для нас путь (см. ниже), не эксперимент.

## Проверенные in-house паттерны (переиспользуем)

- **Фильтр:** `grid/use-filters.ts` + `filters-bar.tsx` + `filter-chip.tsx` — мультиселект-фильтр (проект/отдел/вид работ/категория/сотрудник), **покрыт unit-тестами** (`use-filters.test.ts`). Это рабочий эталон фильтрации в наших фронт-компонентах.
- **Сортировка:** строки `/s/reports` уже **в памяти клиента** → сортировка тривиальна (in-memory `Array.sort`), запросы не нужны.
- **Сегмент-контролы, поповеры** (`<Explainable>`, `Segmented`) — есть.

## Решение (своё, лёгкое, design-system)

1. **`useSortable<T>(rows, columns)`** (shared/) — состояние `{ key, dir }`; клик по заголовку → toggle key/dir (3-такта: asc→desc→none); возвращает отсортированные строки. Чистая, тестируемая (QA).
2. **`<SortHeader label sortKey active dir onSort>`** — кликабельный заголовок со стрелкой ▲▼. DOM-free.
3. **Фильтр отчётов** — обобщить `use-filters`-паттерн на reports (фильтр по категории/отделу/периоду через chip-бар над таблицей). Переиспользовать `filter-chip`.
4. Применить в `reports/breakdown-table.tsx`: заголовки → `<SortHeader>`, верх — фильтр-бар.

**Зона:** всё фронт (мой). design-system `shared/{use-sortable,sort-header}`. Контракт /s/reports не меняется (sort/filter — клиентские).

## Альтернатива (если захотим «мощный» grid)

- **TanStack Table (headless)** — без UI, чистая логика sort/filter/group. НО: ⚠️ **проверить совместимость с Remote DOM/Web Worker** (песочница — не обычный DOM; bundle-size; возможны API, требующие host-DOM). Отдельный **спайк** до внедрения. Скорее всего своё-лёгкое дешевле и безопаснее (наш кейс — простые табличные агрегаты, не миллионы строк).
- **RECORD_TABLE через объект-снимок** — материализовать отчёт в реальный объект `credosTimeReportSnapshot` ради платформенной таблицы. Overkill: дубль данных, сложность, не наш масштаб.

## Рекомендация

**Своё лёгкое:** `useSortable` + `<SortHeader>` + reuse фильтр-паттерна. In-memory, DOM-free, тестируемо, без новых объектов/контрактов. Спайк TanStack — только если упрёмся в сложную группировку/виртуализацию (пока не нужно).

## Фазы
- P1: `useSortable` + `<SortHeader>` → клик-сортировка колонок в breakdown-table.
- P1: фильтр-бар отчётов (категория/отдел) на базе `filter-chip`.
- P2: сохранённые сортировки/фильтры (UI-B паттерн), мультиколоночная сортировка — если попросят.
